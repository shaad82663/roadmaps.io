#Unit 16 — Jenkins Fundamentals & Architecture

Difficulty: Beginner · Goal: Understand how Jenkins is put together before writing a single pipeline.

#16.1 Controller/Agent Architecture

Jenkins runs as a controller (historically called "master" — you'll still see both terms in older docs/plugins) that:

  • Hosts the web UI and REST API
  • Stores job configuration and build history
  • Schedules builds and distributes work to agents

Agents (a.k.a. nodes) are the machines/pods that actually execute build steps. Each agent has one or more executors — a slot that can run one build step at a time. This split matters because:

  • The controller should ideally do no heavy building itself in production — it just orchestrates. Running builds directly on the controller ties up its resources and is a common anti-pattern flagged in interviews.
  • Agents can run different OSes, tool versions, or be ephemeral (spun up per-build and destroyed after), which lets you match a build's environment to what it actually needs.
text
                    ┌─────────────────────┐
                    │   Jenkins Controller │
                    │  (UI, scheduler,     │
                    │   job configs, DB)   │
                    └──────────┬──────────┘
                               │ (SSH / JNLP / Kubernetes API)
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
        ┌───────────┐   ┌───────────┐   ┌───────────────┐
        │  Agent 1  │   │  Agent 2  │   │ Ephemeral Pod │
        │ (Linux,   │   │ (Windows, │   │ (Kubernetes,  │
        │  Node.js) │   │  .NET)    │   │  per-build)   │
        └───────────┘   └───────────┘   └───────────────┘

#16.2 Installing Jenkins (Docker Route)

The fastest way to get a real Jenkins instance running locally — and it plays directly into everything you learned in Phases 1-2:

bash
docker volume create jenkins_home

docker run -d \
  --name jenkins \
  -p 8080:8080 \
  -p 50000:50000 \
  -v jenkins_home:/var/jenkins_home \
  -v /var/run/docker.sock:/var/run/docker.sock \
  jenkins/jenkins:lts
  • Port 8080 is the web UI; 50000 is used for JNLP agent connections.
  • Mounting /var/run/docker.sock lets Jenkins itself spin up Docker containers as build agents/steps ("Docker outside of Docker" pattern) — very common when Jenkins needs to build and push Docker images as part of a pipeline.
  • On first boot, get the unlock password with:
bash
docker exec jenkins cat /var/jenkins_home/secrets/initialAdminPassword

#16.3 Freestyle vs Pipeline vs Multibranch Pipeline Jobs

Job type What it is When to use
Freestyle Configured entirely through the UI, no code Legacy/simple jobs; rarely recommended for new work
Pipeline A single Jenkinsfile-driven pipeline, one branch A project with one long-lived branch or a manually-managed pipeline
Multibranch Pipeline Jenkins auto-discovers every branch/PR in a repo and runs each one's own Jenkinsfile The default modern choice — mirrors how you'd use GitHub Actions across branches

The industry (and interview) consensus: avoid Freestyle for anything beyond a quick throwaway job. Pipeline-as-code (Jenkinsfile committed to the repo) is the expected standard, exactly like a .github/workflows/*.yml file.

#16.4 The Plugin Ecosystem

Jenkins' core is intentionally minimal — almost everything useful comes from plugins. As a baseline for a Node.js pipeline you'll typically install:

  • Git plugin — checkout support
  • Pipeline plugin — Jenkinsfile support (usually pre-installed via the "suggested plugins" set)
  • NodeJS plugin — manages Node.js versions per job, similar to actions/setup-node
  • Credentials Binding plugin — safely injects secrets into pipeline steps
  • Blue Ocean plugin — modern visual pipeline UI

Install via Manage Jenkins → Plugins, or declaratively for reproducible instances using a plugins.txt baked into a custom Jenkins Docker image:

dockerfile
FROM jenkins/jenkins:lts
COPY plugins.txt /usr/share/jenkins/ref/plugins.txt
RUN jenkins-plugin-cli --plugin-file /usr/share/jenkins/ref/plugins.txt
text
# plugins.txt
git:latest
workflow-aggregator:latest
nodejs:latest
credentials-binding:latest
blueocean:latest

#16.5 Blue Ocean vs Classic UI

Blue Ocean gives you a visual, left-to-right pipeline graph (similar to GitHub Actions' run visualization) instead of the classic UI's plain stage-view table. It's largely a UX layer over the same underlying Jenkinsfile — nothing about your pipeline code changes. Worth knowing it exists; don't over-invest time in it, since many orgs still use the classic UI exclusively and Blue Ocean's own development has slowed in recent years.

#16.6 Unit 16 Labs

  1. Docker install. Run Jenkins via the Docker command above, unlock it, and complete the setup wizard with the "suggested plugins" set.
  2. Freestyle job. Create a Freestyle job that clones a small Node.js repo and runs npm test as a build step, just to see the legacy UI-driven flow.
  3. SSH agent. Add a second machine (or another container) as an SSH agent node under Manage Jenkins → Nodes, and confirm a job can be routed to it.
  4. Plugin install. Install the NodeJS plugin, then configure a global Node.js installation under Manage Jenkins → Tools so any job can request "Node 20" without manually installing it.

#16.7 Unit 16 Interview Q&A

  • Q: "Explain Jenkins' controller/agent architecture." A: The controller hosts the UI, stores configuration/history, and schedules work; agents (nodes) execute the actual build steps via executors. Production setups avoid running builds directly on the controller to keep it responsive and to isolate build environments per job.

  • Q: "Freestyle vs Pipeline — which would you use and why?" A: Pipeline (ideally Multibranch), because it's defined as code in a Jenkinsfile committed alongside the application — versioned, reviewable, and portable across environments, unlike Freestyle jobs which live only in Jenkins' UI configuration.

#Unit 17 — Jenkins Pipelines: Declarative vs Scripted

Difficulty: Intermediate · Goal: Write real Jenkinsfiles fluently, the way you already write GitHub Actions YAML.

#17.1 Declarative Pipeline Anatomy

Declarative syntax is structured, opinionated, and closer to what you already know from GitHub Actions YAML — it's the recommended default for nearly all new pipelines.

groovy
pipeline {
    agent any

    stages {
        stage('Install') {
            steps {
                sh 'npm ci'
            }
        }
        stage('Test') {
            steps {
                sh 'npm test'
            }
        }
        stage('Build') {
            steps {
                sh 'npm run build'
            }
        }
        stage('Deploy') {
            steps {
                sh './deploy.sh'
            }
        }
    }
}
  • agent any tells Jenkins to run this on any available agent. You can scope it more tightly (see Unit 19) with agent { label 'node20' }.
  • stages / stage / steps map roughly to GitHub Actions' jobs / job / steps, though Jenkins stages are sequential within one job by default (parallelism needs an explicit parallel block).

#17.2 Scripted Pipeline — When You Still Need Groovy

Scripted Pipeline is full Groovy code, giving you loops, complex conditionals, and dynamic stage generation that Declarative's stricter grammar doesn't allow directly (Declarative can still drop into Groovy via script {} blocks for the same effect — see below).

groovy
node {
    stage('Install') {
        sh 'npm ci'
    }
    stage('Test') {
        sh 'npm test'
    }
    stage('Dynamic Matrix') {
        def versions = ['18', '20', '22']
        for (v in versions) {
            sh "docker run --rm node:${v}-alpine node --version"
        }
    }
}

In practice: most teams write Declarative pipelines and only drop into Groovy where needed via a script {} block inside a stage, rather than writing a fully Scripted pipeline from scratch:

groovy
pipeline {
    agent any
    stages {
        stage('Conditional Logic') {
            steps {
                script {
                    if (env.BRANCH_NAME == 'main') {
                        echo 'Deploying to production'
                    } else {
                        echo 'Skipping deploy for feature branch'
                    }
                }
            }
        }
    }
}

Interview framing: "I default to Declarative for readability, structure, and built-in validation, and drop into a script {} block only when I need Groovy-level logic Declarative doesn't natively support — I'd reach for a full Scripted pipeline only for genuinely complex, highly dynamic pipeline generation."

#17.3 post {} Blocks

post runs cleanup/notification logic after the pipeline (or a stage) finishes, based on the outcome — directly analogous to GitHub Actions' if: failure() / if: always() step conditions, but centralized in one block.

groovy
pipeline {
    agent any
    stages {
        stage('Test') {
            steps {
                sh 'npm test'
            }
        }
    }
    post {
        always {
            junit 'test-results/*.xml'
        }
        success {
            slackSend(color: 'good', message: "Build #${env.BUILD_NUMBER} passed ✅")
        }
        failure {
            slackSend(color: 'danger', message: "Build #${env.BUILD_NUMBER} failed ❌")
        }
        cleanup {
            sh 'docker system prune -f'
        }
    }
}

#17.4 Parameters, Environment Variables & when {}

groovy
pipeline {
    agent any

    parameters {
        choice(name: 'ENVIRONMENT', choices: ['staging', 'production'], description: 'Target environment')
        string(name: 'VERSION', defaultValue: '', description: 'Version to deploy')
    }

    environment {
        NODE_ENV = 'production'
        REGISTRY  = 'my-registry.example.com'
    }

    stages {
        stage('Deploy') {
            when {
                expression { params.ENVIRONMENT == 'production' }
            }
            steps {
                echo "Deploying version ${params.VERSION} to ${params.ENVIRONMENT}"
            }
        }
    }
}

when {} is Jenkins' equivalent of GitHub Actions' if: on a job/step — it decides whether a stage runs at all, based on branch name, parameter values, or arbitrary expressions.

#17.5 A Full Node.js Jenkinsfile

Putting it together — a realistic pipeline for a Node.js service that builds a Docker image (tying back to Phase 2's tagging lessons) and deploys conditionally:

groovy
pipeline {
    agent any

    environment {
        IMAGE_NAME = 'my-node-app'
        REGISTRY   = 'my-registry.example.com'
        GIT_SHA    = "${sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()}"
    }

    stages {
        stage('Install & Test') {
            steps {
                sh 'npm ci'
                sh 'npm test'
            }
        }

        stage('Build Docker Image') {
            steps {
                sh "docker build -t ${REGISTRY}/${IMAGE_NAME}:${GIT_SHA} ."
            }
        }

        stage('Push Image') {
            steps {
                withCredentials([usernamePassword(
                    credentialsId: 'registry-creds',
                    usernameVariable: 'REG_USER',
                    passwordVariable: 'REG_PASS'
                )]) {
                    sh "echo $REG_PASS | docker login ${REGISTRY} -u $REG_USER --password-stdin"
                    sh "docker push ${REGISTRY}/${IMAGE_NAME}:${GIT_SHA}"
                }
            }
        }

        stage('Deploy to Staging') {
            when { branch 'main' }
            steps {
                sh "kubectl set image deployment/my-node-app app=${REGISTRY}/${IMAGE_NAME}:${GIT_SHA} -n staging"
            }
        }
    }

    post {
        failure {
            slackSend(color: 'danger', message: "Pipeline failed on commit ${GIT_SHA}")
        }
    }
}

withCredentials is Jenkins' safe pattern for injecting secrets stored in Manage Jenkins → Credentials into shell steps — the values are masked in console logs automatically.

#17.6 Unit 17 Labs

  1. Declarative pipeline. Write a Jenkinsfile with Install → Test → Build → Deploy stages for a small Node.js repo, committed to the repo root.
  2. post block. Add a post block that sends a notification (Slack webhook or even just an echo/email step) only on failure.
  3. Parameterized pipeline. Add a choice parameter for environment and a when { expression {...} } stage that only runs for production.
  4. Scripted-to-Declarative conversion. Take a small Scripted snippet (like the dynamic Node-version loop above) and convert it into a script {} block inside a Declarative pipeline.

#17.7 Unit 17 Interview Q&A

  • Q: "Declarative vs Scripted — which would you use, and why?" A: Declarative by default — it's more structured, easier to read and review, has built-in validation, and covers the vast majority of real pipeline needs. I'd only reach for Scripted (or a script {} block inside Declarative) when I need Groovy-level dynamic logic that Declarative's stricter syntax can't express directly, like generating stages in a loop.

  • Q: "How do you safely use secrets in a Jenkinsfile?" A: Store them in Jenkins' Credentials store (never hardcode them in the Jenkinsfile), and inject them into a step scope using withCredentials, which also automatically masks the values in console output.

#Unit 18 — Jenkins Shared Libraries, Plugins & Multibranch Pipelines

Difficulty: Advanced · Goal: Stop copy-pasting the same 200 lines of Jenkinsfile across every repo.

#18.1 Why Shared Libraries Exist

Once you have more than a handful of services, most of their Jenkinsfiles end up nearly identical — same Docker build steps, same deploy logic, same Slack notification format. A Shared Library is a separate Git repo containing reusable Groovy code that any Jenkinsfile can import, exactly like extracting a GitHub Actions reusable workflow or composite action (which you covered in Phase 4) — same motivation, different tool.

#18.2 The vars/ and src/ Directory Structure

text
my-shared-library/
├── vars/
│   ├── deployApp.groovy          # becomes a global "deployApp()" step
│   └── buildAndPushImage.groovy  # becomes a global "buildAndPushImage()" step
├── src/
│   └── org/
│       └── mycompany/
│           └── DockerUtils.groovy   # a full Groovy class for more complex logic
└── resources/
    └── org/mycompany/notify-template.json
  • vars/ — each .groovy file here becomes a callable step, named after the file. This is the 90% use case.
  • src/ — standard Groovy/Java-style classes for logic too complex for a single vars/ script, imported with normal import statements.
  • resources/ — static files (templates, JSON, etc.) loadable via libraryResource().

vars/buildAndPushImage.groovy:

groovy
def call(Map config) {
    def imageName = config.imageName
    def registry  = config.registry
    def tag       = config.tag ?: sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()

    sh "docker build -t ${registry}/${imageName}:${tag} ."
    withCredentials([usernamePassword(
        credentialsId: config.credentialsId,
        usernameVariable: 'REG_USER',
        passwordVariable: 'REG_PASS'
    )]) {
        sh "echo \$REG_PASS | docker login ${registry} -u \$REG_USER --password-stdin"
        sh "docker push ${registry}/${imageName}:${tag}"
    }
    return "${registry}/${imageName}:${tag}"
}

Using it from any Jenkinsfile, across any number of repos:

groovy
@Library('my-shared-library@main') _

pipeline {
    agent any
    stages {
        stage('Build & Push') {
            steps {
                script {
                    def image = buildAndPushImage(
                        imageName: 'my-node-app',
                        registry: 'my-registry.example.com',
                        credentialsId: 'registry-creds'
                    )
                    echo "Pushed ${image}"
                }
            }
        }
    }
}

The library is registered once under Manage Jenkins → System → Global Pipeline Libraries, and every team's Jenkinsfile can then call buildAndPushImage(...) with a single line instead of duplicating 15+ lines of shell.

#18.3 Multibranch Pipelines

A Multibranch Pipeline job points at a repository (or an entire GitHub org/organization via the GitHub Organization job type) and automatically:

  • Discovers every branch and open pull request
  • Runs each one's own Jenkinsfile from that branch's HEAD
  • Creates/destroys per-branch jobs automatically as branches are created/deleted

This is the direct equivalent of GitHub Actions triggering on every push/pull_request — except Jenkins manages a distinct job entry per branch, which gives you per-branch build history and status in the UI rather than one flat log stream.

Setup: New Item → Multibranch Pipeline → point at the Git repo URL → Jenkins scans and creates branch jobs on save, and re-scans periodically (or instantly via webhook, see 18.4).

#18.4 Webhooks — Instant Triggers

Without a webhook, Jenkins has to poll the repo on a schedule to notice new commits — slow and wasteful. A webhook makes GitHub/GitLab push straight to Jenkins the instant something happens.

GitHub → Jenkins webhook URL:

text
http://<jenkins-host>:8080/github-webhook/

Configured under the GitHub repo's Settings → Webhooks, with content type application/json and the "push events" (and optionally "pull request events") triggers checked. On the Jenkins side, the GitHub plugin needs "GitHub hook trigger for GITScm polling" enabled on the job (or it's automatic for Multibranch jobs using the GitHub Branch Source plugin).

#18.5 Key Plugins Reference

Plugin Purpose
Pipeline (workflow-aggregator) Core Jenkinsfile/Declarative support
Git / GitHub Branch Source Repo checkout + Multibranch discovery
Credentials Binding withCredentials support
Docker Pipeline docker.build()/docker.withRegistry() DSL steps inside Jenkinsfiles
Blue Ocean Visual pipeline UI
Kubernetes Ephemeral pod-based agents (see Unit 19)

#18.6 Unit 18 Labs

  1. Build a shared library. Create a my-shared-library repo with a single vars/deployApp.groovy step, register it globally, and call it from two different sample Jenkinsfiles.
  2. Multibranch setup. Point a Multibranch Pipeline job at a repo with at least two branches, each containing a slightly different Jenkinsfile, and confirm Jenkins builds each independently.
  3. Webhook wiring. Configure a GitHub webhook pointing at your Jenkins instance (ngrok or similar if running locally) and confirm a push triggers a build within seconds instead of waiting for the polling interval.
  4. Reuse across repos. Call the same vars/ function from three different Jenkinsfiles/repos and confirm a single library update propagates to all three on their next run.

#18.7 Unit 18 Interview Q&A

  • Q: "What problem do Jenkins Shared Libraries solve?" A: They eliminate duplicated pipeline logic across many Jenkinsfiles — reusable steps (like building/pushing a Docker image, or sending a standardized notification) live in one versioned Groovy repo and get imported everywhere, so a fix or improvement in one place propagates to every consuming pipeline. It's the Jenkins equivalent of GitHub Actions reusable workflows/composite actions.

  • Q: "How does a Multibranch Pipeline differ from a regular Pipeline job?" A: A regular Pipeline job runs one fixed Jenkinsfile, typically tied to one branch. A Multibranch Pipeline automatically discovers every branch (and PR) in a repo and runs each one's own Jenkinsfile from that branch, creating/removing per-branch jobs as branches come and go — no manual job creation per branch.

#Unit 19 — Jenkins Distributed Builds & Scaling

Difficulty: Advanced · Goal: Keep Jenkins fast and reliable as team size and build volume grow.

#19.1 Static vs Dynamic/Ephemeral Agents

Type Description Trade-off
Static agent A long-lived VM/machine registered permanently with Jenkins Simple, predictable, but idles (wastes cost) when not building, and can accumulate state/drift between builds
Dynamic/ephemeral agent Spun up on-demand (e.g., a fresh Kubernetes pod) for a single build, then destroyed Clean environment every time, scales to zero cost when idle, but has cold-start latency

Most modern production Jenkins setups favor ephemeral agents (via the Kubernetes plugin, or cloud-specific plugins for EC2/Azure) specifically to avoid "works on this agent but not that one" drift — directly analogous to why GitHub-hosted runners give you a clean VM per job.

#19.2 Labels — Routing Jobs to the Right Agent

Labels are tags assigned to agents (e.g., linux, node20, gpu, windows) that a pipeline can request via agent { label '...' }, ensuring a job only runs where its dependencies actually exist.

groovy
pipeline {
    agent { label 'node20 && linux' }
    stages {
        stage('Test') {
            steps {
                sh 'node --version'
                sh 'npm test'
            }
        }
    }
}

Labels can be combined with boolean expressions (&&, ||, !) exactly as shown, letting you target precisely the capability set a build needs.

#19.3 The Kubernetes Plugin

The Kubernetes plugin lets Jenkins schedule each build as a pod in a Kubernetes cluster — a natural fit if your org already runs services on Kubernetes (which ties directly back to Phase 2's Docker/registry work: the pod's containers are just Docker images).

groovy
pipeline {
    agent {
        kubernetes {
            yaml """
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: node
    image: node:20-alpine
    command: ['cat']
    tty: true
"""
        }
    }
    stages {
        stage('Test') {
            steps {
                container('node') {
                    sh 'npm ci && npm test'
                }
            }
        }
    }
}

Each build gets a brand-new pod with exactly the container image(s) it needs, and Kubernetes destroys the pod when the build finishes — zero long-term agent maintenance, and natural horizontal scaling since the cluster can schedule as many pods as it has capacity for.

#19.4 Managing Build Queues & Executors

Each agent has a configured number of executors (concurrent build slots). If more builds are triggered than there are free executors across all matching agents, they sit in the build queue (visible on the Jenkins dashboard) until a slot frees up.

Symptoms of an under-provisioned setup: builds sitting "queued" for minutes with idle developer time. Fixes:

  • Increase executor count per static agent (bounded by that machine's actual CPU/memory)
  • Add more static agents
  • Switch to the Kubernetes plugin so capacity scales elastically with cluster resources instead of a fixed executor count

#19.5 Scaling Jenkins for Large Orgs

At real scale, a few structural decisions matter beyond individual pipelines:

  • Controller sizing: the controller should be sized for orchestration load (many jobs, UI/API traffic, plugin overhead), not build execution — keep agent any builds off it entirely by not adding executors to the controller itself.
  • Agent pools: dedicated pools per team/tech-stack (e.g., a Node.js pool with the right Node versions pre-baked, a separate .NET pool) reduces cold-start time versus one generic pool.
  • Configuration as Code (JCasC): for reproducible Jenkins instances across environments, the Configuration as Code plugin lets you define the entire Jenkins system config (credentials references, tool installations, security realm, etc.) in a YAML file checked into version control, rather than manual UI clicking — this is the Jenkins equivalent of infrastructure-as-code.

#19.6 Unit 19 Labs

  1. Label routing. Set up two agents with different labels (e.g., node18 and node20) and write a pipeline that uses agent { label 'node20' } to confirm it lands on the correct one.
  2. Kubernetes plugin. If you have access to any Kubernetes cluster (minikube/kind is fine), configure the Kubernetes plugin and run a build inside an ephemeral pod as shown above.
  3. Queue bottleneck simulation. Configure an agent with only 1 executor, trigger 3 builds simultaneously, and observe 2 of them sitting in the queue — then add executors/agents and re-run to see the queue clear faster.
  4. Benchmark. Time a build on a static agent vs the same build on a freshly-scheduled Kubernetes pod agent, and note the cold-start overhead difference.

#19.7 Unit 19 Interview Q&A

  • Q: "Static agents vs dynamic/ephemeral agents — what's the trade-off?" A: Static agents are simple and always warm (no cold-start delay) but idle-cost money and can drift in configuration over time. Ephemeral agents (e.g., Kubernetes pods) give a guaranteed-clean environment per build and scale to zero when idle, at the cost of pod-startup latency on each build.

  • Q: "How would you scale Jenkins for a large organization with many teams?" A: Keep the controller dedicated to orchestration only (no build executors on it), use dedicated or Kubernetes-based ephemeral agent pools sized per team/tech-stack, manage system configuration via Configuration-as-Code (JCasC) instead of manual UI setup, and use Shared Libraries so pipeline logic stays consistent and maintainable across many teams' Jenkinsfiles.
