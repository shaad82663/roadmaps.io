#Unit 05 — Docker Image Optimization & Security

Difficulty: Intermediate · Goal: Understand why production images look different from dev images, then build one yourself.

#5.1 Choosing a Base Image: Alpine vs Distroless vs Slim

The concept. Every image you build starts from a base, and that base is where most of your attack surface and disk weight comes from — not your own code. A node:20 image ships an entire Debian userland: a package manager, a shell, compilers, editors, and dozens of libraries you will never call. Each of those is something that could have a CVE, and something that makes the image bigger to pull. So production images move down a ladder of "less stuff":

Base Size (approx) Shell/Package Manager Best for
node:20 (Debian-based) ~1 GB Yes (bash, apt) Local dev, debugging-heavy images
node:20-slim ~200 MB Yes (minimal apt) A good default balance
node:20-alpine ~120 MB Yes (sh, apk), musl libc Most production Node apps
gcr.io/distroless/nodejs20-debian12 ~120 MB No shell, no package manager Maximum hardening

The one gotcha worth internalizing (because it will bite you eventually): Alpine doesn't use glibc, it uses musl. Most pure-JS packages don't care. But native addons — bcrypt, sharp, anything with a .node binary — are sometimes compiled assuming glibc, and will fail to load or behave subtly differently on musl. This is why you always test your actual dependency tree on the base image you intend to ship, not just assume it'll work because it worked on node:20.

Distroless goes a step further: literally no shell, no ls, no sh at all. Great for security (an attacker with code execution has nowhere to pivot to), painful for debugging (docker exec -it <container> sh just errors out). In practice you debug distroless containers using a :debug variant of the same image, or an ephemeral debug container if you're on Kubernetes.

Hands-on walkthrough.

Start from any small Express app (or scaffold one: npm init -y && npm install express). We'll build it three ways and actually compare the numbers instead of just reading them.

  1. Build a normal single-stage image first, so you can see the "before":

    dockerfile
    # Dockerfile.plain
FROM node:20
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm install
COPY . .
CMD ["node", "server.js"]
    bash
    docker build -f Dockerfile.plain -t myapp:plain .

  2. Now build the multi-stage, Alpine-based version:

    dockerfile
    # Dockerfile.alpine
FROM node:20-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm prune --omit=dev

FROM node:20-alpine AS runtime
WORKDIR /app
ENV NODE_ENV=production
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./package.json
COPY --from=builder /app/server.js ./server.js
EXPOSE 3000
CMD ["node", "server.js"]
    bash
    docker build -f Dockerfile.alpine -t myapp:alpine .

  3. Compare them for real:

    bash
    docker images | grep myapp

    You should see something like myapp:plain at ~1 GB and myapp:alpine at well under 200 MB. That difference is the whole point — same app, radically different footprint, purely from base image choice + not shipping the build toolchain.

  4. If your app uses a native module (npm install bcrypt if you don't already), rebuild on Alpine and actually run it:

    bash
    docker run --rm myapp:alpine node -e "console.log(require('bcrypt'))"

    If this errors, that's the musl/glibc gotcha showing up live, not something you memorized — you now know what it looks like when it happens.

Sanity check. Could you explain to a teammate, in your own words, why a Node image built on node:20 might have a native module fail on node:20-alpine even though the app.js file is byte-for-byte identical? If yes, this section did its job.

#5.2 Running as a Non-Root User & Dropping Capabilities

The concept. Containers don't run as root because Docker "chose" that for you — they run as root because nobody told the process to be anyone else, and root is Linux's default. Container root isn't literally host root (namespaces isolate it), but the moment there's a container-escape bug (a bad kernel exploit, a misconfigured mount, a Docker daemon vulnerability), root-in-container becomes root-on-host. Running as an unprivileged user doesn't fix every vulnerability, but it removes an entire category of "and then it was game over" outcomes for a few lines of Dockerfile.

Hands-on walkthrough.

  1. Prove to yourself what the default actually is:

    bash
    docker run --rm node:20-alpine whoami

    You'll see root. That's true even though your app has zero reason to need root.

  2. Now pin it down. The official Node Alpine image conveniently ships a pre-made low-privilege user called node — use it:

    dockerfile
    FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --omit=dev
COPY . .
USER node
EXPOSE 3000
CMD ["node", "server.js"]
    bash
    docker build -t myapp:nonroot .
docker run --rm myapp:nonroot whoami

    Now it should print node, not root.

  3. If a base image doesn't ship a user for you, create one yourself — this is the pattern to memorize:

    dockerfile
    RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

  4. Separately (this is a runtime flag, not a Dockerfile instruction), strip Linux capabilities the process doesn't need:

    bash
    docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE myapp:nonroot

    This removes every Linux capability except the one needed to bind privileged ports. Try running your app and confirm it still starts and serves traffic on its normal port — dropping capabilities shouldn't change app behavior at all if you only dropped ones it never used.

Sanity check. If someone asks "isn't container root harmless because of namespaces?" — what's your one-sentence pushback? (Hint: it's about what happens if isolation is ever broken, not about normal operation.)

#5.3 Image Vulnerability Scanning with Trivy

The concept. A base image is a snapshot of hundreds of OS packages plus whatever your package-lock.json pulls in — and vulnerabilities in any of them get discovered after you built the image. Scanning tools like Trivy compare every package in your image against public vulnerability databases (NVD, GitHub Advisories) so you find out about a CVE before an attacker does, not after.

Hands-on walkthrough.

  1. Install Trivy and scan the image you just built:

    bash
    brew install trivy   # or see Trivy docs for Linux/Windows
trivy image myapp:nonroot

  2. Read the output like a table, not a wall of text — you care about three columns: Library, Severity, Fixed Version.

    text
    ┌─────────────┬────────────────┬──────────┬───────────────────┐
│   Library   │ Vulnerability  │ Severity │   Fixed Version    │
├─────────────┼────────────────┼──────────┼───────────────────┤
│ openssl     │ CVE-2024-XXXXX │ CRITICAL │ 3.1.4-r0           │
│ busybox     │ CVE-2023-YYYYY │ HIGH     │ 1.36.1-r15         │
└─────────────┴────────────────┴──────────┴───────────────────┘

  3. Pick any actual finding from your scan and fix it. There are basically three moves — practice whichever applies:

    • Bump the base image tag: alpine:3.19 → alpine:3.20
    • Explicitly upgrade the flagged OS package: RUN apk upgrade openssl
    • If it's a transitive npm dependency: npm audit fix, or bump the parent package

  4. Rescan and confirm the finding is gone:

    bash
    trivy image --severity HIGH,CRITICAL --exit-code 1 myapp:nonroot

    The --exit-code 1 flag is what actually makes this useful in CI later — it turns "a scanner ran" into "the build fails if something critical is unpatched."

Sanity check. Would rescanning immediately after docker image prune change your Trivy results for an already-built, already-tagged image? (No — pruning affects local disk, not the contents of a tag you already built. Good to be sure you're not confusing the two.)

#5.4 Handling Secrets at Build Time (BuildKit Secret Mounts)

The concept. Every RUN and COPY in a Dockerfile creates a layer, and layers are permanent — even a later RUN rm somefile doesn't erase it from history, it just hides it in the final filesystem view while the original layer (secret and all) is still sitting there, extractable with docker history or by anyone who gets the image. This is the single most common way Node teams accidentally leak an npm registry token.

Wrong way — this leaks the secret into a layer forever:

dockerfile
COPY .npmrc .npmrc
RUN npm ci
RUN rm .npmrc   # too late — .npmrc already exists in an earlier layer

Right way — BuildKit secret mount:

dockerfile
# syntax=docker/dockerfile:1
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./

RUN --mount=type=secret,id=npmrc,target=/app/.npmrc \
    npm ci

COPY . .
CMD ["node", "server.js"]
bash
DOCKER_BUILDKIT=1 docker build \
  --secret id=npmrc,src=$HOME/.npmrc \
  -t myapp:secretsafe .

The secret is mounted only for that one RUN instruction's duration and is never written to any layer.

Hands-on walkthrough.

  1. Create a fake .npmrc locally to simulate a private registry token:

    bash
    echo "//registry.npmjs.org/:_authToken=FAKE-TOKEN-12345" > .npmrc

  2. Build the "wrong way" Dockerfile above and prove the leak to yourself:

    bash
    docker build -f Dockerfile.wrong -t myapp:leaky .
docker history --no-trunc myapp:leaky | grep FAKE-TOKEN

    You should actually see the token in the output — this is the "aha" moment, seeing it rather than being told about it.

  3. Now build the secret-mount version and run the same check:

    bash
    DOCKER_BUILDKIT=1 docker build --secret id=npmrc,src=$(pwd)/.npmrc -t myapp:safe -f Dockerfile.safe .
docker history --no-trunc myapp:safe | grep FAKE-TOKEN

    This time it should find nothing — the secret never touched a layer.

Sanity check. Why doesn't RUN rm .npmrc in a later step fix the leak from the "wrong way" Dockerfile? Say it out loud in terms of layers, not just "because Docker doesn't allow it."

#Unit 06 — Docker Registries & Image Management

Difficulty: Intermediate · Goal: Know where images live, how they should be versioned, and how to stop them piling up.

#6.1 Where Images Actually Live

The concept. A registry is just a place images get pushed to and pulled from — think of it as a specialized artifact store for image layers, not that different conceptually from an npm registry storing packages. Which one you use is mostly determined by which cloud you already deploy to, because that gives you IAM-based pull permissions for free instead of juggling separate registry credentials.

Registry Typical use case
Docker Hub Public/open-source images, small teams, quick prototyping
AWS ECR Teams already on AWS; tight IAM integration
GCP Artifact Registry Teams on GCP
GitHub Container Registry (GHCR) Teams already using GitHub Actions — same auth token
Harbor Self-hosted, enterprise, on-prem compliance requirements

Hands-on walkthrough. If you have a free Docker Hub account, push something real instead of just reading about it:

bash
docker login -u <your-username>
docker tag myapp:safe <your-username>/myapp:1.0.0
docker push <your-username>/myapp:1.0.0
docker logout
docker pull <your-username>/myapp:1.0.0   # pull it back after logging out, to prove it's really "out there"

#6.2 Why "latest" Is a Trap

The concept. latest looks like it should mean "the newest build," but it's just a tag string like any other — docker build without -t assigns it by default, and nothing stops two completely different commits from both being tagged latest at different points in time. The practical damage:

  • docker pull myapp:latest is non-deterministic across environments and time — you genuinely cannot know what code is running.
  • Rollback becomes guesswork: if latest breaks, what was the previous latest? There's no built-in history to point to.
  • Orchestrators may not even re-pull latest on redeploy unless imagePullPolicy is set correctly, causing silently stale deployments.

The fix is a tagging convention that's both human-readable and immutable:

bash
GIT_SHA=$(git rev-parse --short HEAD)
VERSION=$(node -p "require('./package.json').version")

docker build -t my-registry/myapp:${VERSION} \
             -t my-registry/myapp:${GIT_SHA} \
             -t my-registry/myapp:latest .   # optional convenience tag only, never deployed from directly

Deploy manifests should reference the immutable SHA or semver tag — never latest.

Hands-on walkthrough.

  1. Build the same Dockerfile twice, changing one line of code in between, both tagged latest:

    bash
    docker build -t myapp:latest .
# ...edit server.js slightly...
docker build -t myapp:latest .
docker images myapp   # notice only one "latest" exists — the old one is now untagged, dangling

    This is the non-determinism made visible: you've already lost the ability to run "the previous latest" by name.

  2. Redo it properly with the versioning script above, and confirm both tags point at the same image ID:

    bash
    docker images myapp   # SHA and semver tags both listed, same IMAGE ID

#6.3 Pushing to a Private Registry

The concept. Authentication differs slightly per registry, but the underlying idea is identical everywhere: prove who you are, then push to a path that's namespaced under your account/org/repo.

bash
# AWS ECR (token expires after 12h, so this is usually scripted in CI)
aws ecr get-login-password --region ap-south-1 | \
  docker login --username AWS --password-stdin \
  <account-id>.dkr.ecr.ap-south-1.amazonaws.com

# GitHub Container Registry
echo $GITHUB_TOKEN | docker login ghcr.io -u <github-username> --password-stdin

Tagging for a private registry requires the full registry path as a prefix:

bash
docker tag myapp:1.0.0 <account-id>.dkr.ecr.ap-south-1.amazonaws.com/myapp:1.0.0
docker push <account-id>.dkr.ecr.ap-south-1.amazonaws.com/myapp:1.0.0

Hands-on walkthrough. Pick whichever registry you actually have access to (ECR, GHCR, or a free Docker Hub private repo) and do a real push/pull round trip end-to-end, including logging out and back in on a "fresh" session, so you feel the auth step rather than skip it because you were already logged in from Section 6.1.

#6.4 Keeping a Registry From Growing Forever

The concept. Every CI build produces at least one image, and if nothing ever deletes old ones, storage cost climbs forever for images nobody will ever deploy again. Cleanup happens at two levels: your local Docker host, and the registry itself.

Local cleanup:

bash
docker image prune -f          # remove dangling (untagged) images
docker image prune -a -f       # remove ALL unused images — more aggressive, be careful
docker system prune -a -f      # remove images, containers, networks, build cache

Registry-side retention policy (conceptual — AWS ECR lifecycle policy):

json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Expire untagged images older than 7 days",
      "selection": { "tagStatus": "untagged", "countType": "sinceImagePushed", "countUnit": "days", "countNumber": 7 },
      "action": { "type": "expire" }
    },
    {
      "rulePriority": 2,
      "description": "Keep only the last 20 tagged images",
      "selection": { "tagStatus": "tagged", "tagPrefixList": ["v"], "countType": "imageCountMoreThan", "countNumber": 20 },
      "action": { "type": "expire" }
    }
  ]
}

This kind of policy runs automatically at the registry level — no manual pruning required once it's set.

Hands-on walkthrough. Build 3-4 throwaway variants of myapp locally (small code tweak each time, same tag so old ones go dangling), then:

bash
docker images -f dangling=true    # see the orphaned layers pile up
docker image prune -f             # clean them up
docker images -f dangling=true    # confirm they're gone

Then draft (doesn't need to be applied to a real registry) a retention policy JSON like the one above for a hypothetical repo of yours, just to practice the shape of it.

#Unit 07 — Logging, Resource Limits & Health in Production

Difficulty: Advanced · Goal: Keep containers observable, bounded, and self-healing — and actually see each failure mode happen once so you recognize it instantly later.

#7.1 Logging Drivers & Rotation

The concept. By default Docker's json-file driver writes every stdout/stderr line to a JSON file on the host, with no size cap. A chatty Node app logging on every request will quietly fill a disk over weeks if nobody sets rotation limits.

bash
docker run -d \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  myapp

Or set it daemon-wide in /etc/docker/daemon.json so every container inherits it without remembering the flags each time:

json
{ "log-driver": "json-file", "log-opts": { "max-size": "10m", "max-file": "3" } }

In real production stacks you'll typically forward logs to a central system (CloudWatch, ELK, Loki) via drivers like awslogs, fluentd, or gelf rather than relying on local JSON files at all — but understanding the local default first is what makes those forwarding drivers make sense.

Hands-on walkthrough.

  1. Run a container that logs continuously without a size cap, and watch the log file grow on the host:
    bash
    docker run -d --name chatty node:20-alpine node -e "setInterval(()=>console.log('x'.repeat(1000)), 10)"
docker inspect --format='{{.LogPath}}' chatty
ls -lh $(docker inspect --format='{{.LogPath}}' chatty)   # watch the size climb over a minute
docker rm -f chatty
  2. Now rerun it with rotation and confirm it caps out instead of growing unbounded:
    bash
    docker run -d --name chatty-capped --log-opt max-size=1m --log-opt max-file=2 node:20-alpine node -e "setInterval(()=>console.log('x'.repeat(1000)), 10)"
# after a minute or two:
ls -lh $(docker inspect --format='{{.LogPath}}' chatty-capped)*
docker rm -f chatty-capped

#7.2 Restart Policies

The concept. Restart policies decide what Docker does when a container's main process exits — crucial for a long-running Node service, since you don't want a manual restart at 3am for something that could self-heal.

Policy Behavior
no (default) Never restart automatically
on-failure[:max-retries] Restart only on non-zero exit code, optionally capped
always Always restart, even after a manual docker stop + host reboot
unless-stopped Like always, but stays stopped if you explicitly docker stop'd it before a reboot

The nuance that trips people up: always will bring a container back even if you deliberately stopped it, the moment the host reboots. unless-stopped is almost always the better default for long-running services — it survives crashes and daemon restarts, but respects an intentional stop.

Hands-on walkthrough.

bash
docker run -d --name crasher --restart on-failure:3 node:20-alpine node -e "process.exit(1)"
sleep 5
docker ps -a --filter name=crasher       # RestartCount should show it tried, then gave up after 3
docker inspect crasher --format='{{.RestartCount}}'
docker rm -f crasher

Then repeat with --restart unless-stopped on a normal long-running server, docker stop it deliberately, and confirm it does not come back on its own — that's the behavior you were told about, now something you watched happen.

#7.3 CPU/Memory Limits & the OOM Killer

The concept. Without limits, one misbehaving container (a Node process with a memory leak, say) can starve every other container on the host — or the host itself. Setting limits forces Linux's OOM killer to intervene inside the container's boundary instead of taking down the whole machine.

bash
docker run -d --memory="512m" --memory-swap="512m" --cpus="0.5" myapp
  • --memory caps RAM. --memory-swap equal to --memory disables swap entirely — common in production so a leak gets killed fast instead of silently swapping into unusable latency.
  • When the limit is exceeded, the kernel sends SIGKILL. docker inspect will show:
    json
    "State": { "OOMKilled": true, "ExitCode": 137 }
    137 = 128 + 9 (SIGKILL). This is the single most useful number to instantly recognize in this entire phase.

For Node specifically: --memory caps the container's total memory, but V8's own heap has a separate ceiling. Set --max-old-space-size comfortably below the container limit, or V8 may not GC aggressively enough before the container gets killed first:

dockerfile
CMD ["node", "--max-old-space-size=384", "server.js"]

Hands-on walkthrough. Actually trigger an OOM kill — don't just read about exit code 137, watch it happen:

bash
docker run -d --name leaker --memory=100m node:20-alpine \
  node -e "let a=[]; setInterval(()=>{ a.push(Buffer.alloc(10*1024*1024)); }, 200)"
sleep 15
docker inspect leaker --format='ExitCode={{.State.ExitCode}} OOMKilled={{.State.OOMKilled}}'
docker rm -f leaker

You should see ExitCode=137 OOMKilled=true. Now you'll recognize this signature instantly in a real incident instead of having to look it up.

#7.4 HEALTHCHECK

The concept. A running process isn't the same as a working process — your Node server could be alive but stuck on a dead DB connection pool, still "running" from Docker's point of view. HEALTHCHECK tells Docker (and orchestrators) to actually probe the app, not just check if the PID exists.

dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

Backed by a real endpoint:

javascript
app.get('/health', (req, res) => {
  // optionally check DB connectivity, cache, etc. here
  res.status(200).json({ status: 'ok' });
});

docker ps then shows healthy, unhealthy, or starting in the STATUS column.

Hands-on walkthrough.

  1. Add the health endpoint and HEALTHCHECK above to your app, rebuild, and confirm:
    bash
    docker run -d --name healthdemo -p 3000:3000 myapp:healthcheck
sleep 15
docker ps --filter name=healthdemo   # STATUS should show "(healthy)"
  2. Now deliberately break it (temporarily change the health route to return 500) and rebuild:
    bash
    docker ps --filter name=healthdemo   # watch STATUS flip to "(unhealthy)" after a few failed retries
docker rm -f healthdemo
    Seeing the transition from starting → healthy → unhealthy on a real container is worth far more than memorizing the flag names.

#7.5 Docker Swarm, Briefly

Docker Swarm is Docker's own built-in orchestrator — docker swarm init then docker service create --replicas 3 myapp gets you multi-node scheduling and rolling updates without installing Kubernetes. It's simpler to operate but has a much smaller ecosystem, and most teams today either stay on Compose for small deployments or go straight to Kubernetes when they need real scale. Know it exists and roughly what problem it solves; you're not expected to be a Swarm expert at this stage.

#Unit 08 — Troubleshooting Like You Mean It

Difficulty: Advanced · Goal: Debug confidently under real time pressure — production incident or interview, same muscle.

#8.1 A Debugging Workflow You Can Actually Reuse

The value here isn't memorizing six commands — it's having a default order so you're never standing in front of a broken container wondering what to check first:

bash
docker ps -a                     # 1. Is it running, exited, or restarting?
docker logs --tail 100 -f <id>   # 2. What did the app itself say before it died?
docker inspect <id>              # 3. Exit code, OOMKilled, mounts, env, network config
docker stats <id>                # 4. Live CPU/memory — is it starved or leaking right now?
docker exec -it <id> sh          # 5. Get inside (if a shell exists) and look around live
docker events                    # 6. Host-level timeline of container lifecycle events

docker inspect --format '{{.State.ExitCode}} {{.State.OOMKilled}}' gets you the two most decision-relevant fields directly, without hunting through the full JSON blob.

#8.2 Common Failure Scenarios, Decoded

Symptom Likely cause How to confirm
Exit code 137 OOM killed (or manual docker kill) docker inspect → OOMKilled: true
Exit code 1 Uncaught app exception / non-zero process.exit docker logs shows the stack trace
Exit code 139 Segfault (SIGSEGV) — rare in pure JS, more common with native addons Check native module compatibility (glibc vs musl on Alpine)
"port is already allocated" Another container/process already bound that host port docker ps / lsof -i :<port> on host
"Permission denied" on a mounted volume UID mismatch between container user and host file ownership docker exec <id> id vs ls -la on the host path
Can't resolve another container's hostname Both containers aren't on the same user-defined network (default bridge has no auto DNS) docker network inspect <network>
Container exits immediately with code 0 PID 1 finished and had nothing left to do — common when CMD is a one-shot script, not a server Check your CMD/ENTRYPOINT actually starts a persistent process

#8.3 "Works on My Machine"

The concept. The most common Node-specific version of this bug: a native module (bcrypt, sharp) built against your Mac's or CI runner's libc, then run inside an alpine (musl) container where it silently fails or segfaults. The fix is always the same instinct: reproduce inside the exact target image, not against your local node_modules.

bash
# Reproduce inside the exact image that will run in prod
docker run --rm -v $(pwd):/app -w /app node:20-alpine npm ci
docker run --rm -v $(pwd):/app -w /app node:20-alpine node -e "console.log(require('bcrypt'))"

Other common drift sources: mismatched Node versions between local and image, timezone/locale differences affecting date logic, and env vars that exist in a local .env but were never passed to the container via -e/--env-file.

#8.4 Concept Recap

A short list of ideas worth being able to explain out loud, in your own words, not as memorized definitions:

  • Image vs container — an image is a read-only layered template; a container is a running (or stopped) instance with its own writable layer.
  • Union filesystem — layers stack via copy-on-write (overlay2), so containers share read-only base layers and each gets a thin writable layer of its own.
  • PID 1 — the CMD/ENTRYPOINT process becomes PID 1 inside the container and doesn't get default signal handling like normal processes do; if your app never handles SIGTERM, docker stop hangs until timeout then force-kills it. This is why tini or process.on('SIGTERM', ...) matter for graceful shutdown.
  • Build idempotency — same Dockerfile + same context should give a functionally identical image, which is why pinning versions (node:20.11.1-alpine, not node:latest) matters.
  • Rollback — because tags are immutable and versioned (Unit 06), rollback is just redeploying the previous known-good tag.

#8.5 Full End-to-End Debugging Lab

Do this one slowly — it ties Units 5-8 together instead of testing them in isolation.

  1. Deliberately misconfigure a container so it crash-loops (e.g., have it read a required env var that you never pass in), then use docker logs + docker inspect to find and fix the actual root cause.
  2. On purpose, produce three containers exiting with codes 137 (OOM — reuse the Section 7.3 lab), 1 (uncaught exception), and 0 (a script that just finishes). Write down, for each, exactly which command told you which one it was.
  3. Start a normal container, docker exec -it into it, and look around live: ps aux, check open env vars, poke at the filesystem — all without stopping the container.
  4. Narrate out loud (or in writing) the full flow from docker build through networking and volume mounting for a Node app, as if explaining it to someone who's never seen your project — don't skip a step, including why each step exists, not just what command runs it.

That observe → logs → inspect → live metrics → reproduce → mitigate/rollback shape is the reusable template for debugging almost anything that goes wrong with a container in production.
