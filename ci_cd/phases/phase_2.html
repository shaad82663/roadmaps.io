#Unit 05 — Docker Image Optimization & Security

Difficulty: Intermediate · Goal: Ship images that are small, fast to pull, and hard to exploit.

#5.1 Choosing a Base Image: Alpine vs Distroless vs Slim

Every image you build inherits the attack surface and size of its base. For a Node.js service, you have three realistic tiers:

Base Size (approx) Shell/Package Manager Best for
node:20 (Debian-based) ~1 GB Yes (bash, apt) Local dev, debugging-heavy images
node:20-slim ~200 MB Yes (minimal apt) A good default balance
node:20-alpine ~120 MB Yes (sh, apk), musl libc Most production Node apps
gcr.io/distroless/nodejs20-debian12 ~120 MB No shell, no package manager Maximum hardening

Why this matters beyond disk space:

  • Smaller images pull faster → faster deploys, faster autoscaling, faster CI.
  • Fewer installed packages = fewer CVEs to patch. A full Debian base can carry 100+ known CVEs from packages you never use.
  • Alpine uses musl libc instead of glibc. This occasionally breaks native Node addons (e.g., some versions of bcrypt, sharp) that expect glibc. Always test your exact dependency tree on Alpine before committing to it — this is a very common real-world gotcha for Node engineers.
  • Distroless images have no shell, which means docker exec -it <container> sh simply won't work. That's a deliberate security trade-off (an attacker who gets code execution can't spawn a shell) but it makes debugging harder — you typically debug against a :debug tag of the same distroless image, or attach an ephemeral debug container in Kubernetes.

Multi-stage build combining a full build environment with a lean runtime:

dockerfile
# ---------- Stage 1: builder ----------
FROM node:20-alpine AS builder
WORKDIR /app

# Install deps first (better layer caching)
COPY package.json package-lock.json ./
RUN npm ci

# Copy source and build (TypeScript compile, etc.)
COPY . .
RUN npm run build

# Prune dev dependencies so only prod deps remain
RUN npm prune --omit=dev

# ---------- Stage 2: runtime ----------
FROM node:20-alpine AS runtime
WORKDIR /app
ENV NODE_ENV=production

# Copy only what's needed to run — not the whole build toolchain
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package.json ./package.json

EXPOSE 3000
CMD ["node", "dist/server.js"]

The builder stage (with npm, TypeScript, build tools) never ships. Only runtime's layers end up in the final image.

#5.2 Running as a Non-Root User & Dropping Capabilities

By default, a process inside a container runs as root (UID 0) unless the image says otherwise. This is misleading — container root is not host root because of namespace isolation, but if an attacker breaks out of the container (via a kernel exploit or a misconfigured mount), root-in-container becomes root-on-host. Running as a non-root user is one of the cheapest, highest-leverage hardening steps you can take.

dockerfile
FROM node:20-alpine
WORKDIR /app

COPY package.json package-lock.json ./
RUN npm ci --omit=dev
COPY . .

# The official node:alpine image already ships a low-privilege user called "node"
USER node

EXPOSE 3000
CMD ["node", "server.js"]

If your base image doesn't already provide a user, create one explicitly:

dockerfile
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

Dropping Linux capabilities at run time (not in the Dockerfile, but worth knowing since it's an interview favorite):

bash
docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE my-node-app

This strips every Linux capability except the one needed to bind to a privileged port (<1024). It means even if the process is compromised, it can't, for example, load kernel modules or change file ownership arbitrarily.

#5.3 Image Vulnerability Scanning with Trivy / Docker Scout

Scanning inspects your image's OS packages and language dependencies (including your package-lock.json) against known-vulnerability databases (NVD, GitHub Advisories, etc.).

bash
# Install Trivy (macOS example; see docs for Linux/Windows)
brew install trivy

# Scan a local image
trivy image my-node-app:latest

# Fail a CI pipeline on HIGH or CRITICAL findings only
trivy image --severity HIGH,CRITICAL --exit-code 1 my-node-app:latest

Sample (trimmed) output to know how to read:

text
my-node-app:latest (alpine 3.19.1)
===================================
Total: 3 (HIGH: 2, CRITICAL: 1)

┌─────────────┬────────────────┬──────────┬───────────────────┐
│   Library   │ Vulnerability  │ Severity │   Fixed Version    │
├─────────────┼────────────────┼──────────┼───────────────────┤
│ openssl     │ CVE-2024-XXXXX │ CRITICAL │ 3.1.4-r0           │
│ busybox     │ CVE-2023-YYYYY │ HIGH     │ 1.36.1-r15         │
└─────────────┴────────────────┴──────────┴───────────────────┘

Fixing a finding usually means one of:

  1. Bump the base image tag (alpine:3.19 → alpine:3.20).
  2. Explicitly upgrade the vulnerable package (RUN apk upgrade openssl).
  3. If it's a transitive npm dependency: npm audit fix, or bump the parent package.

Docker Scout (docker scout cves my-node-app:latest) does the same job natively inside Docker Desktop/CLI and is worth knowing exists, but Trivy is the more common interview and industry answer because it's open-source and CI-friendly.

#5.4 Handling Secrets at Build Time (BuildKit Secret Mounts)

A classic Node.js mistake: a private npm registry token or a .npmrc gets COPY'd into an image to allow npm install to authenticate — and it stays in that layer's history forever, even if you delete it in a later RUN command. Anyone with docker history or access to the layer can extract it.

Wrong way (leaks the secret into a layer):

dockerfile
COPY .npmrc .npmrc
RUN npm ci
RUN rm .npmrc   # too late — .npmrc already exists in an earlier layer

Right way — BuildKit secret mount (never persisted in any layer):

dockerfile
# syntax=docker/dockerfile:1
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./

RUN --mount=type=secret,id=npmrc,target=/app/.npmrc \
    npm ci

COPY . .
CMD ["node", "server.js"]

Build it by passing the secret from the host, not baking it into the Dockerfile:

bash
DOCKER_BUILDKIT=1 docker build \
  --secret id=npmrc,src=$HOME/.npmrc \
  -t my-node-app .

The secret is mounted only for the duration of that specific RUN instruction and is never written to any image layer.

#5.5 Unit 05 Labs

  1. Non-root conversion. Take a Dockerfile that runs as root, add a dedicated user, and rebuild. Verify with docker run --rm my-image whoami (expect a non-root username, not root).
  2. Trivy scan-and-fix. Scan any existing image, find one HIGH/CRITICAL finding, fix it (base bump or dependency bump), and rescan to confirm it's gone.
  3. Distroless rebuild. Rebuild your Node app on gcr.io/distroless/nodejs20-debian12 and compare docker images size output against your Alpine build.
  4. BuildKit secret mount. Simulate installing from a private npm registry using a fake .npmrc token, and prove (via docker history --no-trunc <image>) that the secret never appears in any layer.

#5.6 Unit 05 Interview Q&A

  • Q: "How would you harden a Node.js Docker image for production?" A: Use a minimal/distroless base, multi-stage build to exclude dev tools, run as non-root, drop unneeded Linux capabilities, scan with Trivy in CI and fail the build on critical CVEs, and never bake secrets into layers (use BuildKit secret mounts or runtime secret injection instead).

  • Q: "Why is npm ci preferred over npm install inside a Dockerfile?" A: npm ci installs strictly from package-lock.json, is faster (it skips dependency resolution), and fails if the lockfile and package.json are out of sync — giving you reproducible, deterministic builds, which matters a lot for layer caching and for avoiding "it worked yesterday" drift.

#Unit 06 — Docker Registries & Image Management

Difficulty: Intermediate · Goal: Know where images live, how they get versioned, and how they get cleaned up.

#6.1 Docker Hub vs Private Registries

Registry Typical use case
Docker Hub Public/open-source images, small teams, quick prototyping
AWS ECR Teams already on AWS; tight IAM integration
GCP Artifact Registry (GCR successor) Teams on GCP
GitHub Container Registry (GHCR) Teams already using GitHub Actions — same auth token
Harbor Self-hosted, enterprise, on-prem compliance requirements

For most Node.js teams shipping to a cloud provider, the registry choice usually just follows the cloud provider you already deploy to, since it gives you IAM-based pull permissions without managing separate registry credentials.

#6.2 Tagging Strategy: Why "latest" Is a Production Hazard

latest is not a "the newest version" pointer that Docker manages for you — it's just a tag name like any other, and by default docker build without -t assigns it. The danger:

  • Two different commits can both be tagged latest over time, so docker pull my-app:latest is non-deterministic across environments and time — you can't be sure what code is actually running.
  • Rollback becomes guesswork: if latest breaks, what was the previous latest? There's no built-in history.
  • Kubernetes/orchestrators may not re-pull latest on redeploy if imagePullPolicy isn't set correctly, causing stale deployments.

A production-grade tagging convention combines semantic version + commit SHA:

bash
# Build once
docker build -t my-node-app:1.4.2 -t my-node-app:a1b2c3d .

# Push both tags — human-readable AND immutable/traceable
docker push my-node-app:1.4.2
docker push my-node-app:a1b2c3d

A common CI pattern (conceptually, for GitHub Actions or Jenkins covered in later phases):

bash
GIT_SHA=$(git rev-parse --short HEAD)
VERSION=$(node -p "require('./package.json').version")

docker build -t my-registry/my-node-app:${VERSION} \
             -t my-registry/my-node-app:${GIT_SHA} \
             -t my-registry/my-node-app:latest .   # optional convenience tag only, never deployed from directly

Deploy manifests should always reference the immutable SHA or semver tag, never latest.

#6.3 Authenticating & Pushing to a Private Registry

bash
# Docker Hub
docker login -u <username>

# AWS ECR (token expires after 12h, so this is usually scripted in CI)
aws ecr get-login-password --region ap-south-1 | \
  docker login --username AWS --password-stdin \
  <account-id>.dkr.ecr.ap-south-1.amazonaws.com

# GitHub Container Registry
echo $GITHUB_TOKEN | docker login ghcr.io -u <github-username> --password-stdin

Tagging for a private registry requires the full registry path as a prefix:

bash
docker tag my-node-app:1.4.2 <account-id>.dkr.ecr.ap-south-1.amazonaws.com/my-node-app:1.4.2
docker push <account-id>.dkr.ecr.ap-south-1.amazonaws.com/my-node-app:1.4.2

On the pulling side (e.g., another host or a Kubernetes node), the same docker login is required unless the registry allows anonymous pulls or the platform uses an attached IAM role.

#6.4 Image Retention & Cleanup Policies

Registries are not free, and old images accumulate fast in active CI pipelines. Two levels of cleanup:

1. Local Docker host cleanup:

bash
# Remove dangling (untagged) images
docker image prune -f

# Remove all unused images, not just dangling ones (careful — more aggressive)
docker image prune -a -f

# Remove everything unused: images, containers, networks, build cache
docker system prune -a -f

2. Registry-side retention policy (conceptual example — AWS ECR lifecycle policy JSON):

json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Expire untagged images older than 7 days",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 7
      },
      "action": { "type": "expire" }
    },
    {
      "rulePriority": 2,
      "description": "Keep only the last 20 tagged images",
      "selection": {
        "tagStatus": "tagged",
        "tagPrefixList": ["v"],
        "countType": "imageCountMoreThan",
        "countNumber": 20
      },
      "action": { "type": "expire" }
    }
  ]
}

This kind of policy is applied once at the registry/repository level and runs automatically — no manual pruning needed.

#6.5 Unit 06 Labs

  1. Push/pull round trip. Push a tagged image to any private registry you have access to (Docker Hub private repo is fine for practice), then pull it on a second machine or a fresh terminal session after docker logout.
  2. Auth setup. Configure docker login against one private registry (ECR, GHCR, or Docker Hub) and confirm token/credential expiry behavior.
  3. Tagging convention design. Write a short shell script that builds and tags an image with both semver (from package.json) and the current git short SHA.
  4. Cleanup policy. Either write a local cron-able docker image prune script, or draft a retention policy (like the ECR JSON above) for a hypothetical registry.

#6.6 Unit 06 Interview Q&A

  • Q: "Why is tagging images latest risky in production pipelines?" A: latest is mutable and non-deterministic — it doesn't guarantee which build is actually deployed, breaks reliable rollback, and can cause stale deploys if imagePullPolicy isn't configured to always re-pull. Production deployments should reference immutable tags (semver and/or commit SHA).

  • Q: "How do you keep a container registry from growing unbounded?" A: Combine local docker image prune for dev machines/CI runners with a registry-level lifecycle/retention policy that expires untagged images after N days and caps the number of retained tagged versions.

#Unit 07 — Docker Logging, Resource Limits & Health in Production

Difficulty: Advanced · Goal: Keep containers observable, bounded, and self-healing.

#7.1 Logging Drivers & Log Rotation

By default Docker uses the json-file logging driver, writing every line of stdout/stderr to a JSON file on the host — with no rotation by default, which means a chatty Node app (console.log on every request) can silently fill a disk over weeks.

bash
# Run with size-capped, rotated JSON logging
docker run -d \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  my-node-app

Or configure it as the Docker daemon default in /etc/docker/daemon.json so every container inherits it:

json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}

Other drivers worth knowing: journald (integrates with systemd's journal — common on Linux hosts), syslog, and drivers that forward straight to log aggregators (fluentd, awslogs, gelf). In real production stacks, you'll usually ship logs to a central system (CloudWatch, ELK, Loki) rather than relying on local JSON files.

#7.2 Restart Policies

Policy Behavior
no (default) Never restart automatically
on-failure[:max-retries] Restart only on non-zero exit code, optionally capped
always Always restart, even after a manual docker stop + host reboot
unless-stopped Like always, but stays stopped if you explicitly docker stop'd it before a reboot
bash
docker run -d --restart unless-stopped my-node-app

Interview nuance: always will bring a container back up even if you deliberately stopped it and the host reboots — this surprises people. unless-stopped is almost always the right choice for long-running services, because it respects an intentional stop while still surviving crashes and daemon restarts.

#7.3 CPU/Memory Limits & the OOM Killer

Without limits, a single misbehaving container (e.g., a Node process with a memory leak) can starve every other container — or the whole host.

bash
docker run -d \
  --memory="512m" \
  --memory-swap="512m" \
  --cpus="0.5" \
  my-node-app
  • --memory caps RAM usage. --memory-swap set equal to --memory disables swap entirely for the container (common in production so a leaking process is killed fast rather than swapping into unusable latency).
  • --cpus="0.5" caps the container to half a CPU core's worth of processing time.

What happens on OOM: if the process inside the container exceeds its memory ceiling, the Linux kernel's OOM killer sends SIGKILL to the process. docker inspect on that container will show:

json
"State": {
  "OOMKilled": true,
  "ExitCode": 137
}

Exit code 137 = 128 + 9 (SIGKILL). This is one of the single most common "why did my container die" interview and real-world debugging questions — recognize 137 instantly as an OOM kill signature.

For a Node.js app specifically, remember that --memory limits the container's total memory, but Node's own V8 heap has its own --max-old-space-size limit — set the V8 flag comfortably below the container limit, or V8 may not GC aggressively enough before the container gets OOM-killed:

dockerfile
CMD ["node", "--max-old-space-size=384", "server.js"]

(with a 512m container limit, leaving headroom for non-heap memory like buffers and native modules.)

#7.4 HEALTHCHECK Instruction

HEALTHCHECK tells Docker (and orchestrators) how to actively probe whether a container is actually serving traffic correctly — not just "the process is running," but "the process is working."

dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --omit=dev
COPY . .

HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

EXPOSE 3000
CMD ["node", "server.js"]

A minimal Express health endpoint to back it:

javascript
app.get('/health', (req, res) => {
  // Optionally check DB connectivity, cache, etc. here
  res.status(200).json({ status: 'ok' });
});

docker ps will then show healthy, unhealthy, or starting in the STATUS column. Orchestrators (Swarm, Kubernetes via its own probes) use this signal to decide whether to route traffic to a container or restart it.

#7.5 Docker Swarm — A One-Paragraph Orientation

Docker Swarm is Docker's own built-in orchestrator: docker swarm init, then docker service create --replicas 3 my-node-app gives you multi-node scheduling, rolling updates, and service discovery without installing Kubernetes. It's far simpler to operate than Kubernetes but has a much smaller ecosystem and is rarely the default choice for new production systems today — most teams either stay on plain Docker Compose for small deployments or go straight to Kubernetes for anything that needs to scale. Know it exists and roughly what it does; you won't be expected to be a Swarm expert, but "Swarm vs Kubernetes, when would you pick Swarm?" is a fair interview question (answer: small team, want native Docker CLI/UX, don't need Kubernetes' ecosystem).

#7.6 Unit 07 Labs

  1. Restart policy simulation. Run a container with --restart on-failure:3, make the app process process.exit(1) deliberately, and watch docker ps/docker inspect track the restart count.
  2. Trigger an OOM kill. Run a small Node script that intentionally allocates memory in a loop, cap the container at --memory=100m, and confirm via docker inspect that OOMKilled: true and exit code 137 appear.
  3. Add and observe a HEALTHCHECK. Add the Express /health example above, break it deliberately (e.g., return 500), and watch the container flip to unhealthy in docker ps.
  4. Log rotation. Configure --log-opt max-size=1m --log-opt max-file=2 on a container that logs heavily, and confirm old log files get rotated/deleted on disk.

#7.7 Unit 07 Interview Q&A

  • Q: "A container keeps exiting with code 137 — what's happening and how do you fix it?" A: 137 = 128 + SIGKILL(9), almost always an OOM kill. Confirm with docker inspect (OOMKilled: true), then either raise the memory limit if the workload genuinely needs it, fix a memory leak in the app, or tune the app's own memory ceiling (e.g., V8's --max-old-space-size for Node) to stay safely under the container limit.

  • Q: "What's the difference between always and unless-stopped restart policies?" A: Both restart the container on crash or daemon restart. The difference is after a host reboot: always will restart the container even if you had manually stopped it before the reboot; unless-stopped respects that manual stop and leaves it stopped.

#Unit 08 — Docker Troubleshooting & Interview Mastery

Difficulty: Advanced · Goal: Debug confidently, under time pressure, in front of an interviewer or a production incident.

#8.1 The Systematic Debugging Workflow

When something's wrong with a container, work through this order rather than guessing:

bash
docker ps -a                     # 1. Is it running, exited, or restarting?
docker logs --tail 100 -f <id>   # 2. What did the app itself say before it died?
docker inspect <id>              # 3. Exit code, OOMKilled, mounts, env, network config
docker stats <id>                # 4. Live CPU/memory — is it starved or leaking?
docker exec -it <id> sh          # 5. Get inside (if a shell exists) and look around live
docker events                    # 6. Host-level timeline of container lifecycle events

docker inspect is the single richest source of truth — docker inspect <id> --format '{{.State.ExitCode}} {{.State.OOMKilled}}' gets you the two most important fields directly without hunting through JSON.

#8.2 Common Failure Scenarios Decoded

Symptom Likely cause How to confirm
Exit code 137 OOM killed (or manual docker kill) docker inspect → OOMKilled: true
Exit code 1 Uncaught app exception / non-zero process.exit docker logs will show the stack trace
Exit code 139 Segfault (SIGSEGV) — rare in pure JS, more common with native addons Check native module compatibility (e.g., glibc vs musl on Alpine)
docker run fails with "port is already allocated" Another container/process already bound that host port docker ps / lsof -i :<port> on host
"Permission denied" writing to a mounted volume UID mismatch between container user and host file ownership docker exec <id> id vs ls -la on the host path
Container can't resolve another container's hostname Both containers aren't on the same user-defined network (default bridge has no automatic DNS) docker network inspect <network>
Container starts then immediately exits with code 0 The main process (PID 1) finished and had nothing left to do — common when CMD runs a one-shot script instead of a long-running server Check your CMD/ENTRYPOINT actually starts a persistent process

#8.3 "Works on My Machine" — Diagnosing Environment Drift

The most common Node.js-specific cause: native modules built against the wrong libc. A dependency like bcrypt or sharp compiled on your Mac (or a Debian-based CI runner) may not load correctly inside an alpine (musl libc) container.

bash
# Reproduce a "works on my machine" bug by building inside the exact target image
docker run --rm -v $(pwd):/app -w /app node:20-alpine npm ci

# Inspect what actually got installed inside the container vs locally
docker run --rm -v $(pwd):/app -w /app node:20-alpine \
  node -e "console.log(require('bcrypt'))"

Other common drift sources: different Node.js versions between local and image (node --version inside vs outside the container), timezone/locale differences affecting date logic, and missing environment variables that exist in a local .env but were never passed via -e or --env-file to the container.

#8.4 Rapid-Fire Concept Sheet

Use this as flash-card material before an interview:

  • Image vs container: an image is a read-only, layered template; a container is a running (or stopped) instance of that image with its own writable layer.
  • Union filesystem: Docker stacks image layers using a copy-on-write filesystem (overlay2 is the modern default), so multiple containers can share the same underlying read-only layers while each keeps its own thin writable layer.
  • PID 1 in containers: the process specified in CMD/ENTRYPOINT becomes PID 1 inside the container's PID namespace. PID 1 doesn't get default signal handling the way normal processes do, so if your app doesn't explicitly handle SIGTERM, docker stop may hang until the timeout and then force-kill it. This is why tools like tini or Node's built-in signal handling (process.on('SIGTERM', ...)) matter for graceful shutdown.
  • Idempotency of builds: given the same Dockerfile and build context, you should get a functionally identical image — this is why pinning versions (node:20.11.1-alpine instead of node:latest) matters.
  • Artifact immutability: once an image is built and tagged with a specific version/SHA, it should never be overwritten — rebuild and re-tag instead, so every environment can trust that a given tag always means the same bits.
  • Secrets handling: never COPY secrets into a layer; use BuildKit secret mounts at build time and runtime secret injection (env vars from a vault, mounted files, or orchestrator secrets) at run time.
  • Rollback strategy: because tags are immutable and versioned, rollback is just redeploying the previous known-good tag — this is the entire reason the "latest is dangerous" rule from Unit 06 exists.

#8.5 Unit 08 Labs

  1. Restart-loop debugging. Deliberately misconfigure a container (e.g., point it at a nonexistent env var it requires) so it crash-loops, then use docker logs + docker inspect to identify and fix the root cause.
  2. Exit code triage. Produce three containers that exit with codes 137 (OOM), 1 (app crash), and 0 (process finished normally) on purpose, and write down how you distinguished each using only docker inspect/docker logs.
  3. Live exec investigation. Start a container, docker exec -it into it, and inspect running processes (ps aux), open files, and environment variables from inside — without stopping the container.
  4. End-to-end mock walkthrough. Out loud (or in writing), narrate the full flow of docker build → docker run → networking → volume mounting for a Node.js app, as if explaining it to an interviewer, without skipping a step.

#8.6 Scenario-Based Mock Interview Script

Interviewer: "A container keeps restarting in prod. Walk me through your debugging steps."

Strong answer structure:

  1. "First I'd check docker ps -a to confirm it's actually in a restart loop and see the restart count."
  2. "Then docker logs --tail 200 to see what the application printed right before each crash — usually the actual root cause is right there in the stack trace."
  3. "I'd run docker inspect and check the exit code and OOMKilled field specifically — that immediately tells me if it's a memory issue versus an application-level crash."
  4. "If it's memory, I'd check docker stats for a live view, and consider whether it's a genuine leak (fix the code) or just an undersized limit (adjust --memory or the container's replica sizing)."
  5. "If it's an application exception, I'd reproduce locally by running the exact image (not just the source code) with the same environment variables, since drift between the image and local dev is a very common cause."
  6. "Finally, I'd check whether this started right after a deploy — if so, the fastest safe mitigation is rolling back to the previous immutable tag while I root-cause the new one."

That six-step shape (observe → logs → inspect → live metrics → reproduce → mitigate/rollback) is a reusable template for almost any "walk me through debugging X" question in Docker/CI interviews.
