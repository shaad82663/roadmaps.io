#Unit 23 — CI/CD Security: Secrets, SAST/DAST, Supply Chain

#23.1 Secrets Management

Never hardcode credentials in source or in plain pipeline YAML. The maturity ladder, from weakest to strongest:

Approach Risk
Hardcoded in source / committed .env Critical — permanently in git history
CI provider's encrypted secrets (GitHub Secrets, Jenkins Credentials) Good baseline — encrypted at rest, masked in logs
Centralized secrets vault (HashiCorp Vault, AWS Secrets Manager) Better — rotation, fine-grained access policies, audit trail
Short-lived tokens via OIDC (no stored secret at all) Best — nothing to leak because nothing persists
yaml
# GitHub Actions pulling a secret from HashiCorp Vault at runtime instead of storing it directly
- uses: hashicorp/vault-action@v3
  with:
    url: https://vault.mycompany.com
    method: jwt
    role: github-actions-role
    secrets: |
      secret/data/prod/db PASSWORD | DB_PASSWORD

A Node.js habit worth building: never console.log(process.env) in shared code paths, and add a lint rule / pre-commit hook to catch accidental secret-looking strings before they're even committed.

#23.2 Secret Scanning

Secret scanning tools scan git history and diffs for patterns that look like credentials (AWS keys, private keys, API tokens) and block the commit/PR before it merges.

yaml
# .github/workflows/secret-scan.yml
name: Secret Scan
on: [pull_request]

jobs:
  gitleaks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0        # full history — gitleaks needs to scan past commits too
      - uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

Why fetch-depth: 0 matters: a shallow checkout (default is depth=1) only has the latest commit — a secret introduced 3 commits ago in the same PR could slip through unscanned.

#23.3 SAST vs DAST

SAST (Static) DAST (Dynamic)
What it analyzes Source code, without running it A running application, via simulated attacks
When it runs On every commit/PR, fast Usually against a deployed staging environment
Finds SQL injection patterns, hardcoded secrets, insecure deserialization, known-vulnerable code patterns Actual exploitable behavior — auth bypass, XSS that fires in the browser, misconfigured headers
Node.js tools CodeQL, SonarQube, Semgrep, npm audit OWASP ZAP, Burp Suite
yaml
# CodeQL SAST scan for a Node.js repo
name: CodeQL
on:
  push:
    branches: [main]
  pull_request:

jobs:
  analyze:
    runs-on: ubuntu-latest
    permissions:
      security-events: write
    steps:
      - uses: actions/checkout@v4
      - uses: github/codeql-action/init@v3
        with:
          languages: javascript-typescript
      - uses: github/codeql-action/analyze@v3
yaml
# npm's built-in dependency vulnerability check — cheap first line of defense
- run: npm audit --audit-level=high

Interview framing: SAST catches "this code pattern is dangerous" before deploy; DAST catches "this deployed system is actually exploitable" after deploy. Mature pipelines run both — SAST as a fast PR gate, DAST as a slower scheduled/staging-environment job.

#23.4 Software Bill of Materials (SBOM)

An SBOM is a machine-readable inventory of every dependency (direct and transitive) in your build artifact — critical for answering "are we affected by CVE-2024-XXXX?" in minutes instead of days.

bash
# Generate an SBOM for a Node project using Syft
syft packages dir:. -o cyclonedx-json > sbom.json

# Or for a built Docker image
syft packages my-app:1.0 -o spdx-json > sbom.json
yaml
# In a pipeline
- name: Generate SBOM
  uses: anchore/sbom-action@v0
  with:
    image: my-app:1.0
    format: cyclonedx-json
    output-file: sbom.json

- name: Upload SBOM as artifact
  uses: actions/upload-artifact@v4
  with:
    name: sbom
    path: sbom.json

For a Node.js app specifically, npm ls --all --json gives you the raw dependency tree, but a proper SBOM tool (Syft, cdxgen) produces a standardized format (SPDX or CycloneDX) that vulnerability scanners and compliance tooling can actually consume.

#23.5 Supply Chain Attacks — Real Lessons

These are commonly referenced in interviews as case studies:

  • event-stream (2018, npm) — a popular npm package's maintainership was socially engineered away from the original author; the new "maintainer" injected code that targeted a specific cryptocurrency wallet app's users. Lesson: transitive dependencies you've never heard of can run arbitrary code in your build/runtime.
  • SolarWinds (2020) — attackers compromised the build system itself, not the source repo, injecting malicious code during compilation. Lesson: your CI/CD pipeline is as much an attack target as your source code — pipeline integrity matters.
  • xz-utils backdoor (2024) — a long-con social-engineering campaign gained co-maintainer trust over years before inserting a backdoor into a foundational compression library used by OpenSSH indirectly. Lesson: even deeply trusted, boring infrastructure dependencies need scrutiny; "it's a well-known library" isn't sufficient assurance.

Practical Node.js takeaways: pin exact dependency versions (package-lock.json committed, npm ci not npm install in CI), audit new transitive dependencies before major version bumps, and treat your CI runners/build servers as sensitive infrastructure, not throwaway compute.

#23.6 Signing Artifacts & Verifying Provenance

Signing proves an artifact (container image, npm package, binary) genuinely came from your pipeline and wasn't tampered with afterward. Sigstore/cosign does this without requiring you to manage long-lived signing keys yourself (it uses short-lived certificates tied to your CI identity via OIDC).

bash
# Sign a container image (keyless, using OIDC identity from CI)
cosign sign my-registry.io/my-app:1.0

# Verify it later, e.g. before a Kubernetes deploy
cosign verify my-registry.io/my-app:1.0 \
  --certificate-identity=https://github.com/my-org/my-repo/.github/workflows/deploy.yml@refs/heads/main \
  --certificate-oidc-issuer=https://token.actions.githubusercontent.com
yaml
# In a GitHub Actions pipeline
permissions:
  id-token: write     # required for keyless signing via OIDC
  contents: read

steps:
  - name: Install cosign
    uses: sigstore/cosign-installer@v3

  - name: Sign image
    run: cosign sign --yes my-registry.io/my-app:${{ github.sha }}

A deploy step can then refuse to deploy any image that doesn't verify — closing the loop between "this image was actually built by our pipeline" and "we only run verified images in production."

#23.7 Labs

  1. Add a gitleaks step to a pipeline, commit a fake AWS key on a branch, and confirm the PR gets blocked.
  2. Integrate CodeQL (or Semgrep) into a Node.js repo's pipeline and fix one real finding it surfaces.
  3. Generate an SBOM for a sample project with Syft, and grep it to check whether a specific vulnerable package version is present.
  4. Sign a container image with cosign (keyless via OIDC) and write a deploy-gate step that verifies the signature before allowing deployment.

#Unit 24 — RBAC, Compliance & Governance in CI/CD

#24.1 Role-Based Access Control

GitHub Actions / repo-level:

Role Can do
Read View workflows and runs
Triage/Write Trigger workflow_dispatch, re-run failed jobs
Maintain/Admin Edit workflow files, manage secrets, manage environments

Environment-level protection (from Phase 4) is itself a form of RBAC — you can restrict who can approve a deploy to production independent of general repo write access.

Jenkins, via the Role Strategy plugin, applies RBAC at folder level:

text
Folder: team-payments/
  Role: payments-developer
    - Job/Build
    - Job/Read
    - Job/Workspace
  Role: payments-admin
    - Job/Configure
    - Job/Delete
    - Credentials/Manage

This lets a large org give the payments team full control over their own jobs without granting access to, say, the infra team's Jenkins folder — critical once you're past a handful of pipelines.

#24.2 Audit Logging & Traceability

For SOC2/ISO 27001-style compliance, you generally need to answer: who did what, when, and was it authorized?

  • GitHub: Audit log (org settings) records repo/secret/permission changes; combined with required PR reviews + branch protection, gives you "who approved this deploy" traceability.
  • Jenkins: the Audit Trail plugin logs job config changes, credential access, and build triggers with the triggering user.
yaml
# Branch protection config (illustrative — normally set via GitHub UI/API)
required_status_checks:
  contexts: ["ci/test", "ci/security-scan"]
required_pull_request_reviews:
  required_approving_review_count: 2
  require_code_owner_reviews: true
enforce_admins: true         # even admins can't bypass the rules

enforce_admins: true is the detail auditors specifically look for — without it, admins can merge around every control you just configured, which defeats the compliance story.

#24.3 Policy as Code

Rather than relying on tribal knowledge ("everyone knows pipelines need a security-scan stage"), Policy as Code encodes and automatically enforces such rules using tools like OPA (Open Policy Agent) with Conftest.

rego
# policy/pipeline.rego — deny any pipeline missing a security scan stage
package main

deny[msg] {
    not contains_stage(input, "security-scan")
    msg := "Pipeline is missing a required 'security-scan' stage"
}

contains_stage(pipeline, name) {
    some job
    pipeline.jobs[job].name == name
}
bash
# Run as a CI gate against a parsed pipeline definition
conftest test pipeline-config.json --policy policy/
yaml
# Wired into GitHub Actions as a hard gate
- name: Validate pipeline policy
  run: conftest test .github/workflows/deploy.yml --policy policy/

This turns "you're supposed to include a security scan" from a code-review nitpick into a machine-enforced, unbypassable rule across every one of an org's repos.

#24.4 CODEOWNERS & Protected Branches at Scale

text
# .github/CODEOWNERS
# Require specific teams to review sensitive paths

/infra/                 @my-org/platform-team
/.github/workflows/     @my-org/platform-team
/src/payments/          @my-org/payments-team
*.tf                    @my-org/platform-team

Combined with branch protection's "Require review from Code Owners," this guarantees that, e.g., no one can change a GitHub Actions workflow file (a common privilege-escalation vector — modifying CI to exfiltrate secrets) without platform-team sign-off, regardless of who opens the PR.

At scale (200+ pipelines, the classic interview framing), the governance stack layers as:

  1. CODEOWNERS — the right humans review the right code
  2. Branch protection + required status checks — nothing merges without CI passing
  3. Policy as Code (OPA/Conftest) — every pipeline structurally complies with org rules, checked automatically
  4. Audit logs — a durable record for after-the-fact review
  5. RBAC — least-privilege access to secrets, environments, and job configuration

#24.5 Labs

  1. Configure RBAC in a Jenkins folder using the Role Strategy plugin, giving one team build/read-only and another team full admin.
  2. Add a CODEOWNERS file requiring your platform/infra team to review any change under .github/workflows/.
  3. Write an OPA/Conftest policy that fails a pipeline definition missing a required security-scan stage, and wire it into a workflow as a gate.
  4. Audit a sample pipeline against a compliance checklist (secrets scanning present? required reviewers on prod? enforce_admins set?) and document the gaps you find plus how you'd fix each.
