#Unit 0 — Docker Fundamentals & Architecture

#0.1 Containers vs Virtual Machines

As a Node backend engineer you've probably deployed to VMs (EC2, a DigitalOcean droplet) before. The mental shift to containers is about what layer of the stack gets duplicated.

Virtual Machine Container
Isolation unit Full OS (own kernel) Process (shares host kernel)
Boot time Seconds to minutes Milliseconds to ~1s
Size GBs (full OS image) MBs (just app + deps)
Isolation mechanism Hypervisor (hardware-level) Linux namespaces + cgroups
Density per host Low (10s) High (100s–1000s)

Key insight: a container is just a regular Linux process that the kernel has made believe it has its own filesystem, network stack, and process tree, using two kernel features:

  • Namespaces — isolate what a process can see (PID namespace = its own process tree, NET namespace = its own network interfaces, MNT namespace = its own filesystem view, etc.)
  • cgroups (control groups) — limit what a process can use (CPU shares, memory limits, I/O bandwidth)

Docker didn't invent either of these — it made them usable via a friendly CLI/API instead of raw unshare/cgcreate syscalls.

#0.2 Docker Engine Architecture

text
┌─────────────────────────────────────────────┐
│                 docker CLI                    │  (client - what you type)
└───────────────────┬───────────────────────────┘
                     │  REST API (over unix socket / TCP)
┌───────────────────▼───────────────────────────┐
│              dockerd (daemon)                  │  (manages images, networks, volumes)
└───────────────────┬───────────────────────────┘
                     │  gRPC
┌───────────────────▼───────────────────────────┐
│                containerd                      │  (manages container lifecycle)
└───────────────────┬───────────────────────────┘
                     │
┌───────────────────▼───────────────────────────┐
│                   runc                         │  (actually creates the container via
└─────────────────────────────────────────────────┘   namespaces/cgroups, OCI spec)
  • docker CLI — the client you type commands into. It's a thin wrapper that calls the REST API.
  • dockerd — the daemon. Owns images, networks, volumes, and orchestrates containerd.
  • containerd — a separate CNCF project; handles pulling images, managing container lifecycle (start/stop/pause).
  • runc — the low-level OCI-compliant runtime that actually invokes clone() with namespace flags and sets up cgroups.

This layered design is why Kubernetes can also talk to containerd directly without needing dockerd at all (this is why "Docker support removed from Kubernetes" headlines happened — only the dockerd-specific shim was dropped, not container tech itself).

#0.3 Images vs Containers

  • An image is a read-only template — a stack of filesystem layers plus metadata (entrypoint, env, exposed ports).
  • A container is a running (or stopped) instance of an image, with one additional writable layer on top (via a union filesystem, typically overlay2 on Linux).
text
Image layers (read-only, shared across containers):
┌─────────────────────┐
│ Layer 3: COPY . .    │
├─────────────────────┤
│ Layer 2: RUN npm ci  │
├─────────────────────┤
│ Layer 1: FROM node   │
└─────────────────────┘
           +
┌─────────────────────┐
│ Writable layer        │  ← unique per container, deleted with `docker rm`
└─────────────────────┘

This is why container storage is ephemeral by default — anything written to that top writable layer disappears when the container is removed (this is exactly what Unit 3 / Volumes solves).

#0.4 Installing & Verifying Docker

bash
# verify install
docker version        # client + server (daemon) version info
docker info            # daemon-level info: storage driver, cgroup version, # containers/images

# sanity check
docker run hello-world

If docker info fails but docker version shows only the Client block — your daemon isn't running or your user lacks permission (on Linux, add yourself to the docker group: sudo usermod -aG docker $USER, then re-login).

#0.5 What Actually Happens on docker run

This is the single most common Docker interview question. Walk through it precisely:

bash
docker run -d -p 8080:3000 --name api my-node-app:1.0
  1. CLI → daemon: docker CLI sends a REST request to dockerd.
  2. Image resolution: dockerd checks if my-node-app:1.0 exists locally. If not, it pulls it from the configured registry (Docker Hub by default), layer by layer, verifying checksums.
  3. containerd invoked: dockerd asks containerd to create a container from that image.
  4. runc creates the container: containerd (via containerd-shim) invokes runc, which:
    • Creates new Linux namespaces (PID, NET, MNT, UTS, IPC, USER)
    • Sets up cgroups for resource limits
    • Sets up the union filesystem (image layers + new writable layer)
    • Configures the network (creates a veth pair, attaches to the bridge network, assigns an IP)
  5. Entrypoint/CMD executed: The container's main process (PID 1 inside the container's PID namespace) starts — e.g., node server.js.
  6. Port mapping applied: iptables/NAT rules are set up so host port 8080 forwards to container port 3000.
  7. Detached mode: since -d was passed, control returns to your shell immediately; the container runs in the background.

#0.6 Interview Trap Answered

"What's the difference between docker run and docker start?"

docker run = docker create (make a new container from an image) + docker start (run it). docker start only works on an existing, stopped container — it reuses the same writable layer and won't re-pull the image.

#0.7 Labs

  1. Install Docker and run hello-world and nginx; verify the nginx container's IP with docker inspect.
  2. Run docker inspect <container> vs docker inspect <image> — diff the JSON structures side by side.
  3. Write out, in your own words, all 7 steps above for a container you actually ran.
  4. Time docker run vs booting an EC2/VM instance for the same base OS. Compare orders of magnitude.

#Unit 1 — Dockerfile, Images & Layer Caching

#1.1 Core Dockerfile Instructions

dockerfile
FROM node:20-alpine        # base image — always pin a version, never `node:latest` in prod
WORKDIR /app                # sets cwd for subsequent instructions; creates dir if missing
COPY package*.json ./       # copy dependency manifests FIRST (see 1.3 on caching)
RUN npm ci --omit=dev       # install deps in a reproducible, cache-friendly layer
COPY . .                    # now copy the rest of the source
ENV NODE_ENV=production
EXPOSE 3000                 # documentation only — does NOT publish the port (see Unit 2)
CMD ["node", "server.js"]   # default command when container starts
Instruction Purpose
FROM Base image to build on top of
RUN Executes a command at build time, result baked into a new layer
COPY Copies files from build context into the image
ADD Like COPY, but also auto-extracts tarballs and fetches URLs (avoid unless you need that)
WORKDIR Sets the working directory for subsequent instructions
ENV Sets environment variables available at build and run time
EXPOSE Documents which port the app listens on (metadata only)
CMD Default command/args when container starts (overridable at docker run)
ENTRYPOINT Fixed command that always runs; CMD becomes its default arguments

#1.2 Build Context & .dockerignore

When you run docker build ., that . is the build context — the entire directory gets tar'd and sent to the daemon before any instruction runs. If you don't exclude node_modules or .git, every build sends megabytes/gigabytes needlessly and can leak secrets.

text
# .dockerignore
node_modules
npm-debug.log
.git
.env
*.md
coverage
.vscode
Dockerfile

Node-specific gotcha: if you COPY . . without ignoring node_modules, you'll ship your host's node_modules (wrong OS/arch native bindings) into the image instead of the ones installed cleanly inside the container by RUN npm ci.

#1.3 Layer Caching Mechanics

Every instruction in a Dockerfile creates a layer, and Docker caches each layer keyed by (instruction + its inputs). On a rebuild, Docker reuses cached layers until the first instruction whose input changed — everything after that is rebuilt from scratch.

This is why instruction order matters enormously:

dockerfile
# ❌ BAD: any source change invalidates npm install too
COPY . .
RUN npm ci

# ✅ GOOD: dependency layer only invalidates when package*.json changes
COPY package*.json ./
RUN npm ci
COPY . .

With the "good" ordering, editing src/index.js and rebuilding will skip the npm ci layer entirely (cache hit), saving potentially minutes per build — this is the #1 lever for fast CI pipelines.

#1.4 Multi-Stage Builds

Multi-stage builds let you use one image to build (with devDependencies, compilers, TypeScript, etc.) and a slim image to run — only copying the final build artifacts across.

dockerfile
# ---- Stage 1: build ----
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                     # full deps, including devDependencies
COPY . .
RUN npm run build              # e.g., tsc compiles TS -> dist/

# ---- Stage 2: production runtime ----
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
COPY package*.json ./
RUN npm ci --omit=dev          # only prod deps in the final image
COPY --from=builder /app/dist ./dist
USER node                      # non-root (built into the official node image)
EXPOSE 3000
CMD ["node", "dist/server.js"]

Result: the final image contains no TypeScript compiler, no devDependencies, no source .ts files — just compiled JS + prod node_modules. This commonly cuts image size by 60–80% and shrinks the attack surface.

#1.5 CMD vs ENTRYPOINT vs ADD vs COPY

CMD vs ENTRYPOINT — the #1 asked Docker interview question:

dockerfile
ENTRYPOINT ["node"]
CMD ["server.js"]

docker run my-image → runs node server.js. docker run my-image worker.js → runs node worker.js (CMD is overridden, ENTRYPOINT is not).

  • CMD alone: provides a default command, fully replaceable at docker run.
  • ENTRYPOINT alone: the fixed command; anything passed at docker run becomes arguments appended to it.
  • ENTRYPOINT + CMD together: ENTRYPOINT is fixed, CMD provides default arguments that are easy to override — this is the idiomatic pattern for CLI-style images.

ADD vs COPY: Always prefer COPY — it's explicit and predictable. Only use ADD when you specifically need its two superpowers: auto-extracting local .tar.gz archives, or fetching a remote URL (though for URLs, a RUN curl is usually clearer and more cacheable).

#1.6 Labs

  1. Write a Dockerfile for a Node app, deliberately putting COPY . . before RUN npm ci, then fix the ordering and prove the cache-hit difference with docker build timing.
  2. Convert a single-stage Dockerfile into a multi-stage build for a TypeScript app; compare docker images sizes before/after.
  3. Add .dockerignore for node_modules, .git, .env; compare "Sending build context to Docker daemon" size in build logs before/after.
  4. Build an image with ENTRYPOINT ["node"] + CMD ["server.js"], then run it overriding CMD to execute a different script.

#Unit 2 — Docker Networking

#2.1 Network Drivers Overview

Driver Behavior When to use
bridge (default) Container gets its own private IP on a virtual bridge (docker0); NAT'd to host Default for standalone containers
host Container shares the host's network namespace directly — no isolation, no NAT Max performance, single-container-per-host scenarios
none No networking at all Fully isolated / security-sensitive batch jobs
user-defined bridge Like bridge, but with automatic DNS-based service discovery by container name Multi-container apps (recommended default)
bash
docker network ls                 # list networks
docker network create my-net      # create a user-defined bridge
docker network inspect my-net     # see connected containers, subnet, gateway

#2.2 Port Publishing vs Exposing

This trips up almost every backend engineer new to Docker:

  • EXPOSE 3000 in a Dockerfile is documentation only — it does not open any port to the host. It's a hint for humans and tools like Compose.
  • -p 8080:3000 at docker run is what actually creates the host→container port mapping (via iptables NAT rules).
bash
docker run -d -p 8080:3000 my-node-app
#              ^host  ^container
# Traffic to localhost:8080 on the HOST is forwarded to port 3000 INSIDE the container

Without -p, the container is still reachable from other containers on the same Docker network on port 3000 — just not from your host's browser/curl.

#2.3 User-Defined Bridge Networks & DNS

The default bridge network does not provide automatic DNS resolution between containers by name — you'd need to link containers manually or use raw IPs (fragile, since IPs change on restart). A user-defined bridge network solves this: Docker runs an embedded DNS server that resolves container names automatically.

bash
docker network create app-net

docker run -d --name db --network app-net postgres:16
docker run -d --name api --network app-net -p 3000:3000 my-node-app

# inside the `api` container's code, you can now connect via:
# postgresql://db:5432/mydb   <-- "db" resolves via Docker's embedded DNS

#2.4 Multi-Container Service Discovery

js
// Node.js example — connecting to a container named "db" on the same user-defined network
const { Pool } = require('pg');
const pool = new Pool({
  host: 'db',        // container name = hostname, resolved by Docker's DNS
  port: 5432,
  user: 'postgres',
  password: process.env.DB_PASSWORD,
  database: 'mydb',
});

This is precisely the mechanism Docker Compose uses under the hood (Unit 4) — Compose auto-creates a user-defined network per project and every service is reachable by its service name.

#2.5 Labs

  1. Create a user-defined bridge network, run two containers on it, and ping (or curl) one from the other by container name.
  2. Run a web server with -p 8080:80 and confirm curl localhost:8080 works from the host; then run it without -p and confirm it only works from another container.
  3. Compare --network host vs default bridge for a simple HTTP server — benchmark latency difference (usually negligible for most apps but worth seeing).
  4. Deliberately misconfigure two containers on different networks and diagnose the "can't reach" failure using docker network inspect and docker exec ... ping.

#Unit 3 — Docker Storage & Volumes

#3.1 Named Volumes vs Bind Mounts vs tmpfs

Type Managed by Location Use case
Named volume Docker /var/lib/docker/volumes/... (Docker-managed) Persistent app data (DB files) — portable, backed up easily
Bind mount You Any host path you specify Local dev — live-reload source code into a container
tmpfs mount Docker (in-memory) RAM only, never touches disk Secrets/temp data that must never persist
bash
# Named volume
docker run -d -v pgdata:/var/lib/postgresql/data postgres:16

# Bind mount (absolute host path required)
docker run -d -v $(pwd)/src:/app/src -p 3000:3000 my-node-app

# tmpfs
docker run -d --tmpfs /app/tmp my-node-app

Modern, more explicit syntax (--mount, recommended for clarity in production Dockerfiles/scripts):

bash
docker run -d \
  --mount type=volume,source=pgdata,target=/var/lib/postgresql/data \
  postgres:16

#3.2 Volume Lifecycle

bash
docker volume create pgdata          # create explicitly (optional — auto-created on first use)
docker volume ls                     # list all volumes
docker volume inspect pgdata         # see mountpoint, driver, labels
docker volume prune                  # remove ALL unused (unmounted) volumes — careful!
docker volume rm pgdata              # remove a specific one (must not be in use)

Key fact: data in a named volume survives docker rm <container> — the volume is a separate object from the container. It's only deleted if you explicitly docker volume rm it, or run docker rm -v combined with the volume having no other references.

#3.3 Sharing Data Between Containers

bash
docker volume create shared-data

docker run -d --name writer -v shared-data:/data alpine sh -c "while true; do date >> /data/log.txt; sleep 5; done"
docker run -d --name reader -v shared-data:/data alpine sh -c "tail -f /data/log.txt"

Both containers mount the same volume — Docker handles concurrent access at the filesystem level (your application is still responsible for handling concurrent writes safely, e.g., file locks, if that matters for your use case).

#3.4 Backup & Restore

The standard pattern: spin up a throwaway container that mounts the volume plus a host directory, and use tar.

bash
# Backup: tar the volume's contents to a file on the host
docker run --rm \
  -v pgdata:/data \
  -v $(pwd)/backup:/backup \
  alpine tar czf /backup/pgdata-backup.tar.gz -C /data .

# Restore: untar into a (new) volume
docker volume create pgdata-restored
docker run --rm \
  -v pgdata-restored:/data \
  -v $(pwd)/backup:/backup \
  alpine tar xzf /backup/pgdata-backup.tar.gz -C /data

#3.5 Labs

  1. Run Postgres with a named volume, insert data, docker rm -f the container, start a new container with the same volume, and confirm the data is still there.
  2. Bind-mount your local src/ folder into a Node container running with nodemon, edit a file on the host, and observe the live reload.
  3. Back up a volume to a .tar.gz, then restore it into a freshly created volume and verify data integrity.
  4. Deliberately forget to attach a volume to a stateful container, remove the container, and confirm the data loss — then fix it.

#Unit 4 — Docker Compose for Multi-Container Apps

#4.1 Compose File Structure

yaml
# docker-compose.yml
services:
  api:
    build: .                      # build from local Dockerfile
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - DB_HOST=db
      - REDIS_HOST=cache
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    networks:
      - app-net

  db:
    image: postgres:16
    environment:
      - POSTGRES_PASSWORD=secret
      - POSTGRES_DB=mydb
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5
    networks:
      - app-net

  cache:
    image: redis:7-alpine
    networks:
      - app-net

volumes:
  pgdata:

networks:
  app-net:
    driver: bridge

Compose does three things for you automatically that you'd otherwise do manually: creates a dedicated network (so db and cache resolve by name, per Unit 2), creates named volumes as declared, and builds/starts everything with one command.

bash
docker compose up -d          # build (if needed) + start everything, detached
docker compose ps             # status of all services
docker compose logs -f api    # follow logs for one service
docker compose down           # stop and remove containers + network (NOT volumes, by default)
docker compose down -v        # also remove volumes — destructive!

#4.2 depends_on, Healthchecks & Startup Ordering

A common Node.js pitfall: depends_on (without condition) only waits for the dependency container to start, not for the database inside it to be ready to accept connections. Postgres, for example, takes a moment after container-start before it accepts connections.

yaml
depends_on:
  db:
    condition: service_healthy   # waits for the healthcheck to pass, not just container start

Without this, your Node app may crash-loop on boot trying to connect to a DB that technically "started" but isn't accepting connections yet. Two common fixes:

  1. Use condition: service_healthy with a proper healthcheck: (shown above) — the Compose-native solution.
  2. Add retry/backoff logic in your Node app's DB connection code regardless (recommended even in production, since Compose ordering guarantees don't exist in real orchestrators like Kubernetes).

#4.3 Environment Variables & .env Files

Compose automatically loads a .env file from the same directory for variable substitution inside docker-compose.yml:

bash
# .env
DB_PASSWORD=supersecret
APP_PORT=3000
yaml
services:
  api:
    ports:
      - "${APP_PORT}:3000"
    environment:
      - DB_PASSWORD=${DB_PASSWORD}

For multiple environments, use override files instead of duplicating the whole file:

bash
docker compose -f docker-compose.yml -f docker-compose.dev.yml up
yaml
# docker-compose.dev.yml — only the overrides for local dev
services:
  api:
    build:
      target: builder            # use the builder stage, not the slim runtime
    volumes:
      - ./src:/app/src           # bind mount for live reload
    command: npm run dev

#4.4 Scaling Services Locally

bash
docker compose up -d --scale api=3

Note: this only works cleanly when the service doesn't have a fixed host port mapping like "3000:3000" (three containers can't all bind host port 3000). For local scaling experiments, either omit the ports: mapping (access via the internal network only) or map to a range, and put something like Nginx/Traefik in front to load-balance.

#4.5 Labs

  1. Write a docker-compose.yml for a Node API + Postgres + Redis; bring it all up with one command.
  2. Add a healthcheck to the Postgres service and set depends_on.condition: service_healthy on the API; verify the API no longer crash-loops on cold start.
  3. Split into docker-compose.yml (base) + docker-compose.dev.yml (bind mount + dev command), and run both merged for local development.
  4. Scale the API service to 3 replicas (no fixed host port) and verify with docker compose ps that 3 containers are running.
