#Unit 0 — Docker Fundamentals & Architecture

#Concept: What a container actually is

You've deployed Node apps to VMs before — EC2, a droplet, a bare-metal box. The mental shift to containers is about which layer of the stack gets duplicated per app.

Virtual Machine Container
Isolation unit Full OS (own kernel) A process (shares host kernel)
Boot time Seconds to minutes Milliseconds to ~1s
Size GBs (full OS image) MBs (app + deps only)
Isolation mechanism Hypervisor (hardware-level) Linux namespaces + cgroups
Density per host Low (10s) High (100s–1000s)

Here's the part that actually matters for understanding, not memorizing: a container is just a normal Linux process that the kernel has been told to partially blind. Two kernel features make this possible:

  • Namespaces — control what a process can see. A PID namespace gives it its own process tree (it thinks it's PID 1). A NET namespace gives it its own network interfaces. A MNT namespace gives it its own filesystem view.
  • cgroups (control groups) — control what a process can use. CPU shares, memory ceiling, I/O bandwidth.

Docker didn't invent either of these — they're kernel features that existed for years. Docker's real contribution was packaging them behind a friendly CLI and a distributable image format, instead of raw unshare/cgcreate syscalls that almost nobody used directly.

#Concept: The architecture behind docker run

text
docker CLI  →  dockerd (daemon)  →  containerd  →  runc
 (what you        (owns images,      (manages          (actually creates
  type)            networks,          container          the container via
                    volumes)          lifecycle)          namespaces/cgroups)
  • docker CLI — a thin client. Every command you type becomes a REST call.
  • dockerd — the daemon. Owns images, networks, volumes; delegates actual container execution.
  • containerd — a separate CNCF project that handles pulling images and managing container lifecycle (start/stop/pause).
  • runc — the low-level OCI runtime that literally calls clone() with namespace flags and wires up cgroups.

This layering is why Kubernetes dropped dockerd specifically but still runs containers fine — it talks to containerd directly. The underlying container tech didn't go anywhere.

#Concept: Images vs. containers

  • An image = a read-only stack of filesystem layers + metadata (entrypoint, env, exposed ports).
  • A container = a running (or stopped) instance of that image, plus one writable layer on top (via a union filesystem — overlay2 on Linux).
text
Image layers (read-only, shared across every container built from it):
┌─────────────────────┐
│ Layer 3: COPY . .    │
├─────────────────────┤
│ Layer 2: RUN npm ci  │
├─────────────────────┤
│ Layer 1: FROM node   │
└─────────────────────┘
           +
┌─────────────────────┐
│ Writable layer       │  ← unique per container, gone when you `docker rm`
└─────────────────────┘

This single fact explains almost every "why did my data disappear" bug you'll hit later (Unit 3 solves it with volumes) — anything written to that top writable layer dies with the container.

#Hands-On Walkthrough

Open a terminal. Do each step and actually read the output before moving to the next.

bash
# 1. Verify Docker is installed and the daemon is reachable
docker version
docker info | head -20

# 2. Run your first container and watch it pull layer by layer
docker run hello-world

# 3. Run something real, in the background
docker run -d --name my-nginx -p 8080:80 nginx
curl localhost:8080 | head -5          # you should see nginx's welcome HTML

# 4. Look inside the running container
docker inspect my-nginx | less         # scroll through: Network, Mounts, Config
docker exec -it my-nginx sh            # you're now inside the container's shell
   ps aux                              # notice: nginx is PID 1 in here
   exit

# 5. Compare an image's metadata vs a container's metadata
docker inspect nginx | head -30        # image: layers, no "State" block
docker inspect my-nginx | head -30     # container: has "State", "NetworkSettings"

# 6. See the layers themselves
docker history nginx

# 7. Clean up
docker stop my-nginx && docker rm my-nginx

Now trace, out loud, what happened when you ran step 3 — this is the exact chain from the Concept section above:

  1. CLI sent the request to dockerd.
  2. dockerd found nginx:latest locally or pulled it layer by layer from Docker Hub.
  3. dockerd asked containerd to create a container from it.
  4. containerd → runc created new namespaces (PID/NET/MNT/etc.), set up cgroups, mounted the union filesystem, created a network interface.
  5. The image's CMD/ENTRYPOINT (nginx's start script) ran as PID 1 inside the new PID namespace.
  6. Because you passed -p 8080:80, iptables/NAT rules were added so localhost:8080 on your machine forwards to port 80 inside the container.
  7. Because you passed -d, control returned to your shell immediately.

#Practical Takeaway

docker run = docker create + docker start. docker start <name> only works on an existing, stopped container — it reuses the same writable layer and does not re-pull the image. This distinction matters in real debugging: if a container keeps failing right after start, docker start won't fix a bad image — you need to fix the image and re-create the container.

#Try It Yourself

  • Time docker run hello-world vs. spinning up any VM (even a local one) — compare orders of magnitude, not just "it's faster."
  • Run docker stats in one terminal while doing normal work in another, and watch CPU/memory of running containers live.

#Unit 1 — Dockerfile, Images & Layer Caching

#Concept: Core Dockerfile instructions

dockerfile
FROM node:20-alpine        # base image — pin a version, never `node:latest` in prod
WORKDIR /app                # sets cwd for everything after; creates the dir if missing
COPY package*.json ./       # copy dependency manifests FIRST — see caching below
RUN npm ci --omit=dev       # install deps in a reproducible, cache-friendly layer
COPY . .                    # now copy the rest of the source
ENV NODE_ENV=production
EXPOSE 3000                 # documentation only — does NOT publish the port (Unit 2)
CMD ["node", "server.js"]   # default command when the container starts
Instruction What it actually does
FROM Base image to build on top of
RUN Executes a command at build time; the result is baked into a new layer
COPY Copies files from the build context into the image
ADD Like COPY, but also auto-extracts local tarballs and can fetch URLs
WORKDIR Sets the working directory for everything after it
ENV Sets an environment variable, available at build and run time
EXPOSE Documents the port the app listens on (metadata only, doesn't open anything)
CMD Default command/args on container start — fully overridable at docker run
ENTRYPOINT The fixed command that always runs; CMD becomes its default arguments

#Concept: Build context and why .dockerignore matters

docker build . — that . is the build context: the entire directory gets tarred and sent to the daemon before any instruction runs. Skip .dockerignore and you'll send node_modules, .git, and possibly .env secrets into every single build, even the ones that don't need them.

text
# .dockerignore
node_modules
npm-debug.log
.git
.env
*.md
coverage
.vscode
Dockerfile

Node-specific trap: if you COPY . . without ignoring node_modules, you ship your host's node_modules (wrong OS/architecture native bindings, e.g. a Mac-built bcrypt binary inside a Linux container) instead of the ones RUN npm ci installs cleanly inside the container.

#Concept: Why instruction order controls build speed

Every instruction creates a layer, cached by (instruction + its inputs). On rebuild, Docker reuses layers until the first one whose input changed — everything after that rebuilds from scratch, in order, no exceptions.

dockerfile
# ❌ Any source file change invalidates npm ci too — full reinstall every time
COPY . .
RUN npm ci

# ✅ npm ci only reruns when package*.json actually changes
COPY package*.json ./
RUN npm ci
COPY . .

With the second ordering, editing src/index.js and rebuilding skips npm ci entirely — a cache hit. On a real Node project with a few hundred dependencies, that's the difference between a 2-second rebuild and a 90-second one. Multiply that by every CI run in a day and you'll see why this is the single highest-leverage Docker skill for pipeline speed.

#Concept: Multi-stage builds

One image builds (compiles TypeScript, needs devDependencies); a second, slimmer image actually runs. Only the build's output crosses over.

dockerfile
# ---- Stage 1: build ----
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                     # full deps, including devDependencies
COPY . .
RUN npm run build              # tsc compiles TS -> dist/

# ---- Stage 2: production runtime ----
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
COPY package*.json ./
RUN npm ci --omit=dev          # only prod deps in the final image
COPY --from=builder /app/dist ./dist
USER node                      # non-root (built into the official node image)
EXPOSE 3000
CMD ["node", "dist/server.js"]

The final image has no TypeScript compiler, no devDependencies, no .ts source — just compiled JS and prod node_modules. In practice this commonly cuts image size 60–80% and meaningfully shrinks what an attacker could exploit if they got a shell in the container.

#Concept: CMD vs. ENTRYPOINT, ADD vs. COPY

dockerfile
ENTRYPOINT ["node"]
CMD ["server.js"]

  • docker run my-image → runs node server.js

  • docker run my-image worker.js → runs node worker.js (CMD got replaced, ENTRYPOINT didn't)

  • CMD alone — a default, fully replaceable at docker run.

  • ENTRYPOINT alone — fixed; anything passed at docker run is appended as arguments to it.

  • ENTRYPOINT + CMD together — ENTRYPOINT fixed, CMD gives easily-overridden default arguments. This is the idiomatic pattern for CLI-style images.

ADD vs. COPY: default to COPY — explicit, predictable. Only reach for ADD when you need its two extra powers (auto-extracting a local .tar.gz, or fetching a URL) — and even then, a plain RUN curl is usually clearer and more cache-friendly.

#Hands-On Walkthrough

bash
mkdir docker-lab-unit1 && cd docker-lab-unit1
npm init -y
npm install express
cat > server.js <<'EOF'
const express = require('express');
const app = express();
app.get('/', (req, res) => res.send('hello from ' + process.argv[2] || 'default'));
app.listen(3000, () => console.log('listening on 3000'));
EOF

# 1. Bad ordering — prove the cache miss
cat > Dockerfile.bad <<'EOF'
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm ci
CMD ["node", "server.js"]
EOF
docker build -t bad-order -f Dockerfile.bad .
touch server.js && docker build -t bad-order -f Dockerfile.bad .   # watch: npm ci reruns

# 2. Good ordering — prove the cache hit
cat > Dockerfile.good <<'EOF'
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
CMD ["node", "server.js"]
EOF
docker build -t good-order -f Dockerfile.good .
touch server.js && docker build -t good-order -f Dockerfile.good .  # watch: npm ci step says "CACHED"

# 3. .dockerignore effect
echo "node_modules" > .dockerignore
docker build -t good-order -f Dockerfile.good . 2>&1 | grep "Sending build context"

# 4. ENTRYPOINT/CMD override behavior
cat > Dockerfile.entry <<'EOF'
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
ENTRYPOINT ["node"]
CMD ["server.js"]
EOF
docker build -t entry-demo -f Dockerfile.entry .
docker run --rm entry-demo                 # runs node server.js
docker run --rm entry-demo -e "console.log(1+1)"   # ENTRYPOINT stays "node", CMD is replaced

#Practical Takeaway

If you only remember one thing from this unit: put whatever changes least at the top of the Dockerfile, and whatever changes most at the bottom. Dependency manifests change rarely; source code changes constantly. That single ordering decision is responsible for most of the build-speed difference you'll see between a slow CI pipeline and a fast one.

#Try It Yourself

  • Convert a single-stage Dockerfile for a TypeScript app into a multi-stage build; compare docker images sizes before/after.
  • Deliberately break .dockerignore (delete it) and watch the build context size balloon in the build log.

#Unit 2 — Docker Networking

#Concept: The four network modes

Driver What happens When you'd use it
bridge (default) Container gets a private IP on a virtual bridge (docker0); NAT'd to the host Default for a single standalone container
host Container shares the host's network namespace directly — no isolation, no NAT Max performance, single container per host
none No networking at all Fully isolated batch/security-sensitive jobs
user-defined bridge Like bridge, but with automatic DNS resolution by container name Multi-container apps — this is what you'll use 90% of the time
bash
docker network ls
docker network create my-net
docker network inspect my-net     # connected containers, subnet, gateway

#Concept: EXPOSE vs. -p — the thing that confuses everyone

  • EXPOSE 3000 in a Dockerfile is documentation only. It opens nothing. It's a hint for humans and for tools like Compose.
  • -p 8080:3000 at docker run is what actually creates the host→container port mapping, via iptables NAT rules.
bash
docker run -d -p 8080:3000 my-node-app
#              ^host  ^container
# traffic to localhost:8080 on the HOST forwards to port 3000 INSIDE the container

Without -p, the container is still reachable from other containers on the same Docker network, on port 3000 — just not from your host's browser or curl.

#Concept: Why user-defined bridge networks give you DNS for free

The default bridge network does not resolve container names — you'd be stuck hardcoding IPs, which break on every restart. A user-defined bridge network runs an embedded DNS server that resolves container names automatically:

bash
docker network create app-net
docker run -d --name db --network app-net postgres:16
docker run -d --name api --network app-net -p 3000:3000 my-node-app
# inside "api"'s code: postgresql://db:5432/mydb  <-- "db" resolves via Docker's DNS
js
// Node example: connecting to a container named "db" on the same network
const { Pool } = require('pg');
const pool = new Pool({
  host: 'db',        // the container name IS the hostname
  port: 5432,
  user: 'postgres',
  password: process.env.DB_PASSWORD,
  database: 'mydb',
});

This is exactly the mechanism Docker Compose uses under the hood (Unit 4) — it auto-creates a network per project and every service is reachable by its service name.

#Hands-On Walkthrough

bash
# 1. Two containers, same user-defined network, talk by name
docker network create app-net
docker run -d --name web1 --network app-net nginx
docker run -it --rm --network app-net alpine sh
   # inside this shell:
   ping -c 3 web1        # resolves! prints web1's IP
   wget -qO- http://web1 | head -3
   exit

# 2. Prove the default bridge does NOT give you name resolution
docker run -d --name web2 nginx                       # no --network flag = default bridge
docker run -it --rm alpine sh
   ping -c 2 web2         # fails — "bad address"
   exit

# 3. -p vs no -p
docker run -d --name web3 --network app-net -p 8081:80 nginx
curl localhost:8081 | head -3          # works — you published the port
docker run -d --name web4 --network app-net nginx
curl localhost:8082 2>&1 | head -1     # nothing published, nothing to curl on host

# 4. host networking, for comparison
docker run -d --name web5 --network host nginx
curl localhost:80 | head -3            # container's port 80 IS the host's port 80, no mapping needed

# cleanup
docker rm -f web1 web2 web3 web4 web5
docker network rm app-net

#Practical Takeaway

If a Node app in one container can't reach a database in another, the first two things to check are: (1) are they on the same user-defined network (not just both running), and (2) are you using the container name as the hostname, not localhost (a very common mistake — localhost inside a container means that container, not the host machine or a sibling container).

#Try It Yourself

  • Put two containers on different networks on purpose, then diagnose the failure using docker network inspect and docker exec ... ping before fixing it.
  • Benchmark request latency for the same simple HTTP server under bridge vs. host mode — see for yourself how small the real difference usually is.

#Unit 3 — Docker Storage & Volumes

#Concept: Named volumes vs. bind mounts vs. tmpfs

Type Managed by Lives at Best for
Named volume Docker /var/lib/docker/volumes/... Persistent app data (DB files) — portable, easy to back up
Bind mount You Any host path you choose Local dev — live-reloading source code into a container
tmpfs mount Docker (RAM) Never touches disk Secrets/temp data that must never persist
bash
# named volume
docker run -d -v pgdata:/var/lib/postgresql/data postgres:16

# bind mount (needs an absolute host path)
docker run -d -v $(pwd)/src:/app/src -p 3000:3000 my-node-app

# tmpfs
docker run -d --tmpfs /app/tmp my-node-app

More explicit modern syntax, worth using in scripts for clarity:

bash
docker run -d \
  --mount type=volume,source=pgdata,target=/var/lib/postgresql/data \
  postgres:16

#Concept: Why volumes outlive containers

bash
docker volume create pgdata
docker volume ls
docker volume inspect pgdata     # mountpoint, driver, labels
docker volume prune              # removes ALL unused volumes — be careful
docker volume rm pgdata          # removes one specific volume (must not be in use)

The key fact from Unit 0 comes back here: a container's writable layer dies with the container, but a volume is a separate object entirely — it's only deleted if you explicitly docker volume rm it (or docker rm -v when nothing else references it).

#Hands-On Walkthrough

bash
# 1. Prove data survives container removal
docker volume create pgdata
docker run -d --name pg1 -e POSTGRES_PASSWORD=pw -v pgdata:/var/lib/postgresql/data postgres:16
sleep 5
docker exec -it pg1 psql -U postgres -c "CREATE TABLE t(x int); INSERT INTO t VALUES (42);"
docker rm -f pg1                                    # container gone
docker run -d --name pg2 -e POSTGRES_PASSWORD=pw -v pgdata:/var/lib/postgresql/data postgres:16
sleep 5
docker exec -it pg2 psql -U postgres -c "SELECT * FROM t;"   # 42 is still there

# 2. Bind mount for live reload
mkdir -p bindlab/src && cd bindlab
cat > src/index.js <<'EOF'
console.log("version 1");
EOF
docker run --rm -v $(pwd)/src:/app/src -w /app node:20-alpine node src/index.js   # prints "version 1"
echo 'console.log("version 2");' > src/index.js
docker run --rm -v $(pwd)/src:/app/src -w /app node:20-alpine node src/index.js   # prints "version 2" — no rebuild needed
cd ..

# 3. Backup and restore
mkdir -p backup
docker run --rm -v pgdata:/data -v $(pwd)/backup:/backup alpine \
  tar czf /backup/pgdata-backup.tar.gz -C /data .
docker volume create pgdata-restored
docker run --rm -v pgdata-restored:/data -v $(pwd)/backup:/backup alpine \
  tar xzf /backup/pgdata-backup.tar.gz -C /data
docker run -d --name pg3 -e POSTGRES_PASSWORD=pw -v pgdata-restored:/var/lib/postgresql/data postgres:16
sleep 5
docker exec -it pg3 psql -U postgres -c "SELECT * FROM t;"   # 42 should be there too

# cleanup
docker rm -f pg2 pg3
docker volume rm pgdata pgdata-restored

#Practical Takeaway

If you ever see "my container restarted and all my data is gone," the fix is almost always: the container was writing to its own writable layer instead of a mounted volume. Check docker inspect <container> under "Mounts" — if it's empty for a stateful service, that's the bug.

#Try It Yourself

  • Deliberately run a stateful container without a volume, remove it, and confirm the data loss for yourself before fixing it — seeing the failure makes the fix stick better than reading about it.

#Unit 4 — Docker Compose for Multi-Container Apps

#Concept: What Compose actually automates

yaml
# docker-compose.yml
services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - DB_HOST=db
      - REDIS_HOST=cache
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    networks:
      - app-net

  db:
    image: postgres:16
    environment:
      - POSTGRES_PASSWORD=secret
      - POSTGRES_DB=mydb
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5
    networks:
      - app-net

  cache:
    image: redis:7-alpine
    networks:
      - app-net

volumes:
  pgdata:

networks:
  app-net:
    driver: bridge

Compose is doing three things you'd otherwise wire up by hand: creating a dedicated network (so db/cache resolve by name — Unit 2), creating named volumes (Unit 3), and building/starting everything with one command.

bash
docker compose up -d          # build (if needed) + start everything, detached
docker compose ps             # status of every service
docker compose logs -f api    # follow one service's logs
docker compose down           # stop + remove containers and the network (keeps volumes)
docker compose down -v        # also removes volumes — destructive, be deliberate about this

#Concept: The depends_on trap

A very common Node.js pitfall: plain depends_on (no condition) only waits for the dependency's container to start, not for the database inside it to actually accept connections. Postgres takes a moment after container-start before it's ready.

yaml
depends_on:
  db:
    condition: service_healthy   # waits for the healthcheck to pass, not just container start

Without this, your Node app can crash-loop on boot trying to connect to a DB that technically "started" but isn't accepting connections yet. Two real fixes: use condition: service_healthy with a real healthcheck: (shown above), and/or add retry/backoff logic in your app's DB connection code regardless — Compose's ordering guarantees don't exist in real orchestrators like Kubernetes, so the app-level fix is the one that actually matters long-term.

#Concept: Environment variables, .env, and multi-environment setups

Compose auto-loads a .env file from the same directory for variable substitution inside docker-compose.yml:

bash
# .env
DB_PASSWORD=supersecret
APP_PORT=3000
yaml
services:
  api:
    ports:
      - "${APP_PORT}:3000"
    environment:
      - DB_PASSWORD=${DB_PASSWORD}

For dev vs. prod, use an override file instead of duplicating the whole thing:

bash
docker compose -f docker-compose.yml -f docker-compose.dev.yml up
yaml
# docker-compose.dev.yml — only the overrides
services:
  api:
    build:
      target: builder
    volumes:
      - ./src:/app/src
    command: npm run dev

#Hands-On Walkthrough

bash
mkdir compose-lab && cd compose-lab
npm init -y && npm install express pg redis
cat > server.js <<'EOF'
const express = require('express');
const app = express();
app.get('/', (req, res) => res.send('api is up'));
app.listen(3000, () => console.log('listening on 3000'));
EOF
cat > Dockerfile <<'EOF'
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
CMD ["node", "server.js"]
EOF

cat > docker-compose.yml <<'EOF'
services:
  api:
    build: .
    ports:
      - "3000:3000"
    depends_on:
      db:
        condition: service_healthy
    networks: [app-net]
  db:
    image: postgres:16
    environment:
      - POSTGRES_PASSWORD=secret
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5
    networks: [app-net]
volumes:
  pgdata:
networks:
  app-net:
EOF

docker compose up -d
docker compose ps                    # confirm both services are "running" / "healthy"
curl localhost:3000                  # "api is up"
docker compose logs api | tail -10

# prove the depends_on/healthcheck actually matters
docker compose down
sed -i '' -e '/condition: service_healthy/d' -e '/depends_on:/,+1{/db:/d}' docker-compose.yml 2>/dev/null || true
# (if the sed above doesn't apply cleanly on your OS, just manually delete the depends_on
#  block and rerun — watch api's logs for connection-refused errors on a cold start)

docker compose down -v   # cleanup, including volumes

#Practical Takeaway

Compose is the fastest way to reason locally about the exact same networking/volume concepts from Units 2 and 3 — it's not new knowledge, it's automation on top of what you already understand. If Compose behavior ever confuses you, drop back to plain docker network inspect / docker volume inspect on the containers it created — they're ordinary Docker objects underneath.
