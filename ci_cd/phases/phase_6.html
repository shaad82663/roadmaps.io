#20.1 Building & Pushing Docker Images from CI

This unit connects your Docker skills (Phases 1-2) directly into a pipeline. The core loop in any CI system — GitHub Actions or Jenkins — is: checkout code → build image → tag it → push to a registry.

GitHub Actions example:

yaml
name: build-and-push
on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ghcr.io/${{ github.repository }}:${{ github.sha }}
            ghcr.io/${{ github.repository }}:latest

Jenkins declarative equivalent:

groovy
pipeline {
  agent any
  environment {
    IMAGE = "myregistry.io/myapp"
  }
  stages {
    stage('Build Image') {
      steps {
        sh 'docker build -t $IMAGE:$GIT_COMMIT .'
      }
    }
    stage('Push Image') {
      steps {
        withCredentials([usernamePassword(credentialsId: 'registry-creds', usernameVariable: 'USER', passwordVariable: 'PASS')]) {
          sh 'echo $PASS | docker login myregistry.io -u $USER --password-stdin'
          sh 'docker push $IMAGE:$GIT_COMMIT'
        }
      }
    }
  }
}

Notice both use the commit SHA as the primary tag — this is Unit 11's SemVer/immutable-artifact principle applied concretely: every pipeline run produces a uniquely identifiable, traceable image.

#20.2 Reusing Multi-Stage Builds Inside CI

Your Phase 1 multi-stage Dockerfile pays off directly in CI: smaller final images mean faster docker push/pull in every pipeline run, which directly shortens Lead Time for Changes (a DORA metric from Unit 09).

dockerfile
# ---- Build stage ----
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# ---- Runtime stage ----
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package*.json ./
EXPOSE 3000
CMD ["node", "dist/index.js"]

Why this matters specifically in CI (not just locally):

  • The builder stage can be cached as a separate CI layer — if package-lock.json hasn't changed, npm ci layer is reused, and only your app code re-copies/rebuilds.
  • The runtime image that gets pushed to the registry never contains devDependencies, TypeScript source, or build tooling — smaller attack surface (ties back to Phase 2's image-hardening unit) and a much faster push/pull inside the pipeline.

You can also enable BuildKit's inline cache in CI to reuse layers between separate pipeline runs (not just within one build):

yaml
- name: Build with cache
  uses: docker/build-push-action@v5
  with:
    context: .
    push: true
    tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
    cache-from: type=registry,ref=ghcr.io/${{ github.repository }}:buildcache
    cache-to: type=registry,ref=ghcr.io/${{ github.repository }}:buildcache,mode=max

#20.3 Deploying to Kubernetes via kubectl/Helm from a Pipeline

Two common approaches to get a new image running on a Kubernetes cluster from CI:

Approach A — direct kubectl set image (simple, push-based):

yaml
- name: Deploy to Kubernetes
  run: |
    kubectl set image deployment/myapp-deployment \
      myapp=ghcr.io/${{ github.repository }}:${{ github.sha }} \
      --namespace=production
    kubectl rollout status deployment/myapp-deployment --namespace=production

This tells Kubernetes "update this one container's image" and Kubernetes handles the rolling update itself (see Unit 21.1). kubectl rollout status blocks the pipeline until the rollout finishes or fails, so a broken deploy fails the CI job instead of silently leaving a broken rollout running.

Approach B — Helm (templated, versioned deployments):

A minimal Helm values override during a pipeline deploy:

yaml
- name: Deploy with Helm
  run: |
    helm upgrade --install myapp ./charts/myapp \
      --namespace production \
      --set image.repository=ghcr.io/${{ github.repository }} \
      --set image.tag=${{ github.sha }} \
      --wait --timeout 5m

Helm treats your entire Kubernetes config (Deployment, Service, Ingress, ConfigMap, etc.) as one versioned, parameterized chart, so you're not hand-editing YAML per environment — you override only the values that change (like image.tag).

kubectl set image Helm
Best for Quick, simple deployments, single-image updates Multi-resource apps, templated configs, rollback support
Rollback kubectl rollout undo helm rollback myapp <revision>
Config reuse across envs Manual YAML duplication Values files per environment (values-staging.yaml, values-prod.yaml)

#20.4 Wiring Image Scanning as a Hard Gate

Phase 2 introduced Trivy for scanning images locally. In CI, that scan becomes a gate: if it finds critical vulnerabilities, the pipeline stops before the image ever reaches a registry or a cluster.

yaml
- name: Scan image with Trivy
  uses: aquasecurity/trivy-action@0.24.0
  with:
    image-ref: ghcr.io/${{ github.repository }}:${{ github.sha }}
    format: table
    severity: CRITICAL,HIGH
    exit-code: '1'      # non-zero exit fails the pipeline

exit-code: '1' is the key line — it converts a scan report into an actual pipeline gate rather than an informational log. Place this step after build, before push/deploy, so a vulnerable image never reaches production, and ideally before push so you don't even publish a flagged image to your registry.

#20.5 GitOps Basics — ArgoCD/Flux vs Push-Based Deployment

Push-based deployment (what sections 20.3's examples show): your CI pipeline actively runs kubectl/helm commands against the cluster. The pipeline needs cluster credentials, and it pushes changes into the cluster.

GitOps (pull-based): instead of CI pushing to the cluster, a GitOps controller (ArgoCD or Flux) runs inside the cluster, continuously watching a Git repo that holds the desired state (Kubernetes manifests/Helm values). Your CI pipeline's job stops at "build image, push image, update the manifest/values file in a Git repo" — the controller notices the Git change and pulls/applies it itself.

text
Push-based:                          GitOps (pull-based):

CI Pipeline                          CI Pipeline
   │  kubectl apply (needs creds)       │  updates image tag in a Git repo
   ▼                                    ▼
Kubernetes Cluster                  Git repo (source of truth)
                                         │
                                         │ (watched by)
                                         ▼
                                     ArgoCD/Flux (inside cluster)
                                         │  pulls + applies changes
                                         ▼
                                     Kubernetes Cluster

Why teams move to GitOps:

  • No cluster credentials in CI — CI never touches the cluster directly, reducing the blast radius if CI secrets leak.
  • Git becomes the audit log — every deployment is a Git commit, so "what's running in prod" = "what's the latest commit in this repo," always inspectable and revertible with git revert.
  • Self-healing — if someone manually changes something in the cluster (kubectl edit), ArgoCD detects the drift from Git and can automatically reconcile it back.

#Unit 21: Deployment Strategies — Blue-Green, Canary, Rolling

Difficulty: Advanced | Phase: Deployment & Pipeline-as-Code

#21.1 Rolling Deployments

The default Kubernetes Deployment strategy. Pods running the old version are replaced by pods running the new version incrementally, a few at a time, so there's no full-capacity outage window.

text
Time →

Old: [v1][v1][v1][v1]
     [v1][v1][v1][v2]   ← 1 pod replaced
     [v1][v1][v2][v2]   ← 2 pods replaced
     [v1][v2][v2][v2]
     [v2][v2][v2][v2]   ← fully rolled out

Kubernetes Deployment spec controlling the pace:

yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-deployment
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1   # at most 1 pod can be down during rollout
      maxSurge: 1         # at most 1 extra pod can be created above `replicas`
  template:
    spec:
      containers:
        - name: myapp
          image: ghcr.io/org/myapp:abc1234
          readinessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 5

Trade-offs:

  • Pro: zero extra infrastructure cost — no duplicate environment needed.
  • Con: both old and new versions run simultaneously during rollout. If your new version changes an API contract or DB schema incompatibly, you can get errors from in-flight requests hitting mixed versions. This is why backward-compatible changes matter for rolling updates specifically.
  • The readinessProbe is critical — Kubernetes only routes traffic to a new pod once its readiness probe passes, so a broken new version won't receive traffic even though it "started."

#21.2 Blue-Green Deployments

Run two full, identical environments ("Blue" = current live, "Green" = new version), fully deployed and tested in isolation, then cut over traffic all at once by switching a router/load balancer/Service selector.

text
Before cutover:                     After cutover:

  Router → Blue (v1) [LIVE]           Router → Green (v2) [LIVE]
           Green (v2) [staged,           Blue (v1) [idle, kept
             fully tested,                  as instant rollback]
             receiving no traffic]

Kubernetes implementation using two Deployments + one Service whose selector you flip:

yaml
# Both blue and green deployments exist simultaneously
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-blue
spec:
  selector:
    matchLabels: { app: myapp, version: blue }
  template:
    metadata:
      labels: { app: myapp, version: blue }
    spec:
      containers:
        - name: myapp
          image: ghcr.io/org/myapp:v1
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-green
spec:
  selector:
    matchLabels: { app: myapp, version: green }
  template:
    metadata:
      labels: { app: myapp, version: green }
    spec:
      containers:
        - name: myapp
          image: ghcr.io/org/myapp:v2
---
# The Service selector is what you flip to cut over traffic
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
spec:
  selector:
    app: myapp
    version: blue     # change this to "green" to cut over instantly
  ports:
    - port: 80
      targetPort: 3000

Cutover command (just a kubectl patch, no rebuild/redeploy needed):

bash
kubectl patch service myapp-service -p '{"spec":{"selector":{"version":"green"}}}'

Trade-offs:

  • Pro: instant rollback — if Green misbehaves, flip the selector back to Blue immediately; the old version was never torn down.
  • Pro: no mixed-version traffic during the switch (unlike rolling updates) — the cutover is atomic.
  • Con: 2x infrastructure cost while both environments run simultaneously.
  • Con: stateful concerns (DB migrations) are trickier — both versions may need to work against the same DB schema during the overlap window.

#21.3 Canary Releases

Route a small percentage of live traffic to the new version first, monitor error rates/latency, and progressively increase that percentage if healthy — with automated rollback if metrics degrade.

text
Stage 1:  95% → v1   |   5% → v2   (canary)
Stage 2:  75% → v1   |  25% → v2   (if healthy, increase)
Stage 3:  25% → v1   |  75% → v2
Stage 4:   0% → v1   | 100% → v2   (fully rolled out)

              ↓ at any stage, if error rate/latency spikes:
         automatic rollback → 100% → v1

A simplified example using two Kubernetes Deployments behind one Service (weight controlled via replica count ratio, a common lightweight approach without a service mesh):

yaml
# v1 (stable) - 9 replicas, v2 (canary) - 1 replica = ~10% canary traffic
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-stable
spec:
  replicas: 9
  selector:
    matchLabels: { app: myapp }
  template:
    metadata:
      labels: { app: myapp, track: stable }
    spec:
      containers:
        - name: myapp
          image: ghcr.io/org/myapp:v1
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-canary
spec:
  replicas: 1
  selector:
    matchLabels: { app: myapp }
  template:
    metadata:
      labels: { app: myapp, track: canary }
    spec:
      containers:
        - name: myapp
          image: ghcr.io/org/myapp:v2

Both Deployments share the label app: myapp, so a single Service routing to app: myapp load-balances across all 10 pods — roughly 90/10 traffic split. For fine-grained, precise percentage-based routing (not just replica-ratio approximation) and automated metric-based promotion/rollback, production setups use a service mesh or progressive-delivery tool (Istio, Flagger, Argo Rollouts) rather than hand-managing replica counts.

Trade-offs:

  • Pro: limits the blast radius of a bad release to a small % of users before full rollout.
  • Pro: pairs well with real production traffic/metrics — catches issues rolling updates or blue-green (which don't test against live traffic gradually) might miss.
  • Con: more operational complexity — needs good monitoring/alerting to make the "is the canary healthy?" decision, ideally automated.

#21.4 Feature Flags as a Deployment-Decoupling Technique

This directly extends Unit 10.3. Deployment strategies (rolling/blue-green/canary) control which version of your code is running. Feature flags control which features are active, independent of what's deployed. Combining both gives you two independent levers:

text
Deploy   = "is the new code present on the server?"
Release  = "is the new behavior visible to users?"

With feature flags: Deploy ≠ Release

You can deploy a canary (Unit 21.3) that contains a new feature completely hidden behind a flag — meaning even the 5% of users hitting the canary pods see no behavioral difference until you explicitly flip the flag. This separates the risk of new code (crashes, performance regressions — caught by canary analysis) from the risk of new behavior (business logic issues — controlled separately by the flag), letting you roll back a bad feature by flipping a flag in seconds, with zero deployment/rollback needed at all.

#21.5 Interview Scenario: Production Broke After Deploy

How to structure your answer when asked "walk me through your rollback":

"First, I'd confirm it's actually the deploy causing it — check the deployment timestamp against when errors/alerts started. Assuming it is:

  1. Immediate mitigation — if it's a rolling update, kubectl rollout undo deployment/myapp to revert to the last known-good ReplicaSet. If it's blue-green, flip the Service selector back to Blue — that's near-instant. If it's a bad feature behind a flag, flip the flag off first since that's the fastest lever, even before touching the deployment.
  2. Confirm recovery — watch error rates/health checks return to baseline.
  3. Communicate — notify the team/stakeholders, especially if it's user-facing.
  4. Root cause after the fire's out — don't debug live in production; reproduce in staging.
  5. Blameless post-mortem — document what broke, why it wasn't caught by CI/staging, and what gate (test, canary threshold, review) should be added to prevent recurrence."

This shows you know the mechanics (rollout undo, selector flip, flag flip) — not just the concept — and shows a mitigate-first, root-cause-later instinct that interviewers specifically look for.

#Unit 22: Pipeline as Code Best Practices & GitOps

Difficulty: Advanced | Phase: Deployment & Pipeline-as-Code

#22.1 Why Pipeline Config Belongs in Version Control

"Pipeline as Code" means your CI/CD pipeline definition (*.yml for GitHub Actions, Jenkinsfile for Jenkins) lives in the same Git repo as your application code — not configured through a UI that nobody can diff, review, or roll back.

Benefits, directly parallel to why you version-control application code:

  • Code review — a pipeline change goes through a PR like any other change; a reviewer can catch "wait, this removes the test step" before it merges.
  • History & rollback — git log/git blame on your .github/workflows/deploy.yml tells you exactly when and why a deploy step changed.
  • Reproducibility — checking out an old commit gives you the pipeline as it was at that point, not just the app code.
  • No tribal knowledge lock-in — a pipeline configured by clicking through a Jenkins UI is invisible to the rest of the team and undocumented; a Jenkinsfile in the repo is discoverable by anyone who clones it.

#22.2 DRY Pipeline Design at Scale

As pipelines grow, duplication creeps in — the same "checkout, install, lint, test" steps copy-pasted across 10 workflow files. Both GitHub Actions and Jenkins have first-class ways to avoid this (introduced conceptually in Phase 4/5, revisited here from a "pipeline as code hygiene" lens):

GitHub Actions — reusable workflow:

yaml
# .github/workflows/reusable-test.yml
on:
  workflow_call:
    inputs:
      node-version:
        required: true
        type: string

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm ci
      - run: npm test
yaml
# .github/workflows/ci.yml — calls the reusable workflow
jobs:
  test:
    uses: ./.github/workflows/reusable-test.yml
    with:
      node-version: '20'

Jenkins — shared library function (revisited from Phase 5):

groovy
// vars/deployApp.groovy
def call(String imageTag, String environment) {
  sh "helm upgrade --install myapp ./charts/myapp --set image.tag=${imageTag} --namespace ${environment}"
}
groovy
// Jenkinsfile — reuses the shared step across many pipelines
deployApp("${env.GIT_COMMIT}", "staging")

The principle either way: a change to how you deploy should require editing one place, not hunting through a dozen near-identical YAML/Groovy files.

#22.3 Environment Promotion Pattern

This is Unit 11.5's "build once, deploy many" principle, made concrete as an actual pipeline shape: one artifact moves through environments in sequence, with gates between each.

text
   Build (once)
        │
        ▼
    Deploy to DEV ──── auto, on every merge to main
        │
   [automated tests pass?]
        │
        ▼
  Deploy to STAGING ─── auto, if dev tests pass
        │
   [manual approval / smoke tests]
        │
        ▼
   Deploy to PROD ────  requires human approval gate

GitHub Actions Environments make this explicit and enforce the approval gate:

yaml
jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment: staging          # no protection rules — auto-deploys
    steps:
      - run: ./deploy.sh staging ${{ github.sha }}

  deploy-prod:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production       # configured with required reviewers in repo settings
    steps:
      - run: ./deploy.sh production ${{ github.sha }}

Because deploy-prod uses the same ${{ github.sha }} and the same deploy.sh logic as deploy-staging, you're promoting the same artifact forward — never rebuilding between stages.

#22.4 Integrating Terraform (IaC) into CI/CD

So far this phase has focused on deploying application code. Infrastructure (the Kubernetes cluster itself, databases, networking) is commonly managed the same way — as code, in the same pipeline discipline — using Terraform.

A typical Terraform stage added to a pipeline, with a manual approval gate before apply (since infrastructure changes can be destructive):

yaml
jobs:
  terraform-plan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - run: terraform init
      - run: terraform plan -out=tfplan
      - uses: actions/upload-artifact@v4
        with:
          name: tfplan
          path: tfplan

  terraform-apply:
    needs: terraform-plan
    runs-on: ubuntu-latest
    environment: production        # requires manual approval before this job runs
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - uses: actions/download-artifact@v4
        with:
          name: tfplan
      - run: terraform apply tfplan

Note the pattern: plan runs automatically (it's read-only, just shows what would change) and its exact output is passed as an artifact to apply, which is gated behind manual approval — mirroring the "build once, promote/apply the same output" principle applied to infrastructure instead of application artifacts.

#22.5 GitOps Principles: Declarative, Versioned, Pulled Not Pushed

Revisiting Unit 20.5's GitOps concept, formalized as three principles:

  1. Declarative — you describe the desired end state ("3 replicas of image v2 running"), not a sequence of imperative commands ("run this, then that"). Kubernetes manifests are inherently declarative; this is why GitOps tools work naturally with them.
  2. Versioned & immutable — the desired state lives in Git, so every state change is a commit with full history, diff-ability, and revert capability, exactly like Unit 22.1's argument applied to infrastructure state rather than pipeline definitions.
  3. Pulled, not pushed — an agent (ArgoCD/Flux) inside the cluster continuously polls Git and reconciles, rather than an external CI system pushing changes in. This is the key operational shift: your cluster's security boundary no longer needs to trust external CI credentials.
text
Traditional CI/CD (push):                 GitOps (pull):

CI has cluster credentials                CI has NO cluster credentials
CI runs `kubectl apply`                   CI only commits a manifest change to Git
Cluster state = whatever CI last pushed   Cluster state = continuously reconciled
                                              to match Git by an in-cluster agent

The practical effect for you as a developer: your deploy step in CI shrinks down to "update the image tag in a manifests repo and commit" — everything after that (actually rolling it out, monitoring for drift, self-healing) is handled by the GitOps controller, not your pipeline.
