#Unit 09: CI/CD Fundamentals & DevOps Culture

Difficulty: Beginner | Phase: CI/CD Foundation

#9.1 CI vs CD vs Continuous Deployment

These three terms get mixed up constantly, including in interviews. Nail the distinction precisely.

Term What it guarantees What happens automatically Human gate?
Continuous Integration (CI) Every code change is built and tested automatically as soon as it's pushed/merged Build + automated tests run on every commit/PR N/A (this stage ends at "tests passed")
Continuous Delivery (CD) Every change that passes CI is automatically packaged into a release-ready artifact Build → test → package → stage in a deployable state Yes — a human clicks "deploy to prod"
Continuous Deployment (also CD) Every change that passes all pipeline stages goes to production automatically Build → test → package → deploy to prod No — no human in the loop

Think of it as a ladder:

text
CI  →  Continuous Delivery  →  Continuous Deployment
(build+test)   (+ package, always releasable)   (+ auto-release to prod, zero manual gate)

As a Node.js dev, you've lived CI already: every time a PR triggers npm test and npm run lint, that's CI. The moment your team says "and if that passes, it auto-deploys to production with no approval," that's Continuous Deployment — the strictest, most automated end of the spectrum.

Common misconception to avoid: Continuous Delivery does NOT mean auto-deploying to prod. It means you are always in a state where you could deploy to prod with one click. Continuous Deployment is the one where that click is removed entirely.

#9.2 DevOps Culture & DORA Metrics

DevOps is a cultural and technical practice that removes the wall between "Dev" (who writes code) and "Ops" (who runs it in production) so the same team owns a change from commit to production and back.

DORA metrics (from Google's DevOps Research and Assessment team) are the four industry-standard metrics used to measure software delivery performance:

  1. Deployment Frequency (DF) — how often you ship to production.
    • Elite teams: multiple deploys per day. Low performers: once every few months.
  2. Lead Time for Changes (LT) — time from code committed to code running in production.
    • Elite: less than 1 hour. Low: more than 6 months.
  3. Mean Time to Restore (MTTR) — how long it takes to recover from a production incident/failed deploy.
    • Elite: less than 1 hour. Low: more than 1 week.
  4. Change Failure Rate (CFR) — percentage of deployments that cause a production failure requiring a fix (rollback, hotfix, patch).
    • Elite: 0-15%. Low: 46-60%.

Simple mental formulas:

text
Deployment Frequency = (# of production deployments) / (time period)
Lead Time for Changes = (time of deploy) - (time of first commit for that change)
MTTR                  = (time incident resolved) - (time incident detected)
Change Failure Rate   = (# failed deployments) / (total # deployments) × 100

Why this matters to you as an engineer: DORA metrics are what separates "we have a CI/CD pipeline" from "our CI/CD pipeline is actually good." In interviews, being able to say "we improved deployment frequency by doing X" is far stronger than describing YAML syntax.

#9.3 Anatomy of a CI/CD Pipeline

A pipeline is a sequence of automated stages a code change passes through. The canonical shape:

text
 Source  →  Build  →  Test  →  Package  →  Deploy  →  Monitor
   |          |         |          |           |          |
 git push   compile   unit/int   docker      staging/   logs,
 /PR merge  transpile  tests     image or    prod       alerts,
                                  build zip   rollout    metrics

Mapped to a typical Node.js/Express service:

Stage Node.js example
Source Push to feature/add-auth → PR opened against main
Build npm ci (clean install), npm run build (if using TypeScript, transpile to dist/)
Test npm run lint, npm test (Jest/Mocha), npm audit
Package docker build -t myapp:sha123 or npm pack for a library
Deploy Push image to registry → kubectl apply / docker compose up / cloud deploy
Monitor Health checks, log aggregation, error tracking (Sentry), metrics (Prometheus)

Each stage acts as a gate — if Test fails, the pipeline stops and Package/Deploy never run. This is what "fail fast, fail early" means in practice.

#9.4 Shift-Left Testing & Fail Fast

"Shift-left" means moving quality checks (tests, linting, security scans) as early as possible in the pipeline/timeline — ideally to the developer's local machine and the first CI step — rather than discovering problems late (in staging, or worse, in production).

text
Traditional (shift-right, expensive):
  Code → Deploy → QA finds bug → Ops finds it broke prod → Fix

Shift-left (cheap):
  Local lint/test → Pre-commit hook → CI unit tests → CI integration tests → Deploy
  (bug caught here)                                                          ↑
                                                              never gets this far

Practical shift-left tools you'd recognize as a Node dev:

  • Pre-commit hooks (Husky + lint-staged) running eslint and prettier before a commit is even allowed.
  • CI unit tests running on every push, not just before release.
  • Static analysis (ESLint, TypeScript compiler checks) catching type errors before runtime.

The "fail fast" philosophy: it's far cheaper to fail a build in 30 seconds on a laptop than to fail a deployment in production at 2 AM. Every stage of your pipeline should be ordered fastest-and-cheapest-first (lint before unit tests, unit tests before integration tests, integration tests before deploy) so failures surface as early and cheaply as possible.

#9.5 Interview Trap: CI vs CD vs Continuous Deployment

How to answer this cleanly if asked live:

"CI is about integrating and validating code changes constantly — every push triggers a build and automated test suite. Continuous Delivery extends that by ensuring the validated code is always packaged into a deployable artifact, but a human still approves the actual production release. Continuous Deployment removes that human gate entirely — anything that passes the pipeline goes live automatically. The confusion usually comes from both Delivery and Deployment sharing the abbreviation 'CD' — you have to listen for which one is meant from context."

Follow-up they may ask: "Which would you deploy at a bank vs a SaaS startup?" — Banks/regulated industries usually stop at Continuous Delivery (compliance requires a human sign-off); fast-moving SaaS products often run full Continuous Deployment.

#Unit 10: Git Branching Strategies for CI/CD

Difficulty: Beginner | Phase: CI/CD Foundation

#10.1 Trunk-Based Development vs GitFlow vs GitHub Flow

GitFlow — the classic, heavyweight model with long-lived branches:

text
main        ─────●───────────────●──────────  (production releases only)
                  \               /
release/1.2        ●───●───●────●            (stabilization branch)
                   /            
develop     ──●───●───●───●───●───●──────    (integration branch)
              /        \
feature/x    ●──●──●    (merged back into develop)
  • main = always production. develop = integration branch. feature/*, release/*, hotfix/* branches all exist.
  • Pros: very structured, good for scheduled/versioned releases (e.g., mobile apps, enterprise software with release trains).
  • Cons: merge conflicts pile up on long-lived branches, slows down CI/CD velocity, doesn't suit multiple-deploys-per-day teams.

GitHub Flow — simplified, CD-friendly:

text
main   ──●───────●───────●───────●──────  (always deployable)
          \      /\      /\      /
feature/a  ●────●  \    /  \    /
feature/b            ●──●
  • Only main + short-lived feature/* branches. Every feature branch opens a PR, gets reviewed + CI-checked, merges to main, and main is always deployable (often auto-deployed to staging).
  • This is the default for most modern SaaS teams using GitHub Actions.

Trunk-Based Development (TBD) — the most CI/CD-native model:

text
main (trunk) ──●─●─●─●─●─●─●─●─●─●─●─●──   (everyone commits here, directly or via <1 day branches)
  • Developers commit directly to main (or via branches living hours, not days), gated by feature flags for anything incomplete.
  • Requires strong automated testing and feature-flag discipline, since there's no isolation branch to "hide" in-progress work.
  • This is what elite DORA performers use — it's what enables multiple-deploys-per-day.
Model Branch lifespan Best for CI/CD fit
GitFlow Days-weeks (release/hotfix branches) Versioned releases, mobile apps Poor — batches changes, slows delivery
GitHub Flow Hours-days (PR-based) Most SaaS/web teams Good — pairs naturally with Continuous Delivery
Trunk-Based Hours or direct-to-trunk High-velocity teams, elite DORA performers Best — enables Continuous Deployment

#10.2 How Branching Strategy Drives Pipeline Trigger Design

Your branching model directly decides what your CI config's on: (GitHub Actions) or trigger block (Jenkins) looks like.

  • GitFlow → you need separate pipelines/jobs per branch type: develop pushes trigger integration tests + deploy-to-dev; release/* triggers full regression + deploy-to-staging; main triggers production release pipeline.
  • GitHub Flow → simpler: PRs against main trigger the full test suite (required check before merge); merges to main trigger deploy-to-staging, and a tag/release triggers deploy-to-prod.
  • Trunk-Based → every commit to main runs the full pipeline immediately, often followed by automatic deployment gated only by feature flags, not branches.

Example GitHub Actions trigger block illustrating a GitHub-Flow-style setup:

yaml
on:
  pull_request:
    branches: [main]        # run tests on every PR targeting main
  push:
    branches: [main]        # deploy to staging on every merge to main
  release:
    types: [published]      # deploy to prod only when a release/tag is published

#10.3 Feature Flags as an Alternative to Long-Lived Branches

A feature flag (aka feature toggle) is a runtime conditional that lets you merge incomplete or risky code into main while keeping it invisible/inactive in production, until you're ready to switch it on — without a deploy.

Why this replaces long-lived feature branches: instead of keeping a branch alive for 3 weeks while a feature is built (accumulating merge-conflict risk), you merge small pieces continuously behind a flag, and flip the flag when the feature is complete.

Minimal Node.js example — a simple flag check:

javascript
// featureFlags.js
const flags = {
  newCheckoutFlow: process.env.FF_NEW_CHECKOUT === 'true'
};

module.exports = flags;
javascript
// checkoutController.js
const flags = require('./featureFlags');

app.post('/checkout', (req, res) => {
  if (flags.newCheckoutFlow) {
    return handleNewCheckout(req, res);
  }
  return handleLegacyCheckout(req, res);
});

In production, tools like LaunchDarkly, Unleash, or a simple config service manage flags dynamically (no redeploy needed to flip them), and can target flags by user %, region, or account — enabling canary-style rollouts without touching your deployment pipeline at all.

#10.4 Pull Request Workflows & Required Status Checks

A PR workflow is how most teams gate code quality before it reaches main:

  1. Developer opens a PR from feature/x → main.
  2. CI automatically runs (lint, unit tests, build) — these appear as status checks on the PR.
  3. Branch protection rules on main require: at least N approving reviews, all status checks passing, and (optionally) branches to be up to date with main before merge.
  4. Only once all of the above pass can the PR be merged (often only via "Squash and merge" for a clean history).

Example of a GitHub branch protection setup (conceptually, since it's UI-configured, not YAML):

text
Branch protection rule for `main`:
  ✔ Require a pull request before merging
  ✔ Require approvals: 1
  ✔ Require status checks to pass before merging:
        - lint
        - test
        - build
  ✔ Require branches to be up to date before merging
  ✔ Do not allow bypassing the above settings

This is the mechanism that makes "main is always deployable" actually true — it's enforced by tooling, not by developer discipline alone.

#10.5 Interview Angle: Choosing a Branching Strategy

Sample answer:

"It depends on release cadence and team size. For a small team shipping a SaaS product multiple times a day, I'd pick GitHub Flow or trunk-based development with feature flags — it minimizes merge overhead and pairs naturally with continuous deployment. For a product with scheduled, versioned releases — like a mobile app that goes through app-store review — GitFlow's release branches make more sense because you need a stabilization window. I'd avoid GitFlow for a fast-moving backend service; the long-lived develop branch becomes a merge-conflict bottleneck and slows deployment frequency, which hurts DORA metrics."

#Unit 11: Build Systems, Artifacts & Dependency Management

Difficulty: Beginner | Phase: CI/CD Foundation

#11.1 Build Tools Overview

A build tool takes source code and produces a runnable/packaged artifact. Different ecosystems have different standard tools:

Ecosystem Build tool What it does
Node.js/JS npm/yarn/pnpm + bundlers (tsc, webpack, esbuild) Install deps, transpile TS→JS, bundle for prod
Java Maven / Gradle Compile, run tests, package as JAR/WAR
.NET MSBuild Compile, package as DLL/EXE

For your Node.js background specifically, the "build" step usually means:

bash
npm ci                 # clean, reproducible install from package-lock.json
npm run lint           # static analysis
npm test               # unit/integration tests
npm run build          # tsc compile TypeScript → dist/, or bundle with esbuild/webpack

npm ci vs npm install — a common CI interview question:

npm install npm ci
Uses package-lock.json exactly No — may update it Yes — fails if lockfile and package.json mismatch
Speed in CI Slower Faster (deletes node_modules first, installs from lockfile directly)
Use case Local development CI/CD pipelines (reproducibility matters)

Always use npm ci in pipelines, never npm install — it guarantees every build uses the exact same dependency versions, eliminating "works on my machine" drift.

#11.2 Artifact Repositories

An artifact repository stores the packaged output of your build (JARs, npm packages, Docker images, zip files) so it can be pulled by later pipeline stages or other services — instead of being rebuilt from source every time.

Common ones you'll encounter:

Repository Stores Notes
npm registry / GitHub Packages npm packages Public npm registry, or private via GitHub Packages/Verdaccio
Nexus / JFrog Artifactory Generic artifacts (JARs, npm, Docker, etc.) Enterprise-grade, supports many formats in one tool
Container registries (Docker Hub, ECR, GCR, GitHub Container Registry) Docker images The artifact store for anything containerized

Publishing a Node.js package to GitHub Packages (example package.json + workflow snippet):

json
{
  "name": "@your-org/my-lib",
  "version": "1.2.0",
  "publishConfig": {
    "registry": "https://npm.pkg.github.com"
  }
}
bash
npm publish

The key principle: the artifact is built once and stored — every later pipeline stage (staging deploy, prod deploy) pulls that exact same artifact rather than rebuilding, which is the foundation of the "build once, deploy many" principle covered in 11.5.

#11.3 Semantic Versioning & Immutable Artifacts

Semantic Versioning (SemVer) format: MAJOR.MINOR.PATCH (e.g., 2.4.1)

text
MAJOR — incremented for breaking/incompatible API changes       (1.x.x → 2.0.0)
MINOR — incremented for backward-compatible new functionality    (1.2.x → 1.3.0)
PATCH — incremented for backward-compatible bug fixes             (1.2.3 → 1.2.4)

In package.json, the caret/tilde ranges you already use daily are built on this:

json
"dependencies": {
  "express": "^4.18.2",   // ^ = accept MINOR and PATCH updates, not MAJOR
  "lodash": "~4.17.21"    // ~ = accept only PATCH updates
}

Immutable artifacts: once a build artifact is created and tagged (e.g., a Docker image tagged myapp:1.4.2 or myapp:abc1234 using the commit SHA), it must never be overwritten or rebuilt with the same tag. If you need a fix, you produce a new artifact with a new tag/version. This guarantees that "what passed staging is exactly what runs in production" — the same bytes, not a re-build that could differ.

Tagging convention combining SemVer + commit SHA (recommended in production pipelines):

bash
docker build -t myapp:1.4.2 -t myapp:abc1234 -t myapp:latest .
docker push myapp:1.4.2
docker push myapp:abc1234
  • 1.4.2 — human-readable release version.
  • abc1234 (commit SHA) — guarantees a unique, traceable tag per build, even between two commits that don't bump the version.
  • latest — convenient for local dev only; never deploy latest to production (see 11.5's interview trap).

#11.4 Dependency Caching

CI pipelines are billed/measured in minutes, so re-downloading node_modules on every run wastes time and money. Dependency caching stores previously-downloaded dependencies and restores them on the next run if the lockfile hasn't changed.

GitHub Actions example using actions/cache keyed on the lockfile hash:

yaml
- name: Cache node modules
  uses: actions/cache@v4
  with:
    path: ~/.npm
    key: npm-${{ hashFiles('package-lock.json') }}
    restore-keys: |
      npm-

- name: Install dependencies
  run: npm ci

How this works:

  • key is computed from a hash of package-lock.json. If the lockfile hasn't changed since the last run, the cache hits and npm ci runs much faster (dependencies restored from cache instead of the network).
  • restore-keys provides a fallback prefix match if there's no exact hit — useful for partial cache reuse.

Many modern setup actions (actions/setup-node) support caching built-in:

yaml
- uses: actions/setup-node@v4
  with:
    node-version: '20'
    cache: 'npm'

#11.5 Interview Favorite: Build Once, Deploy Many

The principle: compile/package your application into a single immutable artifact exactly once per change, then promote that same artifact through environments (dev → staging → prod) without rebuilding it at any stage.

Why this matters: if you rebuild the artifact separately for staging and for production, you risk subtle differences — a different dependency resolving slightly differently, a different base image patch version, an environment-specific bug that only appears in one build. "Build once, deploy many" eliminates that entire class of bugs by guaranteeing bit-for-bit identical artifacts across environments; only configuration (env vars, secrets, connection strings) changes between environments, never the artifact itself.

text
        ┌─────────────┐
        │  Build once │ →  myapp:abc1234 (immutable artifact)
        └─────────────┘
               │
   ┌───────────┼────────────┐
   ▼           ▼            ▼
 Deploy to   Deploy to    Deploy to
   dev       staging        prod
(same image, different env vars/config only)

Sample interview answer:

"The idea is that whatever you tested in staging is byte-for-byte identical to what runs in production — you're never re-compiling between environments, only re-configuring. I tag Docker images with the commit SHA so the artifact is uniquely identifiable and immutable, then use environment-specific config (via env vars or a config service) to control behavior per environment, rather than baking environment differences into the build."

Question One-line answer
