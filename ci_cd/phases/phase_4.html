#Unit 12 — GitHub Actions Fundamentals

#12.1 Anatomy of a Workflow File

Workflows live in .github/workflows/*.yml. Every workflow file has three top-level keys: name, on (trigger), and jobs.

yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test

Key structural facts:

  • A workflow can have multiple jobs: — by default they run in parallel unless linked with needs: (Unit 13).
  • A job runs on a fresh runner (VM) — nothing persists between jobs unless you explicitly pass artifacts or use caching.
  • Each step either runs a shell command (run:) or invokes a reusable action (uses:).

#12.2 Event Triggers

Trigger Fires on
push Commits pushed to matching branches/tags
pull_request PR opened/updated targeting matching branches
workflow_dispatch Manual trigger via UI/API, supports custom inputs
schedule Cron-based, e.g. nightly builds
release When a GitHub release is published
yaml
on:
  push:
    branches: [main]
    paths:
      - 'src/**'              # only trigger if these paths changed
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        default: 'staging'
        type: choice
        options: [staging, production]
  schedule:
    - cron: '0 2 * * *'       # 2 AM UTC daily

paths: filtering is a common optimization for monorepos — avoid running a Node service's CI when only docs changed.

#12.3 Jobs, Steps & Runners

runs-on selects the runner OS/image. GitHub-hosted runners come pre-installed with common tooling (Node, Docker, git) but versions drift over time — pin explicitly with setup-node rather than relying on the image default.

yaml
jobs:
  build:
    runs-on: ubuntu-latest      # also: windows-latest, macos-latest
    steps:
      - run: node --version     # whatever came pre-installed — don't rely on this

Each step in a job runs sequentially on the same runner/filesystem — so npm ci in step 2 is visible to npm test in step 3. But nothing crosses job boundaries without artifacts (Unit 13) since each job gets a brand-new runner VM.

#12.4 Using Marketplace Actions

Actions are reusable units published to the GitHub Marketplace, referenced as owner/repo@ref.

yaml
- uses: actions/checkout@v4          # clones your repo into the runner
- uses: actions/setup-node@v4        # installs Node + optionally caches npm
  with:
    node-version: '20'
    cache: 'npm'

Security note (relevant for interviews and real audits): always pin to a version tag (@v4) or, for maximum supply-chain safety, a commit SHA (@a1b2c3d...) rather than @main, since a compromised action's main branch could inject malicious code into your pipeline.

#12.5 Reading Logs & Run History

The Actions tab on GitHub shows every workflow run, grouped by workflow file, with pass/fail status, duration, and expandable per-step logs. Key skill: reading a failed step's raw log output to find the actual error (often buried under noisy npm ci output) — click "Search logs" or expand only the failing step rather than scrolling everything.

#12.6 Labs

  1. Write a workflow that runs npm test on every push to main.
  2. Add a workflow_dispatch trigger with a required environment choice input, and trigger it manually from the Actions tab.
  3. Add a scheduled nightly workflow using cron syntax; verify it appears correctly under "Scheduled workflows."
  4. Intentionally break a test, push it, and practice navigating the Actions log viewer to find the exact failing assertion.

#Unit 13 — GitHub Actions Workflow Syntax Deep Dive

#13.1 Contexts & Expressions

Contexts expose structured data you can reference with ${{ }} expression syntax.

Context Example use
github ${{ github.ref }}, ${{ github.sha }}, ${{ github.event_name }}
env ${{ env.NODE_ENV }}
secrets ${{ secrets.NPM_TOKEN }}
needs ${{ needs.build.outputs.version }}
matrix ${{ matrix.node-version }}
yaml
steps:
  - run: echo "Building commit ${{ github.sha }} on branch ${{ github.ref_name }}"
  - run: echo "Event was ${{ github.event_name }}"

#13.2 needs: and Job Dependency Graphs

By default, all jobs run in parallel. needs: makes a job wait for one or more others to succeed first, forming a DAG.

yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build

  test:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  deploy:
    needs: [build, test]        # waits for BOTH to succeed
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying..."

If build fails, test and deploy are skipped automatically — no explicit failure handling needed for the common case.

#13.3 Conditionals with if:

yaml
jobs:
  deploy:
    needs: test
    if: github.ref == 'refs/heads/main'    # only deploy from main
    runs-on: ubuntu-latest
    steps:
      - run: echo "deploying"

  notify-on-failure:
    needs: [build, test]
    if: failure()                          # only runs if a prior needs job failed
    runs-on: ubuntu-latest
    steps:
      - run: echo "Something broke — notify Slack"

Common built-in functions: success() (default implicit condition), failure(), always() (run regardless — useful for cleanup/notifications), cancelled().

#13.4 Artifacts Between Jobs

Since each job is a fresh VM, passing files (build output, coverage reports) requires explicit upload/download.

yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: dist-output
          path: dist/

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: dist-output
          path: dist/
      - run: ls dist/    # the build output from the other job is now here

#13.5 Job Outputs

For small values (a version string, a computed flag) rather than whole files, use job outputs: instead of artifacts — lighter weight.

yaml
jobs:
  version:
    runs-on: ubuntu-latest
    outputs:
      app_version: ${{ steps.get_version.outputs.version }}
    steps:
      - id: get_version
        run: echo "version=$(node -p "require('./package.json').version")" >> "$GITHUB_OUTPUT"

  build:
    needs: version
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building version ${{ needs.version.outputs.app_version }}"

Note the >> "$GITHUB_OUTPUT" pattern — this is the current (post-2022) way to set step outputs; the older ::set-output:: command syntax is deprecated.

#13.6 Labs

  1. Build a 3-job pipeline (build → test → deploy) using needs:, and verify in the Actions UI that they run sequentially, not in parallel.
  2. Add if: github.ref == 'refs/heads/main' to the deploy job and confirm it's skipped on a feature-branch PR.
  3. Upload a build artifact from a build job and download it inside a separate deploy job.
  4. Compute a version string as a job output in one job and consume it via needs.<job>.outputs.* in another.

#Unit 14 — Secrets, Environments & Deployments

#14.1 Secret Scopes

Scope Set at Visible to
Repository secret Repo Settings → Secrets and variables All workflows in that repo
Organization secret Org settings Selected/all repos in the org
Environment secret Repo → Environments → <env> Only jobs targeting that environment
yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying with token"
        env:
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}

Secrets are masked in logs automatically (GitHub redacts any exact-match substring), but never echo a secret directly — masking can be defeated with encoding tricks, and it's just bad practice.

#14.2 GitHub Environments & Protection Rules

An Environment (Settings → Environments) groups deployment-related config: environment-scoped secrets, required reviewers, wait timers, and deployment branch restrictions.

yaml
jobs:
  deploy-prod:
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://myapp.com
    steps:
      - run: ./deploy.sh

If production has a required reviewer configured, this job will pause and show up as "Waiting for approval" in the Actions UI until an authorized person approves it — a native manual gate, no extra tooling needed.

#14.3 OIDC — Keyless Cloud Authentication

Instead of storing long-lived cloud credentials (AWS_ACCESS_KEY_ID etc.) as secrets forever, GitHub Actions can present a short-lived OIDC token that your cloud provider trusts, exchanging it for temporary credentials per-run.

yaml
permissions:
  id-token: write     # required for OIDC — must be explicitly granted
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/github-actions-deploy
          aws-region: us-east-1
      - run: aws s3 sync ./dist s3://my-bucket

Why this matters (common interview question — "how do you avoid storing cloud credentials in CI?"): no static AWS keys sit in GitHub Secrets at all. The IAM role's trust policy is scoped to your specific repo/branch, so even if the workflow file were compromised, there's no long-lived credential to steal — tokens expire in minutes.

#14.4 Approval Gates for Production

Combine Environments + OIDC + branch restrictions for a realistic production deploy flow:

yaml
jobs:
  deploy-staging:
    needs: test
    runs-on: ubuntu-latest
    environment: staging          # no reviewers required — auto-deploys
    steps:
      - run: ./deploy.sh staging

  deploy-prod:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production       # required reviewers configured in repo settings
    steps:
      - run: ./deploy.sh production

Staging deploys automatically on every merge to main; production requires a human click — a very common real-world pattern.

#14.5 Labs

  1. Create a production environment with a required reviewer, and observe a deploy job pause for approval.
  2. Set up OIDC federation to a cloud provider (or simulate with a mock role) and deploy without any static access keys stored as secrets.
  3. Store an NPM_TOKEN as a repository secret and use it in a publish step; verify it's masked in the logs.
  4. Build a two-stage pipeline: auto-deploy to staging, but require manual approval before production.

#Unit 15 — GitHub Actions Advanced Patterns

#15.1 Matrix Builds

Run the same job across multiple combinations of variables (Node versions, OSes) in parallel.

yaml
jobs:
  test:
    strategy:
      matrix:
        node-version: [18, 20, 22]
        os: [ubuntu-latest, macos-latest]
      fail-fast: false        # let all combos finish even if one fails
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci && npm test

This spins up 6 parallel jobs (3 Node versions × 2 OSes) from one job definition.

#15.2 Caching with actions/cache

setup-node's built-in cache: 'npm' covers the common case, but actions/cache gives you full control for custom caching needs (e.g., build output, Docker layers, other package managers).

yaml
- uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-npm-

The key includes a hash of package-lock.json — cache invalidates automatically when dependencies change; restore-keys provides a fallback partial-match cache if no exact key hits.

#15.3 Reusable Workflows vs Composite Actions

Reusable workflow (workflow_call) — an entire callable workflow, good for sharing whole pipelines (build+test+deploy) across repos:

yaml
# .github/workflows/reusable-deploy.yml
on:
  workflow_call:
    inputs:
      environment:
        required: true
        type: string
    secrets:
      DEPLOY_TOKEN:
        required: true

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying to ${{ inputs.environment }}"
        env:
          TOKEN: ${{ secrets.DEPLOY_TOKEN }}
yaml
# caller workflow in another repo
jobs:
  call-deploy:
    uses: my-org/shared-workflows/.github/workflows/reusable-deploy.yml@main
    with:
      environment: production
    secrets:
      DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}

Composite action — a bundle of steps reused within a job (not a whole separate job), good for sharing a handful of common steps:

yaml
# .github/actions/setup-and-lint/action.yml
name: 'Setup and Lint'
runs:
  using: 'composite'
  steps:
    - uses: actions/setup-node@v4
      with:
        node-version: '20'
    - run: npm ci
      shell: bash
    - run: npm run lint
      shell: bash
yaml
# used inside any job's steps
- uses: ./.github/actions/setup-and-lint

Rule of thumb: reusable workflow = share a pipeline; composite action = share a handful of steps.

#15.4 Self-Hosted Runners

GitHub-hosted runners are convenient but limited (no persistent cache beyond actions/cache, capped resources, can't access private-network resources like an internal DB). Self-hosted runners are your own machines/VMs/containers registered to your repo/org.

yaml
jobs:
  build:
    runs-on: [self-hosted, linux, gpu]   # matches runner labels you configured
    steps:
      - uses: actions/checkout@v4
      - run: ./run-gpu-heavy-build.sh

Use when you need: access to private VPC resources, specialized hardware (GPU), persistent local caches across runs, or cost control at high volume. Trade-off: you now own patching, scaling, and securing those machines.

#15.5 Concurrency Groups

Prevents wasted runs when multiple pushes happen in quick succession (e.g., someone pushes 5 times to the same PR in a minute).

yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

This cancels any in-progress run for the same branch/PR when a newer one starts — saving CI minutes and avoiding confusing out-of-order deploy results.

#15.6 Labs

  1. Build a matrix workflow testing across 3 Node versions and 2 OSes; confirm 6 parallel jobs appear in the Actions UI.
  2. Add actions/cache for ~/.npm keyed on package-lock.json hash; measure build time improvement on a cache hit.
  3. Extract a shared "setup + lint" sequence into a composite action, and a full "test + deploy" sequence into a reusable workflow called from two different workflow files.
  4. Add a concurrency block to a workflow, push twice quickly to the same branch, and verify the first run gets cancelled.
