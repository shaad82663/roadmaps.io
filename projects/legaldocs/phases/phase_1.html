#LegalDocs — Phase 1: Knowledge Base Ingestion

Tech stack: Node.js / TypeScript, OpenAI, PostgreSQL + pgvector (from Phase 0), LangChain.js (loaders, splitters, embeddings wrapper) Depends on: Phase 0 (Docker Compose with Postgres+pgvector, Redis, Express+TS skeleton — already done) Feeds into: Phase 2.5 (Conversational Q&A retriever), Phase 4 (Risk Analyzer retriever stack), Phase 4.5 (Legal Research Agent)

#1. Phase Overview

#Objective

Take your ~30 curated legal reference documents (Indian statutes, case judgments, regulatory circulars — PDFs and DOCX files) and turn them into a queryable vector knowledge base stored in pgvector. By the end of this phase, you should be able to run a semantic query like "what are the notice period requirements for termination without cause" and get back the actual relevant chunks of statute/judgment text, ranked by relevance.

#Why this phase is important

Every AI feature you build after this phase is a form of RAG (Retrieval-Augmented Generation) — the LLM's answer is only as good as the text you retrieve and hand it. Concretely:

  • Phase 4's Risk Analyzer will retrieve statute/judgment excerpts to justify why a clause is risky.
  • Phase 4.5's Legal Research Agent will search this same knowledge base as one of its tools.
  • Phase 2.5's conversational Q&A retrieves from a different set of vectors (the user's own uploaded contract), but uses the exact same chunking/embedding techniques you build here.

If this phase is sloppy — chunks that cut a legal provision in half, no useful metadata to filter by, embeddings of noisy OCR text — then no amount of clever prompting later can fix it. This is the foundation phase for every retrieval feature in the app. Get it right once, reuse the pattern everywhere.

#2. Concept Learning

#2.1 Document Loaders

A document loader converts a raw file (PDF, DOCX, TXT, HTML) into a standardized in-memory shape that the rest of your pipeline can work with regardless of original format:

ts
interface Document {
  pageContent: string;
  metadata: Record<string, any>;
}

Why not just fs.readFile()? Because a PDF or DOCX is a binary container format — layout, fonts, embedded images, page breaks. A loader wraps a parsing library (pdf-parse for PDFs, mammoth for DOCX) and gives you back plain text plus metadata (e.g. { source: 'filename.pdf', pdf: { totalPages: 12 } }), so the splitter and embedder downstream never need to know or care what the original file type was.

load() vs lazyLoad()

  • load() reads and parses the entire file and returns the full array of Document objects in memory at once. For a directory loader wrapping many files, this means every file is loaded before you get anything back.
  • lazyLoad() returns an AsyncIterable<Document> — it yields documents one at a time as they become available, so you can process (split → embed → insert) document 1 completely, let it be garbage-collected, then move to document 2.

Analogy: load() is "cook all 30 meals, then bring them all to the table." lazyLoad() is "cook one meal, serve it, then start the next." For 30 documents this isn't strictly necessary to avoid crashing — but it's the correct default habit for any real ingestion pipeline, because "how many reference documents will this system eventually hold" is a number that only grows, and building it lazy from day one means you never have to rewrite this later.

ts
// Eager — all 30 documents fully parsed into memory before processing starts
const docs = await directoryLoader.load();

// Lazy — process and free each document one at a time
for await (const doc of directoryLoader.lazyLoad()) {
  await processDocument(doc);
}

#2.2 Text Splitters

LLMs and embedding models work best on modestly sized, semantically coherent pieces of text — not whole 100-page documents. Two reasons:

  1. Context limits. Embedding models have a maximum input length; if you exceed it the text gets silently truncated.
  2. Embedding quality. A vector embedding is a single point that has to represent the "meaning" of everything you feed it. Embed one paragraph about termination clauses → you get a vector that means "termination clause." Embed an entire 100-page judgment → you get a vector that means, roughly, "average of everything in this document," which matches almost nothing well. Precise retrieval needs precise, small, single-topic chunks.

RecursiveCharacterTextSplitter (length-based)

You configure a chunkSize (e.g. 1000 characters) and chunkOverlap (e.g. 150 characters). It tries a hierarchy of separators — first \n\n (paragraph breaks), then \n, then . , then a space — always preferring to cut at the highest-priority separator it can find near the target size, so it avoids cutting mid-word or mid-sentence when possible. It has no understanding of document structure — it doesn't know what a "section" is, it just knows "roughly N characters, break at a natural point if you can."

chunkOverlap exists because a hard cutoff might separate a sentence's subject from its conclusion. Overlapping the tail of chunk N into the head of chunk N+1 means a query is less likely to miss context that straddled a boundary.

Structure-aware splitting

Instead of "every N characters," you split on the document's own structure — e.g. numbered sections in a statute ("Section 5:", "Section 6:") or paragraph numbers in a judgment. You do this by writing a splitter that detects these markers with a regex and cuts exactly there, regardless of resulting chunk length.

Why this can be better for legal text specifically: a statute section is a complete, self-contained legal unit — the "notice period" section is one indivisible rule. If a character-based splitter cuts that section in half at char #1000, you might retrieve only the first half of the rule and miss the crucial exception clause that came right after. Structure-aware splitting guarantees each chunk is exactly one complete provision.

The tradeoff: chunk sizes become wildly uneven (one section might be a single sentence, another three pages), which can hurt embedding quality at extremes, and it only works if the document actually has clean, consistently-formatted headings — a badly-OCR'd 1960s judgment might not.

This is why the roadmap has you build both for section-structured documents (statutes, judgments) and compare retrieval quality empirically — there's no universal right answer, only what performs best for a given document type.

#2.3 Embeddings & Vector Stores

An embedding model maps a piece of text to a fixed-length vector of numbers (OpenAI's text-embedding-3-small → 1536 numbers) such that texts with similar meaning map to vectors that are close together in that 1536-dimensional space. "Close together" is measured with a distance metric — cosine similarity (angle between vectors) or L2/Euclidean distance (straight-line distance) are the common choices.

A vector store is a database purpose-built to answer: "given this query vector, give me the top-K stored vectors closest to it — fast, even across millions of rows." A naive approach (compute distance to every single row, one by one) works fine for a few thousand rows, but doesn't scale — vector indexes (like HNSW) organize the vectors so nearest-neighbor search is approximate but very fast, trading a small amount of accuracy for massive speed.

pgvector is a PostgreSQL extension that adds:

  • A vector(n) column type to store embeddings directly in a normal table.
  • Distance operators (<-> for L2, <=> for cosine distance, <#> for inner product).
  • Index types (ivfflat, hnsw) for fast approximate nearest-neighbor search at scale.

Why pgvector instead of a dedicated vector database (Pinecone, Weaviate, Qdrant) for this project:

Reason Explanation
One database to operate You already run Postgres (Phase 0). No second system to provision, secure, back up, and monitor.
Relational + vector in one query You can JOIN a vector similarity search with a normal WHERE doc_group = 'labour_law' AND jurisdiction = 'ontario' filter, in a single SQL statement — critical for legal documents where "is this even the right act/jurisdiction" often matters as much as semantic closeness.
Transactional consistency Inserting a document's chunks and its metadata row happens in the same transaction — no risk of the vector store and the relational store drifting out of sync.
Scale fit At dozens to low-thousands of documents (your current and near-future scale), pgvector's HNSW index is more than fast enough. A dedicated vector DB solves a scale problem (tens of millions of vectors, distributed sharding) that this project doesn't have.

#2.4 Chunk Metadata — filtering is half of retrieval

A vector search alone answers "what's semantically similar." It cannot answer "...but only from the Ontario Employment Standards Act, not the Indian Contract Act." That's what metadata columns are for. Every chunk you store should carry, alongside its vector:

  • source_document — original filename, for citation/traceability.
  • doc_type — statute | judgment | regulation | circular.
  • doc_group — a logical bucket used for filtering during retrieval (e.g. labour_law, contract_law, data_protection). This is what Phase 4's retriever filters on.
  • jurisdiction — e.g. india_central, ontario, federal_canada (your roadmap spans Indian legal reference material and Canadian/OPS-context resume work is a separate business, but the LegalDocs app itself targets Indian contracts per your project brief — keep this field generic so it scales to any jurisdiction you add later).
  • section_title — if structure-aware splitting was used, the actual heading/section number, e.g. "Section 73 - Compensation for breach".
  • chunk_index — position of this chunk within its source document, for reconstructing order or debugging.
  • split_strategy — 'recursive' or 'structure_aware' — so you can literally query "show me only the recursive chunks" vs "only structure-aware chunks" during your comparison step.

#2.5 Idempotency and re-ingestion

Legal reference documents get updated (an act is amended, a new judgment matters more). Your ingestion pipeline needs to be safe to re-run: if you run it twice on the same file, you should not get duplicate chunks. The standard approach: compute a content hash (e.g. SHA-256) of the source file, store it, and skip re-processing (or delete-then-reinsert) if the hash hasn't changed. We'll build this in.

#4. Detailed Implementation Plan

#Step 1 — Database schema for the knowledge base

What: A legal_reference_chunks table plus a legal_reference_documents table to track ingested source files (for idempotency and re-ingestion).

Why: Separating "documents" (one row per source file, tracks hash/ingestion status) from "chunks" (many rows per document, holds the actual vectors) lets you answer "has this file changed since I last ingested it" cheaply, without scanning thousands of chunk rows.

How: SQL migration adding the vector extension, both tables, and an HNSW index on the embedding column.

Best practice: Always create the vector index after bulk-loading data for the first time (index build is faster on data already in the table, and for a 30-document corpus this is nearly instant either way, but it's the right habit at scale). Store embedding_model and embedding_dims alongside the vector so a future model upgrade doesn't silently mix incompatible vector spaces in one table.

Common mistake to avoid: Forgetting to CREATE EXTENSION vector; before creating a vector column — Postgres will error immediately with "type vector does not exist."

#Step 2 — Document loader module

What: A module that, given a directory path, lazily yields { document, sourceFile, docType, docGroup, jurisdiction } for every PDF/DOCX in it, using file-path-derived metadata (or a small manifest.json you maintain per folder).

Why: Centralizing "how do we know a file's doc_type/doc_group/jurisdiction" in one place means the rest of the pipeline never has to guess.

How: A manifest.json sitting next to your source files, mapping filename → metadata. This is simpler and more reliable for ~30 curated documents than trying to auto-detect jurisdiction from file content.

Best practice: Fail loudly (throw, don't skip silently) if a file exists in the folder but has no manifest entry — an un-tagged legal document silently ingested with doc_group: null will quietly break every filtered retrieval later.

#Step 3 — Text splitting module (dual strategy)

What: Two splitter functions: splitRecursive(doc) and splitByStructure(doc), both returning arrays of { content, metadata }.

Why: Pluggable, independently testable, and directly comparable — exactly what the roadmap's comparison deliverable needs.

How: RecursiveCharacterTextSplitter from langchain/text_splitter for the first; a custom regex-based splitter for the second, matching patterns like /^(Section|Article|Clause)\s+\d+[A-Za-z]?\.?/gm.

Best practice: Always attach chunk_index and split_strategy to metadata at split time, not later — it's cheap to lose track of ordering once chunks are shuffled into a Promise.all batch.

Common mistake to avoid: Using the same chunkSize for every document type. A dense statute section and a conversational judgment paragraph read very differently — start with one sane default (1000 chars / 150 overlap) and only tune per-type if your comparison step shows a real problem.

#Step 4 — Embedding module

What: A thin wrapper around OpenAIEmbeddings that batches chunk text into groups (OpenAI's embedding endpoint accepts arrays), with retry-on-rate-limit logic.

Why: Embedding 30 documents' worth of chunks one-at-a-time is slow and wasteful; embedding all of them in one giant array risks a token-limit-per-request error. Batching in groups of ~100 is the practical middle ground.

How: embeddings.embedDocuments(texts: string[]) called in chunk()-ed batches.

Best practice: Log token usage and cost per document during ingestion — for a curated, infrequently-re-run pipeline like this, visibility into cost matters more than raw speed.

#Step 5 — Storage module

What: Functions to upsert a document record (with content hash) and bulk-insert its chunk rows with embeddings.

Why: Wrapping this in a transaction per document means a crash halfway through embedding document #14 doesn't leave a half-ingested document #14 permanently corrupting your knowledge base.

How: pg Pool, a transaction per document: delete old chunks for this document (if re-ingesting), insert new ones, update the document's hash/status, commit.

Common mistake to avoid: Inserting chunks one row at a time in a loop of individual INSERT statements — use a single multi-row INSERT ... VALUES (...), (...), (...) (or pg-format/unnest) for performance, even at this small scale; it's the right habit and costs nothing extra to build correctly now.

#Step 6 — Orchestration pipeline

What: ingest.ts — the script that ties loader → splitter → embedder → storage together, iterating documents with lazyLoad().

Why: This is the actual "run this to (re)build the knowledge base" entrypoint you'll invoke via npm run ingest.

How: For each document: check hash against DB, skip if unchanged; otherwise split with both strategies (if structure-aware detects zero sections, fall back to recursive-only and log it); embed all chunks; store in one transaction.

#Step 7 — Retrieval comparison script

What: A small script that takes 3 sample legal queries, runs a similarity search against split_strategy = 'recursive' chunks and separately against split_strategy = 'structure_aware' chunks, and prints both result sets side by side.

Why: This is the roadmap's explicit Phase 1 deliverable — the internal note on which strategy retrieved better.

How: Embed each query once, run two SQL queries differing only in the WHERE split_strategy = ... filter, print top-5 each with their distance score.

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE IF NOT EXISTS legal_reference_documents (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_file     TEXT NOT NULL UNIQUE,
    doc_type        TEXT NOT NULL,        -- statute | judgment | regulation | circular

);

CREATE TABLE IF NOT EXISTS legal_reference_chunks (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id     UUID NOT NULL REFERENCES legal_reference_documents(id) ON DELETE CASCADE,
    source_document TEXT NOT NULL,
    doc_type        TEXT NOT NULL,
    doc_group       TEXT NOT NULL,
    jurisdiction    TEXT NOT NULL,
    section_title   TEXT,                 -- populated for structure_aware chunks
    chunk_index     INTEGER NOT NULL,
    split_strategy  TEXT NOT NULL,        -- 'recursive' | 'structure_aware'
    content         TEXT NOT NULL,
    embedding       vector(1536) NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Filtering columns used constantly by every retriever downstream
CREATE INDEX IF NOT EXISTS idx_chunks_doc_group ON legal_reference_chunks (doc_group);
CREATE INDEX IF NOT EXISTS idx_chunks_jurisdiction ON legal_reference_chunks (jurisdiction);
CREATE INDEX IF NOT EXISTS idx_chunks_split_strategy ON legal_reference_chunks (split_strategy);
CREATE INDEX IF NOT EXISTS idx_chunks_document_id ON legal_reference_chunks (document_id);

-- Build this AFTER your first bulk ingest for best clustering quality.
CREATE INDEX IF NOT EXISTS idx_chunks_embedding_hnsw
    ON legal_reference_chunks
    USING hnsw (embedding vector_cosine_ops);

#6.4 Manifest schema and loader

The manifest is the single source of truth for what each source file is. This avoids fragile content-sniffing logic to guess jurisdiction or doc type.

{
  "indian_contract_act_1872.pdf": {
    "docType": "statute",
    "docGroup": "contract_law",
    "jurisdiction": "india_central",
    "sectionPattern": "section"
  },
  "it_act_2000.pdf": {
    "docType": "statute",
    "docGroup": "data_protection",
    "jurisdiction": "india_central",
    "sectionPattern": "section"
  },
  "sc_judgment_breach_damages_2019.pdf": {

ts
// src/ingestion/manifest.ts
import { readFile } from "fs/promises";
import path from "path";

export type SectionPatternKey = "section" | "paragraph" | "article" | "clause";

export interface DocManifestEntry {
  docType: "statute" | "judgment" | "regulation" | "circular";
  docGroup: string;
  jurisdiction: string;
  /** Which heading style this document uses for structure-aware splitting */
  sectionPattern?: SectionPatternKey;
}

export type Manifest = Record<string, DocManifestEntry>;

/**
 * Loads and validates the manifest.json describing every legal reference
 * document. Fails loudly if a required field is missing - an un-tagged
 * document silently ingested with null metadata breaks every filtered
 * retrieval query downstream, so we refuse to proceed instead.
 */
export async function loadManifest(dir: string): Promise<Manifest> {
  const manifestPath = path.join(dir, "manifest.json");
  const raw = await readFile(manifestPath, "utf-8");
  const manifest: Manifest = JSON.parse(raw);

  for (const [file, entry] of Object.entries(manifest)) {
    if (!entry.docType || !entry.docGroup || !entry.jurisdiction) {
      throw new Error(
        `Manifest entry for "${file}" is missing required fields ` +
          `(docType, docGroup, jurisdiction). Every source document must be tagged.`
      );
    }
  }

  return manifest;
}

#6.5 Lazy document loader

ts
// src/ingestion/loader.ts
import path from "path";
import { readdir, readFile } from "fs/promises";
import { createHash } from "crypto";
import { PDFLoader } from "@langchain/community/document_loaders/fs/pdf";
import { DocxLoader } from "@langchain/community/document_loaders/fs/docx";
import type { Document } from "@langchain/core/documents";
import { loadManifest, type Manifest, type DocManifestEntry } from "./manifest";

export interface LoadedDocument {
  sourceFile: string;
  filePath: string;
  contentHash: string;
  manifestEntry: DocManifestEntry;
  /** Full concatenated text of the document (pages joined) */
  fullText: string;
  pageCount: number;
}

function hashBuffer(buf: Buffer): string {
  return createHash("sha256").update(buf).digest("hex");
}

function loaderForFile(filePath: string) {
  const ext = path.extname(filePath).toLowerCase();
  if (ext === ".pdf") return new PDFLoader(filePath, { splitPages: true });
  if (ext === ".docx") return new DocxLoader(filePath);
  throw new Error(`Unsupported file extension: ${ext} (${filePath})`);
}

/**
 * Lazily yields one fully-loaded document at a time. Using lazyLoad()
 * (rather than eagerly loading every file up front) means each document's
 * pages are parsed, joined, hashed, and handed off to the caller before the
 * next file is even opened - important habit once this corpus grows well
 * past 30 documents.
 */
export async function* lazyLoadLegalDocuments(
  dir: string
): AsyncGenerator<LoadedDocument> {
  const manifest: Manifest = await loadManifest(dir);
  const files = (await readdir(dir)).filter(
    (f) => f.endsWith(".pdf") || f.endsWith(".docx")
  );

  for (const file of files) {
    const manifestEntry = manifest[file];
    if (!manifestEntry) {
      throw new Error(
        `File "${file}" found in ${dir} but has no manifest.json entry. ` +
          `Add one before ingesting.`
      );
    }

    const filePath = path.join(dir, file);
    const rawBytes = await readFile(filePath);
    const contentHash = hashBuffer(rawBytes);

    const loader = loaderForFile(filePath);
    const pages: Document[] = [];
    // Loader-level lazyLoad(): stream pages of THIS file one at a time too.
    for await (const page of loader.lazyLoad()) {
      pages.push(page);
    }

    yield {
      sourceFile: file,
      filePath,
      contentHash,
      manifestEntry,
      fullText: pages.map((p) => p.pageContent).join("\n\n"),
      pageCount: pages.length,
    };
  }
}

#6.6 Dual splitting strategy

ts
// src/ingestion/splitters.ts
import { RecursiveCharacterTextSplitter } from "langchain/text_splitter";
import type { SectionPatternKey } from "./manifest";

export interface SplitChunk {
  content: string;
  chunkIndex: number;
  splitStrategy: "recursive" | "structure_aware";
  sectionTitle: string | null;
}

const RECURSIVE_CHUNK_SIZE = 1000;
const RECURSIVE_CHUNK_OVERLAP = 150;

/**
 * Length-based splitting. No knowledge of document structure - just tries
 * to break on paragraph/sentence boundaries near the target chunk size.
 * This is the safe fallback that works on any document.
 */
export async function splitRecursive(fullText: string): Promise<SplitChunk[]> {
  const splitter = new RecursiveCharacterTextSplitter({
    chunkSize: RECURSIVE_CHUNK_SIZE,
    chunkOverlap: RECURSIVE_CHUNK_OVERLAP,
  });

  const pieces = await splitter.splitText(fullText);

  return pieces.map((content, i) => ({
    content,
    chunkIndex: i,
    splitStrategy: "recursive" as const,
    sectionTitle: null,
  }));
}

// One regex per heading style a source document might use.
const SECTION_PATTERNS: Record<SectionPatternKey, RegExp> = {
  section: /^(?:Section|Sec\.)\s+\d+[A-Za-z]*\.?\s*[-–:]?/gim,
  article: /^Article\s+\d+[A-Za-z]*\.?\s*[-–:]?/gim,
  clause: /^Clause\s+\d+[A-Za-z]*\.?\s*[-–:]?/gim,
  paragraph: /^(?:Para(?:graph)?\.?)\s+\d+[A-Za-z]*\.?\s*[-–:]?/gim,
};

/**
 * Splits on the document's own structural headings (e.g. "Section 73."),
 * so each chunk is exactly one complete legal provision rather than an
 * arbitrary character-count slice that might cut a rule in half.
 *
 * Returns an empty array (rather than throwing) if the pattern finds no
 * matches, signalling to the caller that this document has no detectable
 * structure and should fall back to recursive-only splitting.
 */
export function splitByStructure(
  fullText: string,
  patternKey: SectionPatternKey | undefined
): SplitChunk[] {
  if (!patternKey) return [];

  const pattern = SECTION_PATTERNS[patternKey];
  const matches = [...fullText.matchAll(pattern)];
  if (matches.length < 2) {
    // Fewer than 2 headings found - not enough real structure to split on.
    return [];
  }

  const chunks: SplitChunk[] = [];
  for (let i = 0; i < matches.length; i++) {
    const start = matches[i].index!;
    const end = i + 1 < matches.length ? matches[i + 1].index! : fullText.length;
    const rawChunk = fullText.slice(start, end).trim();
    if (!rawChunk) continue;

    const heading = matches[i][0].trim();

    chunks.push({
      content: rawChunk,
      chunkIndex: i,
      splitStrategy: "structure_aware",
      sectionTitle: heading,
    });
  }

  return chunks;
}

#6.7 Batched embeddings

ts
// src/ingestion/embeddings.ts
import { OpenAIEmbeddings } from "@langchain/openai";

const BATCH_SIZE = 100;

const embeddingsClient = new OpenAIEmbeddings({
  apiKey: process.env.OPENAI_API_KEY,
  model: process.env.EMBEDDING_MODEL ?? "text-embedding-3-small",
});

function chunkArray<T>(arr: T[], size: number): T[][] {
  const out: T[][] = [];
  for (let i = 0; i < arr.length; i += size) out.push(arr.slice(i, i + size));
  return out;
}

/**
 * Embeds an array of chunk texts in batches to stay well under OpenAI's
 * per-request token limits, and to make retrying a single failed batch
 * cheap instead of re-embedding the whole document on any transient error.
 */
export async function embedTexts(texts: string[]): Promise<number[][]> {
  const batches = chunkArray(texts, BATCH_SIZE);
  const allVectors: number[][] = [];

  for (const [i, batch] of batches.entries()) {
    try {
      const vectors = await embeddingsClient.embedDocuments(batch);
      allVectors.push(...vectors);
    } catch (err) {
      console.error(`Embedding batch ${i + 1}/${batches.length} failed:`, err);
      throw err;
    }
  }

  return allVectors;
}

#6.8 Storage with idempotency

ts
// src/ingestion/storage.ts
import { Pool, type PoolClient } from "pg";
import format from "pg-format";
import type { LoadedDocument } from "./loader";
import type { SplitChunk } from "./splitters";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

/**
 * Returns the stored content_hash for a source file, or null if it has
 * never been ingested. Used to skip re-processing unchanged files.
 */
export async function getStoredHash(sourceFile: string): Promise<string | null> {
  const { rows } = await pool.query(
    `SELECT content_hash FROM legal_reference_documents WHERE source_file = $1`,
    [sourceFile]
  );
  return rows[0]?.content_hash ?? null;
}

/**
 * Upserts the document row and replaces all of its chunks in a single
 * transaction. If the process crashes mid-way, the transaction rolls back
 * and the document is left exactly as it was before this run - never in a
 * half-ingested state.
 */
export async function storeDocumentChunks(
  doc: LoadedDocument,
  chunks: SplitChunk[],
  embeddings: number[][],
  embeddingModel: string
): Promise<void> {
  const client: PoolClient = await pool.connect();

  try {
    await client.query("BEGIN");

    const upsertResult = await client.query(
      `INSERT INTO legal_reference_documents
         (source_file, doc_type, doc_group, jurisdiction, content_hash, embedding_model, updated_at)
       VALUES ($1, $2, $3, $4, $5, $6, now())
       ON CONFLICT (source_file) DO UPDATE SET
         content_hash = EXCLUDED.content_hash,
         embedding_model = EXCLUDED.embedding_model,
         updated_at = now()
       RETURNING id`,
      [
        doc.sourceFile,
        doc.manifestEntry.docType,
        doc.manifestEntry.docGroup,
        doc.manifestEntry.jurisdiction,
        doc.contentHash,
        embeddingModel,
      ]
    );
    const documentId: string = upsertResult.rows[0].id;

    // Clear old chunks for this document before inserting the new set -
    // this makes re-ingestion idempotent instead of appending duplicates.
    await client.query(`DELETE FROM legal_reference_chunks WHERE document_id = $1`, [
      documentId,
    ]);

    if (chunks.length > 0) {
      const values = chunks.map((chunk, i) => [
        documentId,
        doc.sourceFile,
        doc.manifestEntry.docType,
        doc.manifestEntry.docGroup,
        doc.manifestEntry.jurisdiction,
        chunk.sectionTitle,
        chunk.chunkIndex,
        chunk.splitStrategy,
        chunk.content,
        `[${embeddings[i].join(",")}]`, // pgvector literal format
      ]);

      const insertSql = format(
        `INSERT INTO legal_reference_chunks
           (document_id, source_document, doc_type, doc_group, jurisdiction,
            section_title, chunk_index, split_strategy, content, embedding)
         VALUES %L`,
        values
      );
      await client.query(insertSql);
    }

    await client.query("COMMIT");
  } catch (err) {
    await client.query("ROLLBACK");
    throw err;
  } finally {
    client.release();
  }
}

export async function closePool(): Promise<void> {
  await pool.end();
}
📝Note

Note: install pg-format (npm install pg-format @types/pg-format) — it safely builds the multi-row VALUES list, which is both faster and safer than looping individual INSERTs.

#6.9 Pipeline orchestration

ts
// src/ingestion/pipeline.ts
import "dotenv/config";
import { lazyLoadLegalDocuments } from "./loader";
import { splitRecursive, splitByStructure, type SplitChunk } from "./splitters";
import { embedTexts } from "./embeddings";
import { getStoredHash, storeDocumentChunks, closePool } from "./storage";

const EMBEDDING_MODEL = process.env.EMBEDDING_MODEL ?? "text-embedding-3-small";
const LEGAL_DOCS_DIR = process.env.LEGAL_DOCS_DIR ?? "./data/legal-docs";

export async function runIngestion(): Promise<void> {
  let processed = 0;
  let skipped = 0;

  for await (const doc of lazyLoadLegalDocuments(LEGAL_DOCS_DIR)) {
    const existingHash = await getStoredHash(doc.sourceFile);
    if (existingHash === doc.contentHash) {
      console.log(`[skip] ${doc.sourceFile} - unchanged since last ingest`);
      skipped++;
      continue;
    }

    console.log(`[ingest] ${doc.sourceFile} (${doc.pageCount} pages)`);

    // Always produce the recursive chunks - the safe universal baseline.
    const recursiveChunks: SplitChunk[] = await splitRecursive(doc.fullText);

    // Attempt structure-aware chunks; fall back silently if no structure found.
    const structureChunks: SplitChunk[] = splitByStructure(
      doc.fullText,
      doc.manifestEntry.sectionPattern
    );
    if (structureChunks.length === 0) {
      console.log(
        `  [info] no structural headings detected for ${doc.sourceFile}, ` +
          `storing recursive chunks only`
      );
    }

    const allChunks = [...recursiveChunks, ...structureChunks];
    const embeddings = await embedTexts(allChunks.map((c) => c.content));

    await storeDocumentChunks(doc, allChunks, embeddings, EMBEDDING_MODEL);

    console.log(
      `  -> stored ${recursiveChunks.length} recursive + ` +
        `${structureChunks.length} structure_aware chunks`
    );
    processed++;
  }

  console.log(`\nDone. Processed ${processed} document(s), skipped ${skipped} unchanged.`);
  await closePool();
}

#6.10 Retrieval comparison script

ts
// scripts/compare-retrieval.ts
import "dotenv/config";
import { Pool } from "pg";
import { OpenAIEmbeddings } from "@langchain/openai";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });
const embeddingsClient = new OpenAIEmbeddings({
  model: process.env.EMBEDDING_MODEL ?? "text-embedding-3-small",
});

const SAMPLE_QUERIES = [
  "What is the required notice period for termination without cause?",
  "What are the limits on indemnification liability?",
  "How long does a confidentiality obligation survive after contract termination?",
];

async function searchByStrategy(queryVector: number[], strategy: string, limit = 5) {
  const { rows } = await pool.query(
    `SELECT source_document, section_title, chunk_index,
            left(content, 200) AS preview,
            embedding <=> $1 AS distance
     FROM legal_reference_chunks
     WHERE split_strategy = $2
     ORDER BY embedding <=> $1
     LIMIT $3`,
    [`[${queryVector.join(",")}]`, strategy, limit]
  );
  return rows;
}

async function main() {
  for (const query of SAMPLE_QUERIES) {
    console.log(`\n=== Query: "${query}" ===`);
    const [queryVector] = await embeddingsClient.embedDocuments([query]);

    const recursiveResults = await searchByStrategy(queryVector, "recursive");
    const structureResults = await searchByStrategy(queryVector, "structure_aware");

    console.log("\n-- recursive --");
    recursiveResults.forEach((r, i) =>
      console.log(
        `${i + 1}. [${r.distance.toFixed(4)}] ${r.source_document} #${r.chunk_index}\n   ${r.preview}...`
      )
    );

    console.log("\n-- structure_aware --");
    structureResults.forEach((r, i) =>
      console.log(
        `${i + 1}. [${r.distance.toFixed(4)}] ${r.source_document} (${r.section_title ?? "n/a"})\n   ${r.preview}...`
      )
    );
  }

  await pool.end();
}

main().catch((err) => {
  console.error(err);
  process.exit(1);
});

#Why this design over alternatives

  • Per-document transaction, not one giant transaction for all 30 documents: a failure on document #22 (a malformed PDF, an API timeout) shouldn't force you to re-embed and re-insert documents 1-21 that already succeeded.
  • Hash-based idempotency at the document level, not the chunk level: far cheaper to check ("has this file changed at all?") than diffing individual chunks, and matches how you'll actually update this corpus in practice — you'll replace an entire statute PDF with an amended version, not hand-edit one paragraph of stored text.
  • Manifest-driven metadata over content-sniffing: for a small, curated, high-stakes corpus (legal reference material feeding a risk analyzer), explicit and auditable beats "clever" auto-detection that might mis-tag a document's jurisdiction.

#8. Best Practices

Industry best practices

  • Version your embedding model in the schema (embedding_model column) so a future upgrade to a new embedding model doesn't silently mix incompatible vector spaces in similarity search.
  • Keep ingestion idempotent and re-runnable — treat npm run ingest as safe to run any time, not a one-shot script.
  • Separate "documents" metadata from "chunks" data — this is standard RAG system design and makes auditing/debugging far easier.

Scalability considerations

  • lazyLoad() throughout keeps memory flat regardless of corpus size.
  • HNSW index scales to millions of rows; for now (few thousand chunks), even a sequential scan would be fast, but building the index now costs nothing and means Phase 4 never needs to revisit this.
  • Batch embedding calls (100 at a time) balance API round-trip overhead against per-request token limits.

Security considerations

  • Never commit .env / API keys. Add data/legal-docs/*.pdf and .docx to .gitignore if these documents are proprietary/licensed, not just secrets.
  • Sanitize file names before using them in any file-system path construction if this pipeline is ever exposed to non-trusted uploads (not a concern for this curated Phase 1 corpus, but worth remembering before Phase 2's user-uploaded contracts).

Performance optimizations

  • Multi-row INSERT ... VALUES instead of per-row inserts.
  • Batch embedding requests instead of one call per chunk.
  • Build the HNSW index once after bulk load, not incrementally per document during a big initial ingest (drop and recreate if doing a very large one-time backfill; for 30 documents this is a non-issue).

Maintainability tips

  • Keep SECTION_PATTERNS centralized and named — as your corpus grows to include Acts with different numbering conventions, you add a pattern key, not new branching logic.
  • Log every skip/fallback decision ([skip], [info] no structural headings detected) — six months from now, an ingestion log is how you'll debug "why does this document have zero structure_aware chunks."

Testing considerations

  • Unit test splitByStructure() against a small fixture string with known section markers — assert exact chunk boundaries and count.
  • Unit test the hash-based skip logic with a mocked DB layer — assert an unchanged file is never re-embedded (this directly protects your OpenAI API cost).
  • Integration test: run the full pipeline against 1-2 real sample documents in a test schema, then assert row counts and that both split strategies produced non-overlapping, order-preserving chunk indices.

#9. Summary

What we learned:

  • How document loaders normalize different file formats into a common Document shape, and why lazyLoad() is the correct default for any pipeline that might scale.
  • The difference between length-based (RecursiveCharacterTextSplitter) and structure-aware text splitting, and why legal text specifically benefits from comparing both rather than picking one blindly.
  • What embeddings and vector similarity actually measure, and why pgvector — living inside the Postgres you already run — is the right choice at this project's scale, over a dedicated vector database.
  • Why metadata (doc_group, jurisdiction, split_strategy) is not an afterthought but half of what makes retrieval useful, since vector similarity alone can't express "only from this jurisdiction."
  • How to design an ingestion pipeline that's safe to re-run (idempotent via content hashing) and safe to partially fail (per-document transactions).

What has been completed:

  • Database schema for the legal reference knowledge base (legal_reference_documents, legal_reference_chunks).
  • A manifest-driven, lazily-loading document loader for PDF/DOCX.
  • Two independent, comparable text-splitting strategies.
  • Batched OpenAI embedding generation.
  • Idempotent, transactional storage into pgvector.
  • A full orchestration pipeline (npm run ingest) you can run against your real ~30 documents.
