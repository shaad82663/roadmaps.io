LegalDocs — Phase 6.5: Multi-Agent Supervisor

#LegalDocs — Phase 6.5: Multi-Agent Supervisor

Tech stack: Node.js / TypeScript, OpenAI, PostgreSQL (Phase 0/1), LangGraph.js (StateGraph, Command for dynamic handoff routing — first real use in the project), @langchain/langgraph-checkpoint-postgres, Zod Depends on: Phase 4 (the risk analysis StateGraph, invoked as a whole), Phase 4.5 (the legal research createReactAgent, invoked ad hoc rather than only automatically per HIGH-risk clause), Phase 5 (the report-generation entrypoint workflow), Phase 6 (the conversational chat agent, and the closure-scoping / checkpointer-namespacing discipline this phase generalizes) Feeds into: Phase 7 (streaming/async progress — a Supervisor run that hands off through several specialists is exactly the kind of longer-running request that benefits from streaming intermediate updates back to the client rather than one blocking response)

#1. Phase Overview

#Objective

Every capability this project has built so far is reached through its own dedicated endpoint, and the client has to know which one to call, in what order, with what IDs: start an analysis here, generate a report there, open a chat session somewhere else. This phase builds a single entry point — a Supervisor — that a user (or a frontend) can talk to in plain language, which itself figures out what's actually being asked and routes the work to whichever specialist can do it: the risk analyzer (Phase 4), the legal research agent (Phase 4.5), the conversational assistant (Phase 6), or the report generator (Phase 5). "Analyze this contract, then get me case law for the indemnification clause, then give me a report" becomes one conversation with one entry point, not four separate API calls the client has to sequence correctly.

#Why this phase is important

Up through Phase 6, this project has been a well-organized collection of capabilities. This phase is where it becomes a system — something that behaves like a single assistant rather than four separate tools that happen to share a database. That distinction matters for two concrete reasons:

  1. It's the natural product shape. A real user doesn't think in terms of "call the analysis endpoint, then the report endpoint" — they think "help me understand this contract." A Supervisor is what lets the product actually match that mental model, and it's the direct payoff of every earlier phase's discipline about giving each capability a clean, well-scoped interface (Phase 4's checkpointed graph, Phase 4.5's tool-using agent, Phase 5's entrypoint workflow, Phase 6's per-session chat agent) — none of that had to be rebuilt to be composed here; it only had to be callable.
  2. It's where "multi-agent," a term you've likely heard used loosely, gets a precise, working definition. A single agent (Phase 4.5's research agent, Phase 6's chat agent) reasons in a loop and calls tools. A multi-agent system is a step up from that: multiple independent agents (or agent-like components), each genuinely capable of running on its own, coordinated by something that decides which one should act next. This phase is where you build that coordination layer for real, on top of components you've already built and already trust individually.

#2. Concept Learning

#2.1 What a Supervisor actually does — and, just as importantly, what it doesn't

A Supervisor, in the multi-agent sense this phase uses, has exactly two jobs: read the conversation so far, and decide which specialist should act next (or that the conversation is done and it's time to reply directly). It does not do any of the domain work itself — it doesn't score risk, it doesn't search case law, it doesn't write report prose. Think of it like a skilled receptionist at a professional services firm: they don't do the accounting or the legal work themselves, but they know the practice well enough to send you to the right person, and they follow up once that person is done to see if anything else is needed.

This separation is deliberate, not incidental. Every specialist this phase routes to is a component you already built, tested, and trust in isolation — Phase 4's graph already knows how to run a full risk analysis correctly; Phase 6's chat agent already knows how to answer questions grounded in one run's findings. The Supervisor's entire value is in not re-implementing any of that, only in deciding, per turn, which of those already-correct components is the right one to invoke.

#2.2 Two competing shapes for "sub-agents," and why this phase picks one

There are two common ways to structure a Supervisor's relationship to its specialists, and it's worth understanding both, because they solve slightly different problems:

Shape A — specialists as tools. The Supervisor is itself a single createReactAgent-style loop (exactly like Phase 4.5's or Phase 6's), and each specialist is wrapped as a tool() the Supervisor's own model can call. This is simple — you already know exactly how to build it, it's the same mechanism from Phase 4.5 — but it has a real limitation here: a tool's result is typically just a string (or a small JSON blob) handed back into the calling agent's context. That's fine when a specialist's output is naturally a short string ("here's what I found"). It's a worse fit when a specialist's outcome needs to update several distinct, structured pieces of shared context at once — for example, after running a full analysis, the Supervisor needs to durably remember which analysisRunId is now active, not just receive a sentence describing that an analysis happened.

Shape B — specialists as peer graph nodes, coordinated via handoffs. The Supervisor and every specialist are all nodes in the same StateGraph, and control passes between them via an explicit handoff: a node's own return value says both "here's what changed" and "here's who should act next." This is a better fit for this phase specifically because the shared context genuinely has multiple structured fields that need to flow precisely between specialists (contractId, analysisRunId, chatSessionId) — not just a conversational transcript.

This phase uses Shape B. Section 2.3 covers the exact mechanism (Command) that makes a handoff work.

#2.3 Command — finally cashing in on Phase 4's foreshadowing

Phase 4's Concept Learning introduced Command and then deliberately set it aside: "Command becomes more valuable once a node's own internal logic is what determines routing dynamically... which you'll see more of once Phase 6.5's Supervisor needs to decide routing based on more complex internal reasoning." This is that moment.

Recall the two things a graph node could do up through Phase 4: return a plain partial-state object (just an update, with routing handled separately by addConditionalEdges), or — this phase's mechanism — return a Command, which bundles a state update and a routing decision into one object, returned from inside the node's own logic:

ts
return new Command({
  goto: "risk_analyzer",       // which node runs next
  update: { pendingInstruction: "Run a full risk analysis on this contract." },
});

Why this fits the Supervisor specifically, where routeByRisk (Phase 4) didn't need it: routeByRisk was a two-way, simple, fixed decision — HIGH or not-HIGH, always computed the same way from one field. The Supervisor's routing decision is genuinely more involved: it's the direct output of an LLM's reasoning about an open-ended conversation, among four-plus possible destinations, and that reasoning naturally also wants to say what it wants the next specialist to do (an instruction), not just which specialist. Splitting "what the LLM decided" (inside the node) from "where that sends execution" (in a separate addConditionalEdges function that would have to re-derive the same decision) would mean either recomputing the LLM's reasoning twice or awkwardly stashing it in a state field just to hand it to a separate router function. Command lets the node that actually made the decision be the same place that acts on it.

A Command can send execution to END (finish the graph run) just as easily as to another named node:

ts
return new Command({
  goto: END,
  update: { messages: [new AIMessage("Here's your risk summary...")] },
});

And critically, specialist nodes use Command too, to hand control back: once risk_analyzer finishes running Phase 4's graph, it returns Command({ goto: "supervisor", update: {...} }) — handing off, not just "finishing" the way a node in a fixed pipeline would. This is what makes the graph a genuine loop between coordinator and specialists, rather than a one-way dispatch.

#2.4 The routing decision itself — one more structured-output classification

Underneath the Command mechanism, the Supervisor still needs to actually decide where to go, and that decision is, at its core, the same kind of thing you've built since Phase 2: a withStructuredOutput() call that classifies the situation into one of a fixed set of outcomes.

ts
const RoutingDecisionSchema = z.object({
  next: z.enum(["risk_analyzer", "research_agent", "chat_assistant", "report_generator", "FINISH"]),
  instructionForAgent: z.string().describe("A short, specific instruction for the chosen specialist"),
  responseIfFinished: z.string().nullable().describe("If next is FINISH, the reply to send now. Null otherwise."),
});

What makes this a routing classification rather than, say, Phase 2's contract-type classification, is simply what it's classifying: not "what kind of document is this" but "what does this conversation need next, given everything that's happened so far." The mechanism — one model call, one schema, one guaranteed-shaped answer — is identical.

#2.5 Revisiting "call the compiled unit from a plain node" vs. true subgraph nesting — now with an actual Supervisor to test it against

Phase 4.5 made a deliberate, explicitly deferred choice: call Phase 4's compiled analysisGraph from inside a plain node function (await analysisGraph.invoke(...)), rather than nesting it as a true LangGraph subgraph sharing the outer graph's state and checkpointer — and named "once there's a Supervisor architecture to nest it inside" as the right point to revisit that decision. Here it is, revisited honestly:

True subgraph nesting requires the inner graph's state schema to either match the outer graph's, or be reconciled by an explicit transformation function at the boundary. Phase 4's AnalysisState has fields — currentClause, currentFinding, the Send()-driven fan-out over an entire clause list — that exist only to support that graph's own internal parallel dispatch. None of that has any meaningful counterpart in the Supervisor's state, which just needs to know "an analysis is running, scoped to this contractId, and here's the analysisRunId once it's done." Building a transformation function to bridge these two shapes would mean writing an adapter that ends up doing almost exactly what "just invoke the compiled graph and read its return value" already does for you — except with more ceremony, and a second thing (the transformation logic) to keep correct as either state schema evolves.

The actual conclusion, now that it's been tested against a real case: invoking an already-compiled, independently useful unit (a graph, an agent, a workflow) from inside a plain node function is the right general pattern for composing this project's specialists — not a stopgap. True subgraph nesting remains the right tool specifically when two graphs share the same fundamental state shape (for instance, if a future phase needed to nest one message-based agent inside another message-based agent) — that's not the situation here, and forcing it wouldn't have bought anything Phase 4.5's original choice didn't already deliver.

#2.6 A federated state: the Supervisor tracks references, not full internal state

Each specialist this phase routes to is already independently durable: Phase 4's analysisGraph has its own checkpointer and its own thread_id per run; Phase 6's chat agent has its own checkpointer and its own thread_id per session. The Supervisor's own state doesn't need to — and shouldn't — duplicate any of that internal detail. It only needs to carry the references that let it (or a specialist) find the right underlying data: which contractId this conversation is about, which analysisRunId is currently active (if any), which chatSessionId is currently active (if any).

ts
export const SupervisorState = Annotation.Root({
  messages: Annotation<BaseMessage[]>({ reducer: (e, i) => e.concat(i), default: () => [] }),
  contractId: Annotation<string | null>({ reducer: (_e, i) => i, default: () => null }),
  analysisRunId: Annotation<string | null>({ reducer: (_e, i) => i, default: () => null }),
  chatSessionId: Annotation<string | null>({ reducer: (_e, i) => i, default: () => null }),
  pendingInstruction: Annotation<string | null>({ reducer: (_e, i) => i, default: () => null }),
});

This is the same principle a microservice architecture follows when passing an order ID between services instead of the order's entire internal state — each specialist remains the sole owner and authority over its own detailed internal history; the Supervisor just needs to know which instance of each to point at.

A sharp, easy-to-miss correctness detail this creates: because LangGraph only updates a state field when a node's return value includes that key at all (an omitted key means "no change," not "clear it to null"), a client sending a follow-up message must never resend contractId/analysisRunId/chatSessionId as part of that turn's input unless it actually wants to change them — sending analysisRunId: null on turn two would overwrite the value Phase 4's specialist worked hard to set on turn one. The correct client behavior is: send only { messages: [...] } on every turn after the first; let the checkpointer be the sole source of truth for everything else.

#2.7 The Supervisor's own memory

The Supervisor conversation itself needs the exact same durable, multi-turn memory Phase 6 built for chat — a user's "analyze this, then get me case law, then send a report" conversation can span several separate HTTP requests, and the Supervisor needs to remember, on request three, that request one already produced an analysisRunId. This reuses the identical PostgresSaver checkpointer backend already running in this project, with its own namespaced thread_id prefix (supervisor-<uuid>) — the same "share the infrastructure, keep the ID namespace distinct" discipline Phase 6 established for chat threads, now applied one layer up.

#3. Different Approaches

#3.1 Sub-agent coordination: specialists as tools vs. specialists as peer nodes with Command handoffs

Approach Pros Cons When to use
A. Specialists as tools on one createReactAgent Reuses a mechanism you already know exactly how to build (Phase 4.5, Phase 6); less new API surface A tool's result is naturally a string/small blob returned into the calling agent's context — an awkward fit when a specialist's outcome needs to durably update several distinct structured fields (an active analysisRunId, a chatSessionId) rather than just add a sentence to context A Supervisor whose specialists' outcomes really are just "a short answer to hand back," with no structured state to track between them
B. Specialists as peer StateGraph nodes, coordinated via Command handoffs (recommended) Each specialist's node return can precisely update the exact structured fields the rest of the system needs (analysisRunId, chatSessionId), not just append to a transcript; the routing decision and its accompanying instruction live in one place, made by whichever node's logic actually decided it A second coordination mechanism (Command) to learn, on top of tool-calling Recommended — this phase's shared context is genuinely structured (multiple distinct IDs that must flow precisely), which is exactly where Approach B's precision earns its keep over Approach A's string-shaped tool results

Recommendation: Approach B, as reasoned through in Concept Learning 2.2–2.3.

#3.2 Routing decision mechanism: fixed keyword/intent rules vs. LLM structured output

Approach Pros Cons When to use
A. Fixed keyword/intent rules (e.g., "contains 'report'" → report_generator) Free, instant, fully deterministic Brittle against real phrasing variety ("can you write this up for me" never says the word "report"); can't handle a single message that implies multiple steps A narrow, tightly-controlled command interface where users only ever issue a small set of known exact phrasings
B. LLM structured-output classification (recommended) Handles the actual variety of how people phrase requests; can reason about sequencing ("analyze, then find case law, then report" implies three sequential routing decisions across three turns of the loop, not one) One more LLM call per routing decision (though a cheap one — this is a short classification, not a long generation) Recommended — the entire point of a natural-language Supervisor is not requiring the user to phrase things in a way a keyword rule would recognize

Recommendation: Approach B, as specified.

#3.3 Composing existing specialists: plain-node invocation vs. true subgraph nesting

Approach Pros Cons When to use
A. Invoke each specialist's already-compiled graph/agent/workflow from inside a plain Supervisor node function (recommended) Each specialist keeps its own independently-correct state schema and checkpointing; no adapter/transformation code to write or maintain; matches the pattern Phase 4.5 already established The specialist's own internal steps aren't checkpointed as part of the Supervisor's own state history (though each specialist remains independently checkpointed and auditable on its own terms — Phase 4's audit endpoints, for instance, are completely unaffected) Composing genuinely independent units whose internal state shapes don't naturally align — true of every specialist in this phase
B. True subgraph nesting, reconciling each specialist's state schema with the Supervisor's via a boundary transformation function Every specialist's internal steps would also appear in the Supervisor's own checkpoint history Requires a real transformation function per specialist, reconciling state shapes that have little in common by design (Concept Learning 2.5); the adapter code ends up duplicating most of what Approach A already gives you Two graphs that already share a fundamentally similar state shape (e.g., nesting one message-based agent inside another)

Recommendation: Approach A, as the now-confirmed general pattern for this project — see Concept Learning 2.5 for the full reasoning, revisited from Phase 4.5's original deferral.

#3.4 Shared state: full internal detail vs. federated references only

Approach Pros Cons When to use
A. Supervisor state carries full internal detail from every specialist (e.g., the entire findings array inline) Everything visible in one place Duplicates data every specialist already stores and owns durably; grows the Supervisor's state (and therefore every checkpoint of it) without bound as more specialists produce more data Never really, for this architecture — it fights the whole reason each specialist has its own storage
B. Supervisor state carries only references (analysisRunId, chatSessionId) (recommended) Small, stable state regardless of how much detail a specialist's own domain produces; each specialist remains the sole source of truth for its own data A specialist node has to do one lookup (by ID) to get its own detail back, rather than finding it already sitting in shared state Recommended — matches this project's established pattern (every phase since Phase 4 has stored its own detailed data in its own table, referenced by ID)

Recommendation: Approach B, as specified in Concept Learning 2.6.

#4. Detailed Implementation Plan

#Step 1 — Database schema for assistant sessions

What: An assistant_sessions table — one row per Supervisor conversation, scoped to a contractId, with its own namespaced thread_id.

Why: The same reasoning as Phase 6's chat_sessions — a stable row the routes look up to resolve which checkpointed conversation a given session ID refers to.

#Step 2 — Extracting a reusable analysis-trigger helper

What: Pull the "load clauses, generate a thread_id, invoke analysisGraph" logic currently inline in Phase 4's POST /analysis/start route out into its own function, runAnalysisForContract(contractId), returning { analysisRunId, findings }. Update Phase 4's route to call this same helper.

Why: The Supervisor's risk_analyzer node needs to do exactly this, and duplicating the logic inline would mean two places that could silently drift out of sync. This is a small, low-risk refactor — Phase 4's route's behavior doesn't change at all, only where the logic physically lives.

Best practice: Whenever a second caller needs logic that currently lives inline in a route handler, that's the concrete signal to extract it — not a violation of "don't build for hypothetical future needs," since the second caller already exists by the time you're extracting it.

#Step 3 — Supervisor state schema

What: SupervisorState (Annotation.Root) — messages (concatenating), contractId, analysisRunId, chatSessionId, pendingInstruction (all replace-reducer).

Why/How: Covered in Concept Learning 2.6.

#Step 4 — Routing schema and the supervisor node

What: RoutingDecisionSchema and the supervisor node function, which runs the classification chain and returns a Command — either handing off to a named specialist (with pendingInstruction set) or finishing with goto: END.

Why/How: Covered in Concept Learning 2.3–2.4.

Common mistake to avoid: Having the supervisor node append a user-facing message on every intermediate handoff (e.g., "Routing you to the risk analyzer..."). Keep intermediate routing silent from the user's perspective — carry the decision via pendingInstruction in state, and only add a message to messages when there's something actually worth showing the user (a specialist's result, or the final FINISH reply).

#Step 5 — The four specialist nodes

What: risk_analyzer, research_agent, chat_assistant, report_generator — each a plain node function that reads what it needs from state (contractId, pendingInstruction, the latest human message), invokes its already-built compiled unit (Step 2's helper, Phase 4.5's runResearchReactAgent, Phase 6's sendChatMessage, Phase 5's generateReportWorkflow), and returns Command({ goto: "supervisor", update: {...} }).

Why/How: Covered in Concept Learning 2.5.

Common mistake to avoid: Letting a specialist node silently do nothing (and return an unhelpful generic message) when a precondition isn't met — e.g., chat_assistant being routed to before any analysis has run. Each specialist should explicitly check its own preconditions and hand back a clear, honest message ("I don't have a completed analysis to discuss yet") rather than erroring or guessing.

#Step 6 — Graph construction

What: Wire all five nodes into one StateGraph(SupervisorState), with a single fixed edge from START to supervisor — every other transition is handled dynamically via each node's own Command.

Why: Since every routing decision is made by the node's own logic at runtime (Concept Learning 2.3), there's no fixed edge to declare between supervisor and any specialist, or back — Command supplies that at runtime.

Best practice: Set an explicit recursionLimit on every .invoke() call — a Supervisor bouncing between itself and specialists is, structurally, exactly the same "the model's own decisions control how many steps happen" situation Phase 4.5 flagged for agent loops, just one level up.

#Step 7 — Session persistence and endpoints

What: assistant_sessions storage functions, and POST /api/assistant/sessions (create, scoped to a contractId) / POST /api/assistant/sessions/:id/messages (send a message, get the latest reply).

Why/How: Same pattern as Phase 6's chat endpoints, one layer up.

Common mistake to avoid: Passing contractId/analysisRunId/chatSessionId explicitly in the request body on every message. Per Concept Learning 2.6's sharp correctness note, only the first message of a session should ever set contractId; every later message should send only the new text and let the checkpointer supply everything else.

#6.3 Extracting the reusable analysis-trigger helper

ts
// src/analysis/runAnalysis.ts
import { randomUUID } from "crypto";
import { Pool } from "pg";
import { analysisGraph } from "./graph";
import { getFindingsForRun } from "./storage";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

/**
 * Extracted from Phase 4's POST /analysis/start route so the Supervisor's
 * risk_analyzer node can trigger the exact same behavior without
 * duplicating this logic - Phase 4's route now calls this too, so there is
 * exactly one place this logic lives.
 */
export async function runAnalysisForContract(contractId: string) {
  const { rows } = await pool.query(
    `SELECT id, clause_type, title, full_text FROM clauses WHERE contract_id = $1`,
    [contractId]
  );
  if (rows.length === 0) {
    throw new Error(`No extracted clauses found for contract ${contractId}.`);
  }

  const analysisRunId = randomUUID();
  const config = { configurable: { thread_id: analysisRunId } };

  await analysisGraph.invoke(
    {
      contractId,
      analysisRunId,
      clauses: rows.map((r) => ({
        id: r.id,
        clauseType: r.clause_type,
        title: r.title,
        fullText: r.full_text,
        docGroups: [],
      })),
    },
    config
  );

  const findings = await getFindingsForRun(analysisRunId);
  return { analysisRunId, findings };
}
ts
// src/routes/analysis.ts (excerpt - the existing route now delegates)
import { runAnalysisForContract } from "../analysis/runAnalysis";

analysisRouter.post("/contracts/:id/analysis/start", async (req, res) => {
  try {
    const { analysisRunId, findings } = await runAnalysisForContract(req.params.id);
    return res.status(201).json({ analysisRunId, findings });
  } catch (err) {
    console.error("Risk analysis failed:", err);
    return res.status(500).json({ error: (err as Error).message });
  }
});

#6.4 Supervisor state schema

ts
// src/supervisor/state.ts
import { Annotation } from "@langchain/langgraph";
import type { BaseMessage } from "@langchain/core/messages";

/**
 * A federated state: the Supervisor tracks REFERENCES to each specialist's
 * own durable data (analysisRunId, chatSessionId), not the detailed data
 * itself - every specialist already owns and persists its own detail
 * (Concept Learning 2.6). Every field below except `messages` uses a
 * replace reducer; an omitted key on any given update leaves it untouched,
 * which is what lets a client send only new message text on later turns
 * without accidentally clearing a previously-set analysisRunId.
 */
export const SupervisorState = Annotation.Root({
  messages: Annotation<BaseMessage[]>({
    reducer: (existing, incoming) => existing.concat(incoming),
    default: () => [],
  }),
  contractId: Annotation<string | null>({ reducer: (_e, i) => i, default: () => null }),
  analysisRunId: Annotation<string | null>({ reducer: (_e, i) => i, default: () => null }),
  chatSessionId: Annotation<string | null>({ reducer: (_e, i) => i, default: () => null }),
  pendingInstruction: Annotation<string | null>({ reducer: (_e, i) => i, default: () => null }),
});

export type SupervisorStateType = typeof SupervisorState.State;

// src/supervisor/nodes/supervisorNode.ts
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate, MessagesPlaceholder } from "@langchain/core/prompts";
import { Command, END } from "@langchain/langgraph";
import { AIMessage } from "@langchain/core/messages";
import { RoutingDecisionSchema } from "../routingSchema";
import type { SupervisorStateType } from "../state";

const model = new ChatOpenAI({
  model: process.env.SUPERVISOR_MODEL ?? "gpt-4o-mini",
  temperature: 0,
});

const routingPrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    "You coordinate a team of contract-analysis specialists:\n" +
      "- risk_analyzer: runs a full risk analysis on the contract (use when " +
      "  none has run yet, or the user explicitly asks to (re-)analyze).\n" +
      "- research_agent: looks up legal grounding (case law/precedent) for a " +
      "  specific clause or concern the user describes, on demand.\n" +
      "- chat_assistant: answers questions about an ALREADY-COMPLETED " +
      "  analysis's findings. Requires an analysis to already exist.\n" +
      "- report_generator: produces a downloadable report from an " +
      "  ALREADY-COMPLETED analysis. Requires an analysis to already exist.\n" +
      "- FINISH: nothing further is needed right now; reply directly.\n\n" +
      "Current state: analysis {analysisStatus}; chat session {chatStatus}.\n" +
      "Choose exactly one next step based on the conversation.",
  ],
  new MessagesPlaceholder("messages"),
]);

const routingChain = routingPrompt.pipe(
  model.withStructuredOutput(RoutingDecisionSchema, { name: "route_request" })
);

/**
 * The Supervisor's only two jobs: read the conversation, decide who acts
 * next. It never does domain work itself. Returns a Command so the routing
 * decision and its accompanying instruction are produced and applied in
 * one place (Concept Learning 2.3), rather than split between this node's
 * return value and a separate conditional-edge function.
 */
export async function supervisorNode(state: SupervisorStateType): Promise<Command> {
  const decision = await routingChain.invoke({
    messages: state.messages,
    analysisStatus: state.analysisRunId ? `active (${state.analysisRunId})` : "none yet",
    chatStatus: state.chatSessionId ? `active (${state.chatSessionId})` : "none yet",
  });

  if (decision.next === "FINISH") {
    return new Command({
      goto: END,
      update: {
        messages: [new AIMessage(decision.responseIfFinished ?? "Let me know how else I can help.")],
      },
    });
  }

  // Intermediate handoffs stay silent to the user - the instruction is
  // carried via pendingInstruction, not appended as a visible message.
  return new Command({
    goto: decision.next,
    update: { pendingInstruction: decision.instructionForAgent },
  });
}

// src/supervisor/nodes/riskAnalyzerNode.ts
import { Command } from "@langchain/langgraph";
import { AIMessage } from "@langchain/core/messages";
import { runAnalysisForContract } from "../../analysis/runAnalysis";
import type { SupervisorStateType } from "../state";

export async function riskAnalyzerNode(state: SupervisorStateType): Promise<Command> {
  if (!state.contractId) {
    return new Command({
      goto: "supervisor",
      update: {
        pendingInstruction: null,
        messages: [new AIMessage("I don't have a contract to analyze in this session yet.")],
      },
    });
  }

  const { analysisRunId, findings } = await runAnalysisForContract(state.contractId);
  const highCount = findings.filter((f: any) => f.risk_level === "HIGH").length;

  return new Command({
    goto: "supervisor",
    update: {
      analysisRunId,
      pendingInstruction: null,
      messages: [
        new AIMessage(
          `I've completed the risk analysis - ${findings.length} clauses reviewed, ` +
            `${highCount} flagged HIGH risk. Want a summary, a downloadable report, or to ask follow-up questions?`
        ),
      ],
    },
  });
}

// src/supervisor/nodes/researchAgentNode.ts
import { Command } from "@langchain/langgraph";
import { AIMessage } from "@langchain/core/messages";
import { runResearchReactAgent } from "../../research/reactAgentVersion";
import { distillAgentOutcome } from "../../research/outcomeParsing";
import type { SupervisorStateType } from "../state";

function lastHumanText(state: SupervisorStateType): string {
  const lastHuman = [...state.messages].reverse().find((m) => m.getType() === "human");
  return typeof lastHuman?.content === "string" ? lastHuman.content : "";
}

/**
 * Invokes Phase 4.5's research agent AD HOC - on demand, for whatever
 * clause/concern the user just described - rather than only automatically
 * per HIGH-risk clause during a full analysis run.
 */
export async function researchAgentNode(state: SupervisorStateType): Promise<Command> {
  const concern = state.pendingInstruction || lastHumanText(state);

  const result = await runResearchReactAgent(concern, "The user is asking for legal research assistance directly.");
  const outcome = await distillAgentOutcome(result.messages);

  const summary = outcome.reviewFlagged
    ? `I couldn't find sufficient legal grounding for this: ${outcome.reviewReason}`
    : `Here's what I found: ${outcome.references.map((r) => `${r.sourceDocument} — ${r.excerpt}`).join("; ")}`;

  return new Command({
    goto: "supervisor",
    update: { pendingInstruction: null, messages: [new AIMessage(summary)] },
  });
}

import { Command } from "@langchain/langgraph";
import { AIMessage } from "@langchain/core/messages";
import { createChatSession, getChatSession } from "../../chat/sessions";
import { sendChatMessage } from "../../chat/agent";
import type { SupervisorStateType } from "../state";

function lastHumanText(state: SupervisorStateType): string {
  const lastHuman = [...state.messages].reverse().find((m) => m.getType() === "human");
  return typeof lastHuman?.content === "string" ? lastHuman.content : "";
}

export async function chatAssistantNode(state: SupervisorStateType): Promise<Command> {
  if (!state.analysisRunId) {
    return new Command({
      goto: "supervisor",
      update: {
        pendingInstruction: null,
        messages: [new AIMessage("I don't have a completed analysis to discuss yet - want me to run one first?")],
      },
    });
  }

  // Reuses Phase 6's chat session per Supervisor conversation - created
  // once, then reused on every later chat-routed turn rather than starting
  // a fresh, memory-less session each time.
  let chatSessionId = state.chatSessionId;
  if (!chatSessionId) {
    const session = await createChatSession(state.analysisRunId, state.contractId!);
    chatSessionId = session.id;
  }

  const session = await getChatSession(chatSessionId);
  const reply = await sendChatMessage(session.thread_id, session.analysis_run_id, lastHumanText(state));

  return new Command({
    goto: "supervisor",
    update: { chatSessionId, pendingInstruction: null, messages: [new AIMessage(reply)] },
  });
}

// src/supervisor/graph.ts
import { StateGraph, START } from "@langchain/langgraph";
import { PostgresSaver } from "@langchain/langgraph-checkpoint-postgres";
import { SupervisorState } from "./state";
import { supervisorNode } from "./nodes/supervisorNode";
import { riskAnalyzerNode } from "./nodes/riskAnalyzerNode";
import { researchAgentNode } from "./nodes/researchAgentNode";
import { chatAssistantNode } from "./nodes/chatAssistantNode";
import { reportGeneratorNode } from "./nodes/reportGeneratorNode";

// The same checkpointer backend already running for Phase 4/5/6 - the
// Supervisor's own thread IDs are namespaced ("supervisor-<uuid>") when
// created, so they never collide with analysis-run or chat-session IDs in
// these shared tables.
const checkpointer = PostgresSaver.fromConnString(process.env.DATABASE_URL!);

/**
 * Only ONE fixed edge (START -> supervisor). Every other transition is
 * decided at runtime by each node's own Command return value - there is
 * nothing to declare between "supervisor" and any specialist, or back,
 * because that decision only exists once the node's own logic has run.
 */
const graph = new StateGraph(SupervisorState)
  .addNode("supervisor", supervisorNode, {
    ends: ["risk_analyzer", "research_agent", "chat_assistant", "report_generator", "__end__"],
  })
  .addNode("risk_analyzer", riskAnalyzerNode, { ends: ["supervisor"] })
  .addNode("research_agent", researchAgentNode, { ends: ["supervisor"] })
  .addNode("chat_assistant", chatAssistantNode, { ends: ["supervisor"] })
  .addNode("report_generator", reportGeneratorNode, { ends: ["supervisor"] })
  .addEdge(START, "supervisor");

export const supervisorGraph = graph.compile({ checkpointer });

export const SUPERVISOR_RECURSION_LIMIT = Number(process.env.SUPERVISOR_RECURSION_LIMIT ?? 12);

// src/supervisor/sessions.ts
import { Pool } from "pg";
import { randomUUID } from "crypto";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

export async function createAssistantSession(contractId: string) {
  const threadId = `supervisor-${randomUUID()}`;
  const { rows } = await pool.query(
    `INSERT INTO assistant_sessions (contract_id, thread_id) VALUES ($1, $2) RETURNING *`,
    [contractId, threadId]
  );
  return rows[0];
}

export async function getAssistantSession(sessionId: string) {
  const { rows } = await pool.query(`SELECT * FROM assistant_sessions WHERE id = $1`, [sessionId]);
  return rows[0] ?? null;
}

// src/routes/assistant.ts
import { Router } from "express";
import { HumanMessage } from "@langchain/core/messages";
import { supervisorGraph, SUPERVISOR_RECURSION_LIMIT } from "../supervisor/graph";
import { createAssistantSession, getAssistantSession } from "../supervisor/sessions";

export const assistantRouter = Router();

assistantRouter.post("/assistant/sessions", async (req, res) => {
  const { contractId } = req.body;
  if (!contractId) {
    return res.status(400).json({ error: "contractId is required." });
  }

  const session = await createAssistantSession(contractId);
  return res.status(201).json({ session });
});

assistantRouter.post("/assistant/sessions/:sessionId/messages", async (req, res) => {
  const { sessionId } = req.params;
  const { message } = req.body;

  if (typeof message !== "string" || message.trim().length === 0) {
    return res.status(400).json({ error: "message is required." });
  }

  const session = await getAssistantSession(sessionId);
  if (!session) {
    return res.status(404).json({ error: "Assistant session not found." });
  }

  const config = {
    configurable: { thread_id: session.thread_id },
    recursionLimit: SUPERVISOR_RECURSION_LIMIT,
  };

  // Only the FIRST message of a session needs to set contractId - every
  // later turn sends only new message text, per Concept Learning 2.6's
  // correctness note: an omitted key leaves the checkpointed value
  // untouched, but an explicitly-included null would overwrite it.
  const priorState = await supervisorGraph.getState(config);
  const input = priorState.values?.contractId
    ? { messages: [new HumanMessage(message)] }
    : { messages: [new HumanMessage(message)], contractId: session.contract_id };

  try {
    const result = await supervisorGraph.invoke(input, config);
    const lastMessage = result.messages[result.messages.length - 1];
    const reply = typeof lastMessage.content === "string" ? lastMessage.content : JSON.stringify(lastMessage.content);
    return res.status(200).json({ reply });
  } catch (err) {
    console.error("Assistant message failed:", err);
    return res.status(500).json({ error: "Failed to process message." });
  }
});

#Request flow — one multi-hop turn

text
POST /api/assistant/sessions/:id/messages   { message: "Analyze this contract and send me a report." }
  │
  ▼
supervisorGraph.invoke({ messages: [HumanMessage] }, { thread_id, recursionLimit })
  │
  ├─ supervisor    — LLM routing decision → Command(goto: "risk_analyzer")
  ├─ risk_analyzer — runs Phase 4's full graph → Command(goto: "supervisor", update: { analysisRunId })
  ├─ supervisor    — sees analysisRunId is now set → Command(goto: "report_generator")
  ├─ report_generator — runs Phase 5's workflow → Command(goto: "supervisor", update: { messages: [download link] })
  ├─ supervisor    — nothing further needed → Command(goto: END, update: { messages: [final reply] })
  ▼
HTTP 200 { reply: "Your report is ready: /api/reports/.../download" }

#Why this design over alternatives

  • Command-based peer nodes over tool-wrapped specialists: the shared context here is genuinely structured (multiple IDs that must flow precisely between specialists), which Command's combined update-and-route return handles more precisely than a tool call's string result — see Concept Learning 2.2–2.3.
  • Plain-node invocation over true subgraph nesting: every specialist's state shape exists to serve that specialist's own internal needs and has little in common with any other specialist's — forcing a shared schema would cost more in adapter code than it would return in checkpoint-visibility benefit, a conclusion reached by actually testing it against this phase's real specialists rather than assumed upfront (Concept Learning 2.5).
  • Federated state (IDs only) over centralizing all detail in the Supervisor: keeps the Supervisor's own state — and therefore every checkpoint of it — small and stable regardless of how much detailed data any one specialist produces, consistent with every specialist already owning its own durable storage.
  • A single fixed edge (START → supervisor) with everything else dynamic: accurately reflects that no other transition in this graph is actually knowable ahead of time — it's a genuine runtime decision, not something a fixed edge could correctly express.

#8. Best Practices

Industry best practices

  • Keep the Supervisor's own logic limited to routing — the moment a Supervisor starts doing domain work itself (rather than delegating), you've lost the clean separation that makes each specialist independently testable and independently correct.
  • Set an explicit recursionLimit on every Supervisor invocation — the same required-not-optional reasoning as any agent loop, now one level up, since a Supervisor bouncing between itself and specialists is exactly that shape of risk.
  • Keep intermediate routing decisions invisible to the end user (Step 4's best practice) — only surface messages that are genuinely worth the user reading.

Scalability considerations

  • Each specialist remains independently scalable exactly as it was before this phase — Phase 4's Send()-based fan-out, Phase 6's per-session agent construction — since the Supervisor only ever calls into them, never replaces their internals.
  • A conversation that legitimately needs many hops (several analyses, several report regenerations) is still bounded by recursionLimit; if real usage ever needs more hops than feels safe to allow in one blocking request, that's a concrete signal for Phase 7's streaming/async work rather than simply raising the limit.

Security considerations

  • Validate that the requesting user owns the contractId a session is created for, once auth exists — the same open item carried forward from every prior phase, now consolidated at a single front door rather than scattered across four.
  • Because the Supervisor is now the primary way users reach every specialist, this is also the single most important place to eventually add rate limiting and request validation — one well-guarded entry point is easier to secure than four separate ones.

Performance optimizations

  • Specialist nodes do no redundant work — risk_analyzer calls the exact same runAnalysisForContract helper Phase 4's own route calls, so there's no separate, slower "Supervisor version" of any capability to maintain or that could drift in performance characteristics from the direct-call path.
  • chat_assistant's session reuse (create once, reuse on every later chat-routed turn) avoids the cost and memory-loss of starting a fresh chat session on every single routed message.

Maintainability tips

  • Adding a fifth specialist later is: write one node function following the same Command-return shape, add it to the graph, add its name to the routing schema's enum and the supervisor prompt's specialist list. No other node changes — the same "extend the enum, add the array entry" pattern every prior phase has followed for its own extensibility points.
  • Because Phase 4's route now calls the same helper the Supervisor calls, any future improvement to how analysis is triggered (e.g., adding a progress callback) benefits both call paths automatically.

Testing considerations

  • Unit test the supervisor node's routing logic directly against constructed message-history fixtures (no real specialists involved) — assert it routes to risk_analyzer when no analysis exists yet and the user asks to analyze, and to chat_assistant once analysisRunId is set and the user asks a findings question.
  • Integration test a full multi-hop conversation against a fixture contract, asserting the final assistant_sessions-scoped conversation ends with both an analysisRunId and (if requested) a generated_reports row, exactly as if each step had been triggered directly through its own Phase 4/5/6 endpoint.
  • Explicitly test the "omitted key vs. explicit null" behavior from Concept Learning 2.6 — send a second turn that does NOT include contractId and assert the checkpointed value survives; this is exactly the kind of subtle correctness bug that's easy to reintroduce during a future refactor.
  • Re-run Phase 4's, Phase 5's, and Phase 6's own existing tests unchanged after this phase — confirm every specialist's direct-call behavior (and audit endpoints) is completely unaffected by also being reachable through the Supervisor.

#9. Summary

What we learned:

  • What a Supervisor actually does (route) and deliberately does not do (any domain work), and why that separation is what makes a multi-agent system meaningfully different from one larger agent with more tools.
  • Command as the mechanism for a node to bundle its own state update with its own routing decision — finally putting to use the concept Phase 4 introduced and deliberately deferred, once there was a genuinely dynamic, LLM-driven routing decision that needed it.
  • A real, tested answer to "plain-node invocation vs. true subgraph nesting," reached by actually examining what reconciling this phase's specific specialists' state shapes would cost, rather than assuming the answer in the abstract.
  • Why a Supervisor's own state should be federated — carrying references to each specialist's independently durable data, not duplicating that data — and the sharp, easy-to-miss correctness implication that follows from LangGraph's "omitted key means no change" update semantics.
  • How every specialist this project already built (Phase 4's graph, Phase 4.5's agent, Phase 5's workflow, Phase 6's chat agent) turned out to be composable into a larger system with zero changes to their own internals — the direct payoff of each of those phases having been given a clean, well-scoped interface in the first place.

What has been completed:

  • assistant_sessions table and storage layer.
  • A reusable runAnalysisForContract() helper, shared between Phase 4's direct route and this phase's risk_analyzer specialist.
  • SupervisorState, a Command-driven routing node, and four specialist nodes, each invoking an already-built, already-correct component from Phases 4 through 6.
  • A compiled, checkpointed Supervisor graph, and the session-creation/message-send endpoints that make it the single front door to the whole project's capabilities.

What is left:

  • Running the hands-on tasks against real multi-turn, multi-hop conversations and validating routing correctness, session reuse, and precondition-handling (tasks 11-15 — yours to execute).
  • Everything from Phase 7 onward, which builds on a Supervisor that currently returns one blocking response per request even when several specialists ran in sequence to produce it.

How this connects to the next phase: Phase 7 (streaming/async progress) is the natural next step precisely because this phase can now produce genuinely multi-hop responses — "analyze, then report" might take real, noticeable time, all inside one blocking HTTP request with nothing shown to the user until it's completely finished. Phase 7 builds on this phase's graph structure directly: LangGraph's .stream() interface (the streaming counterpart to the .invoke() calls used throughout this phase) can emit an update after each node completes — meaning a client could see "analysis complete, 3 HIGH risk clauses found" the moment risk_analyzer hands back to supervisor, well before report_generator finishes. Nothing about this phase's graph structure needs to change for that; only how the client receives updates from it does.
