LegalDocs — Phase 4.5: Legal Research Agent

#LegalDocs — Phase 4.5: Legal Research Agent

Tech stack: Node.js / TypeScript, OpenAI, PostgreSQL (Phase 0/1), LangChain.js (langchain/agents), LangGraph.js prebuilt agents (createReactAgent, ToolNode), Zod Depends on: Phase 1 (legal_reference_chunks — one of the tools this agent can call), Phase 4 (the judgment_lookup stub node this phase replaces, and the graph/checkpointer/audit infrastructure it slots back into unchanged) Feeds into: Phase 5 (the Report Generator surfaces this agent's case-law references and human-review flags in the final report); Phase 6.5 (this agent becomes one of the specialist sub-agents under the Supervisor)

#1. Phase Overview

#Objective

Replace Phase 4's judgment_lookup stub — which always ran the exact same fixed similarity search for every HIGH-risk clause — with a real agent: a component that, given a HIGH-risk clause and the reasoning behind its risk score, decides for itself what to do next. Sometimes that means searching Indian Kanoon for a relevant judgment. Sometimes it means a broader search for general case precedent. Sometimes — when nothing turns up sufficient grounding — it means honestly flagging the clause for human review rather than fabricating a citation just to fill the field.

You'll build this twice, deliberately: first as a hand-rolled LangChain AgentExecutor running a raw ReAct loop (so you see exactly what "an agent" mechanically is, one reasoning step at a time), and then again using LangGraph's createReactAgent() prebuilt (so you can directly compare how much of that mechanism a well-designed abstraction can absorb for you). The second version is what actually plugs into Phase 4's graph.

#Why this phase is important

Every LLM-powered step you've built so far — classification, extraction, risk scoring — has followed a fixed procedure: given this specific input, always run these specific steps, in this specific order. That's appropriate for those tasks, because the right thing to do didn't depend on what was discovered partway through. Legal research is different. Whether Indian Kanoon has something relevant, whether a general precedent search is more fruitful, or whether nothing usable exists at all — none of that is knowable in advance. A fixed node (Phase 4's stub) can only ever do the one thing it was written to do; an agentic node can look at what it just found, decide that it isn't good enough, and try something else — or decide, honestly, that nothing is good enough and say so.

This is also the point where "agents and tools," one of the more talked-about and most-often-misunderstood ideas in modern LLM application development, gets a real, grounded home in your project rather than an abstract definition. Building the same behavior twice — once by hand, once with a framework abstraction — is what turns "an agent is an LLM that can call tools in a loop" from a phrase you've heard into something you've actually watched happen, one iteration at a time.

#2. Concept Learning

#2.1 What actually makes something "an agent," as opposed to a chain or a fixed graph node

Every LLM call you've written so far has had the model produce one final answer to one specific question, in one shot (withStructuredOutput() calls in Phases 2, 3, and 4 are all this shape — even when wrapped in a bigger pipeline, each individual LLM call is "here's your input, give me your one output"). An agent is structurally different: the model is given a goal and a set of tools it's allowed to use, and it is put in a loop where, on each turn, it can either (a) call one of its tools to gather more information, or (b) decide it has enough and produce a final answer. The defining feature isn't that it uses tools — Phase 4.5's judgment_lookup stub also technically "used a tool" (a fixed database query) — it's that the model itself decides, at runtime, whether and which tool to use, and can change its mind based on what a previous tool call returned.

Concretely, contrast the stub from Phase 4 with what this phase builds:

Phase 4's stub This phase's agent
What it does for every HIGH-risk clause Exactly the same fixed similarity search, always Reasons about the specific clause and risk reasoning, then decides
Can it try something else if the first attempt comes up empty? No — one query, whatever it returns is what you get Yes — if search_indian_kanoon returns nothing useful, it can try search_case_precedent next
What happens when nothing relevant exists? Silently returns whatever the query matched best, however weak Can call flag_for_human_review() and say so honestly

#2.2 Tools — giving a model a menu of actions, not just a menu of words

A tool is a function the model can choose to invoke, described to the model via a name, a natural-language description of what it does, and a schema (built with Zod, exactly like your structured-output schemas from earlier phases) describing what arguments it takes. The model never actually executes the function itself — it only ever outputs a request to call a specific tool with specific arguments; your code is the one that actually runs the function and feeds the result back.

ts
const searchIndianKanoon = tool(
  async ({ query }: { query: string }) => {
    // ... your code, not the model's, actually performs this call
    return await callIndianKanoonApi(query);
  },
  {
    name: "search_indian_kanoon",
    description: "Search Indian Kanoon for judgments relevant to a specific legal question or clause concern.",
    schema: z.object({ query: z.string().describe("The legal question or concept to search for") }),
  }
);

This is the exact same underlying mechanism as withStructuredOutput() from Phase 2 (constrained, schema-validated output via the model provider's function-calling API) — the difference is purely how you use the mechanism. withStructuredOutput() uses it to force one final answer into one fixed shape. A tool uses the identical mechanism to let the model request an action, mid-reasoning, whose result it will then see and react to.

This phase gives the agent three tools:

  • search_indian_kanoon(query) — searches India's primary free case-law/statute repository for judgments relevant to the clause's legal concern.
  • search_case_precedent(query) — a broader search (general legal databases/web) for relevant precedent when Indian Kanoon specifically doesn't surface anything useful, or when the concern is more general/commercial than statute-specific.
  • flag_for_human_review() — takes no meaningful arguments; its entire purpose is to give the agent an honest way out when it genuinely cannot find sufficient grounding, rather than forcing it to fabricate a citation to satisfy the output schema.

Why flag_for_human_review matters as much as the search tools: an agent that only ever has "search tools" available will, under pressure to produce some answer, sometimes cite something tenuous rather than admit it found nothing solid. Giving it an explicit, named, legitimate way to say "I don't have good grounding for this" is a deliberate design choice that reduces exactly that failure mode — it's cheaper and more honest to make "I don't know" a first-class action than to hope the model volunteers that admission in free text.

#2.3 The ReAct pattern — Reason, Act, Observe, repeat

ReAct (Reasoning + Acting) is the specific loop structure this kind of agent runs:

  1. Reason: the model looks at the goal and everything so far, and thinks about what to do next (internally, or as an explicit "thought" depending on prompting style).
  2. Act: it either calls a tool, or decides it's done and produces a final answer.
  3. Observe: if it called a tool, your code executes it and appends the result back into the conversation the model sees.
  4. Repeat from step 1, now with that observation available — until the model produces a final answer instead of another tool call, or a safety limit (maxIterations) is hit.
text
Turn 1 — Reason: "This clause concerns indemnification caps under Indian contract law. Let me check Indian Kanoon first."
         Act: search_indian_kanoon("indemnification cap enforceability India")
         Observe: [3 results, none closely on point]

Turn 2 — Reason: "Indian Kanoon didn't surface anything precise. Let me broaden the search."
         Act: search_case_precedent("liability cap enforceability commercial contracts")
         Observe: [1 strong match found]

Turn 3 — Reason: "This precedent is directly relevant and sufficient."
         Act: (final answer — no further tool call)

Why a loop, and why a hard iteration limit matters: nothing in principle stops a model from deciding to call five tools in a row, or from getting into a pattern where it keeps re-trying variations of the same unproductive search. maxIterations (or LangGraph's equivalent recursion limit) is not an edge-case safety net — it is a required, load-bearing setting on every agent you ever build, because an agent loop is the one place in this project where an LLM's own runtime decisions, not your code, control how many LLM calls happen.

#2.4 AgentExecutor — the hand-rolled version, and what it's actually doing for you

AgentExecutor is LangChain's implementation of exactly the loop in 2.3. You give it: a model bound to your tools, a prompt (including a placeholder for "everything that's happened in this loop so far" — called the agent scratchpad), and the tool implementations themselves. It then mechanically runs the reason → act → observe cycle, calling your tool functions for you and feeding results back into the prompt, until the model stops requesting tools.

ts
const agent = createToolCallingAgent({ llm: model, tools, prompt });
const executor = new AgentExecutor({ agent, tools, maxIterations: 5, returnIntermediateSteps: true });

const result = await executor.invoke({ input: "..." });
// result.output          -> the final answer
// result.intermediateSteps -> every (tool call, tool result) pair along the way

Why build this by hand at all, if a prebuilt exists? returnIntermediateSteps: true gives you direct, explicit visibility into every single reasoning/action/observation cycle — you can literally print out what the model tried, in what order, and what each attempt returned. That raw visibility is genuinely valuable the first time you build an agent: it turns "the model decided to do something" from a black box into something you can watch happen step by step, which makes the LangGraph version in 2.5 legible as "the same mechanism, wrapped," rather than unfamiliar magic.

#2.5 createReactAgent() and ToolNode — the same loop, as a compiled graph

LangGraph's createReactAgent() builds the identical ReAct loop, but expresses it as a small, prebuilt StateGraph rather than a bespoke executor class:

text
        START
          │
          ▼
    ┌──────────┐   has the model requested a tool call?
    │  agent    │──────────── no ──────────────────────► END (final answer)
    │ (call LLM)│
    └─────┬────┘
          │ yes
          ▼
    ┌──────────┐
    │  tools    │  (a ToolNode — executes whichever tool(s)
    │ (ToolNode)│   the model requested, in parallel if there are several)
    └─────┬────┘
          │
          └──────────────► back to "agent" (loop)

A ToolNode is a prebuilt LangGraph node that does one job: look at the most recent AI message for any tool-call requests, execute the corresponding tool function(s) — concurrently if the model requested more than one in the same turn — and append the results back into the conversation's message list as tool-result messages. It's the graph-native equivalent of the "observe" step AgentExecutor handles internally for you.

The conditional edge after agent ("has the model requested a tool call, or not?") is the exact same kind of routing function as Phase 4's routeByRisk — inspect the current state (here, the latest message), return the name of the next node.

ts
const reactAgent = createReactAgent({
  llm: model,
  tools,
  messageModifier: systemPrompt, // sets the agent's system-level instructions
});

const result = await reactAgent.invoke({ messages: [new HumanMessage(input)] });

What you get for the same underlying behavior: far less code (the loop, the conditional routing, and the tool-execution step are all already built for you), and — because it's a compiled StateGraph — it composes the same way every other graph in this project does: it can be checkpointed, traced, and (as Phase 6.5 will do) embedded as a subgraph inside a larger graph. AgentExecutor is a self-contained class with its own internal loop; createReactAgent()'s output is a graph, which is why it's the version that actually plugs into Phase 4's larger graph rather than sitting beside it as a separate mechanism.

#2.6 Parsing an agent's run into a structured outcome

An agent's raw output is a list of messages — the back-and-forth of reasoning, tool calls, and tool results, ending in a final AI message. That's perfect for a human reading a trace, but Phase 4's judgment_lookup node needs to return a clean, typed update to currentFinding (specifically, additional legalReferences entries, or a review-flagged marker) — not a raw transcript.

The practical approach: after the agent loop completes, run one more small, fixed, structured-output call (exactly like every prior phase's pattern) whose only job is to read the full message transcript and distill it into a typed AgentOutcome. This keeps the agentic part of the pipeline (genuinely dynamic, tool-using, multi-turn) cleanly separated from the structured-output part (fixed, one-shot, schema-guaranteed) — each piece doing the job it's actually good at.

#3. Different Approaches

#3.1 AgentExecutor vs. createReactAgent() — build one, or both?

Approach Pros Cons When to use
A. AgentExecutor only Full manual visibility into every reasoning step via intermediateSteps; no new LangGraph-specific concepts beyond what Phase 3/4 already taught Doesn't compose with Phase 4's graph as a first-class node the way a compiled graph does; you'd need extra glue code to embed it Good for a standalone research tool outside the graph, or pure learning/debugging
B. createReactAgent() only Composes directly with the rest of the project's LangGraph-based architecture; less code Skips seeing the raw reasoning loop mechanics explicitly, which makes the abstraction feel more like magic Fine if you're already comfortable with the ReAct loop's mechanics from other experience
C. Build both, in sequence (roadmap's choice) You see the mechanism explicitly once (AgentExecutor), then see how much of it a good abstraction absorbs (createReactAgent) — genuinely useful for building calibrated trust in the abstraction; the final code you actually ship is still just the LangGraph version More total code written, though the second version is considerably shorter once the first is understood Recommended — matches your roadmap exactly, and this is a good phase to spend the extra time on since "agents" is a concept worth deeply understanding rather than pattern-matching

Recommendation: Approach C. Section 6 below builds both in full; only the createReactAgent() version is what ends up wired into Phase 4's graph.

#3.2 Tool implementation — real external APIs vs. a mockable interface

Approach Pros Cons When to use
A. Direct hardcoded API calls Simplest code Impossible to unit test the agent's decision-making without hitting real external APIs (slow, costly, non-deterministic responses make assertions brittle) Fine for a quick prototype only
B. A small interface/adapter per tool, with a real implementation and an injectable mock (recommended) Unit tests can supply fixed, deterministic tool responses and assert the agent's routing behavior (e.g. "when Indian Kanoon returns nothing, does it try search_case_precedent next?") without any network calls A small amount of extra structure (one interface per tool) Recommended — agent behavior testing (3.4 in Best Practices) genuinely depends on this

Recommendation: Approach B. Each tool's core logic lives in a plain, injectable async function; the tool() wrapper around it is a thin adapter, not where the logic lives.

#4. Detailed Implementation Plan

#Step 1 — Tool logic, decoupled from the tool() wrapper

What: Three plain async functions (searchIndianKanoon, searchCasePrecedent, flagForHumanReview) containing the actual logic, each then wrapped with LangChain's tool() helper separately.

Why: Keeps the core logic unit-testable and swappable (per 3.2) without needing to construct a full agent just to test one tool's behavior.

Best practice: Every external-API-calling tool should catch its own errors and return a string describing the failure rather than throwing — an agent can reason about "the search failed, let me try a different tool," but it can't recover from an uncaught exception that crashes the whole loop.

Common mistake to avoid: Sending the clause's full, verbatim text (which may include real party names, specific financial terms, etc.) directly to a third-party search API. Extract just the legal concept being searched for (which the model itself formulates as the tool's query argument) rather than passing raw contract content externally.

#Step 2 — The hand-rolled AgentExecutor

What: A createToolCallingAgent() + AgentExecutor combination, with a prompt describing the agent's goal (find grounding for a HIGH-risk clause, or honestly flag if none exists) and a MessagesPlaceholder("agent_scratchpad").

Why/How: Covered in Concept Learning 2.4.

Best practice: Set maxIterations explicitly (this phase uses 5) and returnIntermediateSteps: true — the latter costs nothing and is exactly what you need to inspect the raw reasoning trace while validating this version.

#Step 3 — The LangGraph createReactAgent() version

What: The same three tools, the same system prompt, wrapped with createReactAgent() instead.

Why/How: Covered in Concept Learning 2.5.

Common mistake to avoid: Forgetting to set a recursion/iteration limit here too — createReactAgent()'s underlying graph needs the same safety net AgentExecutor's maxIterations provided; LangGraph exposes this via the recursionLimit option on .invoke().

#Step 4 — Comparing the two on a real example

What: Run the exact same HIGH-risk clause + risk reasoning through both versions and compare: total code needed to set each up, and the shape/readability of each one's trace of what happened.

Why: This comparison is the actual point of building both — a concrete, first-hand answer to "how much does this abstraction save me, and what do I give up."

#Step 5 — Structured outcome parsing

What: AgentOutcomeSchema (Zod) and a small fixed chain that reads the agent's final message transcript and distills it into { references: [...], reviewFlagged: boolean }.

Why/How: Covered in Concept Learning 2.6 — this is the bridge back into Phase 4's typed RiskFindingRecord shape.

#Step 6 — Replacing Phase 4's judgment_lookup stub

What: A new implementation of the judgment_lookup node (same name, same position in the graph, same input/output shape as Phase 4's stub) that internally invokes the createReactAgent() version and applies its parsed outcome to currentFinding.

Why: This is the entire point of the phase — a clean, drop-in replacement, per the design decision made back in Phase 4 (Approach recommendation in section 3.3 there) to give this node its own clearly-scoped identity specifically so this swap wouldn't require touching the graph's structure at all.

Common mistake to avoid: Changing the node's return shape. Phase 4's store_finding node and the aggregate_findings join both depend on currentFinding having the same shape it always has — this phase adds content to legalReferences (or a new reviewFlagged field), it must not change the field names or types Phase 4 already established.

#6.2 Tool logic and definitions

ts
// src/research/tools.ts
import { tool } from "@langchain/core/tools";
import { z } from "zod";

export interface ToolSearchResult {
  source: string;
  excerpt: string;
  url?: string;
}

/**
 * Core logic kept separate from the tool() wrapper (Approach B, section 3.2)
 * so it can be unit tested or mocked without constructing a full agent.
 * Catches its own errors and returns a descriptive string rather than
 * throwing - an agent can reason about a failed search and try a different
 * tool; it cannot recover from an uncaught exception mid-loop.
 */
export async function searchIndianKanoonImpl(query: string): Promise<ToolSearchResult[] | string> {
  const apiKey = process.env.INDIAN_KANOON_API_KEY;
  if (!apiKey) return "Indian Kanoon API is not configured in this environment.";

  try {
    const response = await fetch(process.env.INDIAN_KANOON_API_URL!, {
      method: "POST",
      headers: { Authorization: `Bearer ${apiKey}`, "Content-Type": "application/json" },
      body: JSON.stringify({ query }),
    });

    if (!response.ok) {
      return `Indian Kanoon search failed with status ${response.status}.`;
    }

    const data = await response.json();
    return (data.results ?? []).slice(0, 3).map((r: any) => ({
      source: r.title,
      excerpt: r.snippet,
      url: r.url,
    }));
  } catch (err) {
    return `Indian Kanoon search failed: ${(err as Error).message}`;
  }
}

export async function searchCasePrecedentImpl(query: string): Promise<ToolSearchResult[] | string> {
  const apiKey = process.env.CASE_PRECEDENT_SEARCH_API_KEY;
  if (!apiKey) return "Case precedent search API is not configured in this environment.";

  try {
    const response = await fetch(process.env.CASE_PRECEDENT_SEARCH_API_URL!, {
      method: "POST",
      headers: { Authorization: `Bearer ${apiKey}`, "Content-Type": "application/json" },
      body: JSON.stringify({ query }),
    });

    if (!response.ok) {
      return `Case precedent search failed with status ${response.status}.`;
    }

    const data = await response.json();
    return (data.results ?? []).slice(0, 3).map((r: any) => ({
      source: r.title,
      excerpt: r.snippet,
      url: r.url,
    }));
  } catch (err) {
    return `Case precedent search failed: ${(err as Error).message}`;
  }
}

/**
 * Deliberately takes no meaningful arguments - its entire purpose is to
 * give the agent a legitimate, named way to say "I don't have sufficient
 * grounding for this" rather than fabricating a citation under pressure to
 * produce SOME answer.
 */
export function flagForHumanReviewImpl(reason: string): string {
  return `Flagged for human review: ${reason}`;
}

export const searchIndianKanoonTool = tool(
  async ({ query }: { query: string }) => JSON.stringify(await searchIndianKanoonImpl(query)),
  {
    name: "search_indian_kanoon",
    description:
      "Search Indian Kanoon for judgments or statutes relevant to a specific legal question or " +
      "contract clause concern. Use a focused legal query, not the raw clause text.",
    schema: z.object({
      query: z.string().describe("A focused legal question or concept, e.g. 'enforceability of liquidated damages clauses India'"),
    }),
  }
);

export const searchCasePrecedentTool = tool(
  async ({ query }: { query: string }) => JSON.stringify(await searchCasePrecedentImpl(query)),
  {
    name: "search_case_precedent",
    description:
      "Search general legal databases for broader case precedent when Indian Kanoon specifically " +
      "does not surface sufficient grounding, or when the concern is more general/commercial in nature.",
    schema: z.object({
      query: z.string().describe("A focused legal question or concept to search for precedent on"),
    }),
  }
);

export const flagForHumanReviewTool = tool(
  async ({ reason }: { reason: string }) => flagForHumanReviewImpl(reason),
  {
    name: "flag_for_human_review",
    description:
      "Call this when neither search tool has produced sufficient, relevant grounding for this " +
      "clause's risk assessment. It is better to honestly flag for human review than to cite " +
      "something tenuous or irrelevant.",
    schema: z.object({
      reason: z.string().describe("A brief explanation of why sufficient grounding could not be found"),
    }),
  }
);

export const legalResearchTools = [searchIndianKanoonTool, searchCasePrecedentTool, flagForHumanReviewTool];

#6.3 Hand-rolled AgentExecutor version

ts
// src/research/agentExecutorVersion.ts
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate, MessagesPlaceholder } from "@langchain/core/prompts";
import { AgentExecutor, createToolCallingAgent } from "langchain/agents";
import { legalResearchTools } from "./tools";

const model = new ChatOpenAI({
  model: process.env.RESEARCH_AGENT_MODEL ?? "gpt-4o-mini",
  temperature: 0,
});

const prompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    "You are a legal research assistant. You are given a contract clause that " +
      "has been assessed as HIGH risk, along with the reasoning for that " +
      "assessment. Your job is to find supporting case law or statutory " +
      "grounding for this assessment. Try search_indian_kanoon first for " +
      "India-specific grounding. If that doesn't surface anything sufficiently " +
      "relevant, try search_case_precedent for broader precedent. If neither " +
      "produces sufficient grounding, call flag_for_human_review rather than " +
      "citing something tenuous or irrelevant. Once you have what you need (or " +
      "have determined nothing sufficient exists), give a final summary of what " +
      "you found or why you're flagging this for review.",
  ],
  ["human", "{input}"],
  new MessagesPlaceholder("agent_scratchpad"),
]);

const agent = createToolCallingAgent({ llm: model, tools: legalResearchTools, prompt });

/**
 * returnIntermediateSteps: true gives direct visibility into every
 * (tool call, tool result) pair the loop produced - exactly what makes this
 * version valuable to build even though createReactAgent() is what actually
 * ships: it turns "the model decided to do something" into something you
 * can print and read step by step.
 */
export const legalResearchExecutor = new AgentExecutor({
  agent,
  tools: legalResearchTools,
  maxIterations: Number(process.env.RESEARCH_AGENT_MAX_ITERATIONS ?? 5),
  returnIntermediateSteps: true,
});

export async function runResearchAgentExecutor(clauseText: string, riskReasoning: string) {
  return legalResearchExecutor.invoke({
    input: `Clause:\n${clauseText}\n\nRisk assessment reasoning:\n${riskReasoning}`,
  });
}

#6.4 createReactAgent() version

ts
// src/research/reactAgentVersion.ts
import { ChatOpenAI } from "@langchain/openai";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { HumanMessage } from "@langchain/core/messages";
import { legalResearchTools } from "./tools";

const model = new ChatOpenAI({
  model: process.env.RESEARCH_AGENT_MODEL ?? "gpt-4o-mini",
  temperature: 0,
});

const SYSTEM_PROMPT =
  "You are a legal research assistant. You are given a contract clause that " +
  "has been assessed as HIGH risk, along with the reasoning for that " +
  "assessment. Try search_indian_kanoon first for India-specific grounding. " +
  "If that doesn't surface anything sufficiently relevant, try " +
  "search_case_precedent for broader precedent. If neither produces " +
  "sufficient grounding, call flag_for_human_review rather than citing " +
  "something tenuous or irrelevant.";

/**
 * The identical ReAct loop as the AgentExecutor version above, expressed as
 * a compiled StateGraph instead: an "agent" node (calls the model), a
 * ToolNode (executes whichever tools the model requested, built in), and a
 * conditional edge routing back to "agent" until the model stops requesting
 * tools. Far less code for the same underlying behavior - and because it's
 * a compiled graph, it composes with the rest of this project's LangGraph
 * architecture (traceable, embeddable as a subgraph later) the way
 * AgentExecutor's standalone class does not.
 */
export const legalResearchReactAgent = createReactAgent({
  llm: model,
  tools: legalResearchTools,
  messageModifier: SYSTEM_PROMPT,
});

export async function runResearchReactAgent(clauseText: string, riskReasoning: string) {
  return legalResearchReactAgent.invoke(
    {
      messages: [
        new HumanMessage(`Clause:\n${clauseText}\n\nRisk assessment reasoning:\n${riskReasoning}`),
      ],
    },
    { recursionLimit: Number(process.env.RESEARCH_AGENT_MAX_ITERATIONS ?? 5) * 2 }
    // *2 because each ReAct "turn" is 2 graph steps (agent, then tools) -
    // the recursion limit counts individual node executions, not full turns.
  );
}

#6.5 Structured outcome parsing

ts
// src/research/outcomeParsing.ts
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { z } from "zod";
import type { BaseMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: process.env.RESEARCH_AGENT_MODEL ?? "gpt-4o-mini",
  temperature: 0,
});

export const AgentOutcomeSchema = z.object({
  references: z
    .array(
      z.object({
        sourceDocument: z.string(),
        excerpt: z.string(),
      })
    )
    .describe("Legal references the agent found sufficient grounding in, if any"),
  reviewFlagged: z.boolean().describe("True if the agent could not find sufficient grounding and flagged for human review"),
  reviewReason: z.string().nullable().describe("If reviewFlagged is true, the reason given"),
});
export type AgentOutcome = z.infer<typeof AgentOutcomeSchema>;

const distillPrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    "Read the following legal research agent transcript (its reasoning, tool " +
      "calls, and tool results) and distill it into a structured outcome: any " +
      "legal references it found sufficient, and whether it flagged the clause " +
      "for human review instead.",
  ],
  ["human", "Transcript:\n{transcript}"],
]);

const distillChain = distillPrompt.pipe(
  model.withStructuredOutput(AgentOutcomeSchema, { name: "distill_agent_outcome" })
);

function transcriptFromMessages(messages: BaseMessage[]): string {
  return messages
    .map((m) => `[${m.getType()}] ${typeof m.content === "string" ? m.content : JSON.stringify(m.content)}`)
    .join("\n\n");
}

export async function distillAgentOutcome(messages: BaseMessage[]): Promise<AgentOutcome> {
  return distillChain.invoke({ transcript: transcriptFromMessages(messages) });
}

/**
 * Replaces Phase 4's judgment_lookup stub. Same node name, same position in
 * the graph's edges, same input/output shape (a partial update to
 * currentFinding) - this file is the ONLY thing that changed. Phase 4's
 * graph.ts import is updated to point here instead of the old stub file;
 * nothing about the graph's structure, edges, or checkpointer changes.
 */
export async function judgmentLookupNode(
  state: AnalysisStateType
): Promise<Partial<AnalysisStateType>> {
  const finding = state.currentFinding!;

  const result = await runResearchReactAgent(state.currentClause!.fullText, finding.reasoning);
  const outcome = await distillAgentOutcome(result.messages);

  const updatedFinding = {
    ...finding,
    legalReferences: [...finding.legalReferences, ...outcome.references],
    reviewFlagged: outcome.reviewFlagged,
    reviewReason: outcome.reviewReason,
  };

  return { currentFinding: updatedFinding };
}

#Why this design over alternatives

  • Two implementations, one shipped: building AgentExecutor first and discarding it in favor of createReactAgent() isn't wasted work — it's what makes the second version's brevity legible as "the same mechanism, abstracted," rather than an opaque shortcut you'd have to trust blindly.
  • A distillation call at the boundary, not agentic reasoning threaded through Phase 4's typed state: keeping the free-form, multi-turn agent transcript entirely inside research/ and only crossing back into analysis/'s strictly-typed world via one small structured-output call keeps Phase 4's state schema exactly as clean as it was before this phase — the agent is an implementation detail of one node, not a new kind of thing the rest of the graph needs to understand.
  • A one-node swap, not a graph rewrite: this is the direct payoff of Phase 4's design decision to give the stub its own clearly-scoped node in the first place — the entire upgrade in this phase touches exactly one file's internals plus one import line.

#8. Best Practices

Industry best practices

  • Always set an explicit iteration/recursion limit on any agent loop — this is not optional hardening, it's a required setting given that the model's own decisions, not your code, control how many LLM calls a single request can trigger.
  • Give an agent an explicit, legitimate "I don't know" action (flag_for_human_review) whenever fabricating an answer would be worse than admitting uncertainty — this is a general pattern worth carrying into any future agent you build, not specific to this phase.
  • Keep tool logic and tool wrappers separate (6.2) — it pays for itself the first time you need to test agent decision-making without live external API calls.

Scalability considerations

  • Each HIGH-risk clause triggers its own independent agent loop; since Phase 4's score_clause → judgment_lookup routing already runs per-clause in parallel via Send(), multiple agents can be running concurrently for a single contract with several HIGH-risk clauses — factor this into the same OpenAI rate-limit planning noted in Phase 4's best practices, since an agent loop can itself issue several LLM calls per clause, not just one.

Security considerations

  • Never pass full, verbatim clause text (which may contain real party names or commercial terms) directly into external search tool queries — the model formulates a focused legal concept as the tool argument, which is what actually gets sent externally; review your prompt to ensure it isn't implicitly encouraging the model to paste raw clause text into a query.
  • Treat external search API keys with the same care as your OpenAI key — never log them, and rotate them if a tool's error messages are ever exposed to end users (they should not be).

Performance optimizations

  • The flag_for_human_review tool is essentially free (no external call) — an agent that reaches for it quickly when appropriate is cheaper, not just more honest, than one that exhausts both search tools first. Nothing needs to be added to encourage this beyond the prompt's existing guidance; it falls out naturally once the tool exists.
  • Cache identical tool queries within a single agent run if you notice the model re-issuing a very similar query to the same tool twice — a simple in-memory Map keyed by (toolName, query) scoped to one judgment_lookup invocation is enough; no need for persistent caching across runs given how varied real clause text is.

Maintainability tips

  • Keep the system prompt (used identically by both agent versions in section 6) in one shared constant rather than duplicating it — you built both implementations to compare mechanics, not to maintain two different sets of instructions going forward.
  • Adding a fourth tool later (e.g. a dedicated compliance-database lookup) is: write its logic function, wrap it with tool(), add it to the legalResearchTools array. No other file needs to change — same "adding one array entry" pattern as Phase 2's RunnableBranch and Phase 3's CLAUSE_TYPES.

Testing considerations

  • Unit test each tool's logic function directly (not through the agent) with a mocked fetch, asserting both the success path and the "API not configured / request failed" string-returning path.
  • Integration test the full createReactAgent() loop against constructed scenarios using mocked tool responses (per 3.2's Approach B): assert that when search_indian_kanoon is mocked to return nothing useful, the agent's next tool call is search_case_precedent, not a repeat of the same query; assert that when both searches are mocked empty, the agent calls flag_for_human_review rather than fabricating a reference.
  • Test distillAgentOutcome against a few hand-constructed message transcripts (no live agent run needed) to confirm it reliably extracts reviewFlagged: true from a transcript ending in a flag_for_human_review tool call.
  • Re-run Phase 4's existing integration tests (task 11/12 from Phase 4's hands-on tasks) after wiring this node in, to confirm the surrounding graph's behavior — checkpointing, the audit endpoints, the aggregate_findings join — is genuinely unaffected by this internal swap.

#9. Summary

What we learned:

  • What structurally distinguishes an agent from a chain or a fixed graph node: the model itself deciding, at runtime and based on what it's already observed, whether and which tool to use next — not merely "an LLM call that happens to use a tool."
  • Tools as schema-described functions the model can request but never directly executes, and why giving an agent an explicit "flag for human review" action is a deliberate design choice that reduces fabrication under pressure to produce an answer.
  • The ReAct loop (reason → act → observe → repeat) as the concrete mechanism underlying "an agent," and why an iteration/recursion limit is a required safety setting, not optional polish.
  • AgentExecutor and createReactAgent() as two implementations of the identical loop — one a self-contained class with full manual visibility into every step, the other a compiled StateGraph that composes with the rest of this project's LangGraph architecture — and why the second is what actually ships.
  • The value of keeping an agent's free-form, multi-turn reasoning entirely contained within its own module, crossing back into the rest of the system's strictly-typed world through exactly one small, fixed distillation call.

What has been completed:

  • Three tools (search_indian_kanoon, search_case_precedent, flag_for_human_review) with decoupled, individually testable logic.
  • A hand-rolled AgentExecutor implementation, built and validated for its raw reasoning-trace visibility.
  • A createReactAgent() implementation of the identical behavior, validated against the same test scenarios.
  • Structured outcome distillation (AgentOutcomeSchema + distillAgentOutcome) bridging the agent's message transcript back into Phase 4's typed finding shape.
  • Phase 4's judgment_lookup node fully replaced, in place, with zero changes to the surrounding graph's structure, edges, or checkpointing behavior.

What is left:

  • Running the hands-on tasks against real (or realistically mocked) external search APIs and validating the agent's tool-selection behavior across genuinely varied HIGH-risk clause scenarios (tasks 5, 7, 11-12 — yours to execute).
  • Everything from Phase 5 onward, which consumes this phase's enriched risk_findings (including the new reviewFlagged signal) alongside everything Phase 4 already produced.

How this connects to the next phase: Phase 5 (Report Generator) reads the completed risk_findings table — including, now, this phase's agent-sourced legal references and any reviewFlagged clauses — and turns it into a polished DOCX risk report. A clause flagged for human review by this phase's agent should surface distinctly in that report (e.g. "grounding could not be confirmed — recommend manual legal review") rather than being presented with the same confidence as a fully-grounded finding; keep that distinction in mind as a requirement once you get to Phase 5's report-section design. Structurally, Phase 5 also introduces LangGraph's Functional API (entrypoint/task) as a second, contrasting way to build the same kind of pipeline you've now built twice already (as an AgentExecutor vs. createReactAgent() here; as a StateGraph vs. Functional API there) — continuing this phase's theme of deliberately comparing two ways to build the same thing before settling on one.
