LegalDocs — Phase 5: Report Generator

#LegalDocs — Phase 5: Report Generator

Tech stack: Node.js / TypeScript, OpenAI, PostgreSQL (Phase 0/1), LangGraph.js Functional API (entrypoint, task) — first use in the project, docx (npm), Zod Depends on: Phase 3 (clauses — titles/text surfaced per finding), Phase 4 (risk_findings, the thread_id/analysis_run_id, the checkpointer's completed run), Phase 4.5 (reviewFlagged / reviewReason and agent-sourced legalReferences on each finding) Feeds into: Phase 6 (whatever consumes a completed, downloadable report as a deliverable — a dashboard, a notification, a client-facing share link); Phase 6.5 (the Supervisor's natural terminal action, once an analysis run completes, is to trigger this phase's workflow)

#1. Phase Overview

#Objective

Turn a completed analysis run's risk_findings rows — enriched by both Phase 4's scoring and Phase 4.5's legal grounding — into a single, polished, professional DOCX risk report: a cover page identifying the contract, an executive summary written in plain language, one detailed section per clause finding grouped by risk level, a visually distinct treatment for any clause Phase 4.5 flagged for human review, and an appendix of the specific legal references cited. The output is a real file, stored durably and available for download, not another JSON payload.

This phase also introduces a second way to orchestrate a multi-step pipeline: LangGraph's Functional API (entrypoint / task), used here deliberately in contrast to Phase 3 and 4's StateGraph. Where Phase 4's problem genuinely needed a graph (dynamic fan-out, conditional branching, checkpointed audit history across many parallel branches), this phase's problem is a straight line — fetch data, write a summary, render a document, store it — and building a full StateGraph for a straight line would be paying graph-shaped overhead for a shape that never branches. The Functional API is LangGraph's answer to exactly that mismatch: durable, traceable, individually-retryable steps, expressed as plain async/await code instead of nodes and edges.

#Why this phase is important

Every phase before this one produced data that lived in your database and your API responses — useful to a frontend, invisible to anyone else. This is the first phase whose output is something a person outside your system can actually hold: a document they can email to a client, attach to a compliance file, or print for a signature meeting. That changes the bar. A JSON array of risk findings can afford to look a little rough around the edges; a document with your product's name on it, handed to a lawyer or a business owner, cannot.

It's also the first phase that has to make an editorial decision your data alone doesn't make for you: how much confidence to project. Phase 4.5 gave you a reviewFlagged signal specifically so a clause with no solid legal grounding could be flagged rather than silently cited as if it were fully justified. If this phase's report presents a reviewFlagged clause with the same visual and rhetorical confidence as a fully-grounded HIGH-risk finding, all of Phase 4.5's honesty work is wasted the moment it reaches a reader. Getting that distinction right in the actual rendered document — not just in the underlying data — is the real point of this phase.

#2. Concept Learning

#2.1 Why this phase does not need another StateGraph

Look at what this pipeline actually needs to do: load one analysis run's data, write one summary of it, render one document from it, save that document once. There is no dynamic list to fan out over (Phase 4's Send() had one), no point where the next step depends on a runtime decision (Phase 4's routeByRisk had one), and no need to converge multiple parallel branches back into a shared list (Phase 4's aggregate_findings had one). It is, structurally, four steps in a straight line.

You could still build a StateGraph for this — an Annotation.Root state, four nodes, three fixed edges, START and END. It would work. But every one of those pieces (the reducer question for every field, the edge-wiring, the separate node files) is machinery that exists specifically to support branching, fan-out, and multi-path convergence — none of which this pipeline has. Reaching for StateGraph here would be using the tool that solves "a graph of possible paths" on a problem that only ever has one path.

The Functional API is LangGraph's deliberate alternative for exactly this shape: a linear (or simply-structured) pipeline that still wants LangGraph's operational benefits — tracing, durable checkpointing, per-step retry — without the ceremony of declaring state, nodes, and edges for something that was never going to branch.

#2.2 task — a plain async function that becomes individually traceable and retryable

task() wraps an ordinary async function. The wrapped version behaves identically from a call-site perspective (await someTask(args) still just runs the function and returns its result) — what changes is what happens around the call: LangSmith records it as a distinct traced step (so a report-generation trace shows "fetch data → summarize → render → store" as clearly as Phase 4's node-by-node trace does), and — when the surrounding entrypoint has a checkpointer attached — the task's result is durably recorded against its inputs, so a replayed run can skip re-executing a task whose inputs haven't changed rather than blindly re-running everything from the top.

ts
const fetchReportData = task("fetch_report_data", async (analysisRunId: string) => {
  // ... plain async work, e.g. a database query
});

Nothing here is LLM-specific. A task can wrap an LLM call (like the executive summary step) just as naturally as it wraps a database query or a pure rendering function with no I/O at all — task is a generic "make this step a durable, traceable unit," not a wrapper specifically for model calls.

#2.3 entrypoint — the orchestration is just the function body

entrypoint() wraps the whole workflow. Its body is where the actual difference from StateGraph is most visible: instead of declaring nodes and wiring edges between them, you write plain, ordinary async code — await one task, use its result as the input to the next, if a branch is ever needed, for a loop is ever needed.

ts
export const generateReportWorkflow = entrypoint(
  { checkpointer, name: "generate_report" },
  async (input: { analysisRunId: string }) => {
    const data = await fetchReportData(input.analysisRunId);
    const summary = await generateExecutiveSummary(data);
    const buffer = await renderReportDocx(data, summary);
    const record = await storeReport(input.analysisRunId, data.contractId, buffer, data);
    return record;
  }
);

Read that function top to bottom and you have read the entire pipeline. There is no second file to cross-reference for "what runs after fetchReportData" — the answer is simply "whatever line comes next." This is the direct trade this API makes: you give up StateGraph's ability to express branching and parallel fan-out as first-class, inspectable graph structure, and in exchange you get to write the orchestration in the same style as every other function you've ever written, while still keeping tracing and durability.

#2.4 StateGraph vs. Functional API — a direct comparison

StateGraph (Phase 3, 4) Functional API (this phase)
How you express "what runs after what" Declared edges between named nodes Plain sequential code — the next line
Branching / conditional routing First-class (addConditionalEdges) Ordinary if/else in the function body
Dynamic parallel fan-out over a runtime list First-class (Send()) Ordinary Promise.all over a plain array, same as Phase 3's window extraction
Multiple branches converging First-class (shared downstream node + reducer) N/A — nothing to converge, because nothing branched
Per-step tracing Yes, per node Yes, per task
Durable checkpointing Yes, per superstep, inspectable via getState/getStateHistory Yes, per task result, but there is no separate audit-trail API to browse — durability here is about efficient/correct replay, not a reviewer-facing history feature
Best fit A problem whose shape genuinely branches or fans out — the reason Phase 4 needed it A problem that is fundamentally a sequence of steps — the reason this phase doesn't need more than this

The honest takeaway: neither API is "better" in general. Phase 4's risk analysis is a graph-shaped problem (many parallel clauses, conditional routing per clause, a real audit-trail requirement) and StateGraph is the correct tool for it. This phase's report generation is a sequence-shaped problem, and the Functional API is the correct tool for this — reaching for StateGraph here would mean spending the graph's declarative machinery on a shape that never uses the flexibility it buys you.

#2.5 Grounding the executive summary — and keeping arithmetic out of the model's hands

The report's executive summary is the one genuinely generative step in this pipeline (rendering the DOCX itself is plain code; fetching data is plain code). Same groundedness principle as every prior LLM step in this project: the model should describe what the data says, not invent detail beyond it. But there's a subtler risk here that Phase 2–4's chains didn't have to think about: this step involves counting and aggregating ("3 of 12 clauses were scored HIGH risk," "2 clauses were flagged for human review"), and language models are unreliable arithmetic engines — asking the model to both count the findings and narrate them in the same call invites a mismatch between the prose and the actual numbers.

The fix: compute every hard number in code, before the LLM call, and hand the model those numbers as trusted input it must narrate around, not derive itself.

ts
const stats = {
  totalClauses: data.findings.length,
  highRiskCount: data.findings.filter(f => f.riskLevel === "HIGH").length,
  mediumRiskCount: data.findings.filter(f => f.riskLevel === "MEDIUM").length,
  lowRiskCount: data.findings.filter(f => f.riskLevel === "LOW").length,
  reviewFlaggedCount: data.findings.filter(f => f.reviewFlagged).length,
};
// stats is passed into the prompt as ground truth; the model is asked to
// write a narrative that is CONSISTENT with these numbers, never to
// recompute or restate them from scratch.

This is the same underlying instinct as Phase 4's "cite the specific excerpt that justifies this score" — don't trust the model with a task it's structurally bad at when your own code can do that part perfectly and just hand over the result.

#2.6 Programmatic DOCX generation — documents as a tree of parts, not a template

The docx npm package models a Word document the way docx-the-format actually is under the hood: a Document containing one or more Sections, each section containing an ordered list of block-level children — Paragraphs (which themselves contain TextRuns for styled inline text), Tables (containing TableRows containing TableCells), and heading paragraphs (Paragraph with a heading property). You build this tree entirely in code, then call Packer.toBuffer(document) to serialize it into actual .docx bytes.

ts
const doc = new Document({
  sections: [{
    children: [
      new Paragraph({ text: "Contract Risk Report", heading: HeadingLevel.TITLE }),
      new Paragraph({ text: "Executive Summary", heading: HeadingLevel.HEADING_1 }),
      new Paragraph({ children: [new TextRun(summary.narrative)] }),
    ],
  }],
});
const buffer = await Packer.toBuffer(doc);

Why this over a .docx template with placeholders (the docxtemplater-style approach)? A template is a great fit when a document's structure is fixed and only certain values change (a fixed-format certificate, an invoice with a known set of line-item rows). This report's structure genuinely varies per contract: the number of clause sections is exactly the number of findings, the number of HIGH-risk callouts varies, and whether a "Flagged for Human Review" appendix section appears at all depends on whether reviewFlagged is ever true for this run. Programmatic generation handles a variable number of variably-shaped sections naturally, because it's just a loop building tree nodes; a template would need its own looping/conditional syntax bolted on to do the same thing, which is more indirection for no real gain here.

#2.7 Giving the "flagged for review" distinction real visual weight

Concept Learning 2.5 in Phase 4.5 made the product argument for why reviewFlagged exists; this phase is where that argument has to actually show up on the page. Concretely, that means every reviewFlagged finding's section in the DOCX uses a visually distinct treatment — this phase uses a shaded paragraph background and a leading "⚠ Flagged for Human Review" label — so a reader scanning the document cannot mistake an unconfirmed finding for a fully-grounded one, even if they only skim. The docx library supports paragraph-level shading (shading: { fill: "FFF3CD" }, a soft amber) and per-run bold/color formatting, which is enough to make this distinction unmissable without needing custom styles or a design tool.

#2.8 Where the file actually lives

A generated report is a binary artifact, not a database row — Postgres stores the record of a report (which run it belongs to, its size, its finding counts, where to find it), not the file's bytes themselves. This phase deliberately keeps "where files live" behind one small interface (ReportStorage, Concept 3.4) with a local-filesystem implementation for now — the same instinct as Phase 4.5 keeping tool logic separate from the tool() wrapper: swapping local disk for S3-compatible object storage later should mean writing one new class, not touching the workflow or the routes.

#3. Different Approaches

#3.1 Orchestration: LCEL chain vs. a new StateGraph vs. the Functional API

Approach Pros Cons When to use
A. Plain LCEL .pipe() chain Familiar from Phase 2; works fine for the one genuinely "chain-like" step (the executive summary prompt → structured output) Forces the non-LLM steps (fetching data, rendering a DOCX, writing a file) into awkward RunnableLambda wrappers just to stay inside the same chain — those steps aren't runnables in any meaningful sense, they're just functions Fine if the entire pipeline were LLM calls; this one mostly isn't
B. A new StateGraph Consistent with Phase 3/4's pattern; still traceable and checkpointable Pure ceremony for a linear pipeline — an Annotation.Root state, reducers nobody needs (nothing here ever gets contributed to by more than one branch), and edges that only ever say "then do the next thing" A problem that might grow branches later, or where team convention values one consistent orchestration primitive above all else
C. LangGraph Functional API — entrypoint/task (roadmap's choice) Matches the pipeline's actual shape (a sequence); keeps tracing and durable, replay-safe step caching without declaring state/nodes/edges for a shape that never branches; the orchestration is readable TypeScript, not a wiring diagram A second orchestration primitive to know alongside StateGraph — though this is precisely the point: knowing when not to reach for the graph primitive is as valuable as knowing how to use it Recommended — this phase is deliberately your first contrast case for "which LangGraph API fits this problem's actual shape"

Recommendation: Approach C, as specified. Section 6 builds the full pipeline as task-wrapped steps inside one entrypoint.

#3.2 DOCX generation: programmatic (docx) vs. template-based (docxtemplater)

Approach Pros Cons When to use
A. Template-based (docxtemplater + a pre-designed .docx template with {placeholders}) A designer can own the template's visual polish independently of your code; good fit when structure is fixed This report's structure is not fixed — the number of clause sections, the presence/absence of the review-flagged appendix, and per-section styling (risk-level color coding) all vary per run; templating engines can do loops/conditionals but it turns the template itself into a second place your logic lives, split across two files/formats Fixed-structure documents: certificates, standard-format invoices, cover letters
B. Programmatic (docx npm package) (recommended) Full, type-checked control over a genuinely variable structure; one place (TypeScript) where all the logic lives; straightforward to add new section types later More verbose than filling in a template for a truly fixed layout Recommended — this report's structure is inherently variable (N clause sections, conditional appendix), which is exactly where programmatic generation earns its keep over templating

Recommendation: Approach B, as specified.

#3.3 Executive summary generation: fixed string templates vs. LLM-generated narrative

Approach Pros Cons When to use
A. Fixed string templates ("This contract has {n} HIGH risk clauses...") Zero cost, zero latency, perfectly deterministic Reads as a form letter; can't adapt phrasing to what's actually notable about this contract (e.g. "the indemnification clause is unusually one-sided" vs. a generic risk count) A very budget- or latency-constrained deployment where narrative quality genuinely doesn't matter
B. LLM-generated narrative, grounded in code-computed stats (recommended) Reads like an actual analyst's summary; can highlight what's genuinely notable about this specific contract, not just recite counts One more LLM call, and (per Concept 2.5) requires deliberately keeping arithmetic out of the model's hands to avoid a stats/narrative mismatch Recommended — the counts are computed in code and handed to the model as ground truth specifically so this approach gets the narrative quality without the arithmetic risk

Recommendation: Approach B, with the code-computed-stats safeguard from Concept 2.5 as a hard requirement, not an optional nicety.

#3.4 Report file storage backend

Approach Pros Cons When to use
A. Local filesystem Zero setup, zero new dependencies, fine for local development and small deployments Doesn't survive a container restart in most production deployment models; doesn't scale across multiple app instances Local development, and small/simple production deployments where disk persistence is guaranteed
B. S3-compatible object storage Durable, scalable, standard production choice Requires provisioning a bucket and credentials now, before this phase strictly needs to make that decision Production, once your deployment target is known

Recommendation: Neither, exclusively — instead, define one small ReportStorage interface (save(buffer, filename) → path/URL, read(path) → buffer) now, and implement only the local-filesystem version in this phase. This is the same "interface now, real backend later" move Phase 4.5 made for tool logic — swapping in an S3 implementation later is then a new class, not a rewrite of the workflow or routes.

#4. Detailed Implementation Plan

#Step 1 — Database schema for generated reports

What: A generated_reports table — one row per generated file, recording which analysis run it belongs to, where the file lives, its size, and a few denormalized counts (total findings, HIGH count, review-flagged count) for quick listing without re-reading the file.

Why: This is the record Phase 6 (or any future dashboard) queries to know a report exists and offer it for download, without needing to touch the file itself just to show a list.

Best practice: Store the denormalized counts at generation time rather than recomputing them from risk_findings on every list request — cheap to compute once, here, right when you already have the full findings list in memory.

#Step 2 — Report data aggregation

What: fetchReportData(analysisRunId) — one function that loads the contract's metadata, every clause+finding pair for the run (joined), ordered by risk level (HIGH → MEDIUM → LOW) then by clause order, in a single query.

Why: Every downstream step (the summary prompt, the DOCX renderer) needs the exact same shape of data; computing it once, in one place, keeps the summary and the rendered document provably consistent with each other — they're built from the same object, not two independently-run queries that could drift.

Common mistake to avoid: Re-querying inside the renderer for "just one more field" you forgot to include upstream. Get the aggregation query's shape right once (Step 2's code below includes everything both the summary and the renderer need) rather than patching it incrementally from inside rendering logic.

#Step 3 — Executive summary schema and chain

What: ExecutiveSummarySchema (Zod) and a withStructuredOutput() chain that receives the code-computed stats object (Concept 2.5) plus a compact list of the highest-risk findings, and returns a short narrative plus a one-line headline risk verdict.

Why/How: Covered in Concept Learning 2.5.

Best practice: Pass only the HIGH-risk (and review-flagged) findings' details into the prompt, not every single finding — the summary's job is to highlight what matters, and a wall of low-risk clause detail in the prompt both wastes tokens and dilutes what the model should actually be focusing on.

#Step 4 — The DOCX renderer

What: renderReportDocx(data, summary) — a pure function (no I/O, no LLM calls) that builds the full Document tree: cover page, executive summary section, one subsection per risk level containing one block per finding (with the review-flagged visual treatment applied where relevant), and a closing legal-references appendix.

Why/How: Covered in Concept Learning 2.6–2.7.

Best practice: Keep this function pure and synchronous-in-spirit (everything it needs is already in its two arguments) — it should be trivially unit-testable by handing it a fixture object and asserting on the resulting document's structure, with zero mocking of databases or APIs required.

Common mistake to avoid: Reaching back into the database or calling the LLM from inside the renderer "just this once" for a missing field. If the renderer needs data, that's a signal Step 2's aggregation query is incomplete — fix it there, not by giving the renderer side effects.

#Step 5 — Wrapping each step as a task

What: fetchReportData, generateExecutiveSummary, renderReportDocx, and storeReport each wrapped with task(name, fn).

Why: Covered in Concept Learning 2.2 — this is what makes each step individually traceable and replay-cacheable, without changing how you call them.

#Step 6 — The entrypoint workflow

What: generateReportWorkflow, wiring the four tasks together in the order described in Concept Learning 2.3, with a PostgresSaver-backed checkpointer (the same checkpointer instance Phase 4 already set up — no new infrastructure).

Why: Covered in Concept Learning 2.3.

Best practice: Give the entrypoint a stable, descriptive name ("generate_report") — this is what shows up as the top-level trace name in LangSmith, the same reasoning as naming graph nodes clearly in Phase 3/4.

#Step 7 — Storage abstraction and local filesystem implementation

What: A ReportStorage interface and a LocalFileReportStorage implementation writing into a configured directory.

Why/How: Covered in Concept Learning 2.8 and Approach 3.4.

Common mistake to avoid: Deriving a stored filename from user- or model-influenced text (e.g. the contract's title). Use the report's own generated UUID as the filename, and keep the human-readable contract name only as a Content-Disposition download filename at the HTTP layer — never as the actual on-disk path.

#Step 8 — Storage layer for the generated_reports table

What: insertGeneratedReport(...) and getGeneratedReport(id) / listReportsForContract(contractId) functions.

Why: Same reasoning as every prior phase's storage module — keep SQL in one place, separate from orchestration logic.

#Step 9 — Routes

What: POST /api/analysis/:threadId/report (runs the workflow, returns the generated_reports row) and GET /api/reports/:id/download (streams the file back with correct Content-Type and Content-Disposition).

Best practice: Validate that the given threadId's analysis run actually reached aggregate_findings (i.e., Phase 4's graph completed) before attempting to generate a report — use analysisGraph.getState(config) (already built in Phase 4) to check next is empty, and return a clear 409 if the run is still in progress rather than generating a report from a partial findings list.

#6.4 Report storage abstraction

ts
// src/reports/storage/reportStorage.ts
import { promises as fs } from "fs";
import path from "path";
import { randomUUID } from "crypto";

/**
 * Small interface so "where files live" is swappable (local disk now,
 * S3-compatible object storage later) without touching the workflow or
 * routes that use it - same "interface now, real backend later" move as
 * Phase 4.5's tool logic/wrapper separation.
 */
export interface ReportStorage {
  save(buffer: Buffer): Promise<{ path: string; sizeBytes: number }>;
  read(storedPath: string): Promise<Buffer>;
}

export class LocalFileReportStorage implements ReportStorage {
  constructor(private readonly baseDir: string) {}

  async save(buffer: Buffer): Promise<{ path: string; sizeBytes: number }> {
    await fs.mkdir(this.baseDir, { recursive: true });

    // Filename is a fresh UUID, never derived from user- or model-influenced
    // text - human-readable naming is applied only at the HTTP download
    // layer (Content-Disposition), never as the actual on-disk path.
    const filename = `${randomUUID()}.docx`;
    const fullPath = path.join(this.baseDir, filename);

    await fs.writeFile(fullPath, buffer);

    return { path: fullPath, sizeBytes: buffer.length };
  }

  async read(storedPath: string): Promise<Buffer> {
    return fs.readFile(storedPath);
  }
}

export const reportStorage: ReportStorage = new LocalFileReportStorage(
  process.env.REPORTS_STORAGE_DIR ?? "./storage/reports"
);

#6.5 Report data aggregation

ts
// src/reports/dataAggregation.ts
import { Pool } from "pg";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

export interface ReportFinding {
  clauseId: string;
  clauseTitle: string;
  clauseType: string;
  clauseFullText: string;
  riskLevel: "LOW" | "MEDIUM" | "HIGH";
  riskScore: number;
  reasoning: string;
  fixSuggestion: string;
  legalReferences: { sourceDocument: string; excerpt: string }[];
  reviewFlagged: boolean;
  reviewReason: string | null;
}

export interface ReportData {
  contractId: string;
  analysisRunId: string;
  contractType: string;
  contractFilename: string;
  findings: ReportFinding[];
}

/**
 * The single source of truth both the executive summary chain and the
 * DOCX renderer are built from - computing this once, in one place, keeps
 * the two provably consistent with each other rather than risking two
 * independently-run queries drifting apart.
 */
export async function fetchReportData(analysisRunId: string): Promise<ReportData> {
  const { rows: contractRows } = await pool.query(
    `SELECT c.id AS contract_id, c.contract_type, c.original_filename
     FROM risk_findings rf
     JOIN contracts c ON c.id = rf.contract_id
     WHERE rf.analysis_run_id = $1
     LIMIT 1`,
    [analysisRunId]
  );

  if (contractRows.length === 0) {
    throw new Error(`No findings exist for analysis run ${analysisRunId}.`);
  }

  const { rows: findingRows } = await pool.query(
    `SELECT
        cl.id AS clause_id, cl.title AS clause_title, cl.clause_type,
        cl.full_text AS clause_full_text,
        rf.risk_level, rf.risk_score, rf.reasoning, rf.fix_suggestion,
        rf.legal_references, rf.review_flagged, rf.review_reason
     FROM risk_findings rf
     JOIN clauses cl ON cl.id = rf.clause_id
     WHERE rf.analysis_run_id = $1
     ORDER BY
        CASE rf.risk_level WHEN 'HIGH' THEN 0 WHEN 'MEDIUM' THEN 1 ELSE 2 END,
        rf.created_at ASC`,
    [analysisRunId]
  );

  const findings: ReportFinding[] = findingRows.map((r) => ({
    clauseId: r.clause_id,
    clauseTitle: r.clause_title,
    clauseType: r.clause_type,
    clauseFullText: r.clause_full_text,
    riskLevel: r.risk_level,
    riskScore: r.risk_score,
    reasoning: r.reasoning,
    fixSuggestion: r.fix_suggestion,
    legalReferences: r.legal_references,
    reviewFlagged: r.review_flagged,
    reviewReason: r.review_reason,
  }));

  return {
    contractId: contractRows[0].contract_id,
    analysisRunId,
    contractType: contractRows[0].contract_type,
    contractFilename: contractRows[0].original_filename ?? "Untitled Contract",
    findings,
  };
}

export interface ReportStats {
  totalClauses: number;
  highRiskCount: number;
  mediumRiskCount: number;
  lowRiskCount: number;
  reviewFlaggedCount: number;
}

/**
 * Every hard number the report needs, computed in plain code - handed to
 * the LLM as trusted ground truth (Concept Learning 2.5) rather than asked
 * of the model, since language models are unreliable at exactly this kind
 * of counting/aggregation task.
 */
export function computeReportStats(data: ReportData): ReportStats {
  return {
    totalClauses: data.findings.length,
    highRiskCount: data.findings.filter((f) => f.riskLevel === "HIGH").length,
    mediumRiskCount: data.findings.filter((f) => f.riskLevel === "MEDIUM").length,
    lowRiskCount: data.findings.filter((f) => f.riskLevel === "LOW").length,
    reviewFlaggedCount: data.findings.filter((f) => f.reviewFlagged).length,
  };
}

// src/reports/executiveSummaryChain.ts
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { ExecutiveSummarySchema } from "./schemas";
import type { ReportData, ReportStats } from "./dataAggregation";

const model = new ChatOpenAI({
  model: process.env.REPORT_SUMMARY_MODEL ?? "gpt-4o-mini",
  temperature: 0.2,
});

const summaryPrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    "You are a legal analyst writing the executive summary section of a " +
      "contract risk report. You are given exact, pre-computed statistics - " +
      "treat these as ground truth. Do NOT recompute, restate as your own " +
      "calculation, or contradict them; write a narrative that is consistent " +
      "with them. Highlight what is genuinely notable about the highest-risk " +
      "findings below; do not attempt to describe every clause.",
  ],
  [
    "human",
    "Statistics:\n{stats}\n\n" +
      "Contract type: {contractType}\n\n" +
      "Highest-risk findings (HIGH risk and/or flagged for human review):\n{highlightedFindings}",
  ],
]);

const summaryChain = summaryPrompt.pipe(
  model.withStructuredOutput(ExecutiveSummarySchema, { name: "write_executive_summary" })
);

function formatHighlightedFindings(data: ReportData): string {
  const highlighted = data.findings.filter((f) => f.riskLevel === "HIGH" || f.reviewFlagged);

  if (highlighted.length === 0) {
    return "(No HIGH-risk or flagged findings - this contract's risk profile is low overall.)";
  }

  return highlighted
    .map(
      (f, i) =>
        `${i + 1}. [${f.riskLevel}${f.reviewFlagged ? ", FLAGGED FOR REVIEW" : ""}] ` +
        `${f.clauseTitle}: ${f.reasoning}`
    )
    .join("\n");
}

export async function generateExecutiveSummaryContent(
  data: ReportData,
  stats: ReportStats
) {
  return summaryChain.invoke({
    stats: JSON.stringify(stats, null, 2),
    contractType: data.contractType,
    highlightedFindings: formatHighlightedFindings(data),
  });
}

#6.7 DOCX renderer

ts
// src/reports/docxRenderer.ts
import {
  Document,
  Packer,
  Paragraph,
  TextRun,
  HeadingLevel,
  AlignmentType,
  BorderStyle,
} from "docx";
import type { ReportData, ReportFinding, ReportStats } from "./dataAggregation";
import type { ExecutiveSummary } from "./schemas";

const RISK_COLORS: Record<ReportFinding["riskLevel"], string> = {
  HIGH: "C0392B",
  MEDIUM: "B7950B",
  LOW: "1E8449",
};

const REVIEW_FLAG_SHADING = "FFF3CD"; // soft amber - the review-flagged visual treatment

function buildCoverPage(data: ReportData, stats: ReportStats): Paragraph[] {
  return [
    new Paragraph({
      text: "Contract Risk Report",
      heading: HeadingLevel.TITLE,
      alignment: AlignmentType.CENTER,
    }),
    new Paragraph({
      alignment: AlignmentType.CENTER,
      children: [new TextRun({ text: data.contractFilename, italics: true })],
    }),
    new Paragraph({ text: "" }),
    new Paragraph({
      alignment: AlignmentType.CENTER,
      children: [
        new TextRun(
          `Contract type: ${data.contractType}  |  Clauses analyzed: ${stats.totalClauses}  |  ` +
            `HIGH risk: ${stats.highRiskCount}  |  Flagged for review: ${stats.reviewFlaggedCount}`
        ),
      ],
    }),
    new Paragraph({ text: "" }),
  ];
}

function buildExecutiveSummarySection(summary: ExecutiveSummary): Paragraph[] {
  return [
    new Paragraph({ text: "Executive Summary", heading: HeadingLevel.HEADING_1 }),
    new Paragraph({
      children: [new TextRun({ text: summary.headline, bold: true })],
    }),
    new Paragraph({ text: "" }),
    ...summary.narrative
      .split(/\n{2,}/)
      .map((para) => new Paragraph({ children: [new TextRun(para.trim())] })),
  ];
}

/**
 * Renders one finding as a block of paragraphs. Applies the review-flagged
 * visual treatment (shaded background + explicit warning label) whenever
 * reviewFlagged is true - this is where Phase 4.5's honesty-over-fabrication
 * design decision has to actually show up on the page, not just in the data.
 */
function buildFindingBlock(finding: ReportFinding): Paragraph[] {
  const blocks: Paragraph[] = [];

  blocks.push(
    new Paragraph({
      heading: HeadingLevel.HEADING_3,
      children: [
        new TextRun({
          text: `${finding.clauseTitle} `,
        }),
        new TextRun({
          text: `[${finding.riskLevel} · ${finding.riskScore}/100]`,
          color: RISK_COLORS[finding.riskLevel],
          bold: true,
        }),
      ],
    })
  );

  if (finding.reviewFlagged) {
    blocks.push(
      new Paragraph({
        shading: { fill: REVIEW_FLAG_SHADING },
        border: {
          left: { style: BorderStyle.SINGLE, size: 12, color: "C0392B" },
        },
        children: [
          new TextRun({ text: "⚠ Flagged for Human Review — ", bold: true, color: "8A6D3B" }),
          new TextRun({
            text:
              finding.reviewReason ??
              "Sufficient legal grounding could not be confirmed for this finding. " +
                "Manual legal review is recommended before relying on this assessment.",
            color: "8A6D3B",
          }),
        ],
      })
    );
  }

  blocks.push(new Paragraph({ children: [new TextRun({ text: finding.reasoning })] }));
  blocks.push(
    new Paragraph({
      children: [
        new TextRun({ text: "Suggested fix: ", bold: true }),
        new TextRun({ text: finding.fixSuggestion }),
      ],
    })
  );

  if (finding.legalReferences.length > 0 && !finding.reviewFlagged) {
    blocks.push(
      new Paragraph({
        children: [
          new TextRun({
            text: `Grounded in: ${finding.legalReferences.map((r) => r.sourceDocument).join("; ")}`,
            italics: true,
            size: 20,
          }),
        ],
      })
    );
  }

  blocks.push(new Paragraph({ text: "" }));
  return blocks;
}

function buildRiskLevelSection(
  title: string,
  findings: ReportFinding[]
): Paragraph[] {
  if (findings.length === 0) return [];
  return [
    new Paragraph({ text: title, heading: HeadingLevel.HEADING_1 }),
    ...findings.flatMap(buildFindingBlock),
  ];
}

function buildLegalReferencesAppendix(data: ReportData): Paragraph[] {
  const allRefs = data.findings.flatMap((f) =>
    f.legalReferences.map((r) => ({ clause: f.clauseTitle, ...r }))
  );

  if (allRefs.length === 0) return [];

  return [
    new Paragraph({ text: "Appendix: Legal References", heading: HeadingLevel.HEADING_1 }),
    ...allRefs.flatMap((ref) => [
      new Paragraph({
        children: [
          new TextRun({ text: `${ref.clause} — ${ref.sourceDocument}`, bold: true }),
        ],
      }),
      new Paragraph({ children: [new TextRun({ text: ref.excerpt, italics: true })] }),
      new Paragraph({ text: "" }),
    ]),
  ];
}

/**
 * Pure function: no I/O, no LLM calls. Everything it needs is already in
 * its two arguments, which is what makes it trivially unit-testable with a
 * hand-constructed fixture and zero mocking.
 */
export async function renderReportDocx(
  data: ReportData,
  stats: ReportStats,
  summary: ExecutiveSummary
): Promise<Buffer> {
  const high = data.findings.filter((f) => f.riskLevel === "HIGH");
  const medium = data.findings.filter((f) => f.riskLevel === "MEDIUM");
  const low = data.findings.filter((f) => f.riskLevel === "LOW");

  const doc = new Document({
    sections: [
      {
        children: [
          ...buildCoverPage(data, stats),
          ...buildExecutiveSummarySection(summary),
          ...buildRiskLevelSection("High Risk Clauses", high),
          ...buildRiskLevelSection("Medium Risk Clauses", medium),
          ...buildRiskLevelSection("Low Risk Clauses", low),
          ...buildLegalReferencesAppendix(data),
        ],
      },
    ],
  });

  return Packer.toBuffer(doc);
}

import { task } from "@langchain/langgraph";
import { fetchReportData, computeReportStats } from "./dataAggregation";
import { generateExecutiveSummaryContent } from "./executiveSummaryChain";

  return { data, stats };
});

export const summarizeTask = task(
  "generate_executive_summary",

    data: Awaited<ReturnType<typeof fetchDataTask>>["data"];

    contractId: string;

    return insertGeneratedReport({
      contractId: input.contractId,
      analysisRunId: input.analysisRunId,

      fileSizeBytes: sizeBytes,

      reviewFlaggedCount: input.stats.reviewFlaggedCount,
    });
  }
);
ts
// src/reports/workflow.ts
import { entrypoint } from "@langchain/langgraph";
import { PostgresSaver } from "@langchain/langgraph-checkpoint-postgres";
import { fetchDataTask, summarizeTask, renderTask, storeTask } from "./tasks";

// Reuses the same checkpointer backend Phase 4 already set up - no new
// infrastructure, and durability here means efficient/correct replay
// (skip re-running successful prior steps), not a reviewer-facing audit
// trail feature the way Phase 4's checkpointer additionally provides.
const checkpointer = PostgresSaver.fromConnString(process.env.DATABASE_URL!);

/**
 * The entire pipeline, expressed as plain sequential async code rather than
 * a graph of nodes and edges - appropriate because this pipeline never
 * branches (Concept Learning 2.1). Reading this function top to bottom IS
 * reading the whole workflow; there is no second file to cross-reference
 * for "what runs after what."
 */
export const generateReportWorkflow = entrypoint(
  { checkpointer, name: "generate_report" },
  async (input: { analysisRunId: string }) => {
    const { data, stats } = await fetchDataTask(input.analysisRunId);
    const summary = await summarizeTask({ data, stats });
    const buffer = await renderTask({ data, stats, summary });
    const record = await storeTask({
      buffer,
      contractId: data.contractId,
      analysisRunId: input.analysisRunId,
      stats,
    });
    return record;
  }
);

// src/reports/db.ts
import { Pool } from "pg";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

export interface InsertGeneratedReportInput {
  contractId: string;
  analysisRunId: string;
  filePath: string;
  fileSizeBytes: number;
  findingsCount: number;
  highRiskCount: number;
  reviewFlaggedCount: number;
}

export async function insertGeneratedReport(input: InsertGeneratedReportInput) {
  const { rows } = await pool.query(
    `INSERT INTO generated_reports
       (contract_id, analysis_run_id, file_path, file_size_bytes,
        findings_count, high_risk_count, review_flagged_count)
     VALUES ($1, $2, $3, $4, $5, $6, $7)
     RETURNING *`,
    [
      input.contractId,
      input.analysisRunId,
      input.filePath,
      input.fileSizeBytes,
      input.findingsCount,
      input.highRiskCount,
      input.reviewFlaggedCount,
    ]
  );
  return rows[0];
}

export async function getGeneratedReport(id: string) {
  const { rows } = await pool.query(`SELECT * FROM generated_reports WHERE id = $1`, [id]);
  return rows[0] ?? null;
}

export async function listReportsForContract(contractId: string) {
  const { rows } = await pool.query(
    `SELECT * FROM generated_reports WHERE contract_id = $1 ORDER BY created_at DESC`,
    [contractId]
  );
  return rows;
}

#6.10 Routes

ts
// src/routes/reports.ts
import { Router } from "express";
import { generateReportWorkflow } from "../reports/workflow";
import { getGeneratedReport } from "../reports/db";
import { reportStorage } from "../reports/storage/reportStorage";
import { analysisGraph } from "../analysis/graph";

export const reportsRouter = Router();

reportsRouter.post("/analysis/:threadId/report", async (req, res) => {
  const { threadId } = req.params;

  // Guard: don't build a report from a run that hasn't finished - checked
  // via the same getState() Phase 4 already exposes.
  const config = { configurable: { thread_id: threadId } };
  const state = await analysisGraph.getState(config);

  if (!state.values || state.next.length > 0) {
    return res.status(409).json({
      error: "Analysis run has not completed yet; cannot generate a report from a partial run.",
    });
  }

  try {
    const record = await generateReportWorkflow.invoke(
      { analysisRunId: threadId },
      { configurable: { thread_id: `report-${threadId}` } }
    );
    return res.status(201).json({ report: record });
  } catch (err) {
    console.error("Report generation failed:", err);
    return res.status(500).json({ error: "Failed to generate report." });
  }
});

reportsRouter.get("/reports/:id/download", async (req, res) => {
  const report = await getGeneratedReport(req.params.id);
  if (!report) {
    return res.status(404).json({ error: "Report not found." });
  }

  const buffer = await reportStorage.read(report.file_path);

  res.setHeader(
    "Content-Type",
    "application/vnd.openxmlformats-officedocument.wordprocessingml.document"
  );
  res.setHeader("Content-Disposition", `attachment; filename="risk-report-${report.id}.docx"`);
  return res.status(200).send(buffer);
});

#Why this design over alternatives

  • The Functional API over a new StateGraph: this pipeline never branches, so there is nothing for a graph's edges or reducers to express that plain sequential await code doesn't already express just as clearly — see Concept Learning 2.1 and 2.4's direct comparison.
  • Stats computed in code, never asked of the model: the executive summary's narrative quality benefits from an LLM; its arithmetic reliability does not. Splitting these means the report can never show a summary that contradicts its own numbers.
  • A pure, I/O-free renderer: keeping renderReportDocx free of database or LLM calls is what makes it trivially unit-testable, and it's a forcing function that any data the document needs must already live in the aggregation query's output — no scattered "just one more field" fetches creeping into rendering logic.
  • A ReportStorage interface instead of hardcoded filesystem calls throughout: the same "interface now, real backend later" move as Phase 4.5's tool logic — moving to S3-compatible storage later touches one file, not the workflow or the routes.

#8. Best Practices

Industry best practices

  • Compute every hard number (counts, totals) in code and hand it to the LLM as ground truth; never ask the model to both count and narrate in the same call.
  • Keep the DOCX renderer pure — no I/O, no LLM calls — so it stays trivially unit-testable and any missing-data problem surfaces as an aggregation-query gap, not a rendering-time patch.
  • Give the review-flagged visual treatment real, unmistakable weight (shading, an explicit warning label) — this is the direct, reader-facing payoff of Phase 4.5's flag_for_human_review design decision, and it's easy to accidentally under-deliver on at the rendering stage even after getting the data right.

Scalability considerations

  • Report generation is inherently a single-run, single-document operation with no fan-out — nothing here needs Send() or parallelism; the main scaling lever is running the workflow asynchronously/on a queue for very large contracts, which Phase 7 (streaming/async progress) is the natural place to address.
  • The docx library builds the entire document tree in memory before packing — fine at the scale of a single contract's findings, but worth being aware of if a future direction supports batch/portfolio-wide reports spanning many contracts in one document.

Security considerations

  • Serve /reports/:id/download behind the same auth/ownership checks noted as an open item in prior phases — a report is a complete, readable summary of a contract's legal risk, at least as sensitive as the underlying contract text.
  • Never let a user-influenced string become part of an on-disk file path; the UUID-based filename in LocalFileReportStorage (6.4) is a deliberate guard against path traversal or collision, not an arbitrary choice.

Performance optimizations

  • fetchReportData's single joined query (rather than N+1 queries per clause) keeps report generation's database cost flat regardless of how many clauses a contract has.
  • The Functional API's task-level replay caching (Concept Learning 2.2) means a transient failure late in the pipeline (e.g. a disk write error in storeTask) doesn't force a wasted re-run of the LLM-backed summarizeTask — genuinely useful given the summary call is the most expensive single step in this workflow.

Maintainability tips

  • Adding a new report section (e.g. a "recommended next steps" block) is: write one more buildXSection() function in docxRenderer.ts, call it from renderReportDocx's section array. No other file needs to change.
  • Keep RISK_COLORS and REVIEW_FLAG_SHADING as named constants at the top of the renderer, not inline hex strings scattered through the function bodies — you'll want to tune these once you've seen the report rendered against real contracts.

Testing considerations

  • Unit test computeReportStats() against a fixture findings array with known counts — this is the easiest possible regression to introduce silently, and it's the number the entire executive summary is grounded against.
  • Unit test renderReportDocx() against a hand-constructed ReportData/ReportStats/ExecutiveSummary fixture with at least one reviewFlagged: true finding — assert the resulting document's paragraph count and that the review-flagged block's shading property is actually set (the docx library's Document object is inspectable enough to assert on before packing).
  • Integration test the full entrypoint workflow against a small fixture analysis run — assert a generated_reports row is created with counts matching the fixture, and that the file at file_path is a valid, non-empty .docx.
  • Test the "run not complete" guard (route-level) against both a genuinely in-progress thread_id and a nonexistent one, confirming neither reaches generateReportWorkflow.invoke().

#9. Summary

What we learned:

  • Why a linear, non-branching pipeline is a poor fit for StateGraph's state/node/edge machinery, and how LangGraph's Functional API (entrypoint/task) gives the same tracing and durable-replay benefits to a pipeline that's better expressed as plain sequential code — a genuine contrast case, not just "the same thing with different syntax."
  • Why arithmetic and narrative generation should be split across code and model respectively: compute exact counts in TypeScript, hand them to the LLM as trusted ground truth, and ask it only to narrate consistently with them.
  • Programmatic DOCX generation as building a document as a tree of parts (sections → paragraphs/tables → runs), and why that fits this report's inherently variable structure better than a fixed-placeholder template would.
  • Why the reviewFlagged signal Phase 4.5 introduced only delivers on its purpose if the rendered document gives it real, unmistakable visual weight — the honesty of the underlying data doesn't automatically transfer to the reader unless the rendering layer deliberately carries it through.
  • Keeping a storage decision (local disk vs. object storage) behind one small interface, so today's simplest working choice doesn't become tomorrow's rewrite.

What has been completed:

  • generated_reports table and storage layer.
  • Report data aggregation (fetchReportData, computeReportStats) as the single shared source of truth for both the summary and the rendered document.
  • A grounded, stats-fed executive summary chain.
  • A complete, pure DOCX renderer producing a cover page, executive summary, risk-level-grouped clause sections (with a distinct review-flagged treatment), and a legal-references appendix.
  • The full pipeline as a task-wrapped, checkpointed LangGraph entrypoint workflow.
  • Generation and download endpoints, including a guard against generating from an incomplete analysis run.

What is left:

  • Running the hands-on tasks against a real completed analysis run and visually validating the rendered document, including the replay-caching behavior under a simulated failure (tasks 8, 12-14 — yours to execute).
  • Everything from Phase 6 onward, which surfaces this phase's generated reports to an actual end user.

How this connects to the next phase: This phase closes the loop that began all the way back in Phase 2: raw contract text in, a professional, trustworthy risk report out. Phase 6 builds on top of a report that now durably exists and can be downloaded on demand — whatever that phase's user-facing surface turns out to be (a dashboard, a notification, a shareable link), it consumes exactly the generated_reports row and /reports/:id/download endpoint this phase produced, the same way Phase 4.5 consumed Phase 4's judgment_lookup seam without needing to know how it was implemented underneath. And when Phase 6.5 wraps the full analysis pipeline under a Supervisor, this phase's entrypoint workflow is a natural terminal action for that Supervisor to trigger directly once it decides an analysis run is complete — no restructuring required, for the same reason Phase 4.5's agent slotted into Phase 4's graph without touching its edges: this phase's only real "interface" to the outside world has always been one function call in, one file out.
