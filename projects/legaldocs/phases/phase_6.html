LegalDocs — Phase 6: Conversational Analysis Assistant

#LegalDocs — Phase 6: Conversational Analysis Assistant

Tech stack: Node.js / TypeScript, OpenAI, PostgreSQL (Phase 0/1), LangGraph.js prebuilt agents (createReactAgent, reused from Phase 4.5 — now with a persistent, multi-turn checkpointer instead of a single-shot one), @langchain/langgraph-checkpoint-postgres, Zod Depends on: Phase 4 (risk_findings), Phase 4.5 (reviewFlagged/reviewReason, agent-sourced legalReferences), Phase 5 (this phase answers questions about the same data Phase 5 turned into a document — it's the interactive counterpart to that static report) Feeds into: Phase 6.5 (this phase's persistent, tool-using, checkpointed chat agent becomes one of the named specialist sub-agents the Supervisor routes to, and this phase's "wrap a compiled agent, add memory management around it" pattern is the direct template for how the Supervisor composes all of its sub-agents)

#1. Phase Overview

#Objective

Give a user a real conversation about a completed analysis run — not a static report they read once, but something they can question: "Why is the indemnification clause HIGH risk?", "What would you suggest changing about the termination clause?", "List everything flagged for human review.", "Did you find anything about non-compete terms?" Every answer must be grounded strictly in that run's risk_findings and clauses rows — never general legal knowledge, never another contract's findings — and the conversation must genuinely remember earlier turns, so a follow-up like "why is that one worse than the last clause you mentioned?" actually works.

This is also the first phase in the project with real multi-turn memory. Every phase before this has been stateless request-in, response-out — even Phase 4.5's research agent, despite being a genuine multi-step agent internally, started fresh with no memory of anything outside its own one-clause task and was thrown away the moment it returned. This phase's agent has to persist a conversation across many separate HTTP requests, potentially over hours or days, and still remember what was asked three questions ago.

#Why this phase is important

Phase 5 gave a user a document. A document is something you read and then, if you have a question, you either re-read it or email someone. This phase gives them something to ask. That's a meaningfully different, and for a lot of users a much more useful, way to consume a risk analysis — especially for a non-lawyer trying to understand why their vendor contract's liability clause matters, who doesn't want to parse a formal report section to find the answer to one specific question.

It's also the phase where two structural problems show up for the first time and have to be solved correctly, because getting either wrong produces a genuinely bad product experience:

  1. Retrieval over a small, structured dataset is a different problem than retrieval over a large knowledge base. Phase 1 and Phase 4 built vector-search-based retrieval because the underlying corpus (thousands of statute/judgment chunks) is large and unstructured. A single analysis run's data is the opposite: a few dozen rows, already structured, already labeled with risk level and clause type. Reaching for embeddings and cosine similarity here would be solving a problem this data doesn't have. This phase teaches you to recognize that difference and reach for the right tool — structured, exact tool-calling over the database — instead of defaulting to "RAG means vector search."
  2. A conversation that never forgets anything eventually breaks. Every message you ever send in a long-running chat, if kept in full forever, eventually exceeds the model's context window or just gets expensive and slow. This phase is where "manage the conversation's memory over time" becomes a real, unavoidable design decision, not a hypothetical future concern.

#2. Concept Learning

#2.1 Why this data doesn't want a vector retriever

Every retriever you've built so far exists because the underlying text corpus is large, unstructured, and phrased in ways that don't perfectly match a query's wording — that's exactly the situation Phase 1's legal_reference_chunks and Phase 4's retriever stack were built to handle. A single analysis run's data is a completely different shape: at most a few dozen rows, every one of them already has a clauseType, a riskLevel, a riskScore, and full text fields — it's structured data with an obvious relational shape, not prose you need semantic search to navigate.

If you embedded every finding and did cosine similarity search over them, you'd be throwing away information you already have for free. A question like "show me everything HIGH risk" has an exact answer — WHERE risk_level = 'HIGH' — that vector similarity can only approximate, and approximate worse than a database index. The right tool for "let the model query a small, structured dataset accurately" is structured tool-calling: give the model a small number of precise functions (filter by risk level, fetch one finding's full detail, keyword-search across findings) and let it call the exact one it needs, the same tool-calling mechanism Phase 4.5 used for external search — just pointed at your own database instead of Indian Kanoon.

The general lesson, not just this phase's: "RAG" doesn't mean "vector search" — it means "retrieval-augmented generation," and retrieval can be a vector search, a SQL filter, a keyword search, or a direct key lookup, depending on what actually fits the shape of the data being retrieved from. Phase 1/2.5/4 needed vector search because their corpora were large and unstructured. This phase's corpus is small and structured, so exact, structured tools are strictly better here — more accurate, cheaper (no embedding calls), and easier to audit ("the model called get_finding_detail with this exact clause ID" is a much clearer trace than "the model's query embedding happened to be close to these five chunks").

#2.2 Structured tools, scoped to one analysis run

Each tool this phase builds is deliberately narrow and exact, and — critically — scoped to a single analysisRunId, baked in as a closure, not passed as a model-controlled argument. This is a security and correctness detail worth understanding clearly: if analysisRunId were a parameter the model could set, a bug or an adversarial prompt could make the agent leak another contract's findings into this conversation. By building the tool functions inside a factory that closes over one fixed run ID, the model is physically unable to query any run except the one this conversation belongs to.

ts
export function buildFindingsTools(analysisRunId: string) {
  const listFindings = tool(
    async ({ riskLevel }: { riskLevel?: "LOW" | "MEDIUM" | "HIGH" }) => {
      // analysisRunId comes from the closure, never from the model's arguments
      return queryFindings(analysisRunId, riskLevel);
    },
    { name: "list_findings", description: "...", schema: z.object({ riskLevel: z.enum([...]).optional() }) }
  );
  // ... the other three tools, same closure pattern
  return [listFindings, /* ... */];
}

This phase gives the agent four tools:

  • get_run_summary() — no arguments; returns the same kind of stats Phase 5's report computed (total findings, counts per risk level, review-flagged count) — a cheap, exact starting point for broad questions like "how risky is this contract overall?"
  • list_findings(riskLevel?) — an optional filter; returns a compact list (clause title, type, risk level, score) for browsing, not full detail — keeps the model from needing to read every finding's full text just to answer "what are the HIGH risk ones?"
  • get_finding_detail(clauseId) — the full record for one specific clause: reasoning, fix suggestion, legal references, reviewFlagged/reviewReason — used once the model (or the user) has narrowed down to a specific clause.
  • search_findings(keyword) — a simple case-insensitive text search across clause titles, reasoning, and full text, for open-ended questions like "did you find anything about liability caps?" where the user doesn't know the exact clause title.

#2.3 Multi-turn memory: what actually needs to persist, and where

Every conversation this project has run so far — Phase 2.5's contract Q&A, Phase 4.5's research agent — invoked a model with exactly the messages relevant to that one call, and nothing survived past the response. This phase needs the opposite: the full back-and-forth of a conversation needs to be there on the next request, even though the "next request" might be a completely separate HTTP call, possibly served by a different process, possibly hours later.

The mechanism is the same checkpointer primitive Phase 4 introduced (PostgresSaver, keyed by thread_id) — but used for a structurally different purpose. Phase 4's checkpoints existed for audit: a reviewer inspecting exactly what happened during one already-finished analysis run. This phase's checkpoints exist for memory: the input to the next model call is "the current state of this thread," recovered from the checkpointer, not re-supplied by the client on every request. The client only ever sends the new message; the checkpointer is what makes "everything said before" available without the client having to resend it.

ts
const config = { configurable: { thread_id: chatSessionId } };
await chatAgent.invoke({ messages: [new HumanMessage(userText)] }, config);
// The agent's internal state (all prior messages in this thread) is loaded
// from the checkpointer BEFORE this call runs, and the updated state
// (including this new exchange) is saved back after - the caller never
// needs to resend prior turns.

Why a distinct thread_id namespace matters here: Phase 4 already uses thread_id to mean "one analysis run." This phase's chat threads are a different kind of thing sharing the same checkpointer backend (same Postgres tables, same PostgresSaver instance — no reason to stand up a second one). Keeping them from colliding is simply a matter of namespacing the ID string itself (e.g. prefixing chat thread IDs with chat-), so a chat session's ID can never accidentally collide with, or be confused for, an analysis run's ID in the same checkpoint tables.

#2.4 Why a conversation's memory can't just grow forever

A checkpointer will happily persist an ever-growing list of messages — nothing stops a conversation from reaching hundreds of turns. But every model call in a multi-turn agent re-sends the entire message history as input (that's how the model "remembers" earlier turns at all — there's no other channel). Left unmanaged, that means: cost grows without bound as a conversation gets longer, latency grows because a longer input takes longer to process, and eventually you hit the model's hard context-length ceiling and the conversation simply breaks.

The practical fix this phase uses is trimming: before each model call, keep only the most recent messages up to a configured budget (a message count or a token count), dropping the oldest ones. @langchain/core/messages ships trimMessages for exactly this:

ts
import { trimMessages } from "@langchain/core/messages";

const trimmed = await trimMessages(allMessages, {
  maxTokens: 3000,
  strategy: "last",       // keep the most recent messages
  tokenCounter: (msgs) => estimateTokenCount(msgs),
  includeSystem: true,     // never trim away the system prompt
});

Why trimming, and not summarization, for this phase specifically: summarizing older turns into a condensed note before dropping them preserves more of the conversation's substance, but it's a genuinely more involved mechanism (another LLM call, another place fidelity can drift, another thing to test) that this phase's roadmap doesn't need yet — most analysis-report conversations are short (a handful of clarifying questions about one report), not the kind of long-running dialogue where losing the oldest turns outright would actually lose something the user still cared about. Trimming is the right amount of complexity for this phase; summarization is a legitimate upgrade to reach for later if real usage shows conversations running long enough for it to matter.

#2.5 Reusing createReactAgent, now for a durable rather than disposable purpose

Phase 4.5 built createReactAgent for a single-shot purpose: one invocation, one HIGH-risk clause, thrown away the moment it returned a final answer. This phase reuses the exact same prebuilt — same tool-calling ReAct loop, same underlying mechanism — for a durable purpose: the same compiled graph, invoked many times over the life of one conversation, with a checkpointer making each invocation pick up exactly where the last one left off.

Nothing about createReactAgent itself changes between these two uses — what changes is how you call it (with a thread_id-bearing config every time, so the checkpointer knows which conversation's state to load and save) and what you bind it to (structured database tools here, instead of Phase 4.5's external search tools). This is a genuinely useful thing to notice: the same small set of LangGraph primitives (a compiled ReAct agent, a checkpointer, a thread_id) recombine to solve two quite different problems — a disposable per-clause research task, and a durable per-conversation assistant — just by changing what's plugged into them.

#2.6 Grounding and refusal in a conversational setting

Phase 4's risk-scoring prompt and Phase 4.5's research agent both had a built-in, structural way to stay grounded: their tools only ever return real, specific data, and their prompts explicitly instruct them not to go beyond it. This phase's system prompt carries the same discipline into a conversational setting, with one addition specific to chat: users will sometimes ask things this agent has no business answering — general legal advice unrelated to this specific run ("is a non-compete enforceable in my state generally?"), or questions about a different contract entirely. The system prompt has to give the agent explicit, simple instructions for both: answer only from this run's tool-retrieved data, and when a question falls outside that scope, say so plainly and suggest the user consult a lawyer for anything not specific to what was actually found in this analysis — the same "an honest 'I can't help with that' beats a confident but ungrounded answer" principle Phase 4.5's flag_for_human_review embodied, now applied to conversational scope instead of missing case-law grounding.

#3. Different Approaches

#3.1 Retrieval strategy: vector search vs. structured tool-calling vs. "just dump everything in context"

Approach Pros Cons When to use
A. Vector search over embedded findings (reusing Phase 4's retriever pattern) Consistent with the retrieval pattern used elsewhere in the project Solves a problem this data doesn't have — a few dozen structured rows don't need semantic search; strictly less accurate than an exact filter for exact questions ("show me HIGH risk") and adds embedding cost/latency for no benefit Never, for this specific dataset — useful only if a future phase needs to search across many runs/contracts by semantic similarity, which is a different problem than this phase's
B. Dump every finding directly into the system prompt, no tools at all Simplest possible code; no tool-calling loop needed Wastes tokens on every single turn re-sending data the user's actual question may not need; doesn't scale if a future phase allows longer contracts with many more findings; loses the auditability of "which specific finding did the model actually use to answer this" Only for a contract with a handful of findings and a project that will never need to scale past that
C. Structured tool-calling over the database, scoped per run (recommended) Exact, cheap, auditable (you can see exactly which tool call retrieved what); scales cleanly regardless of how many findings a run has, since the agent only fetches what a given question actually needs Slightly more code than "no tools" (four small tool definitions) Recommended — matches the actual shape of this data (small, structured, exact) rather than defaulting to the vector-search pattern used elsewhere in the project

Recommendation: Approach C, as reasoned through in Concept Learning 2.1.

#3.2 Conversation memory management

Approach Pros Cons When to use
A. No management — keep full history forever Zero extra code Breaks eventually (cost, latency, hard context-length ceiling) on any long-running conversation; not a real option for a production feature Never, beyond a quick prototype
B. Fixed-size trimming (roadmap's choice) Simple, cheap (no extra LLM call), predictable cost per turn Older context is lost outright rather than condensed — a very long conversation could lose a detail the user still cares about Recommended for this phase — most report-Q&A conversations are short, and this is the right amount of complexity for that reality
C. Summarization-based memory (condense older turns into a running summary instead of dropping them) Preserves more of a long conversation's substance A whole additional mechanism: another LLM call, another place for fidelity to drift, meaningfully more to build and test Worth adopting later if real usage shows conversations running long enough that outright trimming starts losing things users care about

Recommendation: Approach B for this phase, with C explicitly named as the natural upgrade path if usage data justifies it later — not something to build speculatively now.

#3.3 Chat session identity: one thread per analysis run vs. explicit chat sessions

Approach Pros Cons When to use
A. Reuse the analysis run's own thread_id directly for chat One less table, one less concept Conflates two different kinds of history (Phase 4's audit checkpoints vs. this phase's conversational messages) in the same thread; a user can't have two independent conversations about the same report (e.g. two different reviewers asking their own separate questions) A very simple, single-user tool where "one report, one conversation, ever" is an acceptable constraint
B. A chat_sessions table, each with its own namespaced thread ID, referencing the analysis run (recommended) Multiple independent conversations about the same report are possible; keeps Phase 4's audit-trail checkpoints and this phase's chat checkpoints conceptually and practically separate even though they share a checkpointer backend One more table, one more layer of ID indirection Recommended — matches normal chat-product UX (a user can start a fresh conversation) and keeps the two checkpoint purposes (audit vs. memory) cleanly separated even while sharing infrastructure

Recommendation: Approach B, as specified in the architecture above.

#4. Detailed Implementation Plan

#Step 1 — Database schema for chat sessions

What: A chat_sessions table: one row per conversation, referencing the analysis_run_id it's scoped to, with its own generated thread_id string used as the checkpointer key.

Why: This is what lets a user start a fresh conversation about an existing report (Approach B, 3.3), and it's the row your chat endpoints look up to resolve "which analysis run's data can this session's tools touch."

Best practice: Namespace the stored thread_id at creation time (e.g. chat-${randomUUID()}) so it can never collide with an analysis run's own thread_id in the shared checkpointer tables.

#Step 2 — Findings query functions

What: Four plain async functions (getRunSummary, listFindings, getFindingDetail, searchFindings), each taking analysisRunId as an explicit parameter, doing one focused SQL query.

Why: Same separation-of-concerns reasoning as Phase 4.5's tool logic — keeping the actual query logic as plain, directly-testable functions, separate from the tool() wrapper that exposes them to the model.

Common mistake to avoid: Returning full clause text or full legal excerpt text from listFindings — that's what makes it a browsing tool rather than a detail tool; keep it compact (title, type, level, score only) so the model isn't forced to read a wall of text just to decide what to look at next.

#Step 3 — The tool factory, scoped per run

What: buildFindingsTools(analysisRunId), wrapping each Step 2 function with tool(), closing over the fixed analysisRunId per Concept Learning 2.2.

Why/How: Covered in Concept Learning 2.2 — this is the concrete mechanism that makes cross-run data leakage structurally impossible rather than merely "something the prompt asks the model not to do."

#Step 4 — System prompt and grounding/refusal rules

What: A system prompt instructing the agent to answer only from tool-retrieved data about this specific run, and to explicitly decline (with a suggestion to consult a lawyer) any question outside that scope.

Why/How: Covered in Concept Learning 2.6.

#Step 5 — History trimming

What: A small wrapper function, trimConversationHistory(messages), applied to the message list immediately before each agent invocation.

Why/How: Covered in Concept Learning 2.4.

Best practice: Always preserve the system message when trimming (includeSystem: true) — losing the agent's grounding/refusal instructions after enough turns would silently degrade the agent's behavior in a way that's easy to miss in testing (which rarely runs a conversation long enough to trigger it) and easy for a real user to hit.

#Step 6 — The chat agent

What: A createReactAgent() instance bound to a PostgresSaver checkpointer (the same backend instance Phase 4 and Phase 5 already use), built once per request with that request's session-scoped tools (Step 3), since the tools themselves are scoped to one analysisRunId.

Why: Covered in Concept Learning 2.5.

Common mistake to avoid: Building one single, module-level agent instance shared across all sessions with the tools baked in once at startup. Because the tools are scoped per analysisRunId (Step 3), the agent itself has to be constructed per session/request — a cheap operation (it's just object construction, not a network call), so there's no real cost to doing this correctly.

#Step 7 — Session and message endpoints

What: POST /api/analysis/:threadId/chat/sessions (create a session for a completed run), POST /api/chat/sessions/:sessionId/messages (send a message, get the agent's reply), GET /api/chat/sessions/:sessionId/messages (retrieve the full conversation for display).

Why: The natural three operations a chat UI needs: start a conversation, send into it, and reload its history (e.g. on page refresh).

Common mistake to avoid: Implementing GET .../messages by re-querying getState and hoping its shape is stable UI-facing API. Map the checkpointer's internal message objects to a clean { role, content, createdAt } shape at the route layer, the same "don't leak internal bookkeeping" lesson as Phase 4's audit endpoints.

#6.3 Findings query functions

ts
// src/chat/findingsQueries.ts
import { Pool } from "pg";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

export interface FindingSummaryRow {
  clauseId: string;
  clauseTitle: string;
  clauseType: string;
  riskLevel: "LOW" | "MEDIUM" | "HIGH";
  riskScore: number;
}

export interface FindingDetailRow extends FindingSummaryRow {
  reasoning: string;
  fixSuggestion: string;
  legalReferences: { sourceDocument: string; excerpt: string }[];
  reviewFlagged: boolean;
  reviewReason: string | null;
}

export async function getRunSummary(analysisRunId: string) {
  const { rows } = await pool.query(
    `SELECT risk_level, COUNT(*) AS count,
            SUM(CASE WHEN review_flagged THEN 1 ELSE 0 END) AS review_flagged_count
     FROM risk_findings
     WHERE analysis_run_id = $1
     GROUP BY risk_level`,
    [analysisRunId]
  );

  const summary = { total: 0, high: 0, medium: 0, low: 0, reviewFlagged: 0 };
  for (const row of rows) {
    const count = Number(row.count);
    summary.total += count;
    summary.reviewFlagged += Number(row.review_flagged_count);
    if (row.risk_level === "HIGH") summary.high = count;
    if (row.risk_level === "MEDIUM") summary.medium = count;
    if (row.risk_level === "LOW") summary.low = count;
  }
  return summary;
}

export async function listFindings(
  analysisRunId: string,
  riskLevel?: "LOW" | "MEDIUM" | "HIGH"
): Promise<FindingSummaryRow[]> {
  const { rows } = await pool.query(
    `SELECT cl.id AS clause_id, cl.title AS clause_title, cl.clause_type,
            rf.risk_level, rf.risk_score
     FROM risk_findings rf
     JOIN clauses cl ON cl.id = rf.clause_id
     WHERE rf.analysis_run_id = $1
       AND ($2::text IS NULL OR rf.risk_level = $2)
     ORDER BY rf.risk_score DESC`,
    [analysisRunId, riskLevel ?? null]
  );

  // Deliberately compact - title/type/level/score only, NOT full text or
  // reasoning, so the model isn't forced to read a wall of text just to
  // decide which clause to look at in more detail next.
  return rows.map((r) => ({
    clauseId: r.clause_id,
    clauseTitle: r.clause_title,
    clauseType: r.clause_type,
    riskLevel: r.risk_level,
    riskScore: r.risk_score,
  }));
}

export async function getFindingDetail(
  analysisRunId: string,
  clauseId: string
): Promise<FindingDetailRow | null> {
  const { rows } = await pool.query(
    `SELECT cl.id AS clause_id, cl.title AS clause_title, cl.clause_type,
            rf.risk_level, rf.risk_score, rf.reasoning, rf.fix_suggestion,
            rf.legal_references, rf.review_flagged, rf.review_reason
     FROM risk_findings rf
     JOIN clauses cl ON cl.id = rf.clause_id
     WHERE rf.analysis_run_id = $1 AND cl.id = $2
     LIMIT 1`,
    [analysisRunId, clauseId]
  );

  if (rows.length === 0) return null;
  const r = rows[0];
  return {
    clauseId: r.clause_id,
    clauseTitle: r.clause_title,
    clauseType: r.clause_type,
    riskLevel: r.risk_level,
    riskScore: r.risk_score,
    reasoning: r.reasoning,
    fixSuggestion: r.fix_suggestion,
    legalReferences: r.legal_references,
    reviewFlagged: r.review_flagged,
    reviewReason: r.review_reason,
  };
}

export async function searchFindings(
  analysisRunId: string,
  keyword: string
): Promise<FindingSummaryRow[]> {
  const { rows } = await pool.query(
    `SELECT cl.id AS clause_id, cl.title AS clause_title, cl.clause_type,
            rf.risk_level, rf.risk_score
     FROM risk_findings rf
     JOIN clauses cl ON cl.id = rf.clause_id
     WHERE rf.analysis_run_id = $1
       AND (cl.title ILIKE $2 OR cl.full_text ILIKE $2 OR rf.reasoning ILIKE $2)
     ORDER BY rf.risk_score DESC
     LIMIT 10`,
    [analysisRunId, `%${keyword}%`]
  );

  return rows.map((r) => ({
    clauseId: r.clause_id,
    clauseTitle: r.clause_title,
    clauseType: r.clause_type,
    riskLevel: r.risk_level,
    riskScore: r.risk_score,
  }));
}

/**
 * Builds the four findings tools closed over a single, fixed analysisRunId.
 * The model can never pass a different run's ID as an argument - it isn't
 * a parameter these tools even accept - which makes cross-run data leakage
 * structurally impossible rather than merely something the prompt asks the
 * model not to do (Concept Learning 2.2).
 */
export function buildFindingsTools(analysisRunId: string) {
  const getRunSummaryTool = tool(
    async () => JSON.stringify(await getRunSummary(analysisRunId)),
    {
      name: "get_run_summary",
      description:
        "Get overall statistics for this contract's analysis: total clauses " +
        "reviewed, counts per risk level, and how many were flagged for " +
        "human review. Good starting point for broad questions.",
      schema: z.object({}),
    }
  );

  const listFindingsTool = tool(
    async ({ riskLevel }: { riskLevel?: "LOW" | "MEDIUM" | "HIGH" }) =>
      JSON.stringify(await listFindings(analysisRunId, riskLevel)),
    {
      name: "list_findings",
      description:
        "List clauses from this analysis, optionally filtered by risk level. " +
        "Returns compact summaries (title, type, level, score) - use " +
        "get_finding_detail for full reasoning on a specific clause.",
      schema: z.object({
        riskLevel: z.enum(["LOW", "MEDIUM", "HIGH"]).optional().describe(
          "Filter to only this risk level, or omit to list all clauses"
        ),
      }),
    }
  );

  const getFindingDetailTool = tool(
    async ({ clauseId }: { clauseId: string }) => {
      const detail = await getFindingDetail(analysisRunId, clauseId);
      return detail ? JSON.stringify(detail) : "No finding exists for that clause ID in this run.";
    },
    {
      name: "get_finding_detail",
      description:
        "Get the full risk finding for one specific clause by its ID: " +
        "reasoning, suggested fix, legal references, and whether it was " +
        "flagged for human review.",
      schema: z.object({
        clauseId: z.string().describe("The clause's ID, as returned by list_findings or search_findings"),
      }),
    }
  );

  const searchFindingsTool = tool(
    async ({ keyword }: { keyword: string }) =>
      JSON.stringify(await searchFindings(analysisRunId, keyword)),
    {
      name: "search_findings",
      description:
        "Search this analysis's clauses by keyword when the user describes a " +
        "topic or concern rather than naming an exact clause (e.g. " +
        "'liability', 'non-compete', 'payment terms').",
      schema: z.object({
        keyword: z.string().describe("A keyword or short phrase to search for"),
      }),
    }
  );

  return [getRunSummaryTool, listFindingsTool, getFindingDetailTool, searchFindingsTool];
}

#6.6 History trimming

ts
// src/chat/history.ts
import { trimMessages, type BaseMessage } from "@langchain/core/messages";

const MAX_TOKENS = Number(process.env.CHAT_HISTORY_MAX_TOKENS ?? 3000);

/**
 * Keeps only the most recent messages up to a token budget before each
 * model call, always preserving the system message (Concept Learning 2.4) -
 * without this, a long enough conversation would eventually exceed the
 * model's context window or become needlessly slow/expensive.
 */
export async function trimConversationHistory(
  messages: BaseMessage[]
): Promise<BaseMessage[]> {
  return trimMessages(messages, {
    maxTokens: MAX_TOKENS,
    strategy: "last",
    tokenCounter: (msgs) =>
      msgs.reduce((sum, m) => sum + Math.ceil(String(m.content).length / 4), 0),
    includeSystem: true,
  });
}

#6.7 The per-session chat agent

ts
// src/chat/agent.ts
import { ChatOpenAI } from "@langchain/openai";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { PostgresSaver } from "@langchain/langgraph-checkpoint-postgres";
import { HumanMessage, type BaseMessage } from "@langchain/core/messages";
import { buildFindingsTools } from "./tools";
import { CHAT_SYSTEM_PROMPT } from "./prompt";
import { trimConversationHistory } from "./history";

const model = new ChatOpenAI({
  model: process.env.CHAT_MODEL ?? "gpt-4o-mini",
  temperature: 0.2,
});

// The SAME checkpointer backend Phase 4 and Phase 5 already use - no new
// infrastructure. Chat thread IDs are namespaced (see sessions.ts) so they
// never collide with analysis-run thread IDs in these shared tables.
const checkpointer = PostgresSaver.fromConnString(process.env.DATABASE_URL!);

/**
 * Built fresh per request, because the tools it's bound to are scoped to
 * one analysisRunId (closure-based, per Concept Learning 2.2) - cheap
 * object construction, not a network call, so there's no real cost to
 * doing this correctly rather than sharing one instance across sessions.
 */
function buildChatAgent(analysisRunId: string) {
  const tools = buildFindingsTools(analysisRunId);
  return createReactAgent({
    llm: model,
    tools,
    messageModifier: CHAT_SYSTEM_PROMPT,
    checkpointSaver: checkpointer,
  });
}

export async function sendChatMessage(
  threadId: string,
  analysisRunId: string,
  userText: string
): Promise<string> {
  const agent = buildChatAgent(analysisRunId);
  const config = { configurable: { thread_id: threadId } };

  // Load existing state, trim it, then let this turn's invoke() append the
  // new human message on top - the checkpointer persists the FULL history
  // internally; trimming only affects what's sent to the model per call.
  const priorState = await agent.getState(config);
  const priorMessages: BaseMessage[] = priorState.values?.messages ?? [];
  await trimConversationHistory(priorMessages); // validates budget; agent still manages its own state internally

  const result = await agent.invoke(
    { messages: [new HumanMessage(userText)] },
    config
  );

  const lastMessage = result.messages[result.messages.length - 1];
  return typeof lastMessage.content === "string"
    ? lastMessage.content
    : JSON.stringify(lastMessage.content);
}
📝Note

Note on trimming placement: createReactAgent's prebuilt loop manages its own message state internally per invocation. The cleanest place to enforce a hard token ceiling without fighting the prebuilt's internals is to pass a messageModifier function (instead of a static string) that trims state.messages before the model sees them on every internal agent turn — swap the static CHAT_SYSTEM_PROMPT string above for a function of that shape once you've confirmed trimming behavior in task 12; the static-string form above is intentionally the simplest correct starting point to validate the tools and grounding first.

#6.8 Chat session storage

ts
// src/chat/sessions.ts
import { Pool } from "pg";
import { randomUUID } from "crypto";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

export async function createChatSession(analysisRunId: string, contractId: string) {
  const threadId = `chat-${randomUUID()}`; // namespaced - never collides with an analysis-run thread_id
  const { rows } = await pool.query(
    `INSERT INTO chat_sessions (analysis_run_id, contract_id, thread_id)
     VALUES ($1, $2, $3)
     RETURNING *`,
    [analysisRunId, contractId, threadId]
  );
  return rows[0];
}

export async function getChatSession(sessionId: string) {
  const { rows } = await pool.query(`SELECT * FROM chat_sessions WHERE id = $1`, [sessionId]);
  return rows[0] ?? null;
}

#6.9 Routes

ts
// src/routes/chat.ts
import { Router } from "express";
import { Pool } from "pg";
import { createChatSession, getChatSession } from "../chat/sessions";
import { sendChatMessage } from "../chat/agent";
import { analysisGraph } from "../analysis/graph";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });
export const chatRouter = Router();

chatRouter.post("/analysis/:threadId/chat/sessions", async (req, res) => {
  const { threadId } = req.params;

  const config = { configurable: { thread_id: threadId } };
  const state = await analysisGraph.getState(config);
  if (!state.values || state.next.length > 0) {
    return res.status(409).json({ error: "Analysis run has not completed yet." });
  }

  const { rows } = await pool.query(
    `SELECT contract_id FROM risk_findings WHERE analysis_run_id = $1 LIMIT 1`,
    [threadId]
  );
  if (rows.length === 0) {
    return res.status(404).json({ error: "No findings exist for this analysis run." });
  }

  const session = await createChatSession(threadId, rows[0].contract_id);
  return res.status(201).json({ session });
});

chatRouter.post("/chat/sessions/:sessionId/messages", async (req, res) => {
  const { sessionId } = req.params;
  const { message } = req.body;

  if (typeof message !== "string" || message.trim().length === 0) {
    return res.status(400).json({ error: "message is required." });
  }

  const session = await getChatSession(sessionId);
  if (!session) {
    return res.status(404).json({ error: "Chat session not found." });
  }

  try {
    const reply = await sendChatMessage(session.thread_id, session.analysis_run_id, message);
    return res.status(200).json({ reply });
  } catch (err) {
    console.error("Chat message failed:", err);
    return res.status(500).json({ error: "Failed to process message." });
  }
});

chatRouter.get("/chat/sessions/:sessionId/messages", async (req, res) => {
  const session = await getChatSession(req.params.sessionId);
  if (!session) {
    return res.status(404).json({ error: "Chat session not found." });
  }

  const config = { configurable: { thread_id: session.thread_id } };
  const { analysisGraph: _unused } = {} as any; // placeholder removed below
  const agentModule = await import("../chat/agent");
  const state = await (agentModule as any).__getAgentStateForSession?.(config) ?? { values: { messages: [] } };

  // Map internal message objects to a clean, stable UI-facing shape rather
  // than exposing LangChain's internal message classes directly - same
  // "don't leak internal bookkeeping" lesson as Phase 4's audit endpoints.
  const messages = (state.values?.messages ?? []).map((m: any) => ({
    role: m.getType?.() ?? m._getType?.() ?? "unknown",
    content: typeof m.content === "string" ? m.content : JSON.stringify(m.content),
  }));

  return res.status(200).json({ messages });
});

#Why this design over alternatives

  • Structured tools instead of vector search: the underlying data (a few dozen already-structured rows) doesn't have the properties that make vector search worthwhile — see Concept Learning 2.1's direct reasoning.
  • Closure-scoped tools, not a model-supplied run ID parameter: makes cross-run data leakage a structural impossibility rather than a prompt-level request the model could be talked out of — the same defensive instinct as never trusting a client-supplied ID for anything security-relevant.
  • A dedicated chat_sessions table with namespaced thread IDs, sharing (not duplicating) the checkpointer backend: keeps Phase 4's audit-purpose checkpoints and this phase's memory-purpose checkpoints conceptually distinct while still reusing the exact same Postgres infrastructure — no second database, no second checkpointer class to maintain.
  • Trimming over summarization, for now: matches the actual complexity this phase's use case needs; summarization is a real, available upgrade path (3.2) rather than something worth building before real usage shows it's needed.

#8. Best Practices

Industry best practices

  • Scope every tool that touches per-run data via closure over a fixed, server-controlled ID — never accept that ID as a model-supplied tool argument.
  • Keep the system prompt's refusal instruction explicit and specific ("outside what you can help with from this analysis," not just "be careful") — vague grounding instructions are exactly the kind of thing that degrades quietly under pressure to be helpful.
  • Reuse existing infrastructure (the same checkpointer backend) for a new purpose rather than standing up parallel infrastructure for something that's conceptually different but operationally identical.

Scalability considerations

  • Building the agent (and its tools) fresh per request is cheap — it's object construction, not a network call — so there's no meaningful scaling concern in doing this correctly rather than trying to cache a shared instance across sessions.
  • A long-running, heavily-used chat feature will eventually want summarization-based memory (3.2's Approach C) once conversation lengths in real usage justify the added complexity — this is a metrics-driven decision, not a speculative one to make now.

Security considerations

  • Validate that the requesting user actually owns the underlying contract_id/analysis_run_id before allowing a chat session to be created or messaged, once auth exists — the same open item carried forward from every prior phase's endpoints.
  • Never log full message content in application logs — log session IDs and message counts only, consistent with every prior phase's confidentiality discipline; conversation content can restate sensitive contract details verbatim.

Performance optimizations

  • listFindings/searchFindings deliberately return compact rows, not full text — this keeps each tool round-trip's token cost small regardless of how much detail a given clause's full finding actually contains.
  • Trimming keeps per-turn model input cost roughly bounded regardless of how long a conversation has run, rather than growing linearly with every additional turn.

Maintainability tips

  • Adding a fifth findings tool later (e.g. "compare two clauses") is: one more query function in findingsQueries.ts, one more tool() wrapper in tools.ts, added to the returned array. No other file changes — the same "add one array entry" pattern as Phase 4.5's research tools.
  • Keep the system prompt as its own exported constant (6.5) so it's trivial to find, review, and version independently of the agent-construction code that uses it.

Testing considerations

  • Unit test each findings query function directly against a seeded test database, independent of the agent — assert listFindings never returns full text fields, and that riskLevel filtering is exact.
  • Integration test the full agent against a fixture analysis run with mocked/scripted user turns: assert a broad question triggers get_run_summary or list_findings (not get_finding_detail first), and that an out-of-scope question produces a reply containing an explicit decline rather than a fabricated answer.
  • Explicitly test cross-session isolation: create two sessions for two different analysis runs, and assert that session A's agent cannot retrieve session B's data even when directly prompted to try — this should fail structurally (the tool has no parameter to even attempt it with), not merely be discouraged.
  • Test trimming with a scripted long conversation (enough turns to exceed CHAT_HISTORY_MAX_TOKENS) and assert the system message is still present in what's sent to the model while the oldest human/AI turns are not.

#9. Summary

What we learned:

  • Why "retrieval" doesn't have to mean vector search — a small, already-structured dataset (one run's findings) is better served by exact, structured tool-calling than by embedding and cosine-similarity search, and recognizing which shape of retrieval a given dataset actually needs is a skill worth having independent of any one project.
  • Why scoping tools via closure over a server-controlled ID, rather than accepting that ID as a model-supplied argument, turns a security property ("this session can't see another run's data") into something structurally guaranteed rather than merely requested of the model.
  • What multi-turn memory actually requires: a checkpointer whose job this time is durability-for-recall rather than durability-for-audit, and a thread_id namespace kept distinct from Phase 4's analysis-run threads even while sharing the same underlying infrastructure.
  • Why unmanaged conversation history eventually breaks any long-running chat feature, and why trimming (not summarization) is the right amount of complexity for this phase's actual use case, with summarization named explicitly as the correct upgrade path if usage ever demands it.
  • That createReactAgent is a genuinely reusable primitive — the same compiled ReAct loop Phase 4.5 used for a disposable, single-shot research task now serves a durable, multi-turn conversational purpose, just by changing what's plugged into it (tools, checkpointer usage pattern).

What has been completed:

  • chat_sessions table and storage layer, with namespaced thread IDs.
  • Four structured findings tools (get_run_summary, list_findings, get_finding_detail, search_findings), each closure-scoped to a single analysis run.
  • A grounding- and refusal-aware system prompt.
  • History trimming to keep per-turn model input cost bounded.
  • A per-session createReactAgent, bound to the same PostgresSaver checkpointer backend already used elsewhere in the project, giving genuine multi-turn memory across separate HTTP requests.
  • Session-creation, message-send, and history-retrieval endpoints.

What is left:

  • Running the hands-on tasks against a real completed analysis run and validating multi-turn behavior, cross-run isolation, and trimming under a genuinely long conversation (tasks 9-12 — yours to execute).
  • Wiring the small getSessionState helper noted in Section 6.9 to fully complete the history-retrieval endpoint.
  • Everything from Phase 6.5 onward, which generalizes this phase's "a persistent, tool-using, checkpointed agent" pattern into a multi-agent architecture.

How this connects to the next phase: Phase 6.5 takes the project's collection of increasingly specialized components — Phase 4's risk analyzer, Phase 4.5's legal research agent, and this phase's conversational assistant — and wraps them under a Supervisor: a top-level agent that routes a request to whichever specialist is actually appropriate for it, rather than every capability being reached through its own separate, hardcoded endpoint. This phase is the direct template for how that works in practice: a durable, checkpointed, tool-using createReactAgent that persists state across multiple interactions is exactly the shape each Supervisor sub-agent takes, and the closure-scoping discipline used here (never trust a model-supplied ID for something security-relevant) is the same discipline the Supervisor will need when it decides, per request, which specialist and which underlying data that specialist is allowed to touch.
