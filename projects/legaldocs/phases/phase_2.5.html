#LegalDocs — Phase 2.5: Conversational Contract Q&A

Tech stack: Node.js / TypeScript, OpenAI, PostgreSQL + pgvector (Phase 0/1), LangChain.js (@langchain/core, @langchain/openai, langchain/memory) Depends on: Phase 1 (chunking/embedding technique reused here), Phase 2 (the contracts table and its raw_text column) Feeds into: Nothing downstream requires this phase's output, but it establishes the memory + basic-retriever patterns Phase 4 will build its much larger retriever stack and audit trail on top of

#1. Phase Overview

#Objective

Let a user ask quick, grounded questions about a contract they've just uploaded — "what's the termination notice period?", followed by "what about for cause?" — before they commit to the slower, more expensive full risk analysis in Phase 4. The answers must be grounded in the actual contract text (not the model's general knowledge), and follow-up questions must correctly use conversation context without the user having to repeat themselves.

#Why this phase is important

Two reasons, one product-shaped and one curriculum-shaped:

  • Product reason: A full risk analysis (Phase 4) is slow and costs real LLM tokens across many clauses. Most users have a quick specific question first ("does this even have a non-compete clause?"). A cheap, fast, conversational Q&A layer in front of that is genuinely useful and reduces wasted full analyses.
  • Curriculum reason: Your roadmap's audit found that LangChain's conversational Memory classes (unit 6) had no home anywhere else in the project, because everything else uses LangGraph's checkpointing for state. This is deliberately the one place you exercise "classic" LangChain memory management, so you understand it before you never need it again (LangGraph checkpointing, which you'll use everywhere from Phase 3 onward, solves the same underlying problem — "remember what happened so far" — in a different, more powerful way).

#2. Concept Learning

#2.1 Why "Memory" is a separate concept from RAG retrieval

It's easy to conflate these because they both involve "remembering things," but they solve different problems:

  • Retrieval answers: "What's in the contract that's relevant to this question?" — it looks into a document.
  • Memory answers: "What did we already talk about in this conversation?" — it looks into the conversation's own history.

A good follow-up answer needs both. Consider:

User: "What's the termination notice period?" (retrieval finds the termination clause, answers "30 days") User: "What about for cause?"

That second question, taken alone, is not answerable — "what about for cause" for what? Only with conversation memory do you know the topic is still "termination notice period," which lets you either (a) answer directly if the retrieved context already covers it, or (b) reformulate the question into a standalone one — "What is the termination notice period for termination for cause?" — before retrieving. That reformulation step is the actual bridge between memory and retrieval, and it's the crux of this phase.

#2.2 Conversation Memory classes

LangChain's classic memory classes exist to solve one specific bookkeeping problem: how much of a growing conversation history do you feed back into the model on each turn, and in what form?

  • BufferMemory — keeps the entire conversation, verbatim, forever. Simple, but token cost grows unbounded as the conversation gets longer, and eventually you'll exceed the model's context window.
  • BufferWindowMemory (your roadmap's choice) — keeps only the last k turns (a sliding window), discarding older ones. Bounded token cost, and for a "quick Q&A before the real analysis" feature, recent context is almost always what matters — nobody is referencing something they asked 40 messages ago in this feature.
  • ConversationSummaryMemory — instead of discarding old turns, periodically asks an LLM to summarize them into a running summary, keeping unbounded conversation length bounded in token cost at the price of an extra LLM call and some information loss.

Why BufferWindowMemory is the right choice here: this is a lightweight "ask a few quick questions" feature, not an extended dialogue. A window of the last 4-6 turns is more than enough context, it's the cheapest option (no extra summarization LLM call), and it matches the roadmap's explicit choice.

A production wrinkle worth understanding clearly: these memory classes were originally designed assuming a long-lived, in-process object that you keep calling .saveContext() on as a conversation progresses in the same process. Your Express server is stateless between HTTP requests — a new request might even be handled by a different server instance if you scale horizontally. So "attaching memory to the chain" here means something slightly different than the textbook example: on every request, you reconstruct a BufferWindowMemory instance from persisted history (loaded from your contract_chats table), use it to format the last k turns for the prompt, and then persist the new turn back to the database yourself — the memory object's job is formatting/windowing logic, not persistence. This is exactly how memory classes are used in any real stateless backend, and understanding this distinction (in-memory object vs. persisted source of truth) is the actual lesson of unit 6, not just "instantiate the class."

#2.3 The "condense question" / history-aware retrieval pattern

A vector similarity search only works well against a self-contained query. "What about for cause?" embedded on its own will retrieve fairly random, unrelated chunks — the query itself doesn't semantically contain "termination." So before retrieving, a conversational RAG system typically:

  1. Takes the raw follow-up question + the recent conversation history.
  2. Asks the LLM (a cheap, fast call) to rewrite it into a standalone question that makes sense with no prior context — "What is the termination notice period for termination for cause?"
  3. Uses that rewritten question for the actual vector similarity search.
  4. Answers using the retrieved context, still with the conversation history available for tone/continuity.

This two-step "rewrite, then retrieve" shape is standard in conversational RAG and is worth understanding as a pattern, not memorizing as a single function name — you'll see the identical shape again in Phase 4.5's agent design, just decided by an agent's own reasoning instead of a fixed prompt.

#2.4 Retrievers — the basic interface

A retriever in LangChain is any object implementing one method conceptually: "given a query string, return the most relevant Document objects." It's deliberately a thin, swappable interface — the same downstream chain code can work whether the retriever underneath does plain cosine similarity, MMR, or a multi-query fan-out (Phase 4 will swap in progressively fancier retrievers behind the exact same interface). This phase builds the basic version: cosine similarity search filtered to one contract's chunks — the roadmap explicitly scopes this phase to "Retrievers - basic" and saves MMR/Multi-Query/Contextual Compression for Phase 4's deeper treatment.

Metadata filtering by contract_id is what makes this retriever scoped rather than searching your entire Phase 1 legal knowledge base. This is a completely separate vector table from Phase 1's legal_reference_chunks — this phase's chunks are the user's own uploaded contract text, not your curated statutes/judgments.

#2.5 Why chunk the contract at all, again?

The same reasoning from Phase 1 applies: a 40-page contract embedded as one vector would produce a useless "average of everything" representation. Chunking the contract into paragraph-sized pieces (using the exact same RecursiveCharacterTextSplitter approach from Phase 1) lets a question like "what's the notice period" retrieve the specific termination clause paragraph, not the whole document.

#3. Different Approaches

#3.1 When to chunk/embed the contract

Approach Pros Cons When to use
A. Eager — chunk/embed immediately after Phase 2's upload analysis First chat message has zero extra latency; indexing happens once, predictably, during upload Every uploaded contract gets embedded even if the user never opens the chat feature — wasted cost for contracts nobody chats with; couples Phase 2's endpoint to Phase 2.5's concern If chat is a near-universal next step after upload in your product flow
B. Lazy — chunk/embed on the first chat request for a contract (roadmap-agnostic, recommended) Zero wasted embedding cost for contracts nobody chats with; keeps Phase 2 and Phase 2.5 fully decoupled — Phase 2 doesn't need to know Phase 2.5 exists First chat message for a contract has a small extra latency hit while indexing happens Recommended — cheaper by default, cleaner separation of concerns, and the one-time latency cost on the first message is a reasonable, explainable tradeoff

Recommendation: Approach B, implemented as an idempotent ensureContractIndexed(contractId) check at the top of the chat endpoint — the same hash/existence-check idempotency pattern you already used in Phase 1, just checking "does this contract already have chunks" instead of "has this file changed."

#3.2 Retriever implementation

Approach Pros Cons When to use
A. LangChain PGVectorStore wrapper configured against a custom table Less code; directly returns a LangChain-native retriever via .asRetriever(); consistent with Phase 4's later retriever stack Requires mapping its expected column names onto your custom schema; slightly more setup/configuration overhead for a table this simple Reasonable if you want to reuse it identically in Phase 4
B. A small custom retriever class extending BaseRetriever Full visibility into exactly what SQL runs and why (genuinely useful at this "basic retriever" learning stage); no schema-mapping friction against your custom contract_chunks table You write ~20 lines of retrieval code yourself instead of configuring a wrapper Recommended for this phase — the roadmap calls this "Retrievers - basic," and building the interface yourself once, by hand, is what makes the wrapped/fancier versions in Phase 4 (MMR, Multi-Query, Contextual Compression) legible as decorators around this same basic idea, rather than opaque magic

Recommendation: Approach B here. In Phase 4 you'll likely construct an actual PGVectorStore for the fuller retriever feature set — that's fine and not a contradiction; this phase is explicitly about understanding the retriever interface at its simplest.

#3.3 Question rewriting — always vs. conditionally

Approach Pros Cons When to use
A. Always rewrite the question through the LLM, even for the first message One consistent code path, no branching Wastes one LLM call on the very first message of a conversation, where there's no history to condense against anyway Simpler codebases where the extra call is a non-issue
B. Rewrite only when conversation history exists; use the raw question otherwise Saves an LLM call on every conversation's first message (the most common case) One small if branch to maintain Recommended — free latency/cost win with no downside

#4. Detailed Implementation Plan

#Step 1 — Database schema

What: contract_chunks (this contract's embedded paragraphs) and contract_chats (the running Q&A transcript).

Why: Separate tables because they have entirely different lifecycles and access patterns — chunks are written once (lazily) and read often via similarity search; chats are appended to on every turn and read back in full, in order, per conversation.

Best practice: ON DELETE CASCADE from both tables back to contracts.id — deleting a contract should cleanly remove its chat history and chunks, not leave orphaned rows.

#Step 2 — Lazy indexing (ensureContractIndexed)

What: A function checking whether contract_chunks already has rows for a given contract_id; if not, it splits the contract's raw_text (reusing Phase 1's RecursiveCharacterTextSplitter logic) and embeds/stores the chunks.

Why: This is what makes the "index on first chat request" approach from section 3.1 idempotent and safe to call at the top of every chat request without re-doing work.

Common mistake to avoid: Checking "does the contract exist" instead of "does the contract already have chunks" — those are different questions, and conflating them means a contract that failed indexing once (e.g. an embedding API blip) can get permanently stuck never being retried.

#Step 3 — The custom retriever

What: A class extending BaseRetriever, parameterized by contractId, implementing the one method retrievers need: given a query, embed it, run a cosine-similarity SQL query filtered to that contract_id, return the top-k chunks as Document objects.

Why: As discussed in 3.2 — full visibility into the retrieval mechanism at this "basic" stage, and a clean interface Phase 4 will later decorate with more advanced strategies.

#Step 4 — Chat history persistence and window memory loading

What: getRecentTurns(contractId, k) reading the last k human/ai message pairs from contract_chats, and a function turning those rows into a BufferWindowMemory instance (via an InMemoryChatMessageHistory seeded with the loaded messages).

Why: This is the "reconstruct memory from persisted state on every stateless request" pattern from Concept Learning 2.2.

Best practice: Order by created_at ASC when loading — conversation history must be replayed in the order it actually happened, or the model will misread who said what.

#Step 5 — Question rewriting (condense) chain

What: A small chain: prompt(history, question) → model → standalone question string, only invoked when history is non-empty (per 3.3).

Why: Makes follow-up questions retrievable on their own, per Concept Learning 2.3.

Common mistake to avoid: Using a high-temperature model for this step — question rewriting is a precise, mechanical transformation, not a creative task. Keep temperature: 0 here too.

#Step 6 — The answer chain

What: A prompt combining the retrieved contract chunks (as grounding context) + the standalone question, instructing the model to answer only from the provided context and to say so explicitly if the contract doesn't address the question — never guess or fall back to general legal knowledge.

Why: Groundedness is the entire point of RAG here — an ungrounded answer about someone's actual contract is actively harmful advice-shaped content, not just a quality issue.

Best practice: Include the source chunk indices in the response payload (not necessarily in the natural-language answer) so a frontend can eventually show "this answer is based on paragraph 14" — cheap to add now, valuable later.

#Step 7 — Orchestration and the endpoint

What: POST /api/contracts/:id/chat wiring: ensure indexed → load window memory → rewrite question (conditionally) → retrieve → answer → persist both the user's question and the model's answer to contract_chats → respond.

Best practice: Persist the user's original question (not the rewritten standalone version) to contract_chats — the rewritten version is an internal implementation detail for retrieval, not something the user said, and mixing the two into the visible transcript would make history replay incoherent on the next turn.

#6.2 Lazy indexing

This reuses Phase 1's splitRecursive splitter and embedTexts embedding function directly — no reason to write a second implementation of either.

ts
// src/chat/indexing.ts
import { Pool } from "pg";
import format from "pg-format";
import { splitRecursive } from "../ingestion/splitters"; // reused from Phase 1
import { embedTexts } from "../ingestion/embeddings";     // reused from Phase 1

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

async function hasChunks(contractId: string): Promise<boolean> {
  const { rows } = await pool.query(
    `SELECT 1 FROM contract_chunks WHERE contract_id = $1 LIMIT 1`,
    [contractId]
  );
  return rows.length > 0;
}

/**
 * Idempotent lazy indexing: chunks and embeds a contract's raw text only if
 * it hasn't been indexed yet. Checking "does contract_chunks have rows" (not
 * "does the contract exist") means a contract that failed indexing once
 * (e.g. a transient embedding API error) will simply be retried on the next
 * chat request instead of getting permanently stuck unindexed.
 */
export async function ensureContractIndexed(
  contractId: string,
  rawText: string
): Promise<void> {
  if (await hasChunks(contractId)) return;

  const chunks = await splitRecursive(rawText);
  if (chunks.length === 0) {
    throw new Error(`No chunks produced for contract ${contractId} - raw text may be empty.`);
  }

  const embeddings = await embedTexts(chunks.map((c) => c.content));

  const values = chunks.map((chunk, i) => [
    contractId,
    chunk.chunkIndex,
    chunk.content,
    `[${embeddings[i].join(",")}]`,
  ]);

  const insertSql = format(
    `INSERT INTO contract_chunks (contract_id, chunk_index, content, embedding) VALUES %L`,
    values
  );
  await pool.query(insertSql);
}

#6.3 Custom basic retriever

ts
// src/chat/retriever.ts
import { BaseRetriever, type BaseRetrieverInput } from "@langchain/core/retrievers";
import type { Document } from "@langchain/core/documents";
import type { CallbackManagerForRetrieverRun } from "@langchain/core/callbacks/manager";
import { OpenAIEmbeddings } from "@langchain/openai";
import { Pool } from "pg";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });
const embeddingsClient = new OpenAIEmbeddings({
  model: process.env.EMBEDDING_MODEL ?? "text-embedding-3-small",
});

export interface ContractRetrieverInput extends BaseRetrieverInput {
  contractId: string;
  k?: number;
}

/**
 * The "basic" retriever for this phase: embed the query, run a cosine
 * similarity search filtered to one contract's chunks, return the top-k as
 * Documents. This is deliberately simple - Phase 4 will build MMR,
 * Multi-Query, and Contextual Compression retrievers as decorators around
 * this exact same underlying idea.
 */
export class ContractRetriever extends BaseRetriever {
  lc_namespace = ["legaldocs", "retrievers", "contract"];

  private contractId: string;
  private k: number;

  constructor(fields: ContractRetrieverInput) {
    super(fields);
    this.contractId = fields.contractId;
    this.k = fields.k ?? 4;
  }

  async _getRelevantDocuments(
    query: string,
    _runManager?: CallbackManagerForRetrieverRun
  ): Promise<Document[]> {
    const [queryVector] = await embeddingsClient.embedDocuments([query]);

    const { rows } = await pool.query(
      `SELECT content, chunk_index, embedding <=> $1 AS distance
       FROM contract_chunks
       WHERE contract_id = $2
       ORDER BY embedding <=> $1
       LIMIT $3`,
      [`[${queryVector.join(",")}]`, this.contractId, this.k]
    );

    return rows.map((row) => ({
      pageContent: row.content,
      metadata: { chunkIndex: row.chunk_index, distance: row.distance },
    }));
  }
}

#6.4 Chat transcript storage

ts
// src/chat/storage.ts
import { Pool } from "pg";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

export interface ChatTurn {
  role: "human" | "ai";
  content: string;
  createdAt: Date;
}

export async function saveChatTurn(
  contractId: string,
  role: "human" | "ai",
  content: string
): Promise<void> {
  await pool.query(
    `INSERT INTO contract_chats (contract_id, role, content) VALUES ($1, $2, $3)`,
    [contractId, role, content]
  );
}

/**
 * Returns the last `k` human/ai turn PAIRS (so up to 2*k rows), in
 * chronological order. Chronological order matters - replaying history out
 * of order will make the model misattribute who said what.
 */
export async function getRecentTurns(contractId: string, k: number): Promise<ChatTurn[]> {
  const { rows } = await pool.query(
    `SELECT role, content, created_at
     FROM contract_chats
     WHERE contract_id = $1
     ORDER BY created_at DESC
     LIMIT $2`,
    [contractId, k * 2]
  );

  return rows.reverse().map((r) => ({
    role: r.role,
    content: r.content,
    createdAt: r.created_at,
  }));
}

#6.5 Window memory from persisted history

ts
// src/chat/memory.ts
import { BufferWindowMemory } from "langchain/memory";
import { InMemoryChatMessageHistory } from "@langchain/core/chat_history";
import { HumanMessage, AIMessage } from "@langchain/core/messages";
import { getRecentTurns, type ChatTurn } from "./storage";

const WINDOW_SIZE = Number(process.env.CHAT_MEMORY_WINDOW ?? 5);

/**
 * Reconstructs a BufferWindowMemory instance for this request from
 * persisted history. This is the stateless-server pattern discussed in
 * Concept Learning 2.2: the memory OBJECT is short-lived and rebuilt every
 * request; the database is the actual source of truth for the conversation.
 */
export async function buildWindowMemory(contractId: string): Promise<{
  memory: BufferWindowMemory;
  turns: ChatTurn[];
}> {
  const turns = await getRecentTurns(contractId, WINDOW_SIZE);

  const messages = turns.map((t) =>
    t.role === "human" ? new HumanMessage(t.content) : new AIMessage(t.content)
  );

  const chatHistory = new InMemoryChatMessageHistory(messages);

  const memory = new BufferWindowMemory({
    k: WINDOW_SIZE,
    chatHistory,
    returnMessages: true,
    memoryKey: "history",
  });

  return { memory, turns };
}

#6.6 Condense-question and answer chains

ts
// src/chat/chains.ts
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate, MessagesPlaceholder } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";
import type { BaseMessage } from "@langchain/core/messages";
import type { Document } from "@langchain/core/documents";

const model = new ChatOpenAI({
  model: process.env.CHAT_MODEL ?? "gpt-4o-mini",
  temperature: 0,
});

// --- Step: rewrite a follow-up question into a standalone one ---

const condensePrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    "Given the conversation history and a follow-up question, rewrite the " +
      "follow-up as a standalone question that makes sense without the " +
      "history. Do not answer it - only rewrite it. If the follow-up is " +
      "already standalone, return it unchanged.",
  ],
  new MessagesPlaceholder("history"),
  ["human", "Follow-up question: {question}\n\nStandalone question:"],
]);

export const condenseQuestionChain = condensePrompt
  .pipe(model)
  .pipe(new StringOutputParser());

export async function toStandaloneQuestion(
  question: string,
  history: BaseMessage[]
): Promise<string> {
  if (history.length === 0) return question;
  return condenseQuestionChain.invoke({ question, history });
}

// --- Step: answer grounded strictly in retrieved contract context ---

const answerPrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    "You answer questions about a specific contract using ONLY the context " +
      "excerpts provided below. If the context does not contain the answer, " +
      "say clearly that the contract does not appear to address this, rather " +
      "than guessing or relying on general legal knowledge.\n\nContext:\n{context}",
  ],
  new MessagesPlaceholder("history"),
  ["human", "{question}"],
]);

export const answerChain = answerPrompt.pipe(model).pipe(new StringOutputParser());

export function formatContext(docs: Document[]): string {
  return docs
    .map((d, i) => `[Excerpt ${i + 1}]\n${d.pageContent}`)
    .join("\n\n");
}

#6.7 Orchestration service

ts
// src/chat/service.ts
import { ContractRetriever } from "./retriever";
import { buildWindowMemory } from "./memory";
import { toStandaloneQuestion, answerChain, formatContext } from "./chains";
import { saveChatTurn } from "./storage";
import { ensureContractIndexed } from "./indexing";

export interface ChatResult {
  answer: string;
  standaloneQuestion: string;
  sourceChunks: { chunkIndex: number; distance: number }[];
}

export async function askContractQuestion(
  contractId: string,
  rawText: string,
  question: string
): Promise<ChatResult> {
  await ensureContractIndexed(contractId, rawText);

  const { memory } = await buildWindowMemory(contractId);
  const { history } = (await memory.loadMemoryVariables({})) as {
    history: import("@langchain/core/messages").BaseMessage[];
  };

  const standaloneQuestion = await toStandaloneQuestion(question, history);

  const retriever = new ContractRetriever({ contractId, k: 4 });
  const docs = await retriever.invoke(standaloneQuestion);

  const answer = await answerChain.invoke({
    context: formatContext(docs),
    history,
    question: standaloneQuestion,
  });

  // Persist the user's ORIGINAL question, not the rewritten standalone
  // version - the rewrite is an internal retrieval detail, not something
  // the user actually said, and storing it would corrupt future replays.
  await saveChatTurn(contractId, "human", question);
  await saveChatTurn(contractId, "ai", answer);

  return {
    answer,
    standaloneQuestion,
    sourceChunks: docs.map((d) => ({
      chunkIndex: d.metadata.chunkIndex,
      distance: d.metadata.distance,
    })),
  };
}

#6.8 Route

ts
// src/routes/chat.ts
import { Router } from "express";
import { Pool } from "pg";
import { askContractQuestion } from "../chat/service";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });
export const chatRouter = Router();

chatRouter.post("/:id/chat", async (req, res) => {
  const { id } = req.params;
  const { question } = req.body as { question?: string };

  if (!question || typeof question !== "string" || question.trim().length === 0) {
    return res.status(400).json({ error: "Request body must include a non-empty 'question'." });
  }

  const { rows } = await pool.query(`SELECT raw_text FROM contracts WHERE id = $1`, [id]);
  if (rows.length === 0) {
    return res.status(404).json({ error: "Contract not found." });
  }

  try {
    const result = await askContractQuestion(id, rows[0].raw_text, question);
    return res.status(200).json(result);
  } catch (err) {
    console.error("Contract chat failed:", err);
    return res.status(500).json({ error: "Failed to answer question." });
  }
});

#Why this design over alternatives

  • Reusing Phase 1's splitter/embedding modules rather than duplicating them: the chunking and embedding logic has nothing contract-specific about it — the same function that split a statute into paragraphs works identically on a contract. Writing a second copy would just be a maintenance liability.
  • A dedicated contract_chunks table, not reusing legal_reference_chunks: these are semantically different corpora (curated static reference law vs. one specific user's uploaded document) with different lifecycles (a contract's chunks are deleted when the contract is deleted; reference chunks never are). Keeping them in separate tables makes both easier to reason about and query correctly.
  • Storing the original question, not the standalone rewrite, in contract_chats: the transcript should reflect what the user actually said — it's both more honest and, practically, prevents the rewritten form from compounding oddly across many turns of history replay.

#8. Best Practices

Industry best practices

  • Keep the answer prompt's groundedness instruction explicit and strong ("only from context, say so if the contract doesn't address it") — this is the single highest-leverage line of prompt engineering in a legal-adjacent RAG feature.
  • Use temperature: 0 for both the condense and answer chains — consistency matters more than variety for both.

Scalability considerations

  • Lazy indexing means embedding cost scales with actual chat usage, not upload volume — important once upload volume is much higher than chat engagement.
  • The HNSW index on contract_chunks.embedding keeps retrieval fast even as the table accumulates chunks across many contracts over time (the WHERE contract_id = $2 filter combined with the index keeps each query scoped and fast regardless of total table size).

Security considerations

  • Validate that the requesting user actually has access to the given contract_id before answering (this project doesn't yet have an auth layer per your roadmap, but note this as a required addition before any multi-tenant deployment — one contract's chat history should never be readable by another user).
  • Never let the answer prompt's context window include chunks from a different contract — the WHERE contract_id = $2 filter in the retriever is the only thing preventing cross-contract data leakage, so treat that filter as a security boundary, not just a relevance optimization.

Performance optimizations

  • Skipping the condense-question LLM call when there's no history (section 3.3) removes an unnecessary round-trip on every conversation's first message — the most common single request this endpoint will ever handle.
  • k=4 retrieved chunks is a reasonable default for quick Q&A; raising it increases context/cost without necessarily improving answer quality for short, specific questions.

Maintainability tips

  • Keep WINDOW_SIZE an environment variable, not a hardcoded constant — you'll likely tune this once you see real usage patterns.
  • The retriever, memory, and chains modules are independently testable — keep them that way rather than inlining logic directly into service.ts.

Testing considerations

  • Unit test toStandaloneQuestion() with a mocked model call, asserting it returns the raw question unchanged when history is empty (no LLM call should even be attempted).
  • Integration test the full askContractQuestion() flow against a small fixture contract with a known termination clause, asserting the answer correctly reflects that clause's content.
  • Explicitly test the "contract doesn't address this" case with a question about a topic absent from your fixture contract, asserting the model doesn't fabricate an answer.

#9. Summary

What we learned:

  • The distinction between conversational memory (what was said) and retrieval (what's in the document), and why conversational RAG needs both working together.
  • BufferWindowMemory and why a sliding window is the right tradeoff for a lightweight Q&A feature, plus the important practical wrinkle of using memory classes correctly against a stateless HTTP server (rebuild from persisted state each request, don't rely on the object's own lifetime).
  • The condense-question / history-aware retrieval pattern, and why a follow-up question needs to be rewritten into a standalone form before it can be usefully embedded and searched.
  • The basic retriever interface — a swappable "query in, relevant documents out" contract — built here by hand so that Phase 4's fancier retrievers (MMR, Multi-Query, Contextual Compression) will read as extensions of this same idea rather than unrelated new concepts.

What has been completed:

  • contract_chunks and contract_chats tables.
  • Idempotent, lazy per-contract indexing reusing Phase 1's splitting/embedding modules.
  • A custom ContractRetriever scoped by contract_id.
  • Persisted, reconstructed conversational memory via BufferWindowMemory.
  • A full conversational RAG chain: condense → retrieve → answer, wired into POST /api/contracts/:id/chat.

What is left:

  • Running the hands-on tasks against real uploaded contracts and validating multi-turn behavior (tasks 10-13 — yours to execute).
  • Access control on the chat endpoint once your project adds authentication (noted above as a required follow-up, out of scope for this phase per your roadmap).

How this connects to the next phase: Phase 3 (Clause Extractor) moves past conversational Q&A into structured, exhaustive extraction — turning the same contracts.raw_text into a complete, typed table of every clause in the document, using withStructuredOutput() + Zod (already familiar from Phase 2) and, for the first time, an actual LangGraph StateGraph (introducing units 0-2 of the LangGraph curriculum).
