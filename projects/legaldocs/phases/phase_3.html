LegalDocs — Phase 3: Clause Extractor

#LegalDocs — Phase 3: Clause Extractor

Tech stack: Node.js / TypeScript, OpenAI, PostgreSQL (from Phase 0), LangChain.js (@langchain/core, @langchain/openai), LangGraph.js (@langchain/langgraph) — first use in the project, Zod Depends on: Phase 2 (the contracts table and its raw_text/contract_type columns) Feeds into: Phase 4 (the Risk Analyzer's Send()-based parallel clause loop runs one risk-scoring pass per row this phase writes to the clauses table); Phase 4.5 (the Legal Research Agent is invoked per HIGH-risk clause, which only exists because this phase named and isolated it first)

#1. Phase Overview

#Objective

Take a contract's full raw text (already stored by Phase 2) and produce a complete, typed, exhaustive list of every distinct clause inside it — each one tagged with a clause type, given a short summary, and stored as its own row in a new clauses table. Where Phase 2 answered "what kind of contract is this, broadly," this phase answers "what are the individual, addressable pieces this contract is made of."

This phase also does something structurally new: because a real contract can run well past what a single LLM call can reliably reason over in one pass, extraction has to run over a sliding window of overlapping text chunks, and the results from those overlapping windows have to be merged and de-duplicated afterward. That two-step shape — extract, then reconcile — is a natural fit for a small, explicit LangGraph graph rather than a single linear LCEL chain, and this is the first phase where you build one.

#Why this phase is important

Phase 4's entire Risk Analyzer is built around the idea of scoring individual clauses, not "the contract" as one blob — the roadmap's Send()-based fan-out explicitly needs a list of discrete clauses to fan out over. If this phase produces messy output (a "clause" that's actually three clauses concatenated, or the same clause duplicated three times because it happened to land in two overlapping windows), Phase 4's parallel risk-scoring will silently inherit that mess: duplicate risk findings, or one finding that's actually a muddled average of three unrelated obligations. Clean, deduplicated, well-typed clauses here are what make everything downstream — risk scoring, the audit trail, the report generator — trustworthy.

This is also the phase where you first confront a real limitation of a single chain call: contract length. A 3-page NDA fits in one LLM call with room to spare. A 40-page vendor master services agreement does not — not because it exceeds the model's raw context window necessarily, but because asking a model to exhaustively enumerate every clause in a huge block of text in one shot degrades in reliability well before you hit a hard token ceiling. The sliding-window approach you'll build here is the standard, practical answer to that problem, and it's a pattern you'll see again (in a more powerful form) with Phase 4's Send()-based fan-out.

#2. Concept Learning

#2.1 Why LangGraph, when you already have LCEL chains

Every chain you built in Phase 2 (preprocess.pipe(parallelStage).pipe(withTypeSpecific).pipe(mergeStep)) is, structurally, a fixed pipeline decided at compose time. You wire the steps together once in code, and every invocation walks that exact same wiring: same steps, same order, no memory of anything beyond what's explicitly piped forward. That's a great fit for "run these known steps once, in this order" — which is exactly what Phase 2 needed.

This phase's problem doesn't quite fit that shape as cleanly, for two reasons that will only get more pronounced from here on:

  1. There's a genuine two-stage process with different concerns at each stage — first extract (possibly messy, possibly duplicated) candidate clauses from each window, then reconcile them into a clean final list. You could still express this as extractStep.pipe(mergeStep) in pure LCEL, and for this phase specifically, that would even work. But —
  2. This is a taste of a pattern that stops being optional very soon. Phase 4's risk analysis needs conditional branching (route HIGH-risk clauses to a research agent, everything else straight to storage), checkpointed state you can inspect and rewind (the audit/undo feature), and parallel fan-out over a dynamic list of clauses whose size you don't know until runtime. None of that is naturally expressible as a .pipe() chain — LCEL has no concept of "shared state multiple steps read and write," no built-in checkpointing, and no conditional-edge routing primitive of its own.

LangGraph's core idea: instead of composing runnables into a fixed pipe, you define an explicit state object (a schema describing everything the graph will track), a set of nodes (each is a function that reads some of that state and returns updates to it), and edges connecting nodes together (which can be fixed, or conditional on the current state). You compile the graph once, and the compiled graph itself becomes a runnable you can .invoke() — but internally, it's walking a graph of nodes reading and writing shared state, not a straight pipe.

Concretely, the difference for this phase:

LCEL chain LangGraph StateGraph
What's passed between steps Whatever the previous step returns, and nothing else, unless you explicitly thread it forward (like Phase 2's RunnablePassthrough) A shared, named state object every node can read from and write partial updates to
Adding a third step later Re-wire the .pipe() chain Add a node, add an edge — the rest of the graph is untouched
Conditional branching RunnableBranch (still fairly linear, decided per-invocation) Native conditional edges — a node's output can determine which node runs next, and this scales to much more complex routing than RunnableBranch comfortably expresses
Memory of past runs / resumability None built in Checkpointers (Phase 4) let you pause, inspect, and resume a graph run

This phase intentionally builds the smallest possible real graph — two nodes, one edge, no branching, no checkpointing yet — specifically so those concepts land as a small, legible first example before Phase 4 asks you to build something considerably larger on the same foundation.

#2.2 State & Schema

A StateGraph needs to know, up front, the shape of the data every node will read and write. In LangGraph.js, you define this with Annotation.Root(...), which is essentially a typed schema declaration — similar in spirit to your Zod schemas from Phase 2, but describing the graph's shared memory rather than an LLM's output shape.

ts
const ClauseExtractionState = Annotation.Root({
  contractId: Annotation<string>(),
  rawText: Annotation<string>(),
  windows: Annotation<ContractWindow[]>(),
  rawExtractedClauses: Annotation<ExtractedClause[]>({
    reducer: (existing, incoming) => existing.concat(incoming),
    default: () => [],
  }),
  mergedClauses: Annotation<ExtractedClause[]>({
    reducer: (_existing, incoming) => incoming,
    default: () => [],
  }),
});

Two things are worth understanding carefully here, because they're easy to misread as boilerplate:

Every field needs a type, same reasoning as Zod in Phase 2 — the graph needs to know what shape of data flows through it so it can validate updates and give you type safety in TypeScript.

The reducer function is the actual new concept. When a node returns a partial state update like { rawExtractedClauses: [...] }, the graph doesn't just overwrite the existing value — it calls that field's reducer with (currentValue, newValue) and stores whatever the reducer returns. The default reducer (if you don't specify one) is "replace" — the new value wins outright, which is what you want for mergedClauses (the merge step's job is to produce one final authoritative list, not append to something). But rawExtractedClauses uses a concatenating reducer, because as you'll see in 2.4, the extraction node processes multiple windows and each window's results should accumulate into one combined list, not overwrite each other.

This is the first real payoff of "shared state" over "piped output": two different nodes (or the same node called multiple times conceptually) can contribute to the same field of state, and the reducer decides how those contributions combine — something a linear .pipe() chain has no clean way to express at all.

#2.3 Nodes & Edges

A node is just a function: (state) => Promise<Partial<State>>. It reads whatever fields of state it needs and returns an object containing only the fields it wants to update — it does not need to return the entire state back. This partial-update convention is what makes the reducer system in 2.2 meaningful: the graph merges your partial return into the full state using each field's reducer.

ts
async function extractClausesNode(
  state: typeof ClauseExtractionState.State
): Promise<Partial<typeof ClauseExtractionState.State>> {
  // ... does work using state.windows ...
  return { rawExtractedClauses: allExtracted };
}

An edge connects one node's completion to the next node's start. The simplest edges are fixed: "after node A finishes, always run node B." LangGraph provides two special pseudo-nodes, START and END, marking where a graph run begins and terminates — every graph needs at least one edge from START and at least one path that reaches END.

ts
const graph = new StateGraph(ClauseExtractionState)
  .addNode("extract_clauses", extractClausesNode)
  .addNode("validate_and_merge", validateAndMergeNode)
  .addEdge(START, "extract_clauses")
  .addEdge("extract_clauses", "validate_and_merge")
  .addEdge("validate_and_merge", END);

const compiledGraph = graph.compile();

compile() validates the graph structure (every node reachable, no dangling edges) and returns a runnable object with the familiar .invoke() / .stream() interface — from the outside, calling a compiled graph looks exactly like calling any other LangChain runnable, which is intentional: it means a graph can itself be embedded as one step inside a larger chain or an even bigger graph later (this is exactly what Phase 5's subgraphs and Phase 6.5's multi-agent supervisor will do).

This phase's graph is deliberately linear (START → extract_clauses → validate_and_merge → END, no branches, no loops) — the branching and looping constructs (conditional edges, Send()) are introduced in Phase 4 once you have a second, more complex graph to justify them.

#2.4 The sliding-window problem, and why it needs two distinct steps

Ask an LLM to "list every clause in this contract" for a 3-page NDA, and it does a genuinely solid job in one call — the whole document fits comfortably in its attention at once. Ask the same question of a 40-page master services agreement, and quality degrades well before you hit any hard context-length error: models get less exhaustive and less precise the more they're asked to hold in view and enumerate at once. The practical fix is the same one Phase 1 used for embeddings, applied to extraction instead: break the document into overlapping windows, run extraction independently on each window, and then reconcile the results.

Why overlapping, not just non-overlapping chunks (as in Phase 1's plain text splitter)? A clause can start near the very end of one window and finish at the start of the next. A non-overlapping split would hand the model half a clause in each of two calls, and neither call would report a complete, correct clause. Overlap (e.g. the last ~15% of window N is repeated as the first ~15% of window N+1) means any clause that happened to straddle a boundary is very likely to appear whole in at least one of the two windows.

The cost of overlap: any clause sitting in the overlapping region will very likely be extracted twice — once by each window that contains it. This is an acceptable, expected cost, not a bug to prevent at the windowing stage. It's exactly why this phase is naturally two steps rather than one:

  1. Extract — run structured extraction independently over every window (some duplication expected, extraction quality is what matters here, not final list length).
  2. Validate and merge — reconcile the (probably duplicated, possibly inconsistent) raw results into one clean, deduplicated final list.

This is precisely the extract_clauses → validate_and_merge two-node graph the roadmap specifies, and it's a materially better fit for LangGraph's shared-state model than for a single LCEL chain: node 1 accumulates raw results from N independent window calls into one field (via the concatenating reducer from 2.2), and node 2 reads that accumulated field, reasons over the whole thing at once, and replaces it with a clean final value in a different field.

#2.5 Token counting and window sizing

To decide where window boundaries go, you need a reasonably accurate way to estimate how many tokens a piece of text will consume — character count alone is a rough proxy (English text averages roughly 4 characters per token, but this varies with formatting, punctuation density, and non-English text). For a project targeting Indian legal documents specifically (some of which mix English with transliterated terms or dense cross-references), a real tokenizer estimate is worth the small dependency cost over a pure character-count heuristic.

js-tiktoken gives you the actual tokenizer OpenAI's models use, so you can count tokens precisely rather than guessing:

ts
import { encodingForModel } from "js-tiktoken";
const enc = encodingForModel("gpt-4o-mini");
const tokenCount = enc.encode(someText).length;

Choosing window size and overlap: this phase uses a 6,000-token window with a 500-token overlap by default. The window size is chosen well under typical model context limits — leaving generous headroom for the system prompt, the schema description injected by withStructuredOutput(), and the model's own output tokens (a contract with unusually dense clause structure could need a large output). The overlap (roughly 8% of the window) is enough to reliably catch clauses straddling a boundary without doubling your LLM call volume the way a 50% overlap would.

A contract under the 6,000-token threshold gets exactly one window — the sliding-window machinery degrades gracefully to "one call, no merging needed" for short contracts, rather than being a special case you have to branch around.

#2.6 Deduplication and merging strategy

Once you have raw extracted clauses from every window, some are genuinely unique and some are near-duplicates of a clause from an adjacent, overlapping window. Two clauses are considered "the same clause" for merging purposes if they share the same clauseType and their fullText values are highly similar — not necessarily identical, since the model might phrase a summary slightly differently between two calls even when quoting the same underlying text.

This phase uses a pragmatic, LLM-free similarity check for merging (token-set overlap on normalized clause text — fast, deterministic, and free), reserving an LLM call only for genuinely ambiguous cases. This keeps the merge step cheap and fast for the common case (obvious duplicates from window overlap) while still handling subtler cases correctly.

#3. Different Approaches

#3.1 Single-pass vs. sliding-window extraction

Approach Pros Cons When to use
A. Always single-pass — one withStructuredOutput() call over the entire raw_text Simplest possible code; no merge step needed Reliability degrades on long contracts well before any hard token limit; risks silently missing clauses on large documents with no signal that anything was dropped Short contracts only (NDAs, simple one-page agreements)
B. Always sliding-window, regardless of length One consistent code path for every contract; no special-casing Adds an unnecessary merge step and slightly higher LLM cost for contracts that would have fit in one call anyway Simpler to reason about if you don't mind the small overhead on short contracts
C. Adaptive — single window if the contract is under the token threshold, sliding windows otherwise (roadmap's approach) No wasted merge-step cost for the (very common) short-contract case; scales correctly for the 40-page case explicitly mentioned in your roadmap Two code paths to test (though the "windowing" function collapsing to one window is the same code path under the hood, not truly a branch) Recommended — the "one window" case is just createSlidingWindows() naturally returning an array of length 1, so this isn't really extra complexity, just correct behavior at both ends of the size spectrum

Recommendation: Approach C. Implement createSlidingWindows() to always run, and let it return a single window for anything under the threshold — this avoids if (isLongContract) {...} else {...} branching in the orchestration code entirely, since the windowing function already produces the right output shape either way.

#3.2 Orchestration: LCEL chain vs. LangGraph StateGraph

Approach Pros Cons When to use
A. Pure LCEL — extractStep.pipe(mergeStep), using RunnableLambda for both Fewer new concepts; you already know this pattern from Phase 2 No shared-state model for accumulating results across an unknown number of window calls (you'd manage that manually inside the RunnableLambda, which works but hides the accumulation logic from the graph/tracing layer); doesn't establish the LangGraph foundation Phase 4 needs Would work fine for this phase alone, in isolation
B. LangGraph StateGraph, 2 nodes (roadmap's choice) Explicit, typed, traceable shared state; the accumulating-reducer pattern (2.2) is a natural fit for "N window extractions feeding into one list"; establishes the exact primitives (state, nodes, edges, compile) Phase 4 builds on immediately One new dependency (@langchain/langgraph) and a handful of new concepts to learn in one sitting Recommended — matches your roadmap exactly, and this phase is deliberately sized to be a gentle first StateGraph before Phase 4's much larger one

Recommendation: Approach B, as specified. This phase is explicitly your roadmap's "first tiny graph" — small enough to fully understand in one sitting, real enough that Phase 4 isn't starting from zero.

#3.3 Deduplication strategy

Approach Pros Cons When to use
A. LLM-based merge only — hand the entire raw list to one more LLM call and ask it to deduplicate and consolidate Handles subtle rephrasing differences well; simple to describe Expensive and slow for contracts with many clauses (the raw list, including duplicates, could be large); another point of possible LLM inconsistency; nondeterministic Fine for small contracts, weaker choice as a general default
B. Pure algorithmic dedup only — normalize + token-set similarity threshold, no LLM involvement Fast, free, fully deterministic, easy to unit test Can miss duplicates where the model phrased the same clause meaningfully differently across two window calls, or wrongly merge two distinct short clauses that happen to share a lot of boilerplate language Good for the common case, weak on genuinely ambiguous edge cases
C. Algorithmic dedup first, LLM tiebreak only for borderline-similarity pairs (roadmap-aligned, recommended) Cheap and fast for the large majority of duplicates (near-identical text from window overlap); reserves the more expensive/careful LLM judgment only for cases genuinely worth it Slightly more code (two similarity thresholds: an "obviously the same" cutoff and an "obviously different" cutoff, with an LLM call only in between) Recommended — best balance of cost, speed, and correctness for a merge step that needs to run on every contract

Recommendation: Approach C. Two thresholds on a normalized token-overlap similarity score: above the high threshold, merge automatically; below the low threshold, treat as distinct automatically; only the narrow middle band pays for an LLM call.

#4. Detailed Implementation Plan

#Step 1 — Database schema for clauses

What: A clauses table, one row per final (post-merge) clause, plus a content_hash-based idempotency guard on the parent contract so re-running extraction on an already-processed contract doesn't create duplicate rows.

Why: This is the table Phase 4 fans its Send()-based parallel risk-scoring out over — one risk-scoring execution per row here.

Best practice: Store window_indices (which window(s) this clause was found in) as jsonb — genuinely useful for debugging "why did the merge step treat these as the same clause" months from now.

Common mistake to avoid: Making clause_type a Postgres ENUM. Exactly like Phase 2's contract_type, keep it free text — new clause types will come up as you process more real contracts, and a text column means that's a data change, not a migration.

#Step 2 — Token-aware sliding window module

What: createSlidingWindows(text, maxTokens, overlapTokens) — splits raw_text into an array of overlapping windows using paragraph boundaries where possible (never splitting a paragraph mid-sentence if it can be avoided), each under maxTokens.

Why: This is the direct fix for the reliability problem described in Concept Learning 2.4 and 2.5.

Best practice: Split on paragraph boundaries (\n\n) first, falling back to sentence boundaries only if a single paragraph itself exceeds the window size (rare, but a contract can have one unusually long defined-terms paragraph) — this keeps each window's content coherent rather than cutting arbitrarily by character count.

Common mistake to avoid: Computing overlap in characters instead of tokens. Since the whole point of windowing is respecting a token budget, sizing the overlap the same way keeps your window-size math internally consistent.

#Step 3 — Zod schema for a single extracted clause

What: ExtractedClauseSchema — the structured shape a single extraction call should return for one window (an array of these).

Why: Same reasoning as Phase 2 — one schema serves as both the LLM's output contract (via withStructuredOutput) and your TypeScript type.

Best practice: Require fullText to be the verbatim clause text (not a paraphrase) — you need the actual original wording for the merge step's similarity comparison, and later, for a compliance reviewer in Phase 4 to trust that "here's the risky clause" quotes the real contract, not the model's rendition of it.

#Step 4 — The extract_clauses node

What: A LangGraph node function that, given state.windows, runs a withStructuredOutput() extraction call concurrently across every window (using Promise.all, the same "why serialize independent work" reasoning as Phase 2's RunnableParallel — just expressed directly here since a graph node is a plain async function, not a composed runnable), and returns the combined raw results.

Why: Windows are fully independent of each other — nothing about extracting clauses from window 3 depends on what window 1 returned — so there's no reason to await them one at a time.

Best practice: Tag every extracted clause with its source windowIndex at extraction time, before anything gets merged — exactly the same "attach metadata now, not later" lesson from Phase 1's chunk indexing.

#Step 5 — The validate_and_merge node

What: A node that takes state.rawExtractedClauses, groups candidates by clauseType, computes pairwise text similarity within each group, merges pairs above the high threshold automatically, asks an LLM to adjudicate pairs in the ambiguous middle band, and returns the final deduplicated list.

Why: This is the reconciliation step from Concept Learning 2.4 and 2.6.

Common mistake to avoid: Comparing every clause against every other clause across the entire contract (O(n²) over all clauses). Grouping by clauseType first narrows comparisons to only clauses that could plausibly be duplicates of each other, which matters once a large contract produces 40+ raw candidate clauses.

#Step 6 — Graph construction and compilation

What: Wire extract_clauses and validate_and_merge into a StateGraph, with the state schema from Concept Learning 2.2, and .compile() it once at module load (not per-request) since a compiled graph has no per-request state of its own.

Why: Compiling once and reusing the compiled graph across requests is both correct (the graph structure doesn't change) and efficient (compilation does real validation work you don't want to repeat every request).

#Step 7 — Storage layer

What: Functions to check whether a contract already has extracted clauses (idempotency, same pattern as Phase 1/2), and to bulk-insert the final merged clause list in one transaction.

Why: Same reasoning as every prior phase — re-running extraction on an unchanged contract should be a cheap no-op, and a crash mid-insert shouldn't leave a half-populated clauses table.

#Step 8 — The extraction endpoint

What: POST /api/contracts/:id/clauses/extract — loads the contract's raw_text, checks idempotency, invokes the compiled graph, stores the result, returns the clause list. A companion GET /api/contracts/:id/clauses for simple retrieval.

Best practice: Return the count of clauses found and how many raw candidates were merged away (rawCount - mergedCount) in the response — a useful sanity signal for you while validating extraction quality against real contracts, and free to compute since you already have both numbers in hand.

#6.4 Token-aware windowing

ts
// src/clauses/tokenizer.ts
import { encodingForModel, type TiktokenModel } from "js-tiktoken";

const MODEL: TiktokenModel = "gpt-4o-mini" as TiktokenModel;
const encoder = encodingForModel(MODEL);

/** Returns an accurate token count for a piece of text, per the actual
 *  tokenizer the extraction model uses - a real count, not a char/4 guess. */
export function countTokens(text: string): number {
  return encoder.encode(text).length;
}
ts
// src/clauses/windowing.ts
import { countTokens } from "./tokenizer";

export interface ContractWindow {
  windowIndex: number;
  text: string;
  tokenCount: number;
}

const MAX_TOKENS = Number(process.env.CLAUSE_WINDOW_MAX_TOKENS ?? 6000);
const OVERLAP_TOKENS = Number(process.env.CLAUSE_WINDOW_OVERLAP_TOKENS ?? 500);

/**
 * Splits contract text into overlapping windows sized in tokens (not
 * characters), so the token-budget math stays internally consistent.
 * Splits on paragraph boundaries first to keep each window's content
 * coherent; only falls back to sentence-level splitting if a single
 * paragraph alone exceeds the window size.
 *
 * A contract under MAX_TOKENS produces exactly one window - the same
 * function handles short and long contracts without a special case.
 */
export function createSlidingWindows(fullText: string): ContractWindow[] {
  const paragraphs = fullText.split(/\n\s*\n/).filter((p) => p.trim().length > 0);

  // Break any single paragraph that alone exceeds MAX_TOKENS into sentences,
  // so no individual unit we place into a window can overflow it by itself.
  const units: string[] = [];
  for (const para of paragraphs) {
    if (countTokens(para) <= MAX_TOKENS) {
      units.push(para);
    } else {
      const sentences = para.split(/(?<=[.!?])\s+/);
      units.push(...sentences);
    }
  }

  const windows: ContractWindow[] = [];
  let current: string[] = [];
  let currentTokens = 0;

  const flush = () => {
    if (current.length === 0) return;
    const text = current.join("\n\n");
    windows.push({ windowIndex: windows.length, text, tokenCount: countTokens(text) });
  };

  for (const unit of units) {
    const unitTokens = countTokens(unit);

    if (currentTokens + unitTokens > MAX_TOKENS && current.length > 0) {
      flush();

      // Build the overlap tail for the next window: walk backward through
      // the just-flushed units until we've collected ~OVERLAP_TOKENS worth.
      let overlapTokens = 0;
      const overlapUnits: string[] = [];
      for (let i = current.length - 1; i >= 0 && overlapTokens < OVERLAP_TOKENS; i--) {
        overlapUnits.unshift(current[i]);
        overlapTokens += countTokens(current[i]);
      }

      current = overlapUnits;
      currentTokens = overlapTokens;
    }

    current.push(unit);
    currentTokens += unitTokens;
  }

  flush();

  return windows;
}

#6.6 Similarity scoring for merge

ts
// src/clauses/similarity.ts

/** Normalizes text for comparison: lowercase, collapse whitespace, strip
 *  punctuation - so trivial formatting differences don't affect similarity. */
function normalize(text: string): Set<string> {
  const cleaned = text
    .toLowerCase()
    .replace(/[^\w\s]/g, " ")
    .replace(/\s+/g, " ")
    .trim();
  return new Set(cleaned.split(" "));
}

/**
 * Jaccard similarity over word sets: |intersection| / |union|. Cheap,
 * deterministic, and free - used to catch the common case of near-identical
 * clause text extracted twice from overlapping windows, without an LLM call.
 */
export function textSimilarity(a: string, b: string): number {
  const setA = normalize(a);
  const setB = normalize(b);

  let intersection = 0;
  for (const word of setA) {
    if (setB.has(word)) intersection++;
  }
  const union = setA.size + setB.size - intersection;

  return union === 0 ? 0 : intersection / union;
}

#6.7 Graph state schema

ts
// src/clauses/state.ts
import { Annotation } from "@langchain/langgraph";
import type { ContractWindow } from "./windowing";
import type { TaggedClause } from "./schemas";

export const ClauseExtractionState = Annotation.Root({
  contractId: Annotation<string>(),
  contractType: Annotation<string>(),
  windows: Annotation<ContractWindow[]>(),

  // Concatenating reducer: the extraction node processes every window and
  // each window's results should ACCUMULATE into this field, not overwrite
  // each other - this is what lets one node's async fan-out over N windows
  // land as a single combined list in shared state.
  rawExtractedClauses: Annotation<TaggedClause[]>({
    reducer: (existing, incoming) => existing.concat(incoming),
    default: () => [],
  }),

  // Replace reducer (the default): the merge step produces one final,
  // authoritative list - it should not be appended to anything prior.
  mergedClauses: Annotation<TaggedClause[]>({
    reducer: (_existing, incoming) => incoming,
    default: () => [],
  }),
});

export type ClauseExtractionStateType = typeof ClauseExtractionState.State;

#6.8 Node 1 — extract_clauses

ts
// src/clauses/nodes/extractClauses.ts
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { WindowExtractionSchema } from "../schemas";
import type { ClauseExtractionStateType } from "../state";
import type { TaggedClause } from "../schemas";

const model = new ChatOpenAI({
  model: process.env.CLAUSE_EXTRACTION_MODEL ?? "gpt-4o-mini",
  temperature: 0,
});

const extractionPrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    "You are a contract analysis expert. Read the contract text window below " +
      "and identify every distinct clause it contains. This window may be a " +
      "partial excerpt of a larger contract - only report clauses that are " +
      "substantially present in THIS window. Quote clause text verbatim; do " +
      "not paraphrase in the fullText field. This is contract type: {contractType}.",
  ],
  ["human", "Contract text window:\n\n{windowText}"],
]);

const extractionChain = extractionPrompt.pipe(
  model.withStructuredOutput(WindowExtractionSchema, { name: "extract_window_clauses" })
);

/**
 * Node 1: runs structured extraction concurrently across every window
 * (windows are fully independent of each other, so there's no reason to
 * await them sequentially - the same "parallelize independent work"
 * reasoning as Phase 2's RunnableParallel, expressed directly here since a
 * graph node is a plain async function rather than a composed runnable).
 *
 * Duplication across overlapping windows is expected and NOT handled here -
 * that is entirely the validate_and_merge node's job.
 */
export async function extractClausesNode(
  state: ClauseExtractionStateType
): Promise<Partial<ClauseExtractionStateType>> {
  const perWindowResults = await Promise.all(
    state.windows.map(async (window) => {
      const result = await extractionChain.invoke({
        windowText: window.text,
        contractType: state.contractType,
      });

      const tagged: TaggedClause[] = result.clauses.map((clause) => ({
        ...clause,
        windowIndices: [window.windowIndex],
      }));

      return tagged;
    })
  );

  return { rawExtractedClauses: perWindowResults.flat() };
}

#6.9 Node 2 — validate_and_merge

ts
// src/clauses/nodes/validateAndMerge.ts
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { z } from "zod";
import { textSimilarity } from "../similarity";
import type { ClauseExtractionStateType } from "../state";
import type { TaggedClause } from "../schemas";

const HIGH_SIMILARITY = Number(process.env.CLAUSE_MERGE_HIGH_SIMILARITY ?? 0.85);
const LOW_SIMILARITY = Number(process.env.CLAUSE_MERGE_LOW_SIMILARITY ?? 0.4);

const model = new ChatOpenAI({
  model: process.env.CLAUSE_EXTRACTION_MODEL ?? "gpt-4o-mini",
  temperature: 0,
});

const TiebreakSchema = z.object({
  sameClause: z.boolean().describe("True if both excerpts describe the same underlying clause"),
});

const tiebreakPrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    "Two clause excerpts were extracted from overlapping windows of the same " +
      "contract. Determine whether they describe the SAME underlying clause " +
      "(even if worded slightly differently) or two DIFFERENT clauses.",
  ],
  ["human", "Excerpt A:\n{textA}\n\nExcerpt B:\n{textB}"],
]);

const tiebreakChain = tiebreakPrompt.pipe(
  model.withStructuredOutput(TiebreakSchema, { name: "clause_tiebreak" })
);

function mergeTwo(a: TaggedClause, b: TaggedClause): TaggedClause {
  // Keep the longer, more complete fullText; union obligations and window indices.
  const primary = a.fullText.length >= b.fullText.length ? a : b;
  return {
    ...primary,
    obligations: Array.from(new Set([...a.obligations, ...b.obligations])),
    sectionReference: a.sectionReference ?? b.sectionReference,
    windowIndices: Array.from(new Set([...a.windowIndices, ...b.windowIndices])),
  };
}

/**
 * Node 2: groups raw candidates by clauseType (narrowing comparisons to
 * only clauses that could plausibly be duplicates), then merges pairs
 * whose text similarity clears HIGH_SIMILARITY automatically, treats pairs
 * below LOW_SIMILARITY as definitely distinct automatically, and only pays
 * for an LLM tiebreak call in the narrow ambiguous band between the two.
 */
export async function validateAndMergeNode(
  state: ClauseExtractionStateType
): Promise<Partial<ClauseExtractionStateType>> {
  const byType = new Map<string, TaggedClause[]>();
  for (const clause of state.rawExtractedClauses) {
    const list = byType.get(clause.clauseType) ?? [];
    list.push(clause);
    byType.set(clause.clauseType, list);
  }

  const merged: TaggedClause[] = [];

  for (const candidates of byType.values()) {
    const consumed = new Set<number>();

    for (let i = 0; i < candidates.length; i++) {
      if (consumed.has(i)) continue;
      let current = candidates[i];

      for (let j = i + 1; j < candidates.length; j++) {
        if (consumed.has(j)) continue;

        const sim = textSimilarity(current.fullText, candidates[j].fullText);

        let isSameClause: boolean;
        if (sim >= HIGH_SIMILARITY) {
          isSameClause = true;
        } else if (sim <= LOW_SIMILARITY) {
          isSameClause = false;
        } else {
          // Ambiguous band - the only place an LLM call is spent.
          const verdict = await tiebreakChain.invoke({
            textA: current.fullText,
            textB: candidates[j].fullText,
          });
          isSameClause = verdict.sameClause;
        }

        if (isSameClause) {
          current = mergeTwo(current, candidates[j]);
          consumed.add(j);
        }
      }

      merged.push(current);
    }
  }

  return { mergedClauses: merged };
}

#6.10 Graph construction

ts
// src/clauses/graph.ts
import { StateGraph, START, END } from "@langchain/langgraph";
import { ClauseExtractionState } from "./state";
import { extractClausesNode } from "./nodes/extractClauses";
import { validateAndMergeNode } from "./nodes/validateAndMerge";

/**
 * The full Phase 3 pipeline as a compiled 2-node graph:
 *   START -> extract_clauses -> validate_and_merge -> END
 *
 * Compiled once at module load (a compiled graph carries no per-request
 * state of its own, so there is no reason to rebuild it per invocation -
 * the same reasoning as reusing a single ChatOpenAI client instance).
 */
export const clauseExtractionGraph = new StateGraph(ClauseExtractionState)
  .addNode("extract_clauses", extractClausesNode)
  .addNode("validate_and_merge", validateAndMergeNode)
  .addEdge(START, "extract_clauses")
  .addEdge("extract_clauses", "validate_and_merge")
  .addEdge("validate_and_merge", END)
  .compile();

#6.11 Storage layer

ts
// src/clauses/storage.ts
import { Pool } from "pg";
import format from "pg-format";
import type { TaggedClause } from "./schemas";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

export async function hasExtractedClauses(contractId: string): Promise<boolean> {
  const { rows } = await pool.query(`SELECT 1 FROM clauses WHERE contract_id = $1 LIMIT 1`, [
    contractId,
  ]);
  return rows.length > 0;
}

export async function getClausesForContract(contractId: string) {
  const { rows } = await pool.query(
    `SELECT * FROM clauses WHERE contract_id = $1 ORDER BY created_at ASC`,
    [contractId]
  );
  return rows;
}

/**
 * Replaces all clauses for a contract in a single transaction - a crash
 * mid-insert rolls back cleanly rather than leaving a half-populated table,
 * same pattern as Phase 1's document-chunk storage.
 */
export async function storeClauses(
  contractId: string,
  clauses: TaggedClause[]
): Promise<void> {
  const client = await pool.connect();
  try {
    await client.query("BEGIN");

    await client.query(`DELETE FROM clauses WHERE contract_id = $1`, [contractId]);

    if (clauses.length > 0) {
      const values = clauses.map((c) => [
        contractId,
        c.clauseType,
        c.title,
        c.fullText,
        c.summary,
        JSON.stringify(c.obligations),
        c.sectionReference,
        JSON.stringify(c.windowIndices),
      ]);

      const insertSql = format(
        `INSERT INTO clauses
           (contract_id, clause_type, title, full_text, summary, obligations, section_reference, window_indices)
         VALUES %L`,
        values
      );
      await client.query(insertSql);
    }

    await client.query("COMMIT");
  } catch (err) {
    await client.query("ROLLBACK");
    throw err;
  } finally {
    client.release();
  }
}

#6.12 Route

ts
// src/routes/clauses.ts
import { Router } from "express";
import { Pool } from "pg";
import { createSlidingWindows } from "../clauses/windowing";
import { clauseExtractionGraph } from "../clauses/graph";
import { hasExtractedClauses, getClausesForContract, storeClauses } from "../clauses/storage";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });
export const clausesRouter = Router();

clausesRouter.post("/:id/clauses/extract", async (req, res) => {
  const { id } = req.params;
  const force = req.query.force === "true";

  const { rows } = await pool.query(
    `SELECT raw_text, contract_type FROM contracts WHERE id = $1`,
    [id]
  );
  if (rows.length === 0) {
    return res.status(404).json({ error: "Contract not found." });
  }

  if (!force && (await hasExtractedClauses(id))) {
    const existing = await getClausesForContract(id);
    return res.status(200).json({ reused: true, clauses: existing });
  }

  try {
    const { raw_text: rawText, contract_type: contractType } = rows[0];
    const windows = createSlidingWindows(rawText);

    const result = await clauseExtractionGraph.invoke({
      contractId: id,
      contractType,
      windows,
    });

    await storeClauses(id, result.mergedClauses);

    return res.status(201).json({
      reused: false,
      windowCount: windows.length,
      rawCandidateCount: result.rawExtractedClauses.length,
      mergedCount: result.mergedClauses.length,
      clauses: result.mergedClauses,
    });
  } catch (err) {
    console.error("Clause extraction failed:", err);
    return res.status(500).json({ error: "Failed to extract clauses." });
  }
});

clausesRouter.get("/:id/clauses", async (req, res) => {
  const clauses = await getClausesForContract(req.params.id);
  return res.status(200).json({ clauses });
});

#Why this design over alternatives

  • Windowing collapses to one window for short contracts, rather than branching: createSlidingWindows() always runs, and a contract under the token threshold simply produces an array of length 1 — the orchestration code (and the graph) never needs to know or care whether it's looking at a 3-page or 40-page contract.
  • The extraction node parallelizes independent window calls: exactly the same "no reason to serialize independent work" reasoning from Phase 2's RunnableParallel, just expressed as a plain Promise.all inside a graph node rather than a composed LCEL runnable — both are valid places for this pattern; which one you reach for depends on whether you're inside a linear chain or a graph node.
  • Grouping by clauseType before computing similarity: keeps the merge step's comparison cost proportional to "how many candidates share a type," not the square of the total candidate count — matters once a large contract produces 40+ raw candidates across windows.
  • The compiled graph is a module-level singleton, not rebuilt per request: a StateGraph.compile() call validates graph structure; doing that once at startup, rather than on every HTTP request, is both faster and correctly reflects that the graph's structure never changes between requests — only the state passed into .invoke() does.

#8. Best Practices

Industry best practices

  • Keep fullText verbatim, never paraphrased, in the extraction schema — every downstream feature (merge similarity, Phase 4's risk quoting, a future compliance reviewer) depends on trusting that field is the actual contract text.
  • Treat the compiled graph the same way you treat a ChatOpenAI client instance — construct once, reuse everywhere, never per-request.
  • Keep temperature: 0 for both the extraction and tiebreak chains — both are precise, low-creativity tasks where consistency matters far more than variety.

Scalability considerations

  • Token-aware windowing means extraction quality and latency scale sensibly with contract length, rather than degrading unpredictably past some undocumented length cliff.
  • The Promise.all fan-out in extract_clauses means a 10-window contract costs roughly the same wall-clock time as a 2-window one (bounded by your OpenAI rate limits, not by serial LLM round-trips) — worth revisiting with a concurrency cap (e.g. p-limit) once contract volume is high enough that many extractions could run simultaneously and collectively exceed your rate limit.

Security considerations

  • Same contract-confidentiality logging discipline as Phase 2: never log full clause text to application logs; log clause counts, types, and contract IDs only.
  • The extraction endpoint should eventually sit behind the same auth/access-control layer noted as a follow-up in Phase 2.5 — a clause list is at least as sensitive as the raw contract text it was derived from.

Performance optimizations

  • The high/low similarity threshold split (Approach C in 3.3) means the LLM tiebreak call — the most expensive part of the merge step — is only ever spent on genuinely ambiguous pairs, not on every duplicate pair from window overlap.
  • Grouping by clauseType before similarity comparison avoids O(n²) comparisons across the entire raw candidate list.

Maintainability tips

  • Adding a new clause type is a one-line addition to CLAUSE_TYPES — no other file needs to change, same pattern as Phase 2's CONTRACT_TYPES.
  • Keep the similarity thresholds (CLAUSE_MERGE_HIGH_SIMILARITY / CLAUSE_MERGE_LOW_SIMILARITY) as environment variables — you'll want to tune these once you've validated merge quality against real contracts with known duplicate clauses.

Testing considerations

  • Unit test createSlidingWindows() against a fixture text sized to require exactly 2 windows — assert both the overlap region's content and that no paragraph is split mid-sentence.
  • Unit test textSimilarity() with known near-duplicate and known-distinct clause text pairs — assert scores land clearly above/below your configured thresholds.
  • Integration test the full graph (clauseExtractionGraph.invoke(...)) against a small fixture contract with a deliberately duplicated clause across two synthetic windows — assert the final mergedClauses list contains exactly one merged entry, not two.
  • Integration test a short, single-window contract end to end — assert windowCount === 1 and that rawCandidateCount === mergedCount (nothing to merge).

#9. Summary

What we learned:

  • Why LangGraph's shared-state model earns its place once a pipeline needs multiple independent calls to accumulate into one place before a distinct reconciliation step runs — a job LCEL's linear .pipe() composition doesn't naturally express.
  • The Annotation.Root state schema, and specifically how per-field reducers decide whether a node's partial update replaces or accumulates into existing state — the concrete mechanic that makes "shared state" meaningful rather than just a bigger prop-drilling problem.
  • Nodes as plain async functions returning partial state updates, and edges (including the START/END markers) as the wiring between them — and that a compiled graph exposes the same .invoke() interface as any other LangChain runnable, which is what makes graphs embeddable inside larger chains or bigger graphs later.
  • Why long documents need token-aware sliding-window extraction rather than a single call, and why overlapping windows trade a predictable, acceptable duplication cost for the guarantee that boundary-straddling clauses are still captured whole somewhere.
  • A pragmatic, cost-aware deduplication strategy: cheap algorithmic similarity scoring for the obvious cases, an LLM call reserved only for the genuinely ambiguous middle band.

What has been completed:

  • clauses table schema.
  • Token-aware sliding-window text splitting with paragraph-preserving overlap.
  • Zod schemas for structured, verbatim clause extraction.
  • A compiled 2-node LangGraph graph (extract_clauses → validate_and_merge) implementing concurrent per-window extraction and similarity/LLM-tiebreak-based deduplication.
  • Idempotent, transactional storage into the clauses table.
  • POST /api/contracts/:id/clauses/extract and GET /api/contracts/:id/clauses endpoints, fully wired.

What is left:

  • Running the hands-on tasks against real short and long contracts and validating extraction/merge quality by manual spot-check (tasks 12-14 — yours to execute).
  • Everything from Phase 4 onward, which consumes the clauses table this phase populates.

How this connects to the next phase: Phase 4 (RAG Risk Analyzer) is the project's core phase, and it starts exactly where this one ends: one row per clause in the clauses table. Where this phase's graph was deliberately small and linear (two nodes, one path, no branching), Phase 4 builds a substantially larger graph on the same StateGraph/Annotation foundation — introducing conditional edges (branching by risk level), the Send() API (fanning out one parallel risk-scoring execution per clause row, directly analogous to this phase's Promise.all over windows but now expressed as a first-class graph construct), checkpointers (persisting graph runs so they can be inspected later), and time-travel state management (letting a compliance reviewer audit and revert a scoring decision). Everything you just learned about state, reducers, nodes, and edges is the prerequisite for all of it.
