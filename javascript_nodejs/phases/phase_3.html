#U12 — Design Patterns for JS

#Why this matters at senior level

Interviewers at 3+ YOE stop asking "what is a closure" and start asking "how would you structure this module" or "why did you pick this pattern." You need to recognize patterns in existing code and justify using them, not just recite definitions.

#1. Singleton Pattern

Ensures a class/module has exactly one instance, with a global access point.

js
class DBConnection {
  static #instance;

  constructor() {
    if (DBConnection.#instance) {
      return DBConnection.#instance;
    }
    this.connection = this.#connect();
    DBConnection.#instance = this;
  }

  #connect() {
    console.log('Establishing new DB connection...');
    return { status: 'connected' };
  }

  static getInstance() {
    if (!DBConnection.#instance) {
      DBConnection.#instance = new DBConnection();
    }
    return DBConnection.#instance;
  }
}

const a = DBConnection.getInstance();
const b = DBConnection.getInstance();
console.log(a === b); // true

Real backend use case: DB connection pools, config loaders, logger instances, Redis client wrappers — anything expensive to create that should be shared app-wide.

Node.js gotcha: require() caching already gives you singleton-like behavior for modules. If you module.exports = new Logger(), every require('./logger') returns the same instance. Interviewers love asking "how does Node give you singletons for free?" — answer: the module cache (require.cache), keyed by resolved file path.

Downsides to mention (shows maturity): Singletons introduce global state, make unit testing harder (need to reset state between tests), and can hide dependencies (implicit coupling instead of explicit DI).

#2. Factory Pattern

Delegates object creation logic to a function/class instead of using new directly, useful when creation logic is conditional or complex.

js
class EmailNotification {
  send(payload) { console.log('Sending email:', payload); }
}
class SmsNotification {
  send(payload) { console.log('Sending SMS:', payload); }
}
class PushNotification {
  send(payload) { console.log('Sending push:', payload); }
}

function notificationFactory(type) {
  switch (type) {
    case 'email': return new EmailNotification();
    case 'sms': return new SmsNotification();
    case 'push': return new PushNotification();
    default: throw new Error(`Unsupported notification type: ${type}`);
  }
}

const notifier = notificationFactory('sms');
notifier.send({ to: '+1234567890', msg: 'OTP: 4821' });

Real backend use case: Strategy selection based on config/env (payment gateway factory: Stripe vs Razorpay vs PayPal), DB driver selection (Postgres vs Mongo repository factory), logger transport selection.

#3. Observer / Pub-Sub Pattern

Decouples producers of events from consumers. Observer = subjects hold a direct reference to observers. Pub-Sub = a broker/event bus sits in between, so publishers and subscribers never know about each other.

js
class EventBus {
  #listeners = new Map();

  on(event, handler) {
    if (!this.#listeners.has(event)) this.#listeners.set(event, []);
    this.#listeners.get(event).push(handler);
    return () => this.off(event, handler); // unsubscribe fn
  }

  off(event, handler) {
    const handlers = this.#listeners.get(event) || [];
    this.#listeners.set(event, handlers.filter(h => h !== handler));
  }

  emit(event, payload) {
    (this.#listeners.get(event) || []).forEach(handler => handler(payload));
  }
}

const bus = new EventBus();
const unsubscribe = bus.on('order.created', (order) => {
  console.log('Send confirmation email for', order.id);
});
bus.on('order.created', (order) => {
  console.log('Update inventory for', order.id);
});

bus.emit('order.created', { id: 'ORD-101' });
unsubscribe();

Real backend use case: This is literally EventEmitter in Node core. Also underlies webhook dispatch systems, domain events in DDD-style architectures, and decoupling side effects (e.g., "on user signup, send welcome email + provision workspace + log analytics" without cramming all of it into the signup controller).

Interview trap: Be ready to explain the difference between Observer and Pub-Sub precisely — Observer is direct coupling (subject knows its observers), Pub-Sub has a mediator (channel/topic) so publisher and subscriber are fully decoupled and can even live in different processes (e.g., Redis Pub/Sub, Kafka).

#4. Module Pattern (IIFE-based encapsulation)

Predates ES modules; still shows up in legacy code and interview "explain this snippet" questions.

js
const CounterModule = (function () {
  let count = 0; // private, closed over

  function increment() { count += 1; return count; }
  function reset() { count = 0; }

  return { increment, reset, getCount: () => count };
})();

CounterModule.increment();
CounterModule.increment();
console.log(CounterModule.getCount()); // 2
console.log(CounterModule.count); // undefined — truly private

Modern equivalent: ES modules give you file-level privacy natively (anything not exported is private), and #privateFields give you class-level privacy — so know why the IIFE trick existed (pre-ES6 JS had no native privacy mechanism).

#U13 — Memory Management & Performance

#Garbage Collection (GC) fundamentals

Mark-and-sweep (the core algorithm V8 uses):

  1. GC starts from "roots" (global object, currently executing function's local variables, call stack).
  2. It marks every object reachable from those roots.
  3. Anything not marked is considered garbage and swept (memory reclaimed).

Key mental model: JS GC is reachability-based, not reference-counting-based (reference counting was an early approach but fails on circular references — two objects referencing each other but unreachable from root still get collected correctly under mark-and-sweep, unlike naive ref-counting).

Generational GC (V8 specifics — ties into U15 too):

  • New Space (young generation): small, where new objects are allocated. Collected frequently via a fast Scavenge algorithm (semi-space copying — two equal halves, live objects copied from one to the other on each GC cycle).
  • Old Space (old generation): objects that survive 2+ scavenge cycles get "promoted" here. Collected less frequently but more expensively via Mark-Sweep-Compact.
  • Rationale ("generational hypothesis"): most objects die young (short-lived request-scoped data, temp variables), so optimizing for fast collection of short-lived objects is a big win.

#Common memory leak sources in Node.js backends (high-frequency interview topic)

1. Forgotten timers/intervals

js
// LEAK: interval keeps a closure alive forever, and the interval itself is never cleared
function startPolling(cache) {
  setInterval(() => {
    cache.refresh(); // `cache` closure never released
  }, 5000);
}

Fix: store the interval id and clearInterval() on shutdown/cleanup; prefer libraries/patterns that tie timer lifecycle to a controller you can dispose of.

2. Detached event listeners

js
// LEAK: emitter grows a new listener every request, never removed
app.get('/status', (req, res) => {
  someLongLivedEmitter.on('data', (chunk) => res.write(chunk)); // never off()
});

Fix: always pair .on() with .off()/.removeListener(), or use .once() where appropriate. Node will actually warn you: MaxListenersExceededWarning — know this warning and what it means (default limit is 10 listeners per event on an EventEmitter).

3. Unbounded caches / module-level Maps

js
// LEAK: a plain object/Map used as a cache with no eviction policy
const requestCache = new Map();
app.use((req, res, next) => {
  requestCache.set(req.id, req.body); // never deleted
  next();
});

Fix: use TTL-based caches (e.g., lru-cache, Redis with expiry) instead of unbounded in-memory Maps.

4. Closures unintentionally retaining large objects

js
function processLargeFile(buffer) {
  const summary = buffer.slice(0, 100); // still references the underlying large buffer's memory in some cases
  return () => console.log(summary);
}

Fix: extract only the primitive/plain data you need out of large structures before closing over it, not the whole reference.

5. Global variables accidentally created (missing let/const/strict mode) Especially dangerous in older CommonJS codebases without 'use strict'.

#Diagnosing leaks (how a senior dev is expected to answer "how would you debug this in prod")

  1. Reproduce with node --inspect + Chrome DevTools Memory tab, or clinic.js / 0x.
  2. Take heap snapshots at two points in time, use the Comparison view to see which object types/allocations are growing unbounded.
  3. Check process.memoryUsage() (rss, heapUsed, heapTotal, external) exposed via a metrics endpoint or logged periodically in production.
  4. Look for detached DOM nodes equivalent in Node — i.e., large buffers/streams retained by forgotten listeners or unresolved promises.

#Debounce vs Throttle (always asked, always implemented live)

js
// Debounce: fire only after `wait` ms of silence since the last call.
// Use case: search-as-you-type, resize handlers.
function debounce(fn, wait) {
  let timeoutId;
  return function (...args) {
    clearTimeout(timeoutId);
    timeoutId = setTimeout(() => fn.apply(this, args), wait);
  };
}

// Throttle: fire at most once every `limit` ms, regardless of call frequency.
// Use case: scroll handlers, rate-limiting outbound API calls.
function throttle(fn, limit) {
  let inThrottle = false;
  return function (...args) {
    if (!inThrottle) {
      fn.apply(this, args);
      inThrottle = true;
      setTimeout(() => { inThrottle = false; }, limit);
    }
  };
}

Be ready to state the difference in one line: debounce delays until activity stops; throttle guarantees a max execution rate during continuous activity.

Know why Array.unshift/splice at index 0 is O(n): every subsequent element must be re-indexed/shifted in memory.

Know why Map/Set beat plain objects for frequent add/delete: Map is optimized for this and doesn't carry prototype chain lookup overhead or string-only key coercion.

#U14 — Generators, Iterators & Proxy/Reflect

#Iterator Protocol

An object is an iterator if it implements .next() returning { value, done }. An object is iterable if it implements Symbol.iterator, which returns an iterator. This is what powers for...of, spread syntax, and destructuring.

js
function createRangeIterator(start, end) {
  let current = start;
  return {
    next() {
      if (current < end) {
        return { value: current++, done: false };
      }
      return { value: undefined, done: true };
    },
  };
}

const it = createRangeIterator(1, 4);
console.log(it.next()); // { value: 1, done: false }
console.log(it.next()); // { value: 2, done: false }

Custom iterable class (common "implement your own iterable" interview task):

js
class Range {
  constructor(start, end) {
    this.start = start;
    this.end = end;
  }
  [Symbol.iterator]() {
    let current = this.start;
    const end = this.end;
    return {
      next() {
        return current < end
          ? { value: current++, done: false }
          : { value: undefined, done: true };
      },
    };
  }
}

for (const n of new Range(1, 5)) console.log(n); // 1 2 3 4
console.log([...new Range(1, 4)]); // [1, 2, 3]

#Generators

A generator function (function*) returns a generator object that is both an iterator and iterable. yield pauses execution and hands control back to the caller — this is the key mental model: generators give you lazy, pausable execution.

js
function* rangeGen(start, end) {
  for (let i = start; i < end; i++) {
    yield i;
  }
}

for (const n of rangeGen(1, 4)) console.log(n); // 1 2 3

// yield can also receive values passed into .next(argument)
function* echo() {
  while (true) {
    const received = yield;
    console.log('received:', received);
  }
}
const gen = echo();
gen.next();          // prime the generator (runs to first yield)
gen.next('hello');   // received: hello

Real backend use case — lazy paginated data fetcher:

js
async function* paginatedFetch(fetchPage) {
  let page = 1;
  while (true) {
    const { items, hasMore } = await fetchPage(page);
    yield* items; // delegate — yields each item individually
    if (!hasMore) return;
    page++;
  }
}

// Consumer only pulls what it needs, doesn't load the entire dataset upfront
for await (const item of paginatedFetch(fetchPageFromApi)) {
  console.log(item);
  if (item.id === 'stop-here') break; // remaining pages never fetched
}

This is exactly the pattern behind cursor-based DB pagination wrappers, streaming large API result sets, and building custom async iterables over Node streams.

Why generators matter for interviews at your level: they're the foundation for understanding async/await itself — async/await is effectively syntactic sugar built on generators + promises under the hood (the "spawn" pattern that libraries like co used pre-native-async).

#Proxy & Reflect

Proxy wraps an object and lets you intercept fundamental operations (get, set, delete, has, etc.) via traps. Reflect provides the default implementations of those same operations, meant to be used inside traps to forward behavior correctly.

js
function createValidatedUser(target) {
  return new Proxy(target, {
    set(obj, prop, value) {
      if (prop === 'age' && (typeof value !== 'number' || value < 0)) {
        throw new TypeError('age must be a non-negative number');
      }
      if (prop === 'email' && !/^\S+@\S+\.\S+$/.test(value)) {
        throw new TypeError('invalid email format');
      }
      return Reflect.set(obj, prop, value); // forward to default behavior
    },
    get(obj, prop) {
      if (!(prop in obj)) {
        console.warn(`Warning: accessing undefined property "${String(prop)}"`);
      }
      return Reflect.get(obj, prop);
    },
  });
}

const user = createValidatedUser({ name: 'Mujtaba' });
user.age = 30;        // OK
user.email = 'x@y.com'; // OK
user.age = -5;         // throws TypeError
user.phone;             // logs warning, returns undefined

Real backend use cases:

  • Runtime validation/schema enforcement layers (lighter alternative to a full validator for internal objects).
  • Logging/auditing every property access on sensitive config objects.
  • Building reactive state systems (this is literally how Vue 3's reactivity system works — Proxy traps on get/set to track dependencies).
  • Default value / auto-vivification objects (return a default instead of undefined).

Interview trap: know why you should call Reflect.set/get instead of directly mutating obj[prop] = value inside a trap — using Reflect ensures correct this binding and default semantics are preserved, especially important with inheritance and getters/setters on the target.

#U15 — V8 Internals & JS Engine Basics

This is the "prove you understand what's under the hood" unit — expect 1-2 pointed questions on this at senior/staff-adjacent interviews, not deep implementation quizzing.

#JIT Compilation Pipeline

V8 doesn't just interpret JS line by line forever. High-level pipeline:

  1. Parser → generates an AST from your source.
  2. Ignition (interpreter) → converts AST to bytecode, executes it immediately. This gets code running fast with low startup cost.
  3. TurboFan (optimizing compiler) → V8 profiles code as it runs ("hot" functions called many times with consistent argument shapes get flagged). TurboFan compiles these hot paths to highly optimized machine code.
  4. Deoptimization → if a previously-optimized function later receives inputs that violate the assumptions TurboFan compiled it for, V8 throws away the optimized code and falls back to Ignition bytecode (deopt), then may re-optimize later if the new pattern stabilizes.

Why this matters practically: writing "monomorphic" code (functions always called with the same shape/type of arguments) is faster than "polymorphic/megamorphic" code (same function called with wildly different argument shapes), because V8 can keep it optimized instead of repeatedly deoptimizing.

#Hidden Classes & Inline Caching

JS objects don't have fixed layouts like C structs, but V8 needs fast property access anyway. It solves this with hidden classes (aka "shapes" or "maps" internally).

js
function Point(x, y) {
  this.x = x;
  this.y = y;
}

const p1 = new Point(1, 2); // gets hidden class C0 -> C1 (after x) -> C2 (after y)
const p2 = new Point(3, 4); // same construction order -> shares hidden class C2 with p1

p1.z = 10; // p1 now transitions to a NEW hidden class C3
// p2 still has hidden class C2 — p1 and p2 now diverge, losing shared optimization

Rule of thumb (frequently tested): always initialize all object properties in the constructor, in the same order, for every instance of a "shape". Adding properties dynamically/conditionally after construction, or in different orders, causes hidden class fragmentation and kills inline caching performance.

Inline caching (IC): once V8 sees a property access (e.g., obj.x) go through a particular hidden class a few times, it caches the memory offset for that property on that hidden class, turning future lookups from a property lookup into a near-direct memory read. If the object's hidden class changes shape, the inline cache "misses" and has to redo the lookup (or worse, go megamorphic if too many shapes appear at one call site).

#Deoptimization triggers to avoid in hot paths (common list to be able to recite)

  • Changing an object's shape after creation (adding/deleting properties dynamically).
  • Using arguments object heavily instead of rest params (harder for V8 to optimize).
  • Mixing types in the same array ([1, 'two', {three: 3}] — becomes a slow "dictionary mode" or generic element kind array instead of a packed, typed one).
  • try/catch around hot loops in older V8 versions (largely mitigated in modern V8, but still worth knowing historically).
  • Using delete obj.prop (converts to dictionary mode) — prefer setting to undefined/null if you must keep the shape.

#A realistic interview scenario to be able to answer

"You have a hot function called millions of times per second in a request handler, processing objects that sometimes have an extra optional field. How would you keep it optimized?"

Good answer: Normalize object shape upfront (e.g., always set optionalField: null if absent instead of omitting the key entirely), avoid delete, keep array element types consistent, and profile with --trace-deopt/--trace-opt flags or node --prof to confirm before optimizing blindly.

#U16 — JS Coding Round Prep

#Must-know polyfills (implement from memory, no lookups)

Function.prototype.myBind

js
Function.prototype.myBind = function (context, ...boundArgs) {
  const fn = this;
  return function (...callArgs) {
    return fn.apply(context, [...boundArgs, ...callArgs]);
  };
};

Array.prototype.myMap

js
Array.prototype.myMap = function (callback, thisArg) {
  const result = [];
  for (let i = 0; i < this.length; i++) {
    if (i in this) {
      result[i] = callback.call(thisArg, this[i], i, this);
    }
  }
  return result;
};

A simplified Promise (very common at senior level)

js
class MyPromise {
  #state = 'pending';
  #value;
  #callbacks = [];

  constructor(executor) {
    const resolve = (value) => this.#settle('fulfilled', value);
    const reject = (reason) => this.#settle('rejected', reason);
    try { executor(resolve, reject); } catch (err) { reject(err); }
  }

  #settle(state, value) {
    if (this.#state !== 'pending') return;
    this.#state = state;
    this.#value = value;
    queueMicrotask(() => this.#callbacks.forEach(cb => cb()));
  }

  then(onFulfilled, onRejected) {
    return new MyPromise((resolve, reject) => {
      const handle = () => {
        try {
          if (this.#state === 'fulfilled') {
            resolve(onFulfilled ? onFulfilled(this.#value) : this.#value);
          } else if (this.#state === 'rejected') {
            if (onRejected) resolve(onRejected(this.#value));
            else reject(this.#value);
          }
        } catch (err) { reject(err); }
      };
      if (this.#state === 'pending') this.#callbacks.push(handle);
      else queueMicrotask(handle);
    });
  }
}

debounce/throttle — see U13 above, but be ready to write both blind, including a cancel() method as a follow-up ask.

#Rapid-Fire Interview Q&A Bank (Phase 3)

Q: What's the difference between Observer and Pub-Sub? A: Observer = subject holds direct references to its observers and notifies them directly. Pub-Sub = a broker/event channel decouples publisher and subscriber entirely; neither knows about the other.

Q: Why do singletons make testing harder? A: Shared global state persists across tests unless explicitly reset, causing test pollution/order-dependence; also hides dependencies that would otherwise be explicit via constructor injection.

Q: What causes a JS memory leak most often in a Node backend? A: Forgotten timers/intervals, detached event listeners on long-lived emitters, and unbounded in-memory caches without TTL/eviction.

Q: Debounce vs throttle — one-liner? A: Debounce waits for silence before firing; throttle guarantees a max firing rate during continuous activity.

Q: What's the difference between an iterator and an iterable? A: Iterable = has Symbol.iterator and can be looped with for...of. Iterator = the object returned by that method, exposing .next().

Q: Why use Reflect inside a Proxy trap instead of direct property access? A: Reflect methods mirror the exact default JS semantics (correct this binding, respects existing getters/setters/inheritance), so forwarding through Reflect avoids subtly breaking default behavior.

Q: What's a hidden class and why does property-add order matter? A: V8's internal representation of an object's "shape." Objects constructed with properties in the same order share hidden classes, enabling fast inline-cached property access. Adding properties dynamically or in inconsistent order causes hidden class transitions/fragmentation, hurting performance.

Q: What triggers V8 deoptimization? A: Changing object shape after creation, mixing types in arrays, heavy use of arguments, and delete on object properties (dictionary mode), among others.

Q: Async/await is sugar over what? A: Generators + Promises — conceptually, await behaves like a generator yielding a promise, with the runtime automatically resuming the generator once the promise settles.
