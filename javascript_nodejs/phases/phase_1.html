#U00: Execution Context, Hoisting & Scope

#1. Execution Context (EC)

An Execution Context is the environment in which JS code is evaluated and executed. Every time a function is invoked, a new EC is created and pushed onto the call stack.

Two phases of every EC:

  1. Creation phase — memory is allocated for variables/functions (hoisting happens here), this binding is set up, scope chain is established.
  2. Execution phase — code runs top to bottom, values are assigned.

Types of EC:

  • Global Execution Context (GEC) — created once, wraps top-level code. Creates the global object (globalThis/window/global) and binds this.
  • Function Execution Context (FEC) — created per function call.
  • Eval Execution Context — rare, from eval().

Call stack: LIFO structure tracking active ECs. When a function returns, its EC pops off. A RangeError: Maximum call stack size exceeded means the stack overflowed (usually infinite/deep recursion).

#2. Hoisting

Hoisting is a result of the creation phase — declarations are processed before execution, so it looks like they're "moved to the top."

Declaration Hoisted? Initialized to Accessible before line?
var Yes undefined Yes (value is undefined)
function decl Yes Fully hoisted (whole function) Yes, callable
let / const Yes (binding created) Uninitialized (TDZ) No — ReferenceError
class Yes (binding created) Uninitialized (TDZ) No — ReferenceError
function expression / arrow assigned to var Only the var part undefined Yes, but calling throws TypeError: not a function

#3. Temporal Dead Zone (TDZ)

The TDZ is the span from the start of a scope until the let/const/class declaration is actually executed. The binding exists (hoisted) but is uninitialized — accessing it throws ReferenceError, not undefined. This is why let/const feel "not hoisted" even though technically they are.

js
console.log(a); // undefined (var hoisted + initialized)
console.log(b); // ReferenceError: Cannot access 'b' before initialization
var a = 1;
let b = 2;

#4. Scope & Scope Chain

  • Global scope — top level.
  • Function scope — var is function-scoped, not block-scoped.
  • Block scope — let/const are block-scoped ({} creates a new scope for them).
  • Lexical scoping: scope is determined by where code is physically written, not by call-time. A nested function has access to variables in its enclosing lexical scopes — this is the scope chain, resolved at parse time (not runtime).
js
function outer() {
  let x = 10;
  function inner() {
    console.log(x); // resolves via scope chain -> outer's x
  }
  inner();
}

Dynamic scope (not used in JS, but interviewers may ask to contrast) would resolve x based on the caller's context instead of where the function was defined. JS never does this for variable scope (only this behaves somewhat "dynamically").

#Interview Q&A

Q: Why does var inside a for loop cause the classic setTimeout closure bug, but let fixes it? A: var is function-scoped — there's only one binding shared across all iterations, so by the time the callbacks fire, the loop has finished and the variable holds its final value. let creates a new binding per iteration (per the spec), so each closure captures its own snapshot.

js
for (var i = 0; i < 3; i++) setTimeout(() => console.log(i), 0); // 3 3 3
for (let j = 0; j < 3; j++) setTimeout(() => console.log(j), 0); // 0 1 2

Q: Is a function declaration inside a block (if {}) hoisted to the block or the function scope? A: Historically inconsistent across engines; in strict mode/ES6, function declarations in blocks are block-scoped, but legacy "Annex B" behavior in non-strict sloppy mode also leaks a var-like binding to the enclosing function scope. Safe interview answer: "Behavior is spec-defined but inconsistent across environments — always use function expressions assigned to let/const inside blocks to avoid ambiguity."

Q: What's the output?

js
console.log(typeof foo); // "function"
function foo() {}
var foo = 5;
console.log(typeof foo); // "number"

A: Function declarations hoist fully (including their body) and take priority over var declarations of the same name during the creation phase; the var foo = 5 assignment only happens at its line during execution, overwriting foo.

#Common Traps

  • Assuming let/const are "not hoisted" — they are hoisted, just uninitialized (TDZ).
  • Redeclaring var in the same scope silently overwrites; redeclaring let/const throws SyntaxError.
  • typeof on a TDZ variable still throws ReferenceError (the one case where typeof on an undeclared-looking identifier isn't safe).

#U01: Type Coercion, Equality & Truthy/Falsy

#1. Implicit Coercion Rules

JS coerces types automatically in certain operator contexts.

  • + operator: if either operand is a string, does string concatenation (after calling ToPrimitive on objects). Otherwise numeric addition.
    js
    1 + '1'      // '11'
1 + {}       // '1[object Object]'
1 + []       // '1'  (array -> '' via ToPrimitive)
[] + []      // ''
[] + {}      // '[object Object]'
  • - * / %: always coerce both operands to Number via ToNumber.
    js
    '5' - 2   // 3
'5' * '2' // 10
  • ToPrimitive algorithm (for objects): tries valueOf() then toString() (for default/number hint), or toString() then valueOf() (for string hint).

#2. Truthy / Falsy

Only 8 falsy values in JS — memorize this list: false, 0, -0, 0n (BigInt zero), '', null, undefined, NaN

Everything else is truthy — including '0', 'false', [], {} (empty array/object are truthy!).

js
if ([]) console.log('runs');   // runs - [] is truthy
if ({}) console.log('runs');   // runs - {} is truthy
if (0) {} else console.log('falsy'); // falsy

Real-world bug this causes: if (count) silently treats count = 0 as "no value," which is wrong when 0 is a legitimate value (e.g., inventory count, page index). Fix: explicit checks — if (count !== undefined) or if (Number.isFinite(count)).

#3. == vs ===

  • === (strict equality): no coercion, compares type + value.
  • == (loose equality): coerces operands to a common type before comparing, following the Abstract Equality Comparison Algorithm.

Key gotchas:

js
null == undefined     // true  (special-cased — only equal to each other)
null === undefined    // false
NaN == NaN             // false (NaN is never equal to anything, incl. itself)
Number.isNaN(NaN)      // true - the only reliable NaN check
[] == false            // true  ([] -> '' -> 0, false -> 0)
'' == 0                // true
'0' == false           // true
[] == ![]              // true (![] is false, [] coerces to 0, false -> 0)

Interview-safe rule: always use ===/!== except the idiomatic x == null (which checks for both null and undefined in one shot — commonly used and accepted).

#4. NaN Handling

  • NaN is the only value in JS not equal to itself.
  • typeof NaN === 'number' (it IS a number type, just "not a number" semantically).
  • Reliable check: Number.isNaN(value) (does NOT coerce). Avoid global isNaN() — it coerces first (isNaN('foo') is true because 'foo' coerces to NaN).

#Interview Q&A

Q: console.log(0.1 + 0.2 === 0.3) — what and why? A: false. Floating point numbers use IEEE-754 double precision, which can't represent 0.1/0.2 exactly, causing rounding errors (0.1+0.2 === 0.30000000000000004). Fix: compare with an epsilon (Math.abs(a-b) < Number.EPSILON) or use integer-based math (cents instead of dollars).

Q: Predict output:

js
console.log([1,2,3] + [4,5,6]); // "1,2,34,5,6"
console.log('5' + 3);           // "53"
console.log('5' - 3);           // 2
console.log('5' + +'3');        // "53"  (unary + coerces '3' to number first, then string concat)

#Common Traps

  • Using == for array/object comparisons — always compares by reference regardless of coercion ([] == [] is false).
  • Forgetting 0 and '' are falsy when validating required form/API fields.
  • Global isNaN() vs Number.isNaN() mixup.

#U02: Closures & Currying

#1. What Is a Closure?

A closure is formed when a function "remembers" the lexical scope it was created in, even after the outer function has returned. Technically: every function in JS carries a reference to its [[Environment]] — the scope in which it was defined — not a copy of the variables, a live reference.

js
function makeCounter() {
  let count = 0;
  return function () {
    return ++count; // closes over 'count'
  };
}
const counter = makeCounter();
counter(); // 1
counter(); // 2 (same 'count' persists across calls)

Key point: closures capture variables by reference, not by value snapshot. This is why the classic var loop bug happens (see U00) — all closures share the same i.

#2. Practical Uses

  • Data privacy / private state: pre-class-fields way to hide internal state (module pattern, IIFE).

    js
    const bankAccount = (function () {
  let balance = 0; // private, inaccessible from outside
  return {
    deposit: (amt) => (balance += amt),
    getBalance: () => balance
  };
})();

  • Memoization: cache expensive computation results in a closed-over object.

    js
    function memoize(fn) {
  const cache = new Map();
  return function (...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}

  • Currying: transforming a function of N args into a chain of functions each taking one arg (or fewer).

    js
    const curry = (fn) => {
  return function curried(...args) {
    if (args.length >= fn.length) return fn(...args);
    return (...more) => curried(...args, ...more);
  };
};
const add3 = (a, b, c) => a + b + c;
const curriedAdd = curry(add3);
curriedAdd(1)(2)(3);   // 6
curriedAdd(1, 2)(3);   // 6
curriedAdd(1, 2, 3);   // 6

    Currying is common in functional pipelines and configuration-style APIs (e.g., partially applying a logger with a fixed prefix).

  • Event handler factories, debounce/throttle (both rely on closures to persist a timer ID across calls — covered in depth in Phase 3).

#3. IIFE (Immediately Invoked Function Expression)

js
(function () {
  // runs immediately, creates an isolated scope
})();

Historically used to avoid polluting global scope before ES6 modules/block scope existed. Still used for one-off initialization or the module pattern above.

#Interview Q&A

Q: Fix this classic bug:

js
for (var i = 1; i <= 3; i++) {
  setTimeout(() => console.log(i), i * 1000);
}
// prints 4, 4, 4 instead of 1, 2, 3

A: Two fixes —

  1. Swap var → let (per-iteration binding).
  2. Wrap in an IIFE to force a new closure per iteration: setTimeout((function(i){ return () => console.log(i); })(i), i*1000).

Q: Implement once(fn) — a function that can only run once.

js
function once(fn) {
  let called = false, result;
  return (...args) => {
    if (!called) { result = fn(...args); called = true; }
    return result;
  };
}

This is a closure over called/result — commonly asked because it demonstrates persisted private state.

#Common Traps

  • Believing closures copy variable values — they close over the variable binding itself.
  • Memory leaks: closures keep referenced variables alive as long as the closure itself is reachable (e.g., a forgotten event listener holding a large object in scope).

#U03: this, call/apply/bind & Arrow Functions

#1. The 4 (+1) Rules of this Binding

this is determined by how a function is called, not where it's defined (except arrow functions — see below). Priority order (highest wins):

  1. new binding — new Foo() → this is the newly created object.
  2. Explicit binding — .call(), .apply(), .bind() → this is whatever is passed.
  3. Implicit binding — called as a method (obj.method()) → this is obj.
  4. Default binding — plain function call (foo()) → this is undefined in strict mode, or the global object in sloppy mode.
js
function show() { console.log(this); }
const obj = { show };

show();               // default: undefined (strict) / global (sloppy)
obj.show();            // implicit: obj
obj.show.call({x:1});  // explicit: {x:1}
new show();            // new binding: newly created object

Classic bug — losing this in a callback:

js
const obj = {
  name: 'A',
  greet() { setTimeout(function () { console.log(this.name); }, 0); } // undefined
};

Fix options: arrow function (setTimeout(() => console.log(this.name), 0)), .bind(this), or store const self = this.

#2. call / apply / bind

Method Invokes immediately? Args format Returns
.call(thisArg, a, b, c) Yes comma-separated function's return value
.apply(thisArg, [a,b,c]) Yes array function's return value
.bind(thisArg, a, b) No comma-separated (partial application) new function with this/args pre-set
js
function greet(greeting) { return `${greeting}, ${this.name}`; }
greet.call({name: 'Sam'}, 'Hi');      // "Hi, Sam"
greet.apply({name: 'Sam'}, ['Hi']);   // "Hi, Sam"
const boundGreet = greet.bind({name: 'Sam'});
boundGreet('Hi');                      // "Hi, Sam"

bind polyfill (frequently asked):

js
Function.prototype.myBind = function (context, ...boundArgs) {
  const fn = this;
  return function (...args) {
    return fn.apply(context, [...boundArgs, ...args]);
  };
};

#3. Arrow Functions & Lexical this

Arrow functions do not have their own this, arguments, or super — they capture these lexically from the enclosing scope at definition time, exactly like a closure captures a variable. This means:

  • .call()/.apply()/.bind() cannot rebind this on an arrow function — the thisArg is silently ignored.
  • Arrow functions should not be used as object methods when you need dynamic this (e.g., obj = { name: 'A', greet: () => console.log(this.name) } — this here is NOT obj, it's whatever enclosing scope the object literal was created in, usually module/global scope).
  • Arrow functions cannot be used as constructors (new arrowFn() throws TypeError).

#Interview Q&A

Q: Why do class methods often need .bind(this) in the constructor, or arrow-function class fields? A: When a method is passed as a callback (e.g., onClick={this.handleClick} in React, or arr.forEach(this.process)), it's invoked as a plain function, losing implicit binding to the instance. Binding in the constructor (this.handleClick = this.handleClick.bind(this)) or using an arrow-function class field (handleClick = () => {...}) locks this to the instance regardless of call-site.

Q: What does this print?

js
const obj = {
  name: 'Obj',
  regular: function () { console.log(this.name); },
  arrow: () => { console.log(this.name); }
};
obj.regular(); // "Obj"
obj.arrow();   // undefined (this is lexical, refers to outer/module scope)

#Common Traps

  • Using arrow functions for object methods that need this.
  • Forgetting bind/arrow in setTimeout, event listeners, array callbacks passed as references.
  • Assuming apply/call change this on arrow functions (they don't).

#U04: Values vs References: Copying & Mutation

#1. Primitives vs Objects

  • Primitives (string, number, boolean, null, undefined, symbol, bigint) are copied by value. Assigning or passing them creates an independent copy.
  • Objects/arrays/functions are copied by reference — but critically, the reference itself is passed by value. This nuance is the #1 source of confusion.
js
function reassign(obj) { obj = { changed: true }; } // reassigning the local reference
const original = { changed: false };
reassign(original);
console.log(original.changed); // false — original untouched

function mutate(obj) { obj.changed = true; } // mutating via the shared reference
mutate(original);
console.log(original.changed); // true — mutation IS visible outside

So: JS is always pass-by-value — for objects, the "value" being passed is the reference (memory address), not the object itself. Reassigning the parameter inside the function only rebinds the local copy of that reference; mutating properties through it affects the shared object.

#2. Shallow Copy vs Deep Copy

Shallow copy — copies top-level properties only; nested objects/arrays are still shared references.

js
const a = { x: 1, nested: { y: 2 } };
const b = { ...a };           // spread — shallow
const c = Object.assign({}, a); // also shallow
b.nested.y = 99;
console.log(a.nested.y); // 99 — nested object shared! bug source.

Deep copy — recursively copies all levels, no shared references.

js
const deep = structuredClone(a); // native, handles Dates/Maps/Sets/circular refs (Node 17+)
// legacy fallback: JSON.parse(JSON.stringify(a)) — loses functions, undefined, Dates become strings, breaks on circular refs
Method Depth Handles functions? Handles circular refs? Handles Date/Map/Set?
{...obj} / Object.assign Shallow N/A (copies ref) N/A N/A (copies ref)
JSON.parse(JSON.stringify()) Deep No (dropped) No (throws) No (Date→string, Map/Set lost)
structuredClone() Deep No (throws) Yes Yes
Lodash _.cloneDeep() Deep Copies ref Yes Yes

#Interview Q&A

Q: Predict the output:

js
function updateArray(arr) {
  arr.push(4);       // mutation - visible outside
  arr = [9, 9, 9];    // reassignment - local only
}
const nums = [1, 2, 3];
updateArray(nums);
console.log(nums); // [1, 2, 3, 4]

Q: You have const state = { user: { settings: { theme: 'dark' } } } and do const copy = {...state}. You then set copy.user.settings.theme = 'light'. Does state.user.settings.theme change? Why, and how do you avoid it? A: Yes, it changes to 'light' too — spread is shallow, so copy.user is the same object reference as state.user. Avoid via structuredClone(state) or manually deep-cloning/using an immutable-update library (Immer) for nested state — this is a very common Redux/React-state interview question, equally relevant to backend config-object mutation bugs.

#Common Traps

  • Assuming spread/Object.assign performs a deep copy.
  • Mutating a function parameter (object/array) thinking it's "safe" because "JS passes by value" — true for the reference, false for the pointed-to data.
  • Using JSON.parse(JSON.stringify()) on objects containing Date, undefined, functions, or circular references and getting silently wrong/broken results.

#U05: Prototypes, Classes, Getters/Setters & Object.create

#1. The Prototype Chain

Every JS object has an internal [[Prototype]] link (exposed via __proto__, or properly via Object.getPrototypeOf()) to another object. When you access a property, the engine walks up this chain until it finds the property or reaches null.

js
const animal = { eats: true };
const dog = Object.create(animal); // dog.[[Prototype]] = animal
dog.barks = true;
console.log(dog.eats);  // true — found via prototype chain
console.log(Object.getPrototypeOf(dog) === animal); // true

__proto__ vs .prototype — the #1 source of confusion:

  • obj.__proto__ — the actual link to the object's prototype (legacy accessor; prefer Object.getPrototypeOf).
  • Fn.prototype — a property that exists only on functions, used as the [[Prototype]] for objects created via new Fn(). It is NOT the function's own prototype.
js
function Dog(name) { this.name = name; }
Dog.prototype.bark = function () { return `${this.name} says woof`; };
const d = new Dog('Rex');
d.bark();                              // "Rex says woof" — found on Dog.prototype
d.__proto__ === Dog.prototype;         // true

#2. Constructor Functions vs ES6 Classes

Classes are (mostly) syntactic sugar over prototype-based inheritance — under the hood, methods still live on .prototype.

js
// Constructor-function style
function Person(name) { this.name = name; }
Person.prototype.greet = function () { return `Hi, I'm ${this.name}`; };

function Student(name, school) {
  Person.call(this, name);       // manual "super" call
  this.school = school;
}
Student.prototype = Object.create(Person.prototype); // set up inheritance
Student.prototype.constructor = Student;

// Class style — equivalent, cleaner
class PersonC {
  constructor(name) { this.name = name; }
  greet() { return `Hi, I'm ${this.name}`; }
}
class StudentC extends PersonC {
  constructor(name, school) {
    super(name);                 // calls PersonC constructor
    this.school = school;
  }
}

Real differences (not just sugar):

  • Class declarations are not hoisted the way function declarations are (they're in TDZ).
  • Class bodies run in strict mode implicitly.
  • Class methods are non-enumerable by default (constructor-function-assigned prototype methods are enumerable unless configured otherwise).
  • Calling a class without new throws TypeError; calling a constructor function without new silently runs with wrong this (global/undefined) — a common source of production bugs pre-ES6.

#3. Object.create()

Creates a new object with a specified prototype directly, without invoking a constructor — useful for prototype-only inheritance or building objects with null prototype (Object.create(null)) to avoid inherited pollution (e.g., safe dictionary/map objects with no toString/hasOwnProperty collisions).

js
const safeDict = Object.create(null); // no prototype chain at all
safeDict.hasOwnProperty; // undefined — no inherited methods, good for pure key-value maps

#4. Getters & Setters

Allow computed/validated property access with normal property syntax (no explicit method call needed).

js
class Circle {
  constructor(radius) { this._radius = radius; } // backing field convention (_prefix)
  get area() { return Math.PI * this._radius ** 2; }
  set radius(value) {
    if (value <= 0) throw new RangeError('Radius must be positive');
    this._radius = value;
  }
  get radius() { return this._radius; }
}
const c = new Circle(5);
c.area;        // computed on access, no ()
c.radius = 10; // runs validation via setter

Prototype pollution risk: if you dynamically merge user-controlled keys into an object (e.g., a naive deep-merge for config), an attacker can inject a __proto__ key to pollute Object.prototype for every object in the app — a well-known Node.js security vulnerability class. Mitigation: use Object.create(null) for untrusted key-value data, or explicitly block __proto__/constructor/prototype keys during merge, or use Object.freeze(Object.prototype).

#Interview Q&A

Q: What's the output?

js
function Foo() {}
Foo.prototype.bar = 1;
const f = new Foo();
console.log(f.bar);                 // 1 (inherited)
console.log(f.hasOwnProperty('bar')); // false (bar is on the prototype, not own)

Q: Implement classic inheritance without class syntax — see the Student/Person example above; commonly asked to test whether you actually understand what class/extends compile down to.

Q: What is prototype pollution and how would you defend against it in an Express API accepting arbitrary JSON bodies? A: See explanation above — validate/whitelist keys via a schema validator (Zod/Joi, covered in Node Backend Engineering phase), avoid naive recursive merge of untrusted input into objects with a prototype chain, or use Object.create(null) / Map for storing arbitrary user-supplied keys.

#Common Traps

  • Confusing __proto__ (instance-level link) with .prototype (function-level blueprint property).
  • Assuming class methods are enumerable like constructor-function prototype methods.
  • Forgetting super() must be called before using this in a subclass constructor (throws ReferenceError otherwise).

#U06: Arrays, Destructuring & Core Methods

#1. map / filter / reduce Internals

  • map: transforms each element 1:1, returns a new array of the same length. Never mutates.
  • filter: returns a new array containing only elements where the predicate returns truthy.
  • reduce: the most general — folds the array into a single accumulated value (which can itself be an array/object). map and filter can both be implemented in terms of reduce.
js
// reduce implementing map
const doubled = [1,2,3].reduce((acc, x) => { acc.push(x*2); return acc; }, []);
// reduce implementing filter
const evens = [1,2,3,4].reduce((acc, x) => x % 2 === 0 ? [...acc, x] : acc, []);

Custom reduce implementation (frequently asked):

js
Array.prototype.myReduce = function (callback, initialValue) {
  let acc = initialValue;
  let startIdx = 0;
  if (acc === undefined) { acc = this[0]; startIdx = 1; } // no initial value -> use first element
  for (let i = startIdx; i < this.length; i++) {
    acc = callback(acc, this[i], i, this);
  }
  return acc;
};

Chaining: each of map/filter creates a new intermediate array — chaining several is readable but does multiple passes over the data. For very large arrays / hot paths, a single reduce (or a manual for loop) avoids the intermediate allocations — a legitimate performance talking point in senior interviews.

#2. Spread & Rest

Same ... syntax, opposite direction:

  • Spread — expands an iterable into individual elements (in array literals, function calls, object literals).
  • Rest — collects remaining elements into an array/object (in destructuring or function parameters).
js
// Spread
const merged = [...arr1, ...arr2];
Math.max(...numbers);
const objCopy = { ...original, updatedField: 1 };

// Rest
function sum(...nums) { return nums.reduce((a,b) => a+b, 0); }
const [first, ...rest] = [1,2,3,4]; // first=1, rest=[2,3,4]
const { id, ...otherFields } = user;

Both spread on objects/arrays are shallow — same caveat as U04.

#3. Destructuring

js
// Array destructuring - position-based, supports skipping & defaults
const [, second, third = 'default'] = [1, 2];

// Object destructuring - key-based, supports renaming & defaults & nesting
const { name: userName, age = 18, address: { city } = {} } = user;

// Function parameter destructuring (very common in Express handlers)
function createUser({ name, email, role = 'user' }) { /* ... */ }

// Swapping without a temp variable
[a, b] = [b, a];

#4. Iterable Protocol (Basics)

An object is iterable if it implements Symbol.iterator — a method returning an iterator ({ next(): {value, done} }). This is what powers for...of, spread, and destructuring on non-array types.

js
const range = {
  from: 1, to: 3,
  [Symbol.iterator]() {
    let current = this.from, last = this.to;
    return {
      next() {
        return current <= last ? { value: current++, done: false } : { value: undefined, done: true };
      }
    };
  }
};
[...range]; // [1, 2, 3] - custom iterable works with spread

(Full depth on generators/custom iterators is covered in Phase 3 — JS Advanced.)

#Interview Q&A

Q: Difference between map and forEach? A: map returns a new array (pure, chainable); forEach returns undefined and is used purely for side effects. Using map when you don't use its return value is a common code-smell flagged in reviews.

Q: What does this print, and why?

js
console.log([1,2,3].map(parseInt)); // [1, NaN, NaN]

A: Classic gotcha — map calls the callback with (element, index, array). parseInt accepts (string, radix). So it's really parseInt(1,0), parseInt(2,1), parseInt(3,2) — index 1 becomes an invalid radix, hence NaN. Fix: [1,2,3].map(n => parseInt(n)) or .map(Number).

Q: Solve — flatten a nested array one level deep, without Array.flat().

js
const flatten = (arr) => arr.reduce((acc, val) => acc.concat(val), []);

#Common Traps

  • Passing a multi-arg native function (parseInt, sometimes isNaN) directly as a map callback — extra args from map collide with the function's signature.
  • Mutating the array inside map/filter callbacks (side effects break the "pure transform" contract and can cause subtle bugs with array methods that rely on stable length, like forEach/map skipping re-indexed elements).
  • Forgetting destructuring defaults only apply when the value is undefined — NOT null. const { x = 5 } = { x: null } → x is null, not 5.
