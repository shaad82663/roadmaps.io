#Unit 28: Scaling Node.js

#28.1 Why scaling matters

Node runs your JS on a single thread. One process = one core used for JS execution (libuv's thread pool handles some I/O off-thread, but your event loop itself is single-threaded). To use multi-core machines, you need to run multiple Node processes.

#28.2 cluster module

  • Spawns multiple worker processes, each with its own event loop, memory, and V8 instance.
  • Master process listens on a port and uses OS-level load balancing (round-robin on Linux, or the OS scheduler) to distribute incoming connections to workers.
  • Workers do not share memory — no shared variables, no shared in-memory cache. If you need shared state, put it in Redis or a DB.
js
const cluster = require('cluster');
const os = require('os');

if (cluster.isPrimary) {
  const cpuCount = os.cpus().length;
  for (let i = 0; i < cpuCount; i++) cluster.fork();

  cluster.on('exit', (worker, code) => {
    console.log(`Worker ${worker.process.pid} died. Restarting...`);
    cluster.fork(); // self-healing
  });
} else {
  require('./server'); // each worker runs the actual app
}

Interview trap: Cluster mode gives you process-level parallelism for handling more concurrent connections — it does not speed up a single CPU-bound request. Two clients hitting the API at the same time can now be served on different cores, but one heavy synchronous computation still blocks that one worker's event loop.

#28.3 worker_threads

  • For CPU-bound work inside a single request (image processing, heavy computation, parsing large JSON, crypto hashing).
  • Unlike cluster, worker_threads can share memory via SharedArrayBuffer.
  • Communication is via postMessage (structured clone by default, unless using SharedArrayBuffer/Atomics).
js
// main.js
const { Worker } = require('worker_threads');

function runHeavyTask(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker('./heavy-task.js', { workerData: data });
    worker.on('message', resolve);
    worker.on('error', reject);
    worker.on('exit', (code) => {
      if (code !== 0) reject(new Error(`Worker stopped with code ${code}`));
    });
  });
}

// heavy-task.js
const { parentPort, workerData } = require('worker_threads');
const result = expensiveComputation(workerData);
parentPort.postMessage(result);

cluster vs worker_threads — the interview answer:

cluster worker_threads
Use case More concurrent requests Offload CPU-heavy work within a request
Memory Isolated processes Can share memory (SharedArrayBuffer)
Overhead Heavier (full process) Lighter (thread)
Crash isolation A worker crash doesn't affect others Same, but easier to leak into main thread if mismanaged

#28.4 PM2

Production process manager built on top of cluster.

  • pm2 start app.js -i max — runs in cluster mode across all cores.
  • Handles auto-restart on crash, zero-downtime reloads (pm2 reload), log aggregation, and monitoring.
  • pm2 reload does rolling restarts (old process keeps serving until new one is ready) vs pm2 restart which kills and starts (brief downtime).

#28.5 Horizontal scaling & stateless design

  • Vertical scaling: bigger machine (more CPU/RAM). Has a ceiling, and single point of failure.
  • Horizontal scaling: more machines/instances behind a load balancer. This is the standard for Node backends.
  • For horizontal scaling to work, your app must be stateless:
    • No in-memory sessions (use Redis-backed sessions or JWTs).
    • No in-memory rate-limit counters (use Redis).
    • No local file writes that later requests depend on (use S3/shared storage).
    • WebSocket connections need sticky sessions or a shared pub/sub layer (Redis adapter) since a client's socket is pinned to one instance.

#28.6 Load balancing

  • Layer 4 (transport): routes based on IP/port, fast, doesn't inspect HTTP content.
  • Layer 7 (application): routes based on HTTP headers/paths, enables smarter routing (path-based, canary deploys), slightly more overhead. (Nginx, ALB, Envoy.)
  • Common algorithms: round robin, least connections, IP hash (useful for sticky sessions).

#Interview Q&A — Unit 28

Q: Your Node API is slow under load on a 8-core machine but CPU usage per core looks low. What's wrong? A: Likely running a single Node process — only using 1 core. Fix: run in cluster mode or with PM2 -i max, or deploy multiple containers behind a load balancer.

Q: When would worker_threads NOT help? A: For I/O-bound work (DB calls, HTTP calls) — those are already async and non-blocking; threads add overhead without benefit. Worker threads only help genuinely CPU-bound synchronous work.

Q: How do you handle WebSocket connections when scaling horizontally? A: Use sticky sessions at the load balancer (so a client always hits the same instance) OR use a shared pub/sub backend (Redis adapter for Socket.io) so any instance can broadcast to any connected client regardless of which instance holds the socket.

Q: What happens to in-flight requests during pm2 reload? A: PM2 does a rolling restart — starts new workers, waits for them to be ready, then gradually shifts traffic away from old workers and kills them only after they've drained existing connections. Zero downtime, unlike a hard restart.

#Unit 29: Caching & Message Queues

#29.1 Why cache

Reduces DB load and latency for frequently-read, infrequently-changed data. The core tradeoff is always staleness vs load.

#29.2 Caching strategies

  • Cache-aside (lazy loading) — most common pattern:
    1. App checks cache first.
    2. On miss, reads from DB, writes result to cache, returns it.
    3. On next read, cache hits.
    js
    async function getUser(id) {
  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);

  const user = await db.users.findById(id);
  await redis.set(`user:${id}`, JSON.stringify(user), 'EX', 3600); // TTL 1hr
  return user;
}
  • Write-through — every write goes to cache and DB simultaneously (keeps cache always fresh, slightly slower writes).
  • Write-behind (write-back) — write to cache immediately, flush to DB asynchronously in batches (fast writes, risk of data loss if cache crashes before flush).
  • TTL / invalidation — the classic hard problem ("there are only two hard things in CS: cache invalidation and naming things"). Options:
    • Time-based expiry (simplest, some staleness window accepted).
    • Explicit invalidation on write (redis.del('user:123') after an update).
    • Versioned/namespaced keys (user:v2:123) so you can invalidate a whole generation at once.

#29.3 Common Redis caching pitfalls (interview favorites)

  • Cache stampede / thundering herd: many requests miss cache simultaneously (e.g., a hot key just expired) and all hammer the DB at once. Fix: request coalescing (only one request repopulates, others wait), or add jitter to TTLs so keys don't all expire at the same moment.
  • Stale cache after failed DB write: always update DB first, then invalidate/update cache — never the other way around, or a race can leave stale cache data even after a DB write succeeds and a concurrent read repopulates cache first.

#29.4 Message queues

Decouple producers from consumers; enable async processing, retries, and load leveling (absorbing traffic bursts).

Kafka

  • Distributed log, partitioned topics, consumers track their own offset.
  • Great for high-throughput event streaming, replay-ability, multiple independent consumer groups reading the same stream.
  • Messages are retained for a configured period regardless of consumption (unlike traditional queues).

RabbitMQ

  • Traditional broker with exchanges/queues/bindings (AMQP protocol).
  • Good for task distribution, routing logic (direct/topic/fanout exchanges), when you need per-message ack/nack and don't need Kafka's replay semantics.

When to use which (interview answer):

  • Need event streaming, multiple consumers replaying history, huge throughput → Kafka.
  • Need flexible routing, simpler ops, classic task queue semantics → RabbitMQ.

#29.5 Job queues — BullMQ

Built on Redis; the practical choice for background jobs in a Node app (emails, PDF generation, image processing, scheduled tasks).

js
const { Queue, Worker } = require('bullmq');

const emailQueue = new Queue('emails', { connection: redisConnection });

// Producer
await emailQueue.add('welcome-email', { userId: 123 }, {
  attempts: 3,
  backoff: { type: 'exponential', delay: 5000 },
});

// Consumer
const worker = new Worker('emails', async (job) => {
  await sendEmail(job.data.userId);
}, { connection: redisConnection, concurrency: 5 });

worker.on('failed', (job, err) => console.error(`Job ${job.id} failed:`, err));

Key features to know for interviews: retries with backoff, concurrency control, delayed jobs, rate limiting per queue, dead-letter handling (failed jobs after max attempts), job priority.

#Interview Q&A — Unit 29

Q: How do you prevent a cache stampede on a very hot key? A: Use a lock/mutex so only one process repopulates the cache while others wait or serve stale data briefly ("stale-while-revalidate"), and/or add random jitter to TTLs to avoid synchronized expiry.

Q: Cache-aside vs write-through — tradeoffs? A: Cache-aside is simpler and only caches what's actually read (efficient), but the first read after expiry is always slow ("cold miss") and there's a brief window of staleness. Write-through keeps cache always warm and consistent but adds write latency and can cache data that's never read.

Q: Why use a message queue instead of calling a service directly? A: Decoupling (producer doesn't need consumer to be up), load leveling (buffer traffic spikes instead of overwhelming downstream), retries/durability (message isn't lost if consumer briefly fails), and independent scaling of producer/consumer.

Q: Kafka vs RabbitMQ — when would you pick each? A: Kafka for high-throughput event streams with multiple independent consumers and replay needs (analytics pipelines, event sourcing). RabbitMQ for simpler task/work queues with flexible routing and lower operational complexity.

#Unit 30: Microservices Architecture

#30.1 Defining service boundaries

The hardest part of microservices isn't the tech — it's drawing boundaries correctly.

  • Use Domain-Driven Design (DDD) bounded contexts: each service owns one business capability (Orders, Payments, Inventory), not a technical layer (don't split into "DB service" / "API service").
  • Rule of thumb: a service should be independently deployable and own its own data. If two "services" always deploy together and share a DB, they're really one service.

#30.2 Data ownership

  • Each microservice owns its own database — no other service reads/writes it directly.
  • Cross-service data needs come through APIs or async events, not shared DB tables (a shared DB re-couples services you just split apart).
  • This introduces eventual consistency — accept that data across services may briefly disagree; design UX/business logic around it (e.g., an order shows "processing" until payment confirmation event arrives).

#30.3 API Gateway pattern

Single entry point that routes requests to the right service, handles cross-cutting concerns (auth, rate limiting, request logging, response aggregation) so individual services don't duplicate that logic.

  • Also solves the "chatty client" problem — client makes one call to the gateway, gateway may fan out to multiple services (Backend-for-Frontend pattern).

#30.4 Sync vs async communication

  • Sync (REST/gRPC): caller waits for response. Simple mental model, but creates tight coupling — if the downstream service is slow/down, the caller is blocked (unless you add timeouts/circuit breakers).
  • Async (message queue/event bus): caller fires an event and moves on; consumer processes independently. Better resilience and decoupling, but harder to reason about (eventual consistency, debugging distributed flows).

#30.5 gRPC

  • Uses Protocol Buffers (binary, strongly typed) over HTTP/2 — faster serialization than JSON, supports streaming (unlike typical REST).
  • Good for internal service-to-service calls where you control both ends (client and server share a .proto contract).
  • Not ideal for public-facing browser APIs (less tooling/debuggability than REST/JSON, browsers need gRPC-Web).
protobuf
// order.proto
service OrderService {
  rpc GetOrder (OrderRequest) returns (OrderResponse);
  rpc StreamOrderUpdates (OrderRequest) returns (stream OrderUpdate);
}
message OrderRequest { string order_id = 1; }
message OrderResponse { string id = 1; string status = 2; }

#30.6 Distributed tracing

When one user request fans out across 5 services, you need to follow it end-to-end.

  • Each request gets a trace ID propagated through headers (traceparent in W3C Trace Context, or custom x-request-id).
  • Each service call within that trace is a span; spans nest to show the call tree and timing.
  • Tools: OpenTelemetry (vendor-neutral SDK), Jaeger/Zipkin (visualization backends).
  • Without this, debugging "why was this request slow" across microservices is nearly impossible — you'd be grepping logs across N services manually.

#30.7 Resilience patterns (commonly asked alongside microservices)

  • Circuit breaker: after N consecutive failures calling a downstream service, "open" the circuit and fail fast (skip the call) for a cooldown period instead of piling up slow/failing calls. Prevents cascading failure.
  • Retry with backoff: retry transient failures, but with exponential backoff + jitter to avoid hammering a struggling service.
  • Timeouts: every network call needs one — an unbounded call can hang a request (and eventually exhaust connections/threads) waiting on a dead dependency.
  • Bulkhead: isolate resource pools per dependency (e.g., separate connection pools) so one slow dependency can't starve resources needed by calls to a healthy one.

#Interview Q&A — Unit 30

Q: How do you keep data consistent across microservices without a shared DB? A: Eventual consistency via async events (e.g., an Order service publishes OrderCreated, Inventory service consumes it and reserves stock). For strict consistency across services, patterns like Saga (choreography or orchestration) coordinate multi-step transactions with compensating actions on failure, since distributed 2PC transactions are generally avoided at scale.

Q: What's a circuit breaker and why do you need one? A: A safeguard that stops calling a failing downstream service after repeated failures, failing fast instead of queuing up requests that will likely fail anyway or hang — this prevents one struggling service from cascading failure across the whole system by exhausting caller resources (threads/connections) waiting on it.

Q: REST vs gRPC for internal service communication? A: gRPC: faster (binary protobuf + HTTP/2 multiplexing), strongly typed contracts, supports streaming — best when you control both client and server. REST/JSON: simpler to debug, universally supported, better for public APIs or when consumers are diverse/external.

Q: How would you debug a slow request that touches 4 microservices? A: Distributed tracing — follow the trace ID across all 4 services' spans to see exactly where time is spent, rather than manually correlating logs by timestamp across services.

#Unit 31: Security Hardening

#31.1 OWASP Top 10 — as it applies to Node APIs

  1. Injection (SQL/NoSQL/command injection) — always use parameterized queries/ORMs, never string-concatenate user input into queries.
  2. Broken authentication — weak session handling, missing rate limits on login, predictable tokens.
  3. Sensitive data exposure — logging secrets/PII, not encrypting data in transit (TLS) or at rest.
  4. XML External Entities (XXE) — less common in Node but relevant if parsing XML from untrusted sources.
  5. Broken access control — missing authorization checks (e.g., checking auth but not ownership: user A can fetch user B's resource by changing an ID in the URL — classic IDOR, Insecure Direct Object Reference).
  6. Security misconfiguration — default credentials, verbose error messages leaking stack traces, unnecessary open ports/headers.
  7. XSS — reflecting unsanitized user input into HTML responses (mainly a frontend concern but APIs serving HTML/templates must sanitize).
  8. Insecure deserialization — trusting JSON.parse/deserialized objects from untrusted sources without validation.
  9. Using components with known vulnerabilities — outdated npm packages with CVEs.
  10. Insufficient logging & monitoring — can't detect or investigate a breach without adequate audit logs.

#31.2 helmet

Sets security-related HTTP headers automatically:

js
const helmet = require('helmet');
app.use(helmet());

Sets things like X-Content-Type-Options: nosniff (prevents MIME sniffing), Strict-Transport-Security (forces HTTPS), removes X-Powered-By (don't leak "Express" to attackers), sets a baseline Content-Security-Policy.

#31.3 Input sanitization & validation

  • Validate at the API boundary (Zod/Joi) — reject malformed input before it touches business logic.
  • Sanitize before rendering (if serving any HTML) to prevent stored/reflected XSS.
  • Never trust client-supplied IDs for authorization — always re-check ownership server-side (if (resource.userId !== req.user.id) throw new ForbiddenError()).

#31.4 SQL/NoSQL injection prevention

js
// BAD — string concatenation
db.query(`SELECT * FROM users WHERE email = '${email}'`);

// GOOD — parameterized
db.query('SELECT * FROM users WHERE email = $1', [email]);

// MongoDB — also vulnerable to operator injection if you don't validate input shape
// { email: { $ne: null } } as a request body would bypass an emailmatch if not validated
UserSchema.findOne({ email: String(req.body.email) }); // coerce type explicitly

#31.5 Dependency vulnerability scanning

  • npm audit (built-in), Snyk, or Dependabot for automated PRs on vulnerable dependencies.
  • Pin versions via lockfile; review before bumping major versions of security-sensitive packages (auth libs, crypto libs).

#31.6 Secrets management

  • Never commit secrets to source control (.env in .gitignore).
  • Env vars are fine for simple deployments, but for production at scale prefer a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager) — supports rotation, access auditing, and avoids secrets sitting in plaintext env files on disk.
  • Rotate secrets periodically; never log secrets (redact in logging middleware).

#31.7 Rate limiting & brute force protection

js
const rateLimit = require('express-rate-limit');
app.use('/api/login', rateLimit({ windowMs: 15 * 60 * 1000, max: 5 })); // 5 attempts / 15 min

Also: account lockout after N failed attempts, CAPTCHA on repeated failures, and always compare passwords with a constant-time hash comparison (bcrypt/argon2 handle this internally).

#Interview Q&A — Unit 31

Q: What's an IDOR vulnerability and how do you prevent it? A: Insecure Direct Object Reference — when an API checks that a user is authenticated but not that they're authorized to access the specific resource (e.g., GET /orders/456 returns another user's order just because the requester is logged in). Fix: always verify resource ownership/permissions server-side against the authenticated user, never trust the client-supplied ID alone.

Q: How do you securely store passwords? A: Never store plaintext or reversibly-encrypted passwords. Hash with a slow, salted algorithm designed for passwords — bcrypt or argon2 (not fast general-purpose hashes like MD5/SHA256 alone, which are crackable via brute force/rainbow tables at scale).

Q: What does helmet actually protect against? A: It's not a single protection — it sets multiple HTTP response headers that mitigate specific attack classes: clickjacking (X-Frame-Options), MIME sniffing attacks, forces HTTPS (HSTS), and reduces information leakage (hides the tech stack fingerprint).

Q: How would you securely manage secrets in a Node app deployed on Kubernetes? A: Avoid baking secrets into images or plain env vars in manifests; use a secrets manager (Vault, AWS Secrets Manager) or Kubernetes Secrets with encryption at rest, inject via env vars or mounted volumes at runtime, and rotate regularly with access scoped per service (least privilege).

#Unit 32: System Design Interview Prep

#32.1 General framework (use this structure every time)

  1. Clarify requirements — functional (what must it do) and non-functional (scale, latency, consistency needs). Ask about read/write ratio, expected QPS, data size.
  2. Back-of-envelope estimation — rough numbers for storage, bandwidth, QPS. Interviewers care that you can estimate, not that you're exact.
  3. High-level API design — the core endpoints/contracts.
  4. Data model — what gets stored, in what kind of store (SQL vs NoSQL and why).
  5. High-level architecture — draw boxes: client → LB → API → cache → DB → queue → workers.
  6. Deep dive on 1–2 hard parts — the interviewer usually steers here (e.g., "how do you handle the hot-key problem").
  7. Scale it — talk about bottlenecks, how you'd scale each component, tradeoffs.

Interviewers are grading your process and tradeoff reasoning, not a single "correct" answer.

#32.2 Design: URL Shortener

  • Core requirement: given a long URL, return a short one; given a short URL, redirect to the long one.
  • API: POST /shorten { longUrl } → { shortUrl }; GET /:code → 302 redirect.
  • ID generation: base62 encode an auto-incrementing counter (simple, no collisions) OR hash the URL (MD5/SHA) and take first 7 chars (risk of collision — need collision check), OR use a distributed ID generator (Snowflake-style) if running multiple write nodes.
  • Data model: { code (PK), longUrl, createdAt, expiresAt, ownerId } in a key-value or SQL store — reads massively outnumber writes here.
  • Caching: cache hot short codes in Redis (cache-aside) since redirects are read-heavy and latency-sensitive.
  • Scale: shard by code prefix if the dataset outgrows a single DB; put a CDN/edge cache in front for extremely hot links.
  • Analytics (common follow-up): don't do click tracking synchronously in the redirect path — fire-and-forget an event to a queue, process click counts asynchronously so redirect latency stays low.

#32.3 Design: Rate Limiter

  • Core requirement: limit each client to N requests per time window.
  • Algorithms (know all four — commonly asked to compare):
    • Fixed window counter: simple, but bursts at window boundaries (client can do 2x limit right at the edge of two windows).
    • Sliding window log: store timestamp of every request, count requests in the last N seconds. Accurate but memory-heavy at scale.
    • Sliding window counter: approximates sliding log using weighted counts of current + previous fixed window — good accuracy/memory tradeoff, most commonly used in practice.
    • Token bucket: bucket refills at a fixed rate, each request consumes a token; allows controlled bursts up to bucket size. Very common in production (also used by AWS/Stripe-style APIs).
  • Implementation: Redis-backed (since rate limiting must be consistent across horizontally-scaled instances — in-memory counters per instance don't work). Use Redis INCR + EXPIRE, or a Lua script for atomicity (avoid race conditions between check-and-increment).
js
// token bucket sketch using Redis + Lua for atomicity
const script = `
  local tokens = tonumber(redis.call('GET', KEYS[1]) or ARGV[1])
  if tokens > 0 then
    redis.call('DECR', KEYS[1])
    return 1
  else
    return 0
  end
`;
  • Where to apply it: at the API gateway/edge (cheaper, protects all downstream services) vs per-service (more granular, more overhead).

#32.4 Design: Chat Application

  • Core requirement: real-time 1:1 and group messaging, delivery/read receipts, online presence.
  • Transport: WebSockets for real-time delivery (fallback to long-polling for constrained clients).
  • Scaling WebSockets: each server holds a subset of open connections; use a pub/sub layer (Redis pub/sub, or Kafka) so a message sent to a user connected on Server B, but published from Server A, gets routed correctly — this is the "fan-out" problem in multi-instance real-time systems.
  • Data model: messages stored per conversation, typically a NoSQL store (Cassandra/DynamoDB) partitioned by conversation ID for fast, high-throughput appends and range reads (e.g., latest 50 messages).
  • Delivery guarantees: at-least-once delivery + client-side dedup (message IDs) is usually the practical choice over exactly-once (which is expensive to guarantee in a distributed system).
  • Presence: heartbeat/ping from client, TTL-based online status in Redis (SETEX user:123:online 30 1), so a dead connection auto-expires to "offline" without needing an explicit disconnect event.

#32.5 Design: Notification System

  • Core requirement: send notifications (push/email/SMS) triggered by events, at scale, without blocking the triggering request.
  • Architecture: triggering service publishes an event to a queue → notification service consumes → fans out to the right channel (push/email/SMS) via provider APIs (FCM/APNs, SendGrid, Twilio).
  • Why async: sending an email inline in the request path adds latency and a new failure mode to unrelated user actions (e.g., "place order" shouldn't fail because the email provider timed out) — always decouple via a queue.
  • Retry/dead-letter: failed sends retry with backoff; after max attempts, move to a dead-letter queue for investigation rather than dropping silently.
  • User preferences & rate limiting: check notification preferences (opted out?) and per-user throttling (don't spam) before sending — usually a lookup against a preferences table/cache before the fan-out step.

#32.6 Cross-cutting concepts interviewers probe (memorize these tradeoffs)

  • CAP theorem: in a network partition, choose Consistency or Availability — you can't have both. Most Node backends lean AP (available, eventually consistent) for user-facing features, CP for things like payments/inventory counts.
  • SQL vs NoSQL for a given design: SQL when you need strong consistency, joins, and transactions (orders, payments, users). NoSQL (key-value/wide-column) when you need massive write throughput, simple access patterns, and horizontal scale (chat messages, activity feeds, click events).
  • Read-heavy vs write-heavy systems: read-heavy → cache aggressively, read replicas. Write-heavy → queue writes, batch/async processing, consider write-optimized stores (LSM-tree based like Cassandra).

#Interview Q&A — Unit 32

Q: How would you generate unique short codes at scale across multiple servers without collisions? A: Avoid pure randomness with no coordination. Use a centralized (or partitioned) counter with base62 encoding, or a Snowflake-style distributed ID generator (timestamp + machine ID + sequence) so multiple nodes can generate unique IDs independently without coordination overhead per request.

Q: Token bucket vs sliding window — when would you pick each? A: Token bucket allows controlled burstiness (good for APIs where occasional spikes are fine, like user-facing apps), while sliding window counter gives smoother, more precise enforcement of a strict rate over time (good for protecting a fragile downstream dependency). Fixed window is simplest but allows boundary bursts, so it's rarely the final answer in an interview — mention it, then explain why you'd upgrade to sliding window or token bucket.

Q: In a chat app, how do you deliver a message to a user connected to a different server instance? A: Use a pub/sub backbone (Redis pub/sub or a Kafka topic) that every server instance subscribes to. When Server A receives a message for a user connected to Server B, it publishes the message; Server B, subscribed to the relevant channel, receives it and pushes it down that user's WebSocket connection.

Q: Why keep notification sending out of the main request path? A: Coupling it synchronously means the core user action (e.g., checkout) inherits the latency and failure modes of a third-party notification provider. Decoupling via a queue keeps the critical path fast and resilient, and lets notification delivery retry independently without affecting the triggering transaction.
