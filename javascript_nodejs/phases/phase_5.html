#Unit 22 — Building REST APIs with Express

#22.1 The Middleware Chain (mental model)

Express is fundamentally a chain of functions with signature (req, res, next). Every request flows through this chain top to bottom, in the order middleware/routes were registered. next() hands control to the next function; not calling it hangs the request; calling it with an argument (next(err)) skips straight to error-handling middleware.

js
app.use(morgan('combined'));      // 1. logging
app.use(express.json());          // 2. body parsing
app.use('/api/users', userRouter);// 3. routing
app.use(notFoundHandler);         // 4. 404
app.use(errorHandler);            // 5. centralized errors (4-arg fn)

Key rule: error-handling middleware must have exactly 4 params (err, req, res, next) — Express detects this by arity, not by name.

#22.2 Router-level vs App-level Middleware

  • app.use(mw) → runs for every request.
  • router.use(mw) → scoped to that router's mount path.
  • app.use('/admin', authMw, adminRouter) → path-scoped, only hits /admin/*.

Use express.Router() to modularize by resource (userRouter, orderRouter) and mount them with a versioned prefix (/api/v1/users). This is what interviewers expect when they ask "how do you structure a large Express app."

#22.3 Layered Architecture (what senior interviews actually probe)

text
routes/        → HTTP concerns only (params, status codes)
controllers/   → orchestrates request → calls service → shapes response
services/      → business logic, framework-agnostic
repositories/  → DB access (Prisma/Sequelize/raw queries)
middlewares/   → auth, validation, error handling

Why this matters: it keeps controllers thin, makes services unit-testable without spinning up Express, and isolates DB swaps to the repository layer. This is the #1 thing interviewers check for "have you worked on production code."

#22.4 Request/Response Lifecycle Details

  • req.params, req.query, req.body — know exactly what each contains and when req.body is undefined (no body-parser middleware, or wrong Content-Type).
  • Response is a stream; res.send()/res.json() implicitly call res.end(). Calling res.json() twice throws ERR_HTTP_HEADERS_SENT — a very common junior bug and a favorite debugging question.
  • res.status(201).json(...) — chainable because most res methods return res.

#22.5 Centralized Error Handling Pattern

js
class AppError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.statusCode = statusCode;
    this.isOperational = true; // distinguishes expected vs programmer errors
  }
}

// controller
const getUser = async (req, res, next) => {
  try {
    const user = await userService.findById(req.params.id);
    if (!user) throw new AppError('User not found', 404);
    res.json(user);
  } catch (err) {
    next(err); // always forward to centralized handler
  }
};

// centralized handler (must be LAST app.use)
app.use((err, req, res, next) => {
  const status = err.statusCode || 500;
  res.status(status).json({
    success: false,
    message: err.isOperational ? err.message : 'Internal Server Error',
  });
  if (!err.isOperational) logger.error(err); // unexpected → log for investigation
});

Async route handlers don't auto-forward rejected promises to next() in Express 4 — you must wrap them (express-async-errors package, or a manual asyncHandler wrapper). Express 5 (still stabilizing in the ecosystem) fixes this natively. Know this distinction — it's a recurring gotcha question.

#22.6 REST Conventions Interviewers Check

  • Correct verb usage: GET (safe, idempotent), POST (create, not idempotent), PUT (full replace, idempotent), PATCH (partial update), DELETE (idempotent).
  • Correct status codes: 200 (ok), 201 (created, with Location header ideally), 204 (no content), 400 (bad request/validation), 401 (unauthenticated), 403 (unauthorized/forbidden), 404, 409 (conflict), 422 (semantic validation failure), 429 (rate limited), 500.
  • Pagination (?page=&limit= or cursor-based), filtering, sorting via query params.
  • Versioning: URI (/v1/) vs header-based — know tradeoffs (URI is simpler/cacheable, header is "cleaner REST" but harder to test manually).

#22.7 Interview Q&A

Q: Difference between app.use() and app.get()/app.post()? use matches all HTTP methods and matches path as a prefix; get/post match exact method + exact path pattern.

Q: How does Express match routes internally? It compiles route paths into regex via path-to-regexp, then iterates the middleware stack in registration order until a match calls next() all the way through or a response is sent.

Q: How do you handle async errors in Express 4 without crashing the process? Wrap handlers: const asyncHandler = fn => (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next); — never let an unhandled rejection escape, since Node will otherwise crash on unhandledRejection in strict configurations.

Q: How would you version an API without breaking existing clients? URI versioning + deprecation headers + maintaining old controllers behind a compatibility shim until sunset date; discuss tradeoffs vs GraphQL-style non-versioned evolution.

#22.8 Practical Task

Build a REST API for a resource (e.g., products) with: layered structure (route → controller → service → repo), centralized error handler, asyncHandler wrapper, proper status codes, and a 404 fallback middleware.

#Unit 23 — Authentication & Authorization

#23.1 Authentication vs Authorization (say this correctly in interviews)

  • AuthN = who are you (login, verifying identity).
  • AuthZ = what are you allowed to do (roles/permissions).

#23.2 Session/Cookie-based Auth

  • Server creates a session (stored in memory/Redis/DB), sends client a Set-Cookie with a session ID.
  • Cookie flags matter: HttpOnly (JS can't read it → mitigates XSS token theft), Secure (HTTPS only), SameSite=Strict/Lax (CSRF mitigation).
  • Stateful: server must look up session on every request → doesn't scale horizontally without a shared store (Redis).

#23.3 Token-based Auth (JWT)

Structure: header.payload.signature, base64url-encoded, not encrypted — never put secrets in the payload (interviewers love catching this misconception).

js
const accessToken = jwt.sign(
  { sub: user.id, role: user.role },
  process.env.ACCESS_SECRET,
  { expiresIn: '15m' }
);
const refreshToken = jwt.sign(
  { sub: user.id, tokenVersion: user.tokenVersion },
  process.env.REFRESH_SECRET,
  { expiresIn: '7d' }
);
  • Access token: short-lived (minutes), sent in Authorization: Bearer <token> header, used to authorize API calls, never persisted server-side (stateless — that's the whole point of JWT).
  • Refresh token: long-lived, stored HttpOnly cookie or secure storage, used only to mint new access tokens, should be persisted/trackable server-side (e.g., a tokenVersion or a stored hash) so it can be revoked on logout/compromise — pure stateless refresh tokens can't be revoked, which is a common design flaw interviewers probe.

Refresh flow:

text
Client access token expires → 401
Client calls /auth/refresh with refresh token (cookie)
Server verifies refresh token, checks tokenVersion/DB record
Server issues new access token (+ optionally rotates refresh token)

Rotation + reuse detection: on every refresh, invalidate the old refresh token and issue a new one; if an old (already-used) refresh token is presented again, it signals theft — revoke the entire token family.

#23.4 JWT vs Session — Interview Comparison Table (know both sides)

Session/Cookie JWT
State Stateful (server store) Stateless (self-contained)
Revocation Easy (delete from store) Hard (needs blocklist/versioning)
Scaling Needs shared store (Redis) Scales naturally (no lookup)
Payload size Small (just an ID) Larger (base64 claims)
Common use Traditional web apps APIs, SPAs, mobile, microservices

#23.5 OAuth2 (conceptual, commonly asked at 3 YOE level)

  • Authorization Code flow (most common, used by "Login with Google"): client redirects to provider → user consents → provider redirects back with a code → server exchanges code + client_secret for tokens (server-to-server, so the secret never touches the browser).
  • Roles: Resource Owner (user), Client (your app), Authorization Server (Google/GitHub), Resource Server (API holding the data).
  • PKCE extension: required for public clients (SPAs, mobile) that can't safely hold a client_secret — prevents authorization code interception attacks.

#23.6 RBAC Middleware

js
const authenticate = async (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return next(new AppError('No token provided', 401));
  try {
    req.user = jwt.verify(token, process.env.ACCESS_SECRET);
    next();
  } catch (err) {
    next(new AppError('Invalid or expired token', 401));
  }
};

const authorize = (...allowedRoles) => (req, res, next) => {
  if (!allowedRoles.includes(req.user.role)) {
    return next(new AppError('Forbidden', 403));
  }
  next();
};

// usage
router.delete('/users/:id', authenticate, authorize('admin'), deleteUser);

For finer-grained control than roles, use permission-based checks (user.can('delete:orders')) — mention this as an evolution when roles get too coarse (e.g., "manager" needing partial admin rights).

#23.7 Password Storage

  • Never store plaintext. Use bcrypt/argon2 (salted, slow hash by design to resist brute force).
  • bcrypt.hash(password, 12) — the cost factor (rounds) trades off security vs latency; know that increasing it exponentially increases hashing time.
  • Never compare passwords with ===; always use the library's compare() (constant-time comparison, mitigates timing attacks).

#23.8 Interview Q&A

Q: Where should you store a JWT on the client and why? HttpOnly cookie over localStorage — localStorage is readable by any injected JS (XSS risk); cookies with HttpOnly aren't accessible to JS at all. Tradeoff: cookies need CSRF protection (SameSite, CSRF tokens).

Q: Can you truly "log out" a JWT? Not by itself — it's valid until expiry. Real logout requires either short expiry + refresh-token revocation, or maintaining a server-side blocklist (defeats some statelessness benefits but is often necessary).

Q: How do you handle a compromised refresh token? Token rotation with reuse detection (see 23.3) — revoke the whole family and force re-login.

#23.9 Practical Task

Implement access + refresh JWT flow with rotation, and RBAC middleware protecting at least one admin-only route.

#Unit 24 — Database Integration

#24.1 SQL vs NoSQL — Decision Framework (not just definitions)

  • SQL (Postgres/MySQL): strong schema, relational integrity, ACID transactions, joins — pick when data has clear relationships and consistency matters (finance, orders, inventory).
  • NoSQL (MongoDB, DynamoDB): flexible schema, horizontal scale, denormalized documents — pick for high write throughput, evolving schemas, or when data is naturally document-shaped (logs, catalogs, activity feeds).
  • Real interviews want you to justify a choice for a given scenario, not recite definitions.

#24.2 ORMs — Prisma / Sequelize / TypeORM

  • Prisma: schema-first (schema.prisma), generates a fully-typed client, migrations via prisma migrate. Great DX, strong TypeScript story.
  • Sequelize: model-first, mature, more "magic," good raw-query escape hatches.
  • TypeORM: decorator-based (works well with NestJS), Active Record or Data Mapper pattern.
  • Know when to drop to raw SQL: complex aggregations, performance-critical queries, or DB features the ORM doesn't expose well.

#24.3 Connection Pooling

  • Opening a new TCP+auth connection per request is expensive; a pool maintains a set of reusable connections.
  • Key config knobs: max connections (bounded by DB's max_connections and your app's concurrency), idleTimeoutMillis, connectionTimeoutMillis.
  • Common production bug: pool exhaustion from not releasing connections (forgotten .release() in raw pg usage, or long-held transactions) → new requests hang until timeout. Interviewers ask about this as a "debug this outage" scenario.
  • In serverless (Lambda), pooling is trickier — each cold start can open a new connection; solutions: connection poolers like PgBouncer, or serverless-specific drivers.

#24.4 Transactions & ACID

  • Atomicity (all-or-nothing), Consistency (valid state transitions), Isolation (concurrent txns don't interfere), Durability (committed = permanent).
  • Multi-step operation example (money transfer) must be wrapped in a transaction:
js
// Prisma example
await prisma.$transaction(async (tx) => {
  await tx.account.update({ where: { id: fromId }, data: { balance: { decrement: amount } } });
  await tx.account.update({ where: { id: toId }, data: { balance: { increment: amount } } });
});
  • Isolation levels (Read Uncommitted → Read Committed → Repeatable Read → Serializable): know the tradeoff — stricter isolation prevents anomalies (dirty reads, non-repeatable reads, phantom reads) but reduces concurrency. Postgres defaults to Read Committed.
  • Deadlocks: two transactions waiting on each other's locks — mitigate with consistent lock ordering, shorter transactions, and retry logic.

#24.5 The N+1 Query Problem

Classic ORM trap:

js
const users = await User.findAll();
for (const user of users) {
  user.posts = await Post.findAll({ where: { userId: user.id } }); // 1 query per user!
}

Fix with eager loading:

js
// Sequelize
const users = await User.findAll({ include: Post });
// Prisma
const users = await prisma.user.findMany({ include: { posts: true } });

This is one of the most-asked backend interview questions — be ready to spot it in a code snippet and fix it, and explain why it degrades badly (100 users → 101 queries instead of 2).

#24.6 Indexing (commonly probed even though "Database" isn't its own unit)

  • Index columns used in WHERE, JOIN, ORDER BY frequently.
  • Tradeoff: faster reads, slower writes (index must be updated), more storage.
  • Composite indexes: column order matters (leftmost-prefix rule).

#24.7 Interview Q&A

Q: How do you prevent SQL injection in Node? Always use parameterized queries/prepared statements (?/$1 placeholders) or an ORM's query builder — never string-concatenate user input into SQL.

Q: What's the difference between optimistic and pessimistic locking? Pessimistic: lock the row on read (SELECT ... FOR UPDATE), blocking others until you finish. Optimistic: no lock; check a version/updatedAt field on write, reject if it changed since read (retry). Optimistic scales better under low contention.

Q: How would you paginate a large dataset efficiently? Prefer cursor-based pagination (WHERE id > lastSeenId LIMIT n) over OFFSET/LIMIT for large tables — offset pagination degrades linearly since the DB still scans and discards skipped rows.

#24.8 Practical Task

Write a transaction that must roll back correctly on partial failure, and find/fix an N+1 query in a given ORM code sample.

#Unit 25 — Validation, Logging & Middleware Patterns

#25.1 Schema Validation at the Boundary

Validate at the edge (before touching business logic) — never trust req.body/req.query/req.params.

js
// Zod example
const createUserSchema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
  age: z.number().int().positive().optional(),
});

const validate = (schema) => (req, res, next) => {
  const result = schema.safeParse(req.body);
  if (!result.success) {
    return next(new AppError(result.error.errors[0].message, 400));
  }
  req.body = result.data; // sanitized/typed
  next();
};

router.post('/users', validate(createUserSchema), createUser);
  • Zod: TypeScript-first, schema = type inference, great DX.
  • Joi: mature, framework-agnostic, widely used in older codebases.
  • Validate query params and route params too (z.coerce.number() for numeric query strings, since query params always arrive as strings).

#25.2 Structured Logging

console.log is unacceptable in production: no log levels, no structure for aggregation, blocks the event loop on large synchronous writes in some setups.

Use Winston or Pino (Pino is faster, JSON-first, popular in high-throughput services):

js
const logger = pino({ level: process.env.LOG_LEVEL || 'info' });
logger.info({ userId, action: 'login' }, 'user logged in');
logger.error({ err }, 'failed to process payment');
  • Log levels: trace/debug/info/warn/error/fatal — set via env var per environment.
  • Log structured JSON, not strings — enables querying in tools like ELK/Datadog/CloudWatch Insights.
  • Never log secrets/PII (passwords, full card numbers, tokens) — redact.
  • Correlate requests with a request ID (generate/propagate via header like X-Request-Id) so you can trace one request across services/logs.

#25.3 CORS

  • Browser-enforced same-origin policy; CORS headers tell the browser which cross-origin requests are allowed.
  • Access-Control-Allow-Origin, -Methods, -Headers, -Credentials.
  • Preflight: browser sends an OPTIONS request first for "non-simple" requests (custom headers, non-GET/POST, Content-Type: application/json in some cases) — server must respond correctly or the actual request never fires.
  • cors npm package handles this; know that credentials: true requires an explicit origin (can't use * with credentials).

#25.4 Rate Limiting

js
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
  standardHeaders: true,
  message: 'Too many requests, please try again later.',
});
app.use('/api/', limiter);
  • Algorithms: fixed window (simple, but bursts at window edges), sliding window (smoother), token bucket (allows bursts up to a cap, refills over time — good general default), leaky bucket (smooths output rate).
  • For distributed systems, rate-limit state must live in Redis (in-memory counters don't work across multiple instances) — this is a favorite "how would you scale this" follow-up.

#25.5 Middleware Ordering Gotchas

  • Body parser before any middleware that reads req.body.
  • CORS before routes.
  • Auth middleware before the routes it protects, but validation typically after auth (so you don't waste validation work on unauthenticated junk) — though some prefer validate-first; be ready to justify either.
  • Error handler always last.

#25.6 Interview Q&A

Q: Why validate on the server if the client already validates? Client-side validation is a UX nicety, not a security boundary — anyone can bypass it (Postman, curl, modified JS). Server-side validation is the actual trust boundary.

Q: How do you avoid leaking stack traces to clients in production? Centralized error handler checks NODE_ENV/err.isOperational and only returns a generic message + logs full detail server-side.

Q: What's the difference between authentication middleware order and validation middleware order, and why does it matter? Running auth first avoids doing expensive validation/DB work for unauthenticated requests (fail fast, cheaper).

#25.7 Practical Task

Add Zod/Joi validation + Pino structured logging with a request-ID correlation header to an existing API, and implement token-bucket-style rate limiting.

#Unit 26 — WebSockets & Real-Time Systems

#26.1 WebSocket vs Polling vs SSE (know when to use each)

Polling SSE WebSocket
Direction Client → Server (repeated) Server → Client only Bidirectional
Protocol Plain HTTP HTTP (long-lived) Own protocol (upgrade from HTTP)
Use case Simple, infrequent updates Live feeds, notifications (one-way) Chat, games, collaborative editing
Overhead High (repeated headers) Low Lowest (persistent connection)

#26.2 The WebSocket Handshake

Client sends an HTTP request with Upgrade: websocket and Connection: Upgrade headers plus a Sec-WebSocket-Key; server responds 101 Switching Protocols with a computed Sec-WebSocket-Accept. After that, it's a persistent full-duplex TCP connection — no more HTTP request/response cycle, just framed messages both ways.

#26.3 Socket.io Essentials

Socket.io is a WebSocket abstraction — it falls back to long-polling if WS is unavailable, and adds rooms, namespaces, auto-reconnect, and acknowledgements on top of raw ws.

js
io.on('connection', (socket) => {
  socket.join(`user:${socket.userId}`); // room per user
  socket.on('message:send', async (data) => {
    const msg = await saveMessage(data);
    io.to(`room:${data.roomId}`).emit('message:new', msg);
  });
  socket.on('disconnect', () => { /* cleanup */ });
});
  • Namespaces (/chat, /admin): separate communication channels over one connection, each with its own event handlers/middleware.
  • Rooms: arbitrary grouping within a namespace for targeted broadcast (io.to(room).emit(...)); rooms aren't a Socket.io "connection" concept, just a server-side tagging mechanism.
  • Auth for sockets: verify the JWT during the handshake (io.use((socket, next) => {...}) middleware), not after connection.

#26.4 Scaling WebSockets Across Instances

Problem: WebSocket connections are stateful and sticky to one server process — if User A is connected to instance 1 and User B to instance 2, instance 1 can't natively emit to User B.

Solution: Redis adapter — every instance subscribes to a Redis pub/sub channel; io.to(room).emit() publishes to Redis, and all instances (including the originating one) receive it and deliver to their locally-connected sockets.

js
const { createAdapter } = require('@socket.io/redis-adapter');
io.adapter(createAdapter(pubClient, subClient));

Also requires sticky sessions at the load balancer (so a client's HTTP polling fallback/reconnects land back on the same instance during the handshake phase) — ip-hash or cookie-based stickiness on the LB.

#26.5 Reliability Considerations

  • Reconnection: Socket.io auto-reconnects by default; design your app state to be resync-able (client requests missed events since lastEventId on reconnect rather than assuming a continuous stream).
  • Heartbeats/ping-pong: detect dead connections (network drop without a clean close) — Socket.io handles this internally via pingInterval/pingTimeout.
  • Backpressure: a slow client can cause the server's send buffer to grow unbounded — Socket.io emits are fire-and-forget by default; for critical messages use acknowledgements (socket.emit('event', data, callback)) with timeouts.

#26.6 Interview Q&A

Q: How would you build a "user is typing" indicator that scales to millions of users? Ephemeral event (not persisted), scoped to a room, broadcast via Redis adapter across instances, with a short client-side debounce/timeout to auto-clear the indicator if no follow-up event arrives.

Q: What happens if a message needs guaranteed delivery (e.g., a chat message) but the WebSocket disconnects mid-send? Persist to DB first (or a queue), then push over WebSocket for real-time delivery; on reconnect, client fetches messages since its last known message ID/timestamp as a fallback — WebSocket delivery is "best effort," DB/queue is the source of truth.

Q: Why can't you just use io.emit() from any instance in a multi-instance deployment without the Redis adapter? Each Node process only "knows about" sockets connected directly to it; without a shared pub/sub layer, emit only reaches locally connected clients.

#26.7 Practical Task

Build a real-time notification/chat channel with Socket.io rooms and JWT-authenticated handshake, then wire the Redis adapter and verify a message sent via instance A is received by a client connected to instance B.

#Unit 27 — Testing Node Applications

#27.1 The Testing Pyramid (backend flavor)

text
        /\        E2E (few) — full stack, slow, brittle
       /  \       Integration (some) — API + real/test DB
      /----\      Unit (many) — pure functions, mocked deps

Interviewers want you to justify the ratio, not just define the layers: heavy unit coverage for business logic, targeted integration tests for critical API flows, minimal E2E.

#27.2 Unit Testing with Jest

js
// userService.test.js
jest.mock('../repositories/userRepository');

describe('userService.createUser', () => {
  it('throws if email already exists', async () => {
    userRepository.findByEmail.mockResolvedValue({ id: 1 });
    await expect(userService.createUser({ email: 'a@b.com' }))
      .rejects.toThrow('Email already in use');
  });

  it('hashes the password before saving', async () => {
    userRepository.findByEmail.mockResolvedValue(null);
    userRepository.create.mockResolvedValue({ id: 1, email: 'a@b.com' });
    await userService.createUser({ email: 'a@b.com', password: 'plain' });
    const savedArg = userRepository.create.mock.calls[0][0];
    expect(savedArg.password).not.toBe('plain'); // was hashed
  });
});
  • jest.mock() for module-level mocking; jest.fn() for individual mock functions; mockResolvedValue/mockRejectedValue for async.
  • Test the service layer, not Express — this is exactly why the layered architecture from Unit 22 matters: services are plain functions, trivially testable without spinning up HTTP.
  • Use dependency injection (pass the repository in, or use a container) rather than hard requires where feasible — makes mocking cleaner and avoids jest.mock path fragility.

#27.3 Integration Testing with Supertest

js
const request = require('supertest');
const app = require('../app'); // exported Express app, not app.listen()

describe('POST /api/users', () => {
  beforeEach(async () => { await resetTestDb(); });

  it('creates a user and returns 201', async () => {
    const res = await request(app)
      .post('/api/users')
      .send({ email: 'test@x.com', password: 'password123' });
    expect(res.status).toBe(201);
    expect(res.body.email).toBe('test@x.com');
    expect(res.body.password).toBeUndefined(); // never leak hash
  });

  it('returns 400 on invalid payload', async () => {
    const res = await request(app).post('/api/users').send({ email: 'bad' });
    expect(res.status).toBe(400);
  });
});
  • Export the Express app without calling .listen() in the testable module — Supertest binds it to an ephemeral port internally.
  • Use a real test database (Dockerized Postgres/Mongo, reset between tests) for integration tests — mocking the DB at this layer defeats the purpose of an integration test (you want to catch real query/schema issues).
  • Seed/reset strategy: transaction rollback per test (fast) or full reset (beforeEach) — know the tradeoff (rollback is faster but doesn't work well with code that itself opens transactions).

#27.4 Mocking External Services

  • Mock third-party HTTP calls (payment gateways, email providers) with tools like nock or MSW — never hit real external APIs in CI.
  • Distinguish: stub (canned response), mock (also asserts it was called correctly), spy (wraps a real implementation, observes calls).

#27.5 What Interviewers Actually Ask

Q: How do you test code that depends on Date.now() or randomness? Inject the dependency (pass a clock/idGenerator) or use Jest's fake timers (jest.useFakeTimers(), jest.setSystemTime()).

Q: How would you test a rate limiter or a queue consumer? Rate limiter: fire N+1 requests in a test and assert the last one gets a 429; for time-window behavior, use fake timers to fast-forward. Queue consumer: test the handler function directly with a crafted job payload, independent of the actual queue infra (mock the queue library, unit-test the processing logic).

Q: What's your strategy for flaky tests? Eliminate shared mutable state between tests, avoid real timers/network, ensure DB resets are deterministic, and never rely on execution order.

Q: Unit vs integration — where do you draw the line for a new PR? Business logic and edge cases → unit tests (fast feedback, exhaustive). At least one integration test per new endpoint verifying the full request→response contract including status codes and auth. E2E reserved for critical user journeys only (login, checkout), run less frequently (e.g., not on every commit).

#27.6 Practical Task

For one API route, write unit tests for its service (all branches: success, validation failure, not-found) with mocked repository, plus a Supertest integration test hitting the real route with a test DB, asserting status code and response shape.
