#Unit 17 — Node Architecture & Event Loop Phases

#17.1 The Big Picture

Node.js is not single-threaded end to end. It's a hybrid:

  • Your JS code runs on a single thread (the "main thread"), driven by the V8 engine.
  • I/O-bound work (file system, DNS lookups, some crypto, compression) is offloaded to libuv, which maintains a thread pool (default size 4, configurable via UV_THREADPOOL_SIZE).
  • Network I/O (sockets, HTTP) on most platforms doesn't even need the thread pool — libuv uses the OS's native async mechanisms (epoll on Linux, kqueue on macOS, IOCP on Windows).

So the correct mental model is: "Node is single-threaded for your JavaScript, but multi-threaded under the hood for I/O."

text
 ┌───────────────────────────┐
 │      Your JS Code          │  ← runs on the Main Thread (V8)
 └──────────────┬─────────────┘
                │
        ┌───────▼────────┐
        │     libuv        │  ← Event Loop lives here
        └───────┬────────┘
     ┌───────────┼────────────┐
     │            │             │
┌────▼───┐  ┌─────▼─────┐  ┌────▼─────┐
│ Thread  │  │  OS Async  │  │ Timers/  │
│  Pool   │  │  (epoll/   │  │ Signals  │
│ (fs,    │  │  kqueue/   │  │          │
│ crypto, │  │  IOCP)     │  │          │
│ dns.lookup)│ (network) │  │          │
└─────────┘  └───────────┘  └──────────┘

#17.2 The Event Loop — Phases in Order

Each iteration of the event loop is called a tick, and it moves through six phases, in this fixed order:

  1. Timers — executes callbacks scheduled by setTimeout() and setInterval() whose threshold has elapsed.
  2. Pending callbacks — executes I/O callbacks deferred to the next loop iteration (e.g., some TCP error callbacks).
  3. Idle, prepare — internal use only, not relevant to app code.
  4. Poll — the workhorse phase:
    • Retrieves new I/O events and executes their callbacks (almost all I/O callbacks fire here).
    • If the poll queue is empty and there are setImmediate() callbacks scheduled, it moves on.
    • If there are timers due to expire, it wraps back to the timers phase.
    • Otherwise, it waits (blocks) here for new events — this is how Node "sleeps" when idle instead of busy-looping.
  5. Check — executes setImmediate() callbacks specifically. This runs right after poll.
  6. Close callbacks — e.g., socket.on('close', ...).

Between every phase transition, and after every individual callback, Node drains two special queues:

  • process.nextTick() queue — highest priority, runs before microtasks.
  • Microtask queue (resolved Promises, queueMicrotask()) — runs right after nextTick queue.

This means the real priority order at any checkpoint is:

text
nextTick queue → microtask queue → next event loop phase's callbacks

#17.3 process.nextTick() vs setImmediate() vs setTimeout(fn, 0)

This is the single most common Node interview trap.

js
console.log('start');

setTimeout(() => console.log('setTimeout'), 0);
setImmediate(() => console.log('setImmediate'));
process.nextTick(() => console.log('nextTick'));
Promise.resolve().then(() => console.log('promise'));

console.log('end');

Output:

text
start
end
nextTick
promise
setTimeout   (or setImmediate — order can flip, see below)
setImmediate
  • nextTick and promise always run before any timer/immediate, because they're drained after the currently executing synchronous code finishes, before the loop even moves to its next phase.
  • nextTick always runs before microtasks (Promise callbacks) — it's Node's own queue, checked first.
  • setTimeout(fn, 0) vs setImmediate() ordering is non-deterministic when called from the main module — it depends on process performance/timer precision (setTimeout's minimum is technically ~1ms, so which one "wins" depends on how much time has passed before the loop starts).
  • BUT, inside an I/O callback (e.g., inside fs.readFile's callback), setImmediate() is always guaranteed to run before setTimeout(), because you're already inside the poll phase, and check (setImmediate) comes immediately next, before looping back to timers.
js
const fs = require('fs');

fs.readFile(__filename, () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate')); // always wins here
});

Why this matters practically: process.nextTick() can starve the event loop if used recursively (it keeps draining itself before I/O can proceed) — this is a real production bug pattern. setImmediate() doesn't have this problem because it's tied to a loop phase, not a queue drained before every phase.

#17.4 Why libuv's Thread Pool Exists

V8 itself has no concept of async I/O — it just runs JS. Node needs a way to do blocking OS calls (like reading a file) without blocking the main thread. libuv solves this with a thread pool:

  • Default size: 4 threads (UV_THREADPOOL_SIZE, max 128).
  • Used by: fs module (most methods), dns.lookup() (not dns.resolve*, which use c-ares async), crypto.pbkdf2, crypto.scrypt, zlib operations, and some os calls.
  • Not used by: network sockets/HTTP — those use the OS's native async I/O notification system directly (epoll/kqueue/IOCP), which is why Node can handle tens of thousands of concurrent connections with just 4 threads in the pool — the pool isn't the bottleneck for network-heavy apps.

Common interview gotcha: "If you increase UV_THREADPOOL_SIZE, does it improve HTTP request handling?" — No, because HTTP/network I/O doesn't use the thread pool at all. It only helps if you're bottlenecked on fs, crypto, dns.lookup, or zlib.

#17.5 Practical Consequence: Don't Block the Event Loop

Because there's one thread running your JS, any CPU-bound synchronous work (a huge JSON.parse, a synchronous crypto hash, a tight loop) blocks everything — no other request can be served, no timer fires, nothing.

js
// BAD - blocks the entire server for all users
app.get('/hash', (req, res) => {
  const hash = crypto.createHash('sha256');
  for (let i = 0; i < 1e9; i++) hash.update('x'); // blocking loop
  res.send(hash.digest('hex'));
});

Fixes: offload to worker_threads, use the async variant of the API (crypto.pbkdf2 instead of pbkdf2Sync), or move it to a separate microservice/queue.

#Interview Q&A — Unit 17

Q: Is Node.js single-threaded? A: The JavaScript execution is single-threaded (one call stack, one event loop thread), but Node itself uses multiple threads under the hood via libuv's thread pool for certain I/O operations, plus the OS's native async I/O for networking.

Q: What's the difference between setImmediate() and setTimeout(fn, 0)? A: setImmediate runs in the check phase, right after the poll phase completes in the current loop iteration. setTimeout(fn, 0) runs in the timers phase of a future iteration. Order at the top level is non-deterministic; inside an I/O callback, setImmediate always fires first.

Q: Why can process.nextTick() be dangerous? A: Because it's drained completely before the event loop can proceed to the next phase — if you recursively schedule nextTick calls, you can starve I/O indefinitely ("I/O starvation").

Q: Does increasing UV_THREADPOOL_SIZE help scale an HTTP API? A: Only if the bottleneck is fs/crypto/dns.lookup/zlib work. Network I/O bypasses the thread pool entirely via epoll/kqueue/IOCP.

#Unit 18 — Module System & Package Management

#18.1 CommonJS Module Resolution Algorithm

When you call require('./foo') or require('lodash'), Node runs a well-defined resolution algorithm:

  1. Core module? (fs, path, http...) → return immediately, these are baked into the binary.
  2. Starts with /, ./, or ../? → treat as a file path:
    • Try exact file (./foo.js, ./foo.json, ./foo.node).
    • Try as a directory: look for package.json's "main" field, or default to ./foo/index.js.
  3. Otherwise (bare specifier like 'lodash') → treat as a node_modules lookup:
    • Look in ./node_modules/lodash.
    • If not found, walk up the directory tree: ../node_modules/lodash, ../../node_modules/lodash, ... until reaching the filesystem root.
    • This is why nested node_modules folders exist (pre-npm3 / edge cases) and why two packages can each depend on a different version of the same dependency without conflict.

#18.2 The Module Cache

Every module is cached by its fully resolved absolute file path after first require(). This has real consequences:

js
// counter.js
let count = 0;
module.exports = { increment: () => ++count };

// a.js
const counter = require('./counter');
counter.increment(); // count = 1

// b.js
const counter = require('./counter');
counter.increment(); // count = 2, SAME instance as a.js

Both a.js and b.js get the exact same object reference — require doesn't re-execute the module file on subsequent calls, it returns the cached module.exports. This is how singletons (like a shared DB connection pool) are commonly implemented in Node without any extra pattern — plain require caching does it for you.

Gotcha: the cache key is the resolved absolute path. Two different relative paths that resolve to the same file (require('./foo') vs require('../src/foo') from different callers) will hit the same cache entry. But if npm installs two different copies of the same package version in nested node_modules (dependency hell), you get two separate cache entries — this is a classic source of "why is my singleton not actually a singleton" bugs in monorepos.

#18.3 How module.exports Actually Works

Node wraps every file in an IIFE-like function before executing it:

js
(function(exports, require, module, __filename, __dirname) {
  // your file's code goes here
});

This is why require, module, exports, __filename, __dirname are available without importing them — they're injected function parameters, not globals.

Common trap: exports is just a local variable that initially points to the same object as module.exports. Reassigning exports = {...} breaks the link (it now points somewhere else, but module.exports still points to the original object) — only mutating exports (exports.foo = ...) or reassigning module.exports directly works as expected.

js
exports = { foo: 'bar' };       // BROKEN — module.exports unchanged, foo is lost
module.exports = { foo: 'bar' }; // CORRECT
exports.foo = 'bar';             // CORRECT (mutation, not reassignment)

#18.5 CJS/ESM Interop Pitfalls (real production pain)

  • require()-ing an ESM-only package from CJS code fails — you must use dynamic import() (which returns a Promise) instead.
  • import-ing a CJS package from ESM generally works, but only the default export is guaranteed reliably — named exports from CJS packages are inferred via static analysis and sometimes miss exports that are added dynamically (e.g., via Object.assign(module.exports, ...)).
  • Mixed default/named export interop is the #1 cause of "X is not a function" or undefined errors when migrating a codebase to ESM.
js
// If 'some-esm-only-pkg' only ships ESM:
// const pkg = require('some-esm-only-pkg'); // ❌ throws ERR_REQUIRE_ESM

// Fix — dynamic import (works in CJS too):
const pkg = await import('some-esm-only-pkg'); // ✅ (inside an async function)

#18.6 package.json Essentials for Interviews

  • "main" — CJS entry point (legacy).
  • "exports" — modern field; explicitly defines the public API surface and can differ per environment (require vs import vs browser), and — importantly — anything not listed in "exports" becomes unreachable from outside, even if the file physically exists (this is used to prevent consumers from deep-importing internal files).
  • "type": "module" — makes .js files in that package be treated as ESM by default.
  • dependencies vs devDependencies vs peerDependencies: peerDependencies say "the consumer must provide this" (classic for plugins expecting a specific React/host version) rather than the package bundling its own copy.

#18.7 Semver & Lockfiles

  • ^1.2.3 → allows updates that don't change the leftmost non-zero digit (so 1.x.x but not 2.0.0; for ^0.2.3 it only allows 0.2.x, quirk of pre-1.0 semver rules).
  • ~1.2.3 → allows only patch updates (1.2.x).
  • Lockfiles (package-lock.json, pnpm-lock.yaml, yarn.lock) pin the exact resolved tree (including transitive deps) so npm install is reproducible across machines/CI — without a lockfile, two installs of the same package.json weeks apart can silently pull different transitive versions.

#18.8 npm vs pnpm vs yarn (quick comparison)

Disallows phantom deps No (flat hoisting lets you accidentally require an undeclared transitive dep) Yes with PnP Yes — strict node_modules structure catches this

#Interview Q&A — Unit 18

Q: Why does require() return the same object every time for the same module? A: Node caches modules by resolved absolute path after first execution — the file body runs once, and module.exports is cached and returned on every subsequent require() call.

Q: What breaks if you write exports = {...} instead of module.exports = {...}? A: exports is a local reference to the same object module.exports points to initially. Reassigning exports just repoints the local variable — module.exports (what the caller actually receives) is untouched, so your new object is silently lost.

Q: Can you require() an ESM-only package from CommonJS? A: Not with static require() — it throws ERR_REQUIRE_ESM. You need dynamic import(), which is async and returns a Promise.

Q: What's a "phantom dependency" and how does pnpm prevent it? A: A dependency your code imports but never declares in package.json — it "works" only because npm's flat node_modules hoisting happens to place it at the top level. pnpm uses a strict, non-flat structure so only declared dependencies are resolvable, surfacing the bug immediately.

#Unit 19 — File System & Streams

#19.1 Buffers — The Foundation

A Buffer is Node's way of handling raw binary data (before Uint8Array was standard in JS, this was Node's only tool; it now extends Uint8Array). Strings in JS are always UTF-16 internally, but files/network data are raw bytes — Buffers bridge that gap.

js
const buf = Buffer.from('hello', 'utf8'); // <Buffer 68 65 6c 6c 6f>
buf.toString('utf8'); // 'hello'
buf.toString('hex');  // '68656c6c6f'
buf.toString('base64'); // 'aGVsbG8='

Key facts:

  • Buffers are allocated outside the V8 heap (they're backed by raw memory), which is why they don't count against --max-old-space-size limits the same way regular objects do.
  • Buffer.alloc(n) zero-fills; Buffer.allocUnsafe(n) doesn't (faster, but may contain old memory — never send allocUnsafe data to a client without overwriting it fully first — that's a real security bug class, "buffer over-read").

#19.2 The Four Stream Types

Streams let you process data incrementally instead of loading it all into memory. All four types are EventEmitters.

Type Direction Example
Readable Source you read from fs.createReadStream(), HTTP request (server-side)
Writable Destination you write to fs.createWriteStream(), HTTP response
Duplex Both readable and writable, independent TCP sockets
Transform Duplex where output is derived from input zlib.createGzip(), crypto cipher streams

#19.3 Why Streaming Matters (the actual reason, not the buzzword)

js
// BAD - loads entire file into memory before sending
app.get('/download', (req, res) => {
  const data = fs.readFileSync('huge-5gb-file.csv');
  res.send(data); // 5GB in RAM, blocks event loop while reading sync
});

// GOOD - constant, small memory footprint regardless of file size
app.get('/download', (req, res) => {
  fs.createReadStream('huge-5gb-file.csv').pipe(res);
});

The readable stream reads a chunk (default highWaterMark: 64KB for files), pushes it into an internal buffer, and the writable side drains it as fast as it can consume — memory usage stays roughly constant no matter the file size.

#19.4 Backpressure — the Concept Interviewers Actually Care About

Backpressure happens when a Readable stream produces data faster than a Writable stream can consume it (e.g., reading from a fast local SSD, writing to a slow network socket).

  • writable.write(chunk) returns false when its internal buffer has exceeded highWaterMark — this is the signal to pause.
  • If you ignore this and keep writing anyway, memory balloons unboundedly — this is one of the most common causes of an OOM crash in a naive Node file-processing service.
  • .pipe() handles backpressure automatically — it listens for that false return, calls readable.pause(), and resumes once the writable emits 'drain'.

Manual (advanced/interview-favorite) implementation of what .pipe() does internally:

js
readable.on('data', (chunk) => {
  const ok = writable.write(chunk);
  if (!ok) {
    readable.pause();
    writable.once('drain', () => readable.resume());
  }
});
readable.on('end', () => writable.end());

#19.5 Building a Transform Stream

js
const { Transform } = require('stream');

class UppercaseTransform extends Transform {
  _transform(chunk, encoding, callback) {
    this.push(chunk.toString().toUpperCase());
    callback(); // MUST be called, or the stream stalls forever
  }
}

fs.createReadStream('in.txt')
  .pipe(new UppercaseTransform())
  .pipe(fs.createWriteStream('out.txt'));

Gotcha interviewers love: forgetting to call callback() inside _transform/_write — the stream simply hangs with no error, which is a nasty silent production bug.

#19.6 Streaming a Large CSV Without Loading It Fully (lab-style example)

js
const fs = require('fs');
const readline = require('readline');

async function processLargeCsv(path) {
  const rl = readline.createInterface({
    input: fs.createReadStream(path),
    crlfDelay: Infinity,
  });

  let lineCount = 0;
  for await (const line of rl) {
    lineCount++;
    // process one line at a time, constant memory
  }
  return lineCount;
}

for await...of over a readable (or readline interface wrapping one) is the modern idiomatic pattern — it automatically manages backpressure by only pulling the next chunk once the previous iteration's await resolves.

#19.7 fs Module: Sync vs Callback vs Promises API

js
const fs = require('fs');
const fsp = require('fs/promises');

fs.readFileSync(path);                 // blocks event loop — avoid in servers
fs.readFile(path, (err, data) => {});  // callback style, non-blocking
const data = await fsp.readFile(path); // Promise-based, modern preferred style

Sync variants run entirely on the main thread and block everything — acceptable only at startup (e.g., reading a config file once before the server starts accepting requests), never inside a request handler.

#Interview Q&A — Unit 19

Q: What is backpressure and how does .pipe() handle it? A: Backpressure is when a writable destination can't consume data as fast as a readable source produces it. .pipe() detects when write() returns false (internal buffer full), pauses the readable, and resumes it once the writable emits 'drain' — preventing unbounded memory growth.

Q: Why use Buffer.alloc() over Buffer.allocUnsafe() when handling user-facing data? A: allocUnsafe doesn't zero out the underlying memory — it can contain leftover data from previous allocations, which is a real information-disclosure risk if that buffer is ever sent to a client before being fully overwritten.

Q: What happens if you forget to call callback() in a custom Transform stream's _transform? A: The stream stalls indefinitely — no error is thrown, it just silently hangs, because the stream's internal machinery is waiting for that callback signal to pull the next chunk.

Q: When is fs.readFileSync acceptable in a production server? A: Only during startup/bootstrap (e.g., reading a config file before the HTTP server starts listening) — never inside a request handler, since it blocks the single JS thread for every concurrent request.

#Unit 20 — Process, Child Processes & EventEmitter

#20.1 The process Object

Global object giving access to the running Node process:

js
process.env.NODE_ENV       // environment variables
process.argv                // ['node', 'script.js', 'arg1', 'arg2']
process.pid                 // process ID
process.exitCode = 1;       // set exit code without force-exiting (lets pending I/O finish)
process.exit(1);            // force-exit immediately — can cut off pending async work
process.cwd()                // current working directory

process.exitCode vs process.exit(): interviewers often ask this. process.exit() terminates synchronously and can drop unflushed stdout writes or unfinished async callbacks. Setting process.exitCode and letting the event loop drain naturally is the safer pattern in production code.

#20.2 Signals & Graceful Shutdown

js
process.on('SIGTERM', gracefulShutdown); // sent by orchestrators (k8s, PM2, Docker stop)
process.on('SIGINT', gracefulShutdown);  // Ctrl+C

async function gracefulShutdown() {
  console.log('Shutting down gracefully...');
  server.close(() => {          // stop accepting new connections, let existing finish
    db.disconnect().then(() => {
      process.exit(0);
    });
  });

  // safety net: force exit if graceful shutdown hangs
  setTimeout(() => process.exit(1), 10_000).unref();
}

This pattern — stop accepting new work, drain existing work, then exit — is exactly what's expected in any container-orchestrated deployment (k8s sends SIGTERM, then SIGKILL after a grace period if the process hasn't exited).

#20.3 EventEmitter — the Pattern Underlying Almost All of Node Core

http.Server, streams, process itself — all extend EventEmitter. It's the pub-sub backbone of Node.

js
const { EventEmitter } = require('events');

class OrderService extends EventEmitter {
  placeOrder(order) {
    // ...business logic...
    this.emit('order:placed', order);
  }
}

const service = new OrderService();
service.on('order:placed', (order) => console.log('Send confirmation email', order.id));
service.on('order:placed', (order) => console.log('Update inventory', order.id));
service.placeOrder({ id: 42 });

Key facts:

  • Listeners are called synchronously, in registration order, on the emit call — emit() doesn't return until every listener has run (or thrown).
  • emitter.once(event, fn) — auto-removes after first invocation.
  • Default max listeners: 10 per event — exceeding it logs a MaxListenersExceededWarning, which is usually a signal of a memory leak (listeners being added repeatedly without ever being removed, e.g., inside a request handler that attaches a new listener on every request).
  • An EventEmitter that throws inside an 'error' event listener (or has no 'error' listener at all when 'error' is emitted) will crash the process — 'error' is a special-cased event name in Node; always attach an 'error' handler on emitters you don't fully control (sockets, streams).
js
emitter.on('error', (err) => {
  console.error('handled:', err.message); // prevents process crash
});

#20.4 Child Processes — spawn vs exec vs fork

Method Use case Output handling Shell
spawn() Long-running or large-output processes (e.g., streaming ffmpeg output) Streams via stdout/stderr (no buffering limit) No shell by default
exec() Short commands, need the full output as a string Buffers entire output in memory (has a max buffer size, default ~1MB) Runs through a shell (/bin/sh) — supports pipes/glob, but shell-injection risk if interpolating user input
execFile() Like exec but for running an executable directly Buffered No shell — safer than exec for untrusted input
fork() Spawning another Node.js script specifically IPC channel built-in (process.send() / .on('message')) No shell
js
const { spawn } = require('child_process');
const child = spawn('ffmpeg', ['-i', 'input.mp4', 'output.avi']);

child.stdout.on('data', (data) => console.log(`stdout: ${data}`));
child.stderr.on('data', (data) => console.error(`stderr: ${data}`));
child.on('close', (code) => console.log(`exited with code ${code}`));

Security interview point: exec('rm -rf ' + userInput) is a classic shell injection vulnerability because exec runs through /bin/sh, letting ;, &&, backticks etc. execute arbitrary commands. Always prefer execFile/spawn with an argument array (not a concatenated string) when any part of the command comes from user input.

#20.5 fork() and IPC — Real Use Case

js
// parent.js
const { fork } = require('child_process');
const child = fork('worker.js');

child.send({ task: 'heavy-computation', payload: 42 });
child.on('message', (result) => console.log('Got result:', result));

// worker.js
process.on('message', (msg) => {
  const result = doHeavyWork(msg.payload); // runs in a separate process, separate event loop
  process.send(result);
});

This offloads CPU-bound work to a fully separate process (separate V8 instance, separate event loop, separate memory) — the parent's event loop is never blocked. This is the older sibling of worker_threads (which is lighter-weight, shares memory via SharedArrayBuffer, but doesn't isolate crashes the way a separate process does — a crash in a fork()-ed child doesn't take down the parent, but a crash in a worker thread can, depending on how it's handled).

#20.6 cluster Module — Brief Preview (expanded fully in Phase 6)

js
const cluster = require('cluster');
const os = require('os');

if (cluster.isPrimary) {
  os.cpus().forEach(() => cluster.fork()); // one worker process per CPU core
} else {
  require('./server'); // each worker runs its own HTTP server, OS load-balances between them
}

Uses fork() under the hood — each worker is a full separate Node process sharing the listening port via the OS.

#Interview Q&A — Unit 20

Q: What's the difference between process.exit() and process.exitCode = n? A: process.exit() terminates immediately and synchronously, potentially cutting off unflushed stdout writes or pending async callbacks (like a database close). Setting process.exitCode lets the event loop drain naturally and the process exits on its own once there's no more work — the safer choice in production.

Q: Why is exec() more dangerous than spawn()/execFile() for user-controlled input? A: exec() runs the command through a shell (/bin/sh), so shell metacharacters in unsanitized input (;, &&, backticks) can execute arbitrary commands — classic shell injection. execFile/spawn take an argument array and don't invoke a shell, so injection via those characters doesn't work the same way.

Q: What happens if an EventEmitter emits 'error' with no listener attached? A: Node treats 'error' as special — if there's no listener, it throws the error and crashes the process. Always attach an .on('error', ...) handler on any emitter you don't fully control.

Q: fork() vs worker_threads — when would you pick one over the other? A: fork() spawns a full separate OS process — total isolation (a crash doesn't affect the parent), higher memory overhead, IPC via message passing (serialized). worker_threads run in the same process, can share memory via SharedArrayBuffer, much lighter weight — better for CPU-bound tasks where you want speed and don't need process-level isolation.

#Unit 21 — Debugging & Diagnostics in Node

#21.1 Global Error Handlers

js
process.on('uncaughtException', (err, origin) => {
  console.error('Uncaught exception:', err, origin);
  // Node's official guidance: log it, then exit.
  // The process is in an undefined state after an uncaught sync exception —
  // do NOT try to "resume normal operation".
  process.exit(1);
});

process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled rejection at:', promise, 'reason:', reason);
  // Since Node 15+, unhandled rejections crash the process by default
  // (previously just a warning) — treat this the same as uncaughtException.
});

Key interview point: uncaughtException is a last-resort safety net, not a routine error-handling mechanism. Relying on it to "catch" errors instead of proper try/catch / .catch() / error-first callback handling is a red flag in code review — by the time it fires, you may have corrupted in-memory state, half-completed transactions, etc. The correct pattern is: handle errors as close to the source as possible; use uncaughtException/unhandledRejection only to log-and-exit so a process supervisor (PM2, k8s) can restart a clean instance.

#21.3 Diagnosing a Hung Process

If a process appears frozen (not responding, no crash):

  1. Attach --inspect to a running process without restarting it: kill -USR1 <pid> on Linux/macOS enables the inspector on a live process. Then attach via chrome://inspect.
  2. Check the Call Stack — if it's stuck in a while(true) loop or synchronous CPU-heavy code, DevTools will show exactly where.
  3. If it's stuck waiting on I/O that never resolves (e.g., a DB query with no timeout), the call stack will typically look idle rather than pinned — check for missing await/dangling Promises, or a connection pool that's exhausted (all connections checked out, none released — classic "pool leak").
  4. process._getActiveHandles() / process._getActiveRequests() (undocumented but commonly used in debugging) list what's keeping the event loop alive — useful for finding "why won't this process exit."

#21.4 Graceful Shutdown for an HTTP Server (full pattern, tying Units 20 & 21 together)

js
const server = app.listen(3000);

let isShuttingDown = false;

async function shutdown(signal) {
  if (isShuttingDown) return;
  isShuttingDown = true;
  console.log(`${signal} received, starting graceful shutdown`);

  server.close((err) => {
    if (err) {
      console.error('Error closing server:', err);
      process.exit(1);
    }
    console.log('HTTP server closed — no longer accepting new connections');
  });

  try {
    await db.close();          // close DB pool
    await redisClient.quit();  // close cache connection
    console.log('All connections closed cleanly');
    process.exit(0);
  } catch (err) {
    console.error('Error during cleanup:', err);
    process.exit(1);
  }

  // hard safety-net timeout in case something hangs
  setTimeout(() => {
    console.error('Forced shutdown after timeout');
    process.exit(1);
  }, 10_000).unref();
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));

server.close() stops the server from accepting new connections but lets in-flight requests finish — this is exactly what's needed for zero-downtime deploys under a rolling-update orchestrator.

#21.5 Other Diagnostic Tools Worth Knowing (name-drop level for interviews)

  • node --prof — generates a V8 CPU profile log for later analysis (node --prof-process).
  • clinic.js (clinic doctor, clinic flame) — third-party but widely used for diagnosing event-loop delay, CPU flame graphs, memory issues in one command.
  • --max-old-space-size — tune V8's heap ceiling; relevant when discussing OOM crashes in containers with limited memory.
  • Heap snapshots (via --inspect Memory tab, or heapdump package) — compare two snapshots taken minutes apart to spot growing object counts = memory leak fingerprint.

#Interview Q&A — Unit 21

Q: Should you use uncaughtException as your main error-handling strategy? A: No — it's a last-resort safety net. By the time it fires, the process may be in a corrupted state (partial writes, half-open transactions). Best practice: log the error and exit, letting a process manager restart a clean instance; real error handling belongs in try/catch and proper async error propagation.

Q: How would you diagnose a Node process that's hung but not crashed? A: Attach the inspector to the live process (kill -USR1 <pid> then chrome://inspect), check the call stack for a blocking sync loop, or check for exhausted resources (DB pool with no released connections) if it looks idle rather than pinned. process._getActiveHandles() can reveal what's keeping the loop alive.

Q: What does server.close() actually do, and why is it central to graceful shutdown? A: It stops the server from accepting new incoming connections but does not forcibly terminate in-flight requests — they're allowed to complete normally. This is what enables zero-downtime rolling deploys, since the old process finishes its current work before exiting.

#Phase 4 Rapid-Fire Interview Round

Use these as flash-card style self-testing before an interview.

  1. Q: What are the 6 phases of the event loop, in order? A: Timers → Pending callbacks → Idle/prepare → Poll → Check → Close callbacks.

  2. Q: Which queue is drained first — process.nextTick or Promise microtasks? A: process.nextTick queue, always.

  3. Q: Does require() execute a module file more than once? A: No — only on first call; subsequent calls return the cached module.exports.

  4. Q: What's the danger of Buffer.allocUnsafe()? A: Doesn't zero-fill memory — can leak old heap data if sent to a client before being overwritten.

  5. Q: How does .pipe() prevent OOM crashes on large file transfers? A: Automatic backpressure handling — pauses the readable when the writable's buffer is full, resumes on 'drain'.

  6. Q: Why does exec('cmd ' + userInput) risk shell injection but execFile/spawn with an args array doesn't? A: exec invokes a shell that interprets metacharacters (;, &&, backticks); execFile/spawn pass arguments directly to the executable without shell interpretation.

  7. Q: What crashes the process if left unhandled: an EventEmitter emitting 'error' with no listener, or emitting any other custom event with no listener? A: Only 'error' — it's special-cased in Node to throw if unhandled. Other events with no listeners are silently no-ops.

  8. Q: Does UV_THREADPOOL_SIZE affect how many concurrent HTTP requests Node can handle? A: No — network I/O uses the OS's native async mechanism (epoll/kqueue/IOCP), not the libuv thread pool. The thread pool only matters for fs, crypto, dns.lookup, zlib.

  9. Q: What's the correct order of operations in a graceful shutdown handler? A: Stop accepting new connections (server.close()) → let in-flight requests finish → close DB/cache connections → exit; with a hard timeout fallback in case cleanup hangs.

  10. Q: Why is relying on uncaughtException for routine error handling considered bad practice? A: The process state is undefined after an uncaught synchronous exception; it should only be used to log-and-exit cleanly, not to "catch and continue."
