#Unit 07 — Promises & Async/Await

#1. Why Promises exist

Before Promises, async work was handled with callbacks — leading to "callback hell" (deeply nested, hard to compose, hard to error-handle consistently because every callback needed its own if (err) check). Promises give you:

  • A single, standard shape for "a value that will exist later."
  • Composability (.then chains, Promise.all, etc.).
  • Guaranteed error propagation through the chain (no manual err-checking at every step).

#2. The three states

A Promise is a state machine with exactly one transition allowed:

text
pending  --resolve(value)-->  fulfilled   (settled, immutable)
pending  --reject(reason)-->  rejected    (settled, immutable)

Once settled (fulfilled or rejected), a Promise's outcome is locked forever — calling resolve() again does nothing. This immutability is why Promises are safe to pass around and hand to multiple consumers.

#3. .then() internals — what interviewers actually probe

js
promise.then(onFulfilled, onRejected)

Key facts most 3-YOE devs get wrong:

  • .then() always returns a new Promise — this is what makes chaining work.
  • If onFulfilled returns a plain value, the new promise resolves with that value.
  • If onFulfilled returns a thenable/Promise, the new promise "adopts" its state (waits for it).
  • If onFulfilled throws, the new promise rejects with that error — this is how .catch() at the end of a chain catches errors from any step above it.
  • .catch(fn) is literally sugar for .then(undefined, fn).
  • .finally(fn) runs regardless of outcome, receives no arguments, and passes through the original value/error unless it itself throws.
js
fetchUser(id)
  .then(user => fetchOrders(user.id))   // returns a promise -> chain waits
  .then(orders => processOrders(orders)) // returns a value -> next .then gets it
  .catch(err => logger.error(err))       // catches ANY rejection above
  .finally(() => connection.release());  // always runs

#4. async/await — what it desugars to

async function always returns a Promise. await pauses execution of that function (not the whole thread) until the awaited promise settles, then either returns the value or throws the rejection reason as a normal JS exception.

js
// async/await
async function getUser(id) {
  try {
    const res = await fetch(`/users/${id}`);
    if (!res.ok) throw new Error('not found');
    return await res.json();
  } catch (err) {
    logger.error(err);
    throw err; // re-throw so caller's promise rejects
  }
}

// roughly equivalent .then() version
function getUser(id) {
  return fetch(`/users/${id}`)
    .then(res => {
      if (!res.ok) throw new Error('not found');
      return res.json();
    })
    .catch(err => {
      logger.error(err);
      throw err;
    });
}

Interview gotcha: await inside a try/catch catches rejections just like synchronous throws — this is the #1 reason teams migrate away from .then() chains for readability, not because .then() is "wrong."

#5. Sequential vs Parallel await — the #1 real-world bug

js
// BAD — sequential, 300ms + 300ms + 300ms = ~900ms
const a = await fetchA(); // 300ms
const b = await fetchB(); // 300ms
const c = await fetchC(); // 300ms

// GOOD — parallel, ~300ms total (if independent)
const [a, b, c] = await Promise.all([fetchA(), fetchB(), fetchC()]);

This is a very common senior-level review comment: "these calls don't depend on each other, why are they sequential?" Know how to spot it in a code review and fix it.

#6. Promise combinators — know the exact failure semantics

Combinator Resolves when Rejects when Use case
Promise.all all settle fulfilled any one rejects (fails fast) All results required, all-or-nothing
Promise.allSettled all settle (either state) never Want results even if some fail (e.g. batch email sends)
Promise.race first to settle (fulfilled OR rejected) — Timeouts, "whichever comes first"
Promise.any first to fulfill all reject (AggregateError) First success wins, ignore individual failures (e.g. multi-CDN fetch)
js
// Timeout pattern using Promise.race — very common interview ask
function withTimeout(promise, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error('Timeout')), ms)
  );
  return Promise.race([promise, timeout]);
}
js
// allSettled for "best effort batch" jobs
const results = await Promise.allSettled(userIds.map(sendWelcomeEmail));
const failed = results.filter(r => r.status === 'rejected');

#7. Implementing a mini Promise from scratch (common interview ask)

js
class MyPromise {
  #state = 'pending';
  #value;
  #callbacks = [];

  constructor(executor) {
    const resolve = (value) => this.#settle('fulfilled', value);
    const reject = (reason) => this.#settle('rejected', reason);
    try { executor(resolve, reject); } catch (e) { reject(e); }
  }

  #settle(state, value) {
    if (this.#state !== 'pending') return; // immutability
    this.#state = state;
    this.#value = value;
    queueMicrotask(() => this.#callbacks.forEach(cb => cb()));
  }

  then(onFulfilled, onRejected) {
    return new MyPromise((resolve, reject) => {
      const run = () => {
        try {
          if (this.#state === 'fulfilled') {
            resolve(onFulfilled ? onFulfilled(this.#value) : this.#value);
          } else {
            if (onRejected) resolve(onRejected(this.#value));
            else reject(this.#value);
          }
        } catch (e) { reject(e); }
      };
      if (this.#state === 'pending') this.#callbacks.push(run);
      else queueMicrotask(run);
    });
  }
}

This demonstrates: pending/settled immutability, microtask scheduling, and chain adoption. You won't need to write this perfectly in an interview, but you should be able to explain each piece.

#8. Common pitfalls checklist

  • Forgetting to return a promise inside .then() — breaks chaining, causes "silent" out-of-order execution.
  • Unhandled rejections — a rejected promise with no .catch triggers process.on('unhandledRejection') in Node; in modern Node (>=15) this crashes the process by default.
  • Mixing await with .then() unnecessarily — pick one style per function.
  • forEach with async callbacks does not await — arr.forEach(async x => await doThing(x)) runs all iterations concurrently and forEach doesn't wait for any of them. Use for...of with await, or Promise.all(arr.map(...)).

#9. Interview Q&A

Q: What happens if you await a non-promise value? A: It's wrapped in Promise.resolve(value) and resolves immediately (on the next microtask tick) — no error.

Q: Does await block the Node.js thread? A: No. It pauses only the enclosing async function's execution; the event loop keeps processing other work (other requests, timers, I/O callbacks) while waiting.

Q: Why does Promise.all "fail fast"? A: It rejects as soon as the first passed promise rejects, without waiting for the others to settle — the other promises keep running in the background but their results are discarded by Promise.all.

Q: How do you catch errors from Promise.all? A: Wrap in try/catch with await, or attach .catch() — but you lose the individual per-item success/failure detail (use allSettled if you need that).

#Unit 08 — Event Loop & Concurrency Model

#1. The mental model

JavaScript is single-threaded for your code execution, but the runtime (browser or Node/libuv) is multi-threaded under the hood for I/O. The event loop is the mechanism that lets a single-threaded language handle concurrent I/O without blocking.

Three key structures:

  • Call stack — synchronous execution, LIFO.
  • Microtask queue — Promise callbacks (.then, catch, finally), queueMicrotask, async/await continuations.
  • Macrotask queue(s) — setTimeout, setInterval, I/O callbacks, setImmediate (Node only).

#2. The golden rule

After every single macrotask (or after each top-level script), the JS engine fully drains the microtask queue before doing anything else — including rendering (browser) or moving to the next macrotask (Node).

This is why:

js
console.log('1');
setTimeout(() => console.log('2'), 0);
Promise.resolve().then(() => console.log('3'));
console.log('4');

// Output: 1, 4, 3, 2

1 and 4 run synchronously. 3 (microtask) always beats 2 (macrotask), even with setTimeout(fn, 0).

Critical nuance: if a microtask queues another microtask, that new one is also drained before the next macrotask — microtasks can starve macrotasks if you keep chaining them. This is a real production bug pattern (infinite .then() recursion delaying I/O).

#3. Node's event loop phases (beyond the browser model)

Node (via libuv) breaks the loop into ordered phases per tick:

text
┌───────────────────────────┐
│           timers          │  <- setTimeout / setInterval callbacks
├───────────────────────────┤
│     pending callbacks     │  <- deferred I/O callbacks (some system errors)
├───────────────────────────┤
│       idle, prepare       │  <- internal use
├───────────────────────────┤
│           poll            │  <- retrieve new I/O events, execute I/O callbacks
├───────────────────────────┤
│           check           │  <- setImmediate() callbacks
├───────────────────────────┤
│      close callbacks      │  <- e.g. socket.on('close', ...)
└───────────────────────────┘

Microtasks (Promise callbacks, process.nextTick) are drained between every phase transition, not just once per full loop.

#4. process.nextTick vs setImmediate vs setTimeout(fn, 0)

This is the single most-asked Node interview question in this unit.

  • process.nextTick(fn) — Node-specific, higher priority than Promise microtasks. Runs immediately after the current operation completes, before the event loop continues to any phase. Overusing it can starve I/O entirely.
  • setImmediate(fn) — runs in the check phase, i.e. after the current poll phase completes. Good for "run after I/O, in this loop iteration."
  • setTimeout(fn, 0) — runs in the timers phase; effectively setTimeout(fn, 1) due to minimum clamping. Ordering relative to setImmediate is not guaranteed in the main module (depends on process startup timing) but is guaranteed inside an I/O callback:
js
const fs = require('fs');

fs.readFile(__filename, () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
});
// Inside an I/O callback, 'immediate' ALWAYS logs before 'timeout'
// because we're already past the timers phase and heading into poll->check

Priority order per tick (highest to lowest): process.nextTick queue → microtask (Promise) queue → the relevant macrotask phase.

#5. Blocking the event loop — the real production killer

Any synchronous, CPU-heavy operation blocks everything — all pending requests, all timers, all I/O callbacks — because there's only one thread running JS.

js
// This blocks the entire server for ~2s, killing throughput for ALL clients
app.get('/hash', (req, res) => {
  const hash = crypto.pbkdf2Sync(req.query.pw, 'salt', 1_000_000, 64, 'sha512');
  res.send(hash.toString('hex'));
});

Fixes:

  • Use async variants (crypto.pbkdf2 instead of Sync).
  • Offload to worker_threads for CPU-bound work.
  • Chunk large synchronous loops using setImmediate to yield control back to the loop between chunks.

#6. Interview Q&A

Q: Is Node.js single-threaded? A: The JS execution thread is single-threaded, but Node itself uses a libuv thread pool (default size 4) for certain async operations (file system, DNS lookup, some crypto) and the OS kernel handles network I/O asynchronously via epoll/kqueue/IOCP — so Node as a whole is not single-threaded, only your JS callback execution is.

Q: Why is Promise.then faster than setTimeout(fn, 0)? A: Because microtasks are drained completely before the loop even considers moving to the next macrotask phase — they're always processed first, every tick.

Q: What's a real symptom of event-loop blocking in production? A: Rising p99 latency across unrelated endpoints, health-check timeouts, and increasing event loop lag metrics (measurable via perf_hooks.monitorEventLoopDelay()), even though CPU usage might look "fine" on average.

#Unit 09 — Modules: CommonJS vs ESM

#1. CommonJS (CJS) — how require() actually works

  • Synchronous loading — file is read and executed on require() call, blocking until done.
  • Each module is wrapped in a function at load time:
    js
    (function(exports, require, module, __filename, __dirname) {
  // your module code
});
    This is why module, exports, require, __filename, __dirname are available without importing them.
  • Caching: require() caches by resolved absolute file path in require.cache. Second require() of the same file returns the same object reference, doesn't re-execute the file.
  • Circular require: returns whatever has been exported so far (a partial/incomplete exports object) — a classic bug source.
js
// a.js
console.log('a starting');
exports.done = false;
const b = require('./b');
console.log('in a, b.done =', b.done);
exports.done = true;

// b.js
console.log('b starting');
exports.done = false;
const a = require('./a'); // circular
console.log('in b, a.done =', a.done); // logs false — a hasn't finished yet!
exports.done = true;

#2. ESM (import/export) — how it differs fundamentally

  • Static structure: import/export declarations are parsed and resolved before any code executes (hoisted), enabling static analysis — this is what makes tree-shaking possible.
  • Live bindings, not copies: an imported value is a live reference to the exporting module's binding. If the exporter later reassigns the exported variable, importers see the updated value automatically (unlike CJS, which copies the exports object reference at require-time).
  • Loading is asynchronous under the hood (even for local files) — this is why top-level await is legal in ESM but not CJS.
  • import.meta.url replaces __filename/__dirname (which don't exist in ESM).
js
// counter.mjs
export let count = 0;
export function increment() { count++; }

// main.mjs
import { count, increment } from './counter.mjs';
console.log(count); // 0
increment();
console.log(count); // 1 — live binding, reflects the update

#3. Enabling each in Node

  • CJS is default for .js files unless package.json has "type": "module".
  • .mjs is always ESM; .cjs is always CJs, regardless of package.json.
  • "type": "module" in package.json flips the default for .js files to ESM.

#4. Interop pitfalls (real-world pain, frequently asked)

  • ESM can import a CJS module — Node wraps module.exports as the default export, and does static analysis via cjs-module-lexer to also expose named exports where possible (works for simple exports.x = ... patterns, fails for dynamically-computed exports).
    js
    // CJS module: module.exports = { foo, bar }
import pkg from 'cjs-package';       // default import always works
import { foo, bar } from 'cjs-package'; // works only if statically analyzable
  • CJS cannot require() a pure ESM module synchronously — you must use dynamic import() (which returns a Promise) instead.
    js
    // inside a CJS file
async function load() {
  const esmMod = await import('./pureEsmModule.mjs');
  esmMod.default();
}
  • __dirname is undefined in ESM — the common fix:
    js
    import { fileURLToPath } from 'url';
import path from 'path';
const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);
  • package.json "exports" field can restrict/remap what's importable from a package, and can define different entry points for require vs import (conditional exports) — this is a frequent source of "works with require but not import" bugs when consuming third-party packages.

#5. Tree-shaking — why it needs ESM

Bundlers (webpack, esbuild, Rollup) can only safely eliminate unused exports when imports/exports are statically declared (ESM). CJS's dynamic, runtime nature (require() can be called conditionally, exports can be mutated at any time) makes static dead-code elimination unsafe in general.

#6. Interview Q&A

Q: Why does Node treat .js as CJS by default even in 2026? A: Backward compatibility — the vast npm ecosystem historically shipped CJS; changing the default would break millions of existing packages.

Q: What's the practical difference in how "the same value" behaves between CJS and ESM exports? A: CJS exports a copy of the reference to the exports object at require-time (mutating the module's internal variable after export doesn't reflect unless you're mutating properties of an already-shared object); ESM exports live bindings, so re-assignments in the source module are always visible to importers.

Q: You need to require() a package but get "ERR_REQUIRE_ESM" — what do you do? A: Either switch your file to ESM (.mjs or "type": "module") and use static/dynamic import, or use dynamic import() inside your CJS file (returns a Promise), since synchronous require() fundamentally cannot load an ESM-only module.

#Unit 10 — Error Handling Patterns

#1. Baseline: Error and subclassing

js
class AppError extends Error {
  constructor(message, statusCode = 500, code = 'INTERNAL_ERROR', isOperational = true) {
    super(message);
    this.name = this.constructor.name;
    this.statusCode = statusCode;
    this.code = code;
    this.isOperational = isOperational; // distinguishes "expected" vs "bug" errors
    Error.captureStackTrace(this, this.constructor);
  }
}

class ValidationError extends AppError {
  constructor(message, details) {
    super(message, 400, 'VALIDATION_ERROR');
    this.details = details;
  }
}

class NotFoundError extends AppError {
  constructor(resource) {
    super(`${resource} not found`, 404, 'NOT_FOUND');
  }
}

Error.captureStackTrace excludes the constructor call itself from the stack trace, giving a cleaner trace pointing at the actual throw site.

#2. Operational vs Programmer errors — the key production distinction

  • Operational errors: expected failure modes — invalid input, failed DB connection, 3rd-party API timeout, resource not found. Handle gracefully, respond to the client, keep the process alive.
  • Programmer errors: bugs — undefined is not a function, reading a property of null. These indicate the app is in an unknown state. The standard guidance (Node core team recommendation) is to let the process crash and restart it via a process manager (PM2, Kubernetes) rather than try to keep running in a corrupted state.

The isOperational flag above lets a central handler decide: log-and-respond vs log-and-crash.

#3. Sync vs async error propagation

js
// Sync throw — caught by surrounding try/catch normally
function parseConfig(json) {
  try {
    return JSON.parse(json);
  } catch (e) {
    throw new AppError('Invalid config JSON', 400, 'BAD_CONFIG');
  }
}

// Async — must await, or the rejection is "swallowed" silently
async function loadUser(id) {
  const user = await db.users.findById(id); // if this rejects, it propagates
  if (!user) throw new NotFoundError('User');
  return user;
}

Classic bug: forgetting await on an async call inside a try/catch — the promise rejection happens after the try/catch block has already exited, so it's never caught there, and (depending on Node version) may become an unhandled rejection.

js
// BUG — missing await
try {
  doAsyncThing(); // fires and forgets, returns a promise immediately
} catch (e) {
  // never runs for doAsyncThing's rejection
}

#4. Global safety nets (last line of defense, not primary handling)

js
process.on('unhandledRejection', (reason, promise) => {
  logger.error('Unhandled Rejection', reason);
  // In modern Node, unhandled rejections already terminate by default —
  // this handler is for logging/alerting before the crash, or converting
  // to a controlled shutdown.
  process.exit(1);
});

process.on('uncaughtException', (err) => {
  logger.error('Uncaught Exception', err);
  // Never attempt to "resume" — the process state is unknown.
  process.exit(1);
});

These are safety nets, not a substitute for handling errors at the call site. Relying on them for normal control flow is an anti-pattern.

#5. Centralized Express error-handling middleware

js
// error-handler.js — must have exactly 4 params for Express to recognize it
function errorHandler(err, req, res, next) {
  const isOperational = err.isOperational ?? false;
  const statusCode = err.statusCode || 500;

  logger.error({
    message: err.message,
    stack: err.stack,
    isOperational,
    path: req.path,
  });

  res.status(statusCode).json({
    error: {
      code: err.code || 'INTERNAL_ERROR',
      message: isOperational ? err.message : 'Something went wrong',
    },
  });
}

// wrapper to avoid try/catch boilerplate in every async route
const asyncHandler = (fn) => (req, res, next) => {
  Promise.resolve(fn(req, res, next)).catch(next);
};

app.get('/users/:id', asyncHandler(async (req, res) => {
  const user = await userService.findById(req.params.id);
  if (!user) throw new NotFoundError('User');
  res.json(user);
}));

app.use(errorHandler); // registered LAST, after all routes

Interview point: Express (pre-v5) does not automatically catch rejected promises from async route handlers — an unhandled rejection inside an async route handler will not reach your error middleware unless you either wrap it (asyncHandler) or explicitly .catch(next). Express 5 fixes this by auto-forwarding rejected promises to next().

#6. Error-first callbacks (legacy Node style — still shows up in interviews)

js
fs.readFile('config.json', (err, data) => {
  if (err) return callback(err); // always check err FIRST
  callback(null, JSON.parse(data));
});

Convention: first argument is always null on success, or an Error on failure. This predates Promises and is still the pattern for some core Node APIs and older libraries.

#7. Retry & circuit breaker patterns (senior-level expectation)

js
async function withRetry(fn, { retries = 3, delayMs = 200 } = {}) {
  let lastErr;
  for (let attempt = 0; attempt <= retries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      lastErr = err;
      if (attempt < retries) {
        await new Promise(r => setTimeout(r, delayMs * 2 ** attempt)); // exponential backoff
      }
    }
  }
  throw lastErr;
}

Know the concept of a circuit breaker (e.g. via the opossum library) even if you don't implement one from scratch: after N consecutive failures, stop calling a failing dependency for a cooldown period instead of hammering it — protects both the caller and the struggling downstream service.

#8. Interview Q&A

Q: Should you catch an error you can't meaningfully handle? A: No — let it propagate to a centralized handler. Catching-and-swallowing (or catching-and-logging without re-throwing) at a low level hides failures from callers and makes debugging much harder.

Q: What's the danger of process.exit() inside a request handler? A: It kills the entire server process — all other in-flight requests are dropped. It should only be used at the top level (e.g. in uncaughtException/unhandledRejection handlers), ideally after a graceful shutdown sequence closing DB connections, not from inside business logic.

Q: Difference between isOperational errors and letting the process crash? A: Operational errors (bad input, 404, timeout) are anticipated and safe to handle inline — respond and move on. Non-operational (programmer) errors mean the app's internal state is unknown, so the safer move is to crash and let an orchestrator (PM2/K8s) restart into a known-good state.

#Unit 11 — Functional Programming Patterns

#1. First-class functions — the foundation

In JS, functions are values: they can be assigned to variables, stored in data structures, passed as arguments, and returned from other functions. This single property is what makes callbacks, .then(), middleware chains, and higher-order functions (HOFs) possible at all.

js
const handlers = { onSuccess: (data) => console.log(data), onError: (err) => console.error(err) };

#2. Pure functions & referential transparency

A function is pure if:

  1. Same input always produces same output.
  2. No observable side effects (no mutating external state, no I/O, no random/date dependence).
js
// Impure — depends on external state, has a side effect
let taxRate = 0.1;
function addTax(price) { return price + price * taxRate; }

// Pure — all inputs explicit, no external reads/writes
function addTax(price, rate) { return price + price * rate; }

Why it matters in backend code: pure functions are trivially unit-testable (no mocking needed), safely parallelizable/memoizable, and easier to reason about in code review because their behavior is fully described by their signature.

#3. Immutability in JS — no built-in enforcement, know the strategies

JS has no true immutable data structures built in (aside from primitives, which are already immutable).

  • Object.freeze(obj) — shallow freeze only; nested objects remain mutable.
  • Spread/Object.assign for shallow copies when "updating" — common in Redux-style reducers.
  • Libraries like immer (structural sharing, "draft" mutation API) or immutable.js for deep, efficient immutability at scale.
js
// Shallow-freeze gotcha
const config = Object.freeze({ db: { host: 'localhost' } });
config.db.host = 'prod-db'; // succeeds! nested object isn't frozen

#4. Function composition — building pipelines

js
const compose = (...fns) => (x) => fns.reduceRight((acc, fn) => fn(acc), x);
const pipe = (...fns) => (x) => fns.reduce((acc, fn) => fn(acc), x);

const trim = (s) => s.trim();
const toLower = (s) => s.toLowerCase();
const removeSpaces = (s) => s.replace(/\s+/g, '-');

const slugify = pipe(trim, toLower, removeSpaces);
slugify('  Hello World  '); // 'hello-world'

compose applies right-to-left (mathematical convention, like f(g(x))); pipe applies left-to-right (reads like a sequence of steps — generally preferred for readability in application code).

#5. Higher-order functions in practice

A HOF either takes a function as an argument, returns a function, or both.

js
// Returns a function — classic middleware/decorator pattern
function withLogging(fn) {
  return (...args) => {
    console.log(`Calling ${fn.name} with`, args);
    const result = fn(...args);
    console.log(`Result:`, result);
    return result;
  };
}

const add = (a, b) => a + b;
const loggedAdd = withLogging(add);
loggedAdd(2, 3); // logs call + result, still returns 5

This is the exact shape of Express middleware, Redux middleware, and decorator-style utilities — recognizing this pattern is a strong signal in interviews.

#6. Currying — partial application for reusable, specialized functions

js
const curry = (fn) => {
  return function curried(...args) {
    if (args.length >= fn.length) return fn(...args);
    return (...more) => curried(...args, ...more);
  };
};

const multiply = curry((a, b, c) => a * b * c);
multiply(2)(3)(4);     // 24
multiply(2, 3)(4);     // 24
multiply(2, 3, 4);     // 24

// Practical use: specialized, reusable validators
const validateRange = curry((min, max, value) => value >= min && value <= max);
const isValidAge = validateRange(0, 120);
isValidAge(35); // true

#7. Refactor example (imperative → functional pipeline — common lab exercise)

js
// Imperative
function processOrders(orders) {
  const result = [];
  for (const order of orders) {
    if (order.status === 'completed') {
      const withTax = order.total * 1.1;
      result.push({ id: order.id, total: withTax });
    }
  }
  return result;
}

// Functional pipeline
const isCompleted = (order) => order.status === 'completed';
const addTax = (order) => ({ ...order, total: order.total * 1.1 });
const toSummary = (order) => ({ id: order.id, total: order.total });

const processOrders = (orders) =>
  orders.filter(isCompleted).map(addTax).map(toSummary);

Each step is independently testable, and the intent reads top-to-bottom without tracking a mutable accumulator by hand.

#8. Where "pure FP" meets real backend code — practical honesty

Backend code is inherently side-effect-heavy (DB writes, HTTP calls, logging). The realistic goal isn't "make everything pure" — it's:

  • Keep business logic / transformations pure and isolated (easy to unit test).
  • Push side effects (I/O) to the edges of your functions/modules.
  • Use composition to build readable pipelines out of small, named, single-purpose functions instead of one large imperative block.

#9. Interview Q&A

Q: Why are pure functions easier to test? A: No mocking of external state/dependencies is needed — you just assert output for given input, and the test can't be flaky due to shared/mutable state or execution order.

Q: What's the tradeoff of heavy immutability (e.g. via immer or spreading nested objects) in a hot path? A: Extra allocations/copies have a real performance and GC cost; for very large objects or hot loops this can matter, so it's a deliberate tradeoff (correctness/predictability vs raw throughput), not a free lunch.

Q: compose vs pipe — which do most codebases prefer and why? A: pipe (left-to-right) is generally preferred in application/backend code because it reads in the same order the operations execute, matching how most people reason about a sequence of steps; compose (right-to-left) mirrors mathematical function composition notation and shows up more in FP-library internals.
